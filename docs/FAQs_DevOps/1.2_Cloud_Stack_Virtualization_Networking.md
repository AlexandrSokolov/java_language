## Cloud stack - Cloud, Virtualization, Networking

### What does "cloud" mean?
<details><summary>Show answer</summary>

The word rides two independent axes and never says which one it means:

- **Who owns the hardware?** you own it → on-prem; you rent it → **IaaS = cloud** (you still run the software).
- **Who runs the software?** you run it → self-host; the vendor runs it → **SaaS = cloud** (you only consume it).

So "cloud" points at either axis. Name the axis before answering.

Trigger: *cloud names either the operator (SaaS) or the hardware (IaaS).*

</details>

### What does cloud cost developers?
<details><summary>Show answer</summary>

**IaaS — you rent the box (but still run your own software): you lose control of the machine.**
- **Can't pick / tune / reach it** — no choosing the box, no pinning the JVM to it, no SSH.

  This is the first trade-off the move to cloud imposes on Java developers and performance engineers:
  capacity and automation in exchange for control over the machine underneath the JVM.

**SaaS — the vendor runs the software too: on top of the machine loss, you lose control of the software.**
- **Missing feature** — can't add it; you ask the vendor and hope.
- **Outage** — can't dig in; you wait for them to recover.
- **Hard to debug** — no OS metrics, internals, or server logs; slowdowns are opaque.

Trigger: *IaaS = lose the box; SaaS = lose the box **and** the software.*

</details>

### Why can't you reach the hardware in cloud?
<details><summary>Show answer</summary>

Because a hypervisor is in the way, by design.

Cloud sells one physical machine to many tenants, so no tenant can own it. The way to share it safely is to stop
every guest OS from touching hardware directly and route all access through one mediating layer. That layer is the
hypervisor — and it is the same layer that costs you control over the machine.

Trigger: *many tenants on one box forces a layer between you and it — that layer is virtualization.*

Detail: [what virtualization must satisfy](#what-defines-virtualization),
[what changes for the kernel](#what-changes-for-a-guest-os),
[what it costs you](#who-controls-virtualization-overhead).

</details>

### What defines virtualization?
<details><summary>Show answer</summary>

Three conditions, each protecting one thing:

1. **Fidelity** — looks real — programs on the guest run as if on real hardware.
2. **Mediation/control** — stays boxed in — a hypervisor controls all hardware access; the guest can't reach around it.
3. **Low overhead** — costs little — running the guest takes only a tiny slice of extra resources.

Trigger: *the guest must not notice, must not escape, must not pay much.*

</details>

### What changes for a guest OS?
<details><summary>Show answer</summary>

It loses the privilege it normally has.

Unvirtualized (your laptop): the kernel runs in a privileged mode and talks to hardware directly — that is what a
switch into kernel mode buys you.

Virtualized: direct hardware access by the guest OS is forbidden. The stack becomes host OS → hypervisor → guest OS →
your deployment (often containers). The hypervisor is the indirection layer that intercepts what the guest kernel
used to do itself.

</details>

### Who controls virtualization overhead?
<details><summary>Show answer</summary>

Mostly not you — but you get one choice up front.

- **Choose-time (your lever):** providers offer different VM types, so **picking the right VM** is a real
  performance choice for some workloads. This is the one knob you turn.
- **Run-time (not you):** once chosen, the hypervisor sits between your JVM and the hardware and mediates every
  access on its own logic. You don't operate it or tune it — part of your performance is decided by a layer you
  picked but don't run.

Trigger: *pick the VM once; after that the hypervisor runs the show.*

</details>

### Is virtualization still slow?
<details><summary>Show answer</summary>

No longer — and if you remember it as slow, you were probably running a different kind.

- **Type-2 (hosted)** — hypervisor runs as an app *on top of* your OS (VirtualBox, VMware Workstation), fighting the
  host for CPU and RAM. This is the "disaster on my laptop" version.
- **Type-1 (bare-metal)** — hypervisor runs *directly on* the hardware, no host OS underneath (KVM, Xen, ESXi,
  Hyper-V). Plus CPU extensions (Intel VT-x / AMD-V) let most guest instructions run natively instead of emulated.

Cloud runs type-1 with hardware assist, and heavy cloud demand funded years of work cutting the overhead. That's why
the old "virtualization is slow" reputation is outdated.

Trigger: *type-2 on your laptop was slow; type-1 with hardware assist in the cloud is cheap.*

</details>

### How are cloud VM types categorized?
<details><summary>Show answer</summary>

By **which resource the workload leans on hardest** — providers pre-shape the ratio of CPU, memory, storage, and
accelerators, and you pick the shape that matches your profile.

| Type                       | Leans on           | Typical workload                                                           |
|----------------------------|--------------------|----------------------------------------------------------------------------|
| General purpose            | balanced           | starting point; spans tiny (1 vCPU / 4 GB) to bare metal (64 CPU / 256 GB) |
| Compute optimized          | CPU                | high CPU load                                                              |
| Memory optimized           | RAM                | large in-memory data sets — databases, caches, analytics                   |
| Accelerated computing      | special hardware   | graphics, heavy calculation, generative AI                                 |
| Storage optimized          | local disk I/O     | transactional databases, Spark-style jobs                                  |
| High-performance computing | scale + throughput | complex simulation, deep learning                                          |

Trigger: *start balanced, then name the resource you run out of first.*

Note: even one series spans a wide range — a "general purpose" family can go from a single-vCPU VM up to a bare
metal machine, so the category names a ratio, not a size.

</details>

### Which providers offer VM families?
<details><summary>Show answer</summary>

The three large public clouds, all with the same category idea under different names:

- **AWS (Amazon EC2)** — the widest option set; also rents **bare metal servers**, not only VMs.
- **Azure** — e.g. D-series for general purpose, L-series for storage optimized.
- **Google Cloud** — a comparable set: general purpose, ultra-high memory, compute intensive, and GPU-backed
  workloads.

Trigger: *same shapes everywhere, different letters.*

</details>

### What governs VM choice for a workload?
<details><summary>Show answer</summary>

Three goals — **cost, reliability, scalability** — and the VM type is the one lever that sets all three.

- **Fit the VM to the workload.** CPU-heavy work → compute-optimized VM, and so on. But the profile is a guess until
  proven: **measure it** in a production-like setup before committing, or you're sizing for a hunch.
- **Don't force one platform.** A system has parts with different shapes: put elastic, spiky work on cloud; put
  steady, latency-sensitive work on bare metal, where performance stays predictable and no neighbor VM interferes.

Trigger: *fit the shape, prove it by measurement, split the system if one platform can't serve both halves.*

</details>

### Why isn't cloud compute permanent?
<details><summary>Show answer</summary>

To save money. Your app isn't tied to one fixed machine, so the provider can add and remove machines
to match demand:

- **scale up** — give more power to the machines you already have;
- **scale out** — add or drop whole machines (K8s "nodes") from the group.

Pushed to the limit: **spot compute** — spare capacity sold cheap, with one catch: it's not reserved.
The moment a full-price buyer wants it, your machine is shut down with no warning.

The field calls this **ephemeral** — short-lived, can vanish anytime.

Trigger: *pay less by never claiming a spot → so you never own one.*

</details>

### Cloud machines aren't permanent — what does that cost you?
<details><summary>Show answer</summary>

A known address that stays up. No dedicated IP, no always-on compute — a workload can move, vanish, or multiply
between calls. So an address can't be what you reason about.

[Reason about **traffic**](#if-not-the-address-what-do-you-reason-about) 
instead: not "where is that host" but "what kind of call is this, and which way is it going." 

**ephemeral** — the field's word for short-lived, can vanish anytime.

</details>

### If not the address, what do you reason about?
<details><summary>Show answer</summary>

To reason about **traffic**: not "where is that host" but "what kind of call is this, and which way is it going."

**Root idea: don't hand out IP addresses (they keep changing) — hand out a name (it stays fixed).**

The name points at a fixed front door, and the front door forwards each call to whichever copies are alive right now.
The name and the front door don't change; only the copies change.

One axis: **who is allowed to look the name up?**

- **North → south (ingress)** — caller is outside (internet, other cloud). It can't be told anything, so the name is
  **public**. Example: `api.mycompany.com` → a load balancer (usually the cloud provider's) with one fixed public IP →
  the copies behind it. Copies scale and move freely; the address isn't theirs, so it doesn't change.
- **East → west** — caller is inside (service-to-service). It can be told, so the name is **private**. Example: a
  Kubernetes *Service* named `payment-service`, a DNS name visible only inside the cluster, resolving to whatever
  copies exist right now.

Same trick both times: **one fixed name in front of changing copies.** The only difference is who is allowed to look
the name up — outsiders (public) or insiders only (private).

</details>
