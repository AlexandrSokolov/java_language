### Preferred unit of app packaging?
<details><summary>Show answer</summary>

Images. One image bundles everything needed to run — userspace OS pieces, the JVM, and the application itself — so the
deployed artifact is self-contained and identical everywhere. This holds even when the target is bare metal, not a
container platform.

</details>

### How are container images built?
<details><summary>Show answer</summary>

Four approaches, ordered by how much the build tool knows about your app:

1. [Plain Dockerfile](#plain-dockerfile--how-does-it-work) — you write the layers by hand.
2. [Layered Dockerfile](#splitting-a-jar-across-image-layers) — still by hand, but split by rate of change.
3. [Multistage Dockerfile](#multistage-build--what-does-it-buy-you) — build and runtime split into two stages.
4. [Jib](#build-tool-driven-image-creation) — the Java build system emits the image directly.

Trigger: the tool sees more of your project → it can make better layering decisions.

</details>

### Plain Dockerfile — how does it work?
<details><summary>Show answer</summary>

Each instruction produces one layer: an immutable filesystem change, kept in the build cache and reusable as a
building block for later images.

```
FROM registry.access.redhat.com/ubi8/openjdk-17:1.13-1
RUN mkdir /app
COPY target/animals-demo-1.0-SNAPSHOT.jar /app   # whole jar = one layer
WORKDIR /app
CMD ["java", "-jar", "animals-demo-1.0-SNAPSHOT.jar"]
```

Next: [Plain Dockerfile — what does it cost?](#plain-dockerfile--what-does-it-cost)

</details>

### Plain Dockerfile — what does it cost?
<details><summary>Show answer</summary>

Root cause: **you own every layering decision, and nothing checks your work.**

- **The jar is one fat layer.** `COPY app.jar` puts your code and every dependency in a single layer, so a one-line
  change invalidates the whole thing. Push and pull cost the full size each time, even though the libraries did not
  move. See [Splitting a jar across image layers](#splitting-a-jar-across-image-layers).
- **Base images go stale silently.** A pinned `FROM` tag keeps building fine long after the base picks up CVEs. Nothing
  in the build tells you. Fix: rebuild on a schedule, or let a scanner watch the registry.
- **Every project writes its own.** Layering, user, entrypoint, JVM flags — repeated per repo, drifting apart.
  Fix: a templated base image or a shared build step.

What it does *not* cost you: control. The file says exactly what happens, which is why it stays the fallback when
anything non-Java is in the image.

</details>

### Splitting a jar across image layers?
<details><summary>Show answer</summary>

A fat jar has parts that change at very different rates: the loader almost never, third-party libraries rarely,
your own classes on every commit. One `COPY` throws them into one layer, so the slow parts get invalidated by the
fast ones. Split them and only the small top layer churns.

Spring Boot writes the split into the jar at package time as `layers.idx` — default buckets: `spring-boot-loader`,
`dependencies`, `snapshot-dependencies`, `application`. The buckets are configurable in the Maven plugin if they do
not match your churn. `layertools` reads the index and unpacks each bucket into its own folder; one `COPY` per folder
gives one layer per bucket.

```
FROM bellsoft/liberica-openjdk-alpine:17 as builder   # extract needs a JVM, must not ship
ARG JAR_FILE=target/*.jar
COPY ${JAR_FILE} application.jar
RUN java -Djarmode=layertools -jar application.jar extract

FROM bellsoft/liberica-openjdk-alpine:17
COPY --from=builder dependencies/ ./            # slow-changing, first
COPY --from=builder snapshot-dependencies/ ./
COPY --from=builder spring-boot-loader/ ./
COPY --from=builder application/ ./             # your classes, last
```

Inspect the result with `dive` or `docker history`.

Next: [Layered Dockerfile — what does it cost?](#layered-dockerfile--what-does-it-cost)

</details>

### Layered Dockerfile — what does it cost?
<details><summary>Show answer</summary>

Root cause: **you get Jib's layering while keeping the Dockerfile, and pay for the seams that creates.**

- **The split lives in two places.** `layers.idx` decides the buckets, the `COPY` lines consume them. Reconfigure the
  buckets and the Dockerfile silently copies folders that no longer exist, or misses new ones. Jib has one place.
- **It is framework-bound.** `layertools` is a Spring Boot jarmode. A plain jar or another framework gives you no
  index, so you either split by hand or drop back to one layer.
- **Order is yours to get right.** Slow-changing folders must come first. Nothing enforces it; put `application/`
  ahead of `dependencies/` and the layering silently buys nothing.
- **A builder stage is now mandatory.** Extract needs a JVM and must not ship, so the simplest Dockerfile is gone —
  this is the case where multistage genuinely earns its place.

What it buys over Jib: the layering is *visible*. The `COPY` lines are the decision, in a file you read and change,
and `dive` shows whether you got what you intended.

</details>

### Multistage build — what does it buy you?
<details><summary>Show answer</summary>

The build itself runs inside a container, so the build environment is standardized instead of depending on the
machine. The first stage compiles and produces the jar; its layers are thrown away, and only the jar is copied into
the final image.

```
FROM maven:3.9-openjdk-17 AS builder          # stage 1: named, discarded later
COPY src /usr/src/app/src
COPY pom.xml /usr/src/app
RUN mvn -f /usr/src/app/pom.xml package

FROM registry.access.redhat.com/ubi8/openjdk-17:1.13-1
COPY --from=builder /usr/src/app/target/animals-demo-1.0-SNAPSHOT.jar /app   # only the jar crosses over
WORKDIR /app
CMD ["java", "-jar", "animals-demo-1.0-SNAPSHOT.jar"]
```

Next: [Multistage — what does it cost?](#multistage--what-does-it-cost)

</details>

### Multistage — what does it cost?
<details><summary>Show answer</summary>

Root cause: **it standardizes the build environment — which a real CI system already does, better.**

- **Redundant where CI exists.** A pinned build agent plus a pipeline definition in the repo gives the same
  reproducibility, and publishes the jar as a versioned artifact besides. Stage one publishes nothing.
- **It collapses build and deploy into one step.** No jar exists outside the image, so you cannot rebuild an image
  without recompiling, and you cannot promote one tested artifact through environments.
- **The dependency cache fights the layer cache.** Maven wants a warm `~/.m2`; the layer cache is not that, and keeping
  it warm needs mount tricks. A CI agent workspace just has it.
- **Layering is untouched.** The final stage is still hand-written, so the fat-jar-layer problem from the Dockerfile
  card is fully intact.

Where it genuinely earns its place: no CI at all (`docker build .` is the whole pipeline), or a build that needs tools
that must not ship — native toolchain, protoc, GraalVM native-image, npm for a bundled frontend.

</details>

### Build-tool-driven image creation?
<details><summary>Show answer</summary>

Jib runs inside Maven/Gradle, so it knows the project structure and splits the image by rate of change: dependencies,
resources, classes. Only changed layers are rebuilt — and since libraries change far less often than your own code,
the big layers stay put, cutting rebuild and startup time in the dev loop. It also refreshes dependencies and the
lower image layers on every build, which helps with security patching.

Next: [Build-tool-driven builds — what do they cost?](#build-tool-driven-builds--what-do-they-cost)

</details>

### Build-tool-driven builds — what do they cost?
<details><summary>Show answer</summary>

Root cause: **the build tool can only see what it understands — your language.**

- **One pipeline per language.** Java services build one way, Node and Python another. A Dockerfile is the common
  format every team and every reviewer already reads; a Java-only plugin splits that.
- **Anything outside the JVM app is unreachable.** Native steps, extra binaries, arbitrary files — no path to them.
  Fix: fall back to a Dockerfile for those images.
- **Do not confuse it with buildpacks.** Spring's `bootBuildImage` is Paketo Buildpacks, not Jib. Buildpacks *detect*
  your JRE, entrypoint and memory flags and can produce images in the gigabytes. Jib's default base is distroless
  Java (~200 MB) and you set base and entrypoint yourself.

Reality check on size: the floor for any JVM service is the JVM plus its userspace, roughly 150–200 MB. Nothing in
this cluster gets under it — that needs a native binary (GraalVM, ~60–80 MB), which is exactly when multistage becomes
worth it again.

</details>

### One shared image, jar mounted at runtime?
<details><summary>Show answer</summary>

The model: build one image with the JVM and everything installed, run it for every service, mount a different jar at
container start. Its stated payoffs — no per-service image build, and a live container whose jar you can pull out or
swap for diagnosis.

Judgement: **the storage argument is false, the speed argument is real, and the price is that the image stops being
the deployable unit.**

- **Layers already give you the sharing.** Images are content-addressed: one shared JVM base is stored once in the
  registry and pulled once per node. Ten services cost one base plus ten jar layers, not ten full images. The scheme
  saves registry churn, not disk.
- **Version becomes tag + whatever got mounted.** Two containers from the same tag can run different code. Rollback is
  no longer "deploy the previous tag" but "get the old jar back onto the volume" — an out-of-band mutation.
- **The tooling goes blind.** Scanners, SBOMs, provenance and admission policy all read image contents and find a bare
  JVM. "Which services ship the vulnerable library?" becomes unanswerable from the registry.
- **JVM upgrades move the whole fleet at once.** A feature if you patch aggressively; otherwise the reason nobody
  upgrades for years.
- **The mount is a new runtime failure mode.** Wrong jar, stale jar, missing volume. A self-contained image cannot fail
  that way.
- **The diagnostic gain is not exclusive.** `docker cp` extracts a jar from any image; `kubectl debug`, heap dumps and
  JFR work the same either way. What the mount adds is hot-swapping a running deployment — convenient in dev, a
  liability in prod.

Read it as a rational adaptation to a slow registry or a slow pipeline. It works, and the cost lands late — in audit,
rollback and supply-chain questions — not in daily operation.

</details>

### What isolates one container from another?
<details><summary>Show answer</summary>

Two Linux kernel features, each on its own axis:

- **Namespaces — visibility.** What a process can *see* and reach on the host: its own view of the filesystem,
  process tree, network stack, users. Outside the namespace, nothing exists as far as the process is concerned.
- **cgroups — consumption.** How much the process may *take*: CPU shares and memory limits above all. Visibility says
  nothing about greed, so this is a separate mechanism.

Trigger: **see vs. take.** Namespaces answer the first, cgroups the second — neither substitutes for the other.

</details>

### Containers vs. VMs — where's the line?
<details><summary>Show answer</summary>

One difference forces the rest: **there is no hypervisor.** A container's processes run directly on the host kernel;
a VM's run on a guest kernel that a hypervisor sits under.

What falls out of that:

- **Lightweight and fast to start** — no guest OS to boot, no hypervisor indirection on every call. This is what makes
  orchestrating thousands of them practical.
- **Weaker isolation** — the kernel is shared, so the boundary is a kernel feature, not a hardware-backed one. A VM's
  boundary survives a kernel bug; a container's is that kernel.
- **One kernel for everyone** — every container on a host runs the host's kernel, so the guest OS you "choose" is only
  userspace on top of it.

</details>
