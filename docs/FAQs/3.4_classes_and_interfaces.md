## Classes and Interfaces - Inheritance and Composition

### What kinds of property can an object have?
<details><summary>Show answer</summary>

Every property is one of three kinds. 
Only the first is a **dimension** (axis) — an independent way the object varies.

- **Independent** — you pick its value freely, and it combines with every other property → this *is* a **dimension**.
- **Derived** — its value is computed from the other properties, so you never pick it freely → **not** a dimension.
- **Branch-local** — it exists only inside one branch/subtype, not across all objects → **not** a dimension.

Note: a dimension is a *way the object varies*, not the object itself — a t-shirt is the **object**; 
size/color/sleeve are its **dimensions**.

</details>

### How do you test if a property is a dimension?
<details><summary>Show answer</summary>

One question: **can I set this value freely, or is it already decided?**

- **Free** to pick, combines with everything else → **dimension**.
- **Decided by the other properties** (derived) → not a dimension.
- **Tied to one branch** (exists only for one subtype) → not a dimension.

Free choice is the whole signal. If anything else fixes the value — a formula over the other properties, or the
branch you're in — it isn't an independent dimension.

</details>

### Which t-shirt properties are dimensions?
<details><summary>Show answer</summary>

A t-shirt comes in two kinds — `CrewNeck` and `Polo` — which lets all three property kinds show up on one object:

```java
class TShirt { Size size; Color color; }       // every shirt
class CrewNeck extends TShirt { }               // no collar, no buttons
class Polo     extends TShirt { int buttonCount; }   // branch-local property
```

- `size` (S/M/L), `color` — free, independent picks, on **every** shirt, freely combined → **dimensions**.
- `chestWidthCm` — fixed by `size`; choose M and it's already settled → **derived**, not a dimension.
- `buttonCount` — exists only on `Polo`; a crew-neck has no button count at all → **branch-local**, not a dimension.

The cut: a **dimension** applies to *every* object in the group; a **branch-local** property applies to *only one
branch* and is meaningless on the others; a **derived** property is fixed by the other properties.

</details>

### What ultimately decides whether a property is a dimension?
<details><summary>Show answer</summary>

**The domain rules — nothing else.** Whether a property is a dimension is not fixed by the property's name; 
it's fixed by *your* domain. The same property can be an independent **dimension** in one system and **not** one in
another (derived, or branch-local), depending entirely on whether your rules let its value be chosen freely.

</details>

### How can a dimension be realized in code?
<details><summary>Show answer</summary>

A dimension is not automatically anything — you **choose** how to realize it, and there are two ways:

- **As a field** — state held *on* the type. `Color color;` — the object *has* a value on that dimension.
- **As a subclass tree** — branches *of* the type. `Circle extends Shape`, `Square extends Shape` — each value
  becomes a separate subtype.

This is the decision that actually matters, and it's separate from "is this a dimension?" Knowing a property is a
dimension does **not** tell you to subclass on it — most dimensions should stay **fields**.
([what is a dimension](#what-kinds-of-property-can-an-object-have))

The explosion mistake lives *here*, not in the state question: it's turning a **second** dimension into a **tree**.
Two dimensions as fields never multiply; two dimensions as trees do — every branch of one must be repeated under
every branch of the other. So the rule isn't about how many dimensions exist, it's about how many you **make into
trees**. ([the one-tree limit](#how-many-dimensions-may-inheritance-carry) ·
[why two trees explode](#how-does-inheritance-scale-across-multiple-dimensions))

</details>

### Property: `teamSize` on a manager — axis?
<details><summary>Show answer</summary>

**Not an axis — branch-local data.** `teamSize` only means something *inside* the Manager branch; 
a non-manager has no team size. The value is tied to the branch, so it can't combine freely with other roles.


Adding it as a field in `Manager` is ordinary specialization — a subtype is *expected* to carry more state than its
parent. No multiplication, no explosion.

</details>

### Property: pay type (salaried / hourly) — axis?
<details><summary>Show answer</summary>

**A new axis.** Pay type combines **freely** with every role — a Manager can be salaried or hourly, an Engineer can
be salaried or hourly, and the role never fixes the choice. Every (role × payType) pairing is valid.


Model it by subclassing and you get `SalariedManager`, `HourlyEngineer`, … = role × payType subclasses → explosion.
That free combination across all branches is exactly what makes it a separate dimension.

</details>

### Property: `bonusFormula` — axis?
<details><summary>Show answer</summary>

**It depends on your domain rules — there's no fixed answer.**

- If the role **fixes** the formula (the Manager formula exists *because* you're a Manager, and pairing a
  senior-manager formula with an entry engineer is meaningless) → **branch-local**, a field on the branch.
- If **any formula can attach to any role** as a free, independent choice → it combines freely across all objects →
  an **independent dimension**.

The word "bonusFormula" tells you nothing. Only your combination rule does. 
The trap: "every employee has *a* bonus formula" feels like it applies to all — 
but applying-to-all is not the test; **free combination** is.

</details>

### Property: `shiftType` — axis?
<details><summary>Show answer</summary>

- Rule: *managers never work nights.* → the role **constrains** the value; not freely combinable → **branch-local**.
- Rule: *any role can be assigned either shift.* → values combine freely with every branch → **new axis**.

Identical property, opposite verdict, decided only by whether the domain permits free combination. 
This is the sharpest proof that "axis-ness" lives in the domain rules, never in the property name.

</details>

### How many dimensions may inheritance carry?
<details><summary>Show answer</summary>

**At most one as a tree.** A dimension can be realized two ways — a field, or a subclass tree
([the two ways](#how-can-a-dimension-be-realized-in-code)). Only **one** dimension may be the tree (the *subtype
axis* — kind of employee, kind of shape). Every other dimension stays a **field**.

Fields don't count against the limit: a field is state *on* the type, not a branch *of* it. A type can hold any
number of dimensions as fields.

```java
class Employee {          // payType, region, remote: dimensions held as FIELDS — no limit
    PayType payType;
    Region  region;
    boolean remote;
}
class Manager  extends Employee { ... }   // the ONE tree splits on: role
class Engineer extends Employee { ... }
```

Here `role` is the single tree — it isn't a field, it's what the subtypes branch on. payType, region, remote ride as
fields; `Manager` and `Engineer` each simply *have* them.

Make a *second* dimension into a tree — a Salaried/Hourly tree crossing the role tree — and the two trees multiply:
role × payType subclasses → explosion. The fields never caused that; only a second tree does.

So "inheritance is for one dimension" never meant the type may vary one way. It meant: **only one dimension may be a
tree; every other dimension stays a field.**
([Why two trees explode](#how-does-inheritance-scale-across-multiple-dimensions) ·
[How it's solved](#how-is-the-multiple-axes-modeling-problem-solved))

</details>

### How does inheritance scale across multiple dimensions?
<details><summary>Show answer</summary>

Inheritance expresses exactly **one** is-a chain, so it fits **one** axis cleanly — pick the axis, one subclass per
value.

It breaks down the moment behavior varies along **several independent axes** at once. A single tree can't hold them
side by side, so you're forced to make a class per *combination* — and combinations are the **product** of the axes,
not the sum.

Example — a coffee drink varies along four axes:
- base: Espresso / Tea / HotChocolate
- milk: none / whole / oat / soy
- size: small / medium / large
- extras: caramel / whip / extra shot

One class per combination means `LargeOatEspressoWithCaramel`, `SmallSoyTeaWithWhip`, … — 3 × 4 × 3 × … subclasses.
Add one new milk and a whole branch of the tree doubles. The axes are independent (milk has nothing to do with
size), but inheritance forces them into one rigid tree → **class explosion**.

The escape: keep inheritance for the *one* axis you genuinely subtype on, and move every freely-combining dimension
**off the tree and onto the type** — as a plain field when it's just data, or a strategy/component object when it
carries its own behavior. Independent dimensions held as fields don't multiply, so the product collapses back to one
axis. ([How is it solved?](#how-is-the-multiple-axes-modeling-problem-solved))

</details>

### How is the multiple-axes modeling problem solved?
<details><summary>Show answer</summary>

By keeping inheritance to its **one safe axis** and adding every other axis by other means:

**Runtime → composition.** Assemble the object from independent pieces — each axis becomes a field set per instance,
so combinations are chosen at runtime and adding an option is one new value, not a new branch:
- **Strategy** — a pluggable behavior chosen at runtime (the "how it behaves" axis).
- **Decorator** — wraps an object to stack behavior on a base, same interface (the "extras" axis).

**Design time → mixins.** Declare each extra axis as an **additional interface** the class implements. A class can
implement many, each in its own slot, so none competes for the single inheritance slot and no diamond arises — you
lift the one-axis limit without making the hierarchy dangerous.

[→ mixin vs composition trade-off](3.6_classes_and_interfaces.md#mixin-vs-composition--whats-the-trade-off)

</details>

### How does functional programming reframe Strategy and Decorator?
<details><summary>Show answer</summary>

Strategy and Decorator are, at heart, just *passing a function* — once functions are first-class you don't need a
class to vary behavior:
- **Strategy → a function parameter** — pass the algorithm directly (`x -> x * x`) instead of a `Strategy` interface
  plus implementing classes.
- **Decorator → a wrapper function** — `withLogging(f)` returning an enhanced function is function composition, not a
  wrapper class.

So FP reframes a chunk of the GoF catalog: where OOP reaches for a class to vary behavior along an axis, FP often just
passes a function — same goal, far less ceremony. The patterns shrink to functions.

</details>

### How can object-oriented designs mitigate the impact of inheritance on encapsulation?
<details><summary>Show answer</summary>

In priority order — the first is the default, the others are for when you must inherit or want to
reinforce the boundary:

1. **Favor composition over inheritance.** The primary answer: hold the other object as a field and
   delegate to it, instead of inheriting its implementation. You depend only on its public contract,
   so its internals can't couple to you — the encapsulation-weakening problem simply doesn't arise.
2. **Inherit only when the class is designed for it.** When you genuinely need inheritance, extend only
   classes that are *documented for extension* or *under your own control* — so the self-use you depend
   on is part of the contract, not an accident.
3. **Program to interfaces, not implementations.** Depend on contracts rather than concrete types, so
   clients are insulated from implementation and you retain freedom to change it.

The thread: #1 *avoids* the coupling, #2 *contains* it to a boundary the author supports, #3 *limits*
how far any implementation dependency can spread.

</details>

### What is composition, and what does it solve?
<details><summary>Show answer</summary>

**Composition** is one underlying idea: build behavior by combining or delegating to separate parts, instead of
inheriting a fixed implementation. You've now seen it answer *two* distinct inheritance failures — it takes a different
form for each:

- **Object composition (delegation)** → fixes the **encapsulation** problem. Hold another object as a field and
  delegate to its **public contract**; you never touch its internals, so the coupling that makes inheritance fragile
  can't form. [→ inheritance and encapsulation](3.3_classes_and_interfaces.md#what-is-the-relationship-between-inheritance-and-encapsulation)
- **Behavioral composition** → fixes the **single-axis** problem. Assemble an object from independent pieces, one per
  axis, so independent dimensions combine without a combinatorial class tree.
  [→ how the multiple-axes problem is solved](#how-is-the-multiple-axes-modeling-problem-solved)

**It isn't only an OO technique.** Behavioral composition's purest form is *composing functions*: pass behavior as a
function, or chain functions together, with no class hierarchy at all — same principle, less ceremony.
[→ functional reframing of Strategy and Decorator](#how-does-functional-programming-reframe-strategy-and-decorator)

So one principle, two payoffs: delegation buys you encapsulation, behavioral/functional composition buys you
free combination along independent axes. Both replace *inheriting an implementation* with *assembling from parts*.

</details>

### If composition is preferable, why not avoid inheritance entirely?
<details><summary>Show answer</summary>

Because "prefer composition" is a *default*, not an absolute — and inheritance retains genuine
advantages composition can't replicate cleanly. The guidance is narrower than it sounds: it warns
against **implementation inheritance across a boundary you don't control**, not against all inheritance.

Where inheritance is still the right tool:

- **A genuine, stable is-a relationship.** When the subtype truly *is* the supertype and substitutes
  for it everywhere (LSP holds), inheritance models it directly. Forcing composition here adds
  indirection to express a relationship the language already has a keyword for.
- **Polymorphism is the goal.** If the point is to treat many subtypes uniformly through a supertype
  reference, inheritance (or interface implementation) is the mechanism that provides it. Composition
  doesn't give you subtype substitutability.
- **Designed and documented for extension.** When the superclass author has opened the class for it —
  self-use documented, hooks provided — the coupling is part of the contract, so the main objection
  (depending on undocumented internals) goes away.
- **Less boilerplate for true specialization.** Composition requires forwarding every delegated method;
  for a real subtype that extends a stable base, inheritance avoids that forwarding cost.

What composition buys that inheritance can't — and why it's the *default*:

- It avoids the encapsulation-weakening coupling entirely (you depend only on a public contract).
- It composes along **multiple independent axes** without combinatorial class explosion.
- The wrapped object can be swapped or chosen at runtime; an inherited superclass is fixed at
  compile time.

So the rule isn't "never inherit" — it's "**don't reach for inheritance by default for code reuse.**"
Use it when there's a real is-a, substitutability holds, or the class is built for extension; reach for
composition for everything else — which, in practice, is most of the time.

</details>

### How can behavior be added to an object without extending its class?
<details><summary>Show answer</summary>

The **Decorator** pattern. Instead of subclassing to add behavior, you *wrap* the object in another
that implements the **same interface**, adds its own behavior, and delegates the rest to the wrapped
instance.

Because the decorator depends only on the **interface contract** — never the wrapped object's
internals — it sidesteps the fragile-base-class problem entirely and preserves encapsulation. It's the
composition answer to "add behavior" that the previous cards' guidance points toward.

</details>

### Can you give an example of the Decorator pattern?
<details><summary>Show answer</summary>

```java
// Common interface
public interface Coffee {
  double cost();
  String description();
}

// Base implementation
public class SimpleCoffee implements Coffee {
  @Override public double cost()        { return 2.0; }
  @Override public String description() { return "Simple coffee"; }
}

// Abstract decorator — holds and implements the same interface
public abstract class CoffeeDecorator implements Coffee {
  protected final Coffee coffee;
  protected CoffeeDecorator(Coffee coffee) { this.coffee = coffee; }
}

// Concrete decorators — add behavior, delegate the rest
public class MilkDecorator extends CoffeeDecorator {
  public MilkDecorator(Coffee coffee) { super(coffee); }
  @Override public double cost()        { return coffee.cost() + 0.5; }
  @Override public String description() { return coffee.description() + ", milk"; }
}

public class SugarDecorator extends CoffeeDecorator {
  public SugarDecorator(Coffee coffee) { super(coffee); }
  @Override public double cost()        { return coffee.cost() + 0.2; }
  @Override public String description() { return coffee.description() + ", sugar"; }
}
```

Stacking — wrap a base object in successive decorators; each adds its bit and delegates the rest:
```java
Coffee coffee = new SimpleCoffee();
coffee = new MilkDecorator(coffee);
coffee = new SugarDecorator(coffee);

coffee.description();  // "Simple coffee, milk, sugar"
coffee.cost();         // 2.7

// equivalently, nested in one expression:
coffee = new SugarDecorator(new MilkDecorator(new SimpleCoffee()));
```

`Milk` and `Sugar` combine freely without a `MilkSugarCoffee` class — the "extras" axis composing
without class explosion. Each decorator depends only on the `Coffee` interface, never on another
decorator's internals, so encapsulation holds.

</details>

### Compare decoration with inheritance
<details><summary>Show answer</summary>

Three ways to add behavior — inheritance, OOP Decorator, and the functional form:

**Inheritance**
- *Pro:* simple, direct; behavior fixed at compile time; one object, easy to debug.
- *Con:* a separate class per combination → **class explosion** (`CoffeeWithMilkAndSugar`, …);
  combinations fixed at compile time; couples to superclass internals.

**Decorator (OOP composition)**
- *Pro:* combine behaviors by **stacking** at runtime — `Milk` + `Sugar` with no `MilkSugarCoffee`
  class; open-ended; depends only on the interface, so encapsulation holds.
- *Con:* not free — many small wrapper objects; the wrapped object's **identity is lost** (`==`/
  `instanceof` see the decorator); debugging steps through delegation layers; order-sensitive
  decorators must be applied in the right order.

**Functional alternative**
A third option models decoration as a `T -> T` transform and composes the functions — best when the decorated thing is
a value, not a stateful object. 
[→ when functional decoration beats the OOP Decorator](#functional-decoration-vs-the-oop-decorator)

**When each wins:**
- inheritance for a small, fixed set along *one* axis;
- OOP Decorator when features combine freely and need a shared interface;
- functional composition when the decoration is a simple transform and you want minimal ceremony.

</details>

### Functional decoration vs the OOP Decorator?
<details><summary>Show answer</summary>

Both add behavior in layers without subclassing, but they decorate **different kinds of things**.

**The real marker: is the decorated thing a *value/transform*, or a *stateful, behavioral object*?**

- **Functional wins when you're transforming a value.** If each layer takes the thing and returns an enhanced version
  (`T -> T`), decoration is just function composition — chain the transforms with `andThen`. No interface, no abstract
  decorator, no wrapper classes. Adding a layer is one new function.
- **OOP Decorator wins when you're wrapping a behavioral object** — one with identity, state, or a multi-method
  contract you must preserve. The wrapper holds the real object and intercepts calls to it; the object graph stays
  intact.

**Where functional decoration actively loses, even though it looks simpler:**

- **Identity must be preserved.** A `T -> T` transform returns a *new* value each layer; callers holding a reference
  to the original, or relying on `==`/`instanceof`, are broken. The OOP Decorator wraps the same instance.
- **The layer carries rich state.** A decorator adding a buffer, a cache, or a connection across calls needs somewhere
  to keep that state. A pure transform is stateless; forcing state into closures gets ugly fast.
- **The contract is genuinely multi-method and you need all of it.** Collapsing a many-method type into "rebuild the
  whole thing each layer" stops being cheaper than a wrapper once the type is non-trivial — you reconstruct everything
  on every call.
- **Each layer intercepts behavior, not data.** Logging, access control, or caching *around a service* is wrapping a
  behavioral contract — the OOP Decorator's home ground. Functional fits when each layer *transforms a value*.

**The practical heuristic** (true most of the time, for the right reason): if you genuinely have a single-method
functional interface to decorate, functional almost always wins — because that situation only arises when the thing is
already a value/transform. The moment the thing is a stateful or multi-method object, reach for the OOP Decorator
despite its extra ceremony. [→ functional decoration code example](#what-does-functional-decoration-look-like-in-code)

</details>

### What does functional decoration look like in code?
<details><summary>Show answer</summary>

Model the decorated thing as an immutable value and each decoration as a `T -> T` transform, then **compose** the
transforms. No interface, no abstract decorator, no wrapper classes:

```java
// The decorated thing is a value, not a behavioral object
record Coffee(double cost, String description) { }

// Each decoration is a Coffee -> Coffee transform
UnaryOperator<Coffee> milk  = c -> new Coffee(c.cost() + 0.5, c.description() + ", milk");
UnaryOperator<Coffee> sugar = c -> new Coffee(c.cost() + 0.2, c.description() + ", sugar");

// Stacking decorators becomes andThen composition
Coffee result = milk.andThen(sugar).apply(new Coffee(2.0, "Simple coffee"));
// result -> Coffee[cost=2.7, description="Simple coffee, milk, sugar"]
```

Each `andThen` is one more layer; adding a feature is a **new function, not a new class**. Compare the OOP Decorator
for the same example — interface, abstract `CoffeeDecorator`, and a concrete class per feature — all of which collapse
here into two lambdas, *because* `Coffee` is pure data. The instant `Coffee` needed identity or state, this approach
would break and the wrapper-class version would be the right one.

</details>

### Notification design #0 — the starting tree
<details><summary><strong>Show details</strong></summary>

<details><summary>Show question</summary>

We model notifications. The one subtype axis is *kind of notification* — each kind sends differently and carries
different data, so a subclass tree is justified. Here is the starting hierarchy:

```java
abstract class Notification {
    Recipient recipient;        // every notification has one
    abstract void send();
}
class Email extends Notification { ... }   // the ONE tree: kind of notification
class Sms   extends Notification { ... }
class Push  extends Notification { ... }
```

</details>

<details><summary>Show answer</summary>

One tree, one dimension (kind). Everything we add from here, we place by asking: **field on the parent, field/branch
on a subclass, or a second tree?** — and a second tree is the thing we must avoid.
([the one-tree limit](#how-many-dimensions-may-inheritance-carry))

</details>

</details>

### Notification design #1 — adding `priority`
<details><summary><strong>Show details</strong></summary>

<details><summary>Show question</summary>

Every notification needs a `priority` — `LOW`, `NORMAL`, or `HIGH`. Any notification of any kind can be any priority.
Model it for the existing `Notification` hierarchy.

</details>

<details><summary>Show answer</summary>

```java
abstract class Notification {
    Recipient recipient;
    Priority  priority;         // LOW / NORMAL / HIGH — parent field
    abstract void send();
}
class Email extends Notification { ... }
class Sms   extends Notification { ... }
class Push  extends Notification { ... }
```

**Parent field.** Every kind can carry any priority, and the value is a free, independent pick — it combines with
every kind. That makes it an **independent dimension**, and an independent dimension is held as a **field**, never a
tree. ([how to test](#how-do-you-test-if-a-property-is-a-dimension))

It goes on the **parent** because it applies to *all* kinds — putting it on `Email`/`Sms`/`Push` separately would
duplicate the same field three times. A dimension shared by every object lives at the top. No second tree, no
multiplication.

</details>

</details>

### Notification design #2 — adding `subject`
<details><summary><strong>Show details</strong></summary>

<details><summary>Show question</summary>

We need to add a `subject` to notifications. Where does it go in the existing `Notification` hierarchy?

</details>

<details><summary>Show answer</summary>

**First ask the domain question — does *every* kind have a subject, or only some?** Placement depends entirely on the
answer; the property name alone can't decide it. ([what ultimately decides](#what-ultimately-decides-whether-a-property-is-a-dimension))

**Case A — only Email has a subject (the usual rule).** SMS and push have none, so `subject` is **branch-local** —
a way only the Email branch varies. It lives on the `Email` subclass.

```java
class Email extends Notification {
    String subject;             // branch-local — only Email
    ...
}
class Sms  extends Notification { ... }   // no subject
class Push extends Notification { ... }   // no subject
```

**Case B — every kind carries a subject** (e.g. your system shows one in the SMS/push title). Then it applies to all
objects → it's a shared property → it goes on the **parent**.

```java
abstract class Notification {
    String subject;             // on the base — every kind has one
}
```

The deciding test is "does it exist on *every* object, or only one branch?" — applies-to-all → parent; one-branch →
that subclass. Either way it's still just a **field**: no new tree, no multiplication.

</details>

</details>

### Notification design #3 — adding `deviceToken`
<details><summary><strong>Show details</strong></summary>

<details><summary>Show question</summary>

A push notification needs a `deviceToken`. A push can target either Android or iOS, and the token format differs by
platform. Where does this go in the existing `Notification` hierarchy?

</details>

<details><summary>Show answer</summary>

Two things arrive together — separate them.

**The token itself is branch-local to Push.** Email and SMS have no device token, so it lives on the `Push`
subclass, not the parent — same as any one-branch property. ([the three kinds](#what-kinds-of-property-can-an-object-have))

**Platform (Android / iOS) is a second dimension — do *not* make it a tree.** It's tempting to split
`AndroidPush` / `IosPush` because the format differs, but platform varies *independently* of kind, so a platform
tree would cross the kind tree → `AndroidPush`, `IosPush`, and later the same split repeated anywhere else → the
start of explosion. ([the one-tree limit](#how-many-dimensions-may-inheritance-carry))

Keep platform as a **field**, and if the format genuinely needs different behavior, hold that behavior as a strategy
object — still a field, still no second tree.

```java
class Push extends Notification {
    String   deviceToken;       // branch-local to Push
    Platform platform;          // ANDROID / IOS — a field, NOT a subclass split
    ...
}
enum Platform { ANDROID, IOS }
```

The format difference is real, but it's handled by the field's value (or a strategy keyed off it), never by a second
tree. ([how it's solved](#how-is-the-multiple-axes-modeling-problem-solved))

</details>

</details>

### Notification design #4 — transactional vs marketing
<details><summary><strong>Show details</strong></summary>

<details><summary>Show question</summary>

Every notification can be either **transactional** (password reset, receipt) or **marketing** (promo, newsletter).
The two differ in send-time rules: marketing must honor unsubscribe and quiet hours; transactional ignores both.
This applies to email, SMS, and push alike. Where does this go in the existing `Notification` hierarchy?

</details>

<details><summary>Show answer</summary>

This is a **second independent dimension** — *category* (transactional / marketing) — and it cuts across *every*
kind: any kind can be either, freely combined. That independence is the warning sign.
([what makes it a dimension](#how-do-you-test-if-a-property-is-a-dimension))

**A subclass tree is wrong** — it would cross the kind tree and multiply: `TransactionalEmail`, `MarketingEmail`,
`TransactionalSms`, `MarketingSms`, `TransactionalPush`, `MarketingPush` = kind × category. Two trees, the classic
explosion. ([the one-tree limit](#how-many-dimensions-may-inheritance-carry))

**A plain enum field is not enough either** — the categories carry *different behavior* (honor unsubscribe + quiet
hours, or skip both), not just a label. A bare `category` field would push that logic into `if`/`switch` branches
inside `send()`.

**So hold the behavior as a strategy on a field** — the second dimension rides as a component, not a tree:

```java
abstract class Notification {
    Priority    priority;
    SendPolicy  policy;         // second dimension as a STRATEGY field — not a subclass
    abstract void send();
}
interface SendPolicy { boolean maySendNow(Recipient r); }   // honor unsubscribe + quiet hours, or skip
class TransactionalPolicy implements SendPolicy { ... }     // ignores both
class MarketingPolicy     implements SendPolicy { ... }     // enforces both
```

Now kind stays the one tree, category is a field holding behavior, and the two compose: any kind × any policy with no
new classes. This is the general fix — a freely-combining dimension with behavior becomes a **strategy field**.
([how it's solved](#how-is-the-multiple-axes-modeling-problem-solved))

</details>

</details>
