## Classes and Interfaces - Inheritance 2


### What happens if a constructor invokes an overridable method?
<details><summary>Show answer</summary>

When a constructor invokes an overridable method, the superclass constructor runs **before** the subclass constructor.
As a result, the subclass’s overriding method is called on a **partially constructed object**.


If the overridden method relies on initialization performed in the subclass constructor,
it may observe uninitialized state and behave incorrectly, leading to subtle bugs and broken invariants.


**Constructors must not invoke overridable methods, directly or indirectly**.

</details>

### In which situations — besides constructors — should overridable methods not be invoked?
<details><summary>Show answer</summary>

Overridable methods should also **not** be invoked from methods that behave like constructors.

In particular if you do decide to implement either `Cloneable` or `Serializable`
in a class that is designed for inheritance:
- **`clone`**
- **`readObject`**

These methods can be executed on **partially initialized objects**, similar to constructors.
Invoking overridable methods from them may call subclass code before the subclass state is fully initialized,
leading to incorrect behavior and broken invariants.

</details>

### In Java, how can inheritance be restricted for concrete classes that are not designed for extension?
<details><summary>Show answer</summary>

There are two accepted ways to do this:

- **Declare the class `final`**  
  This is the simplest and most straightforward approach, ensuring that the class cannot be subclassed.

- **Restrict constructors and provide static factory methods**  
  Make all constructors private or package‑private and expose public static factory methods instead.
  This prevents classes outside the package from subclassing the class,
  while still allowing subclasses to be defined and used within the package if necessary,
  providing more flexibility than simply declaring the class `final`.

</details>

### How can Java define a type that allows multiple implementations?
<details><summary>Show answer</summary>

Java provides **two mechanisms** to define a type that allows multiple implementations:

- **Interfaces**  
  Define a contract that multiple, unrelated classes can implement.

- **Abstract classes**  
  Define a partially implemented type that subclasses can extend and complete.


[**Prefer interfaces over abstract classes**.](#interfaces-or-abstract-classes-how-should-a-type-with-multiple-implementations-be-defined)

</details>

### Interfaces or abstract classes: how should a type with multiple implementations be defined?
<details><summary>Show answer</summary>

A key difference between **interfaces** and **abstract classes** is how they fit into the class hierarchy:

- **Abstract classes**  
  A class must **extend** the abstract class to implement its type.
  Since Java allows only **single inheritance**, this severely limits their use as general type definitions.

- **Interfaces**  
  Any class can **implement** an interface, regardless of its position in the class hierarchy,
  as long as it provides the required methods and follows the contract.

</details>

### What is a mixin, and how is it used in Java?
<details><summary>Show answer</summary>

A **mixin** is a type that a class can implement in addition to its primary type to declare that
it provides some **optional behavior**. A mixin does not define what an object *is*, but rather what it *can do*.

For example, the `Comparable` interface is a classic mixin.
When a class implements `Comparable`,
it declares that its instances can be ordered relative to other instances of the same type.
Ordering is not the class’s primary purpose, but an additional capability that can be "mixed in".

Interfaces are ideal for defining mixins because a class can implement **multiple interfaces**,
regardless of its position in the class hierarchy.
This allows optional behavior to be added without affecting inheritance.

Abstract classes cannot serve this purpose because Java supports only **single inheritance**.
A class cannot extend more than one abstract class,
and there is no natural place in the hierarchy to insert a mixin without restricting or breaking existing designs.

</details>

### How can Java combine the flexibility of interfaces with reusable implementation code?
<details><summary>Show answer</summary>

Java combines the flexibility of interfaces with reusable implementation code in **two complementary ways**.

- **Interfaces with default methods**  
  An interface defines the type and may provide **default methods** that supply reusable behavior,
  including partial algorithm definitions.
  This allows a form of code reuse and even Template‑Method‑like behavior **without any abstract class**,
  while preserving multiple inheritance of behavior through interfaces.

- **Skeletal implementation classes provide reusable code**
  An abstract skeletal implementation class implements an interface and supplies default implementations
  for most methods, typically built on top of a small set of primitive methods that subclasses must provide.
  Extending such a class removes much of the boilerplate involved in implementing an interface
  and is a direct application of the **Template Method pattern**.

  By convention, skeletal implementation classes are named `AbstractInterface`,
  where `Interface` is the interface being implemented.

  The Java Collections Framework follows this pattern:
    - `AbstractCollection`
    - `AbstractSet`
    - `AbstractList`

</details>

### What should be considered before adding a default method to an interface?
<details><summary>Show answer</summary>

**Risk to existing implementations**

Adding default methods to existing interfaces is risky,
as it can silently break logic at runtime despite successful compilation.


As a consequence:
- **Use only when absolutely necessary**
  Adding default methods to existing interfaces should be avoided unless the need is critical.
  If unavoidable, carefully evaluate whether any existing implementation could be affected.

- **Best use case: new interfaces**  
  Default methods are most effective when defining **new interfaces**,
  where they can provide standard behavior and reduce implementation effort without risking backward compatibility.

</details>

### What is the proper purpose of an interface in Java?
<details><summary>Show answer</summary>

**Use interfaces only to define types.**

When a class implements an interface, the interface defines a **type**
that clients use to interact with instances of that class.
Implementing an interface should clearly communicate **what operations a client can perform** on those instances.


Using interfaces for any purpose other than defining such contracts is inappropriate
and leads to unclear or misleading APIs.

</details>

### What are the design options for exposing constants in Java?
<details><summary>Show answer</summary>

- **Add constants to an existing class or interface**  
  If the constants are strongly tied to a specific abstraction,
  they should be defined directly on that class or interface.  
  *Example:* `Integer.MIN_VALUE`, `Integer.MAX_VALUE`.

- **Use an enum type**  
  If the constants represent a fixed set of related values, they should be exported as an `enum`,
  which provides type safety and expressiveness.

- **Use a non‑instantiable utility class**  
  If the constants don’t naturally belong to a specific type,
  define them in a utility class with a private constructor and public static final fields.

</details>

### What are the drawbacks of using interfaces to expose constants?
<details><summary>Show answer</summary>

This practice is known as the **Constant Interface Pattern**, and it is a poor use of interfaces.

Using an interface solely to export constants abuses interfaces by employing them for something other
than **defining types**. Constants used by a class are an **implementation detail**,
and implementing a constant interface causes this detail to leak into the class’s public API.

While a few constant interfaces exist in the Java platform (for example, `java.io.ObjectStreamConstants`),
they are best regarded as **anomalies** and should not be emulated.

#### Interfaces as Constant Containers – Clarified Note

Most problems attributed to “constant interfaces” do not stem from storing constants in an interface,
but from a **completely incorrect use of inheritance**,
where such interfaces are implemented or extended by classes merely to access their constants.
This is a misuse: these interfaces are neither marker interfaces nor do they define any behavioral contract.
Consequently, all classic criticisms (type abuse, misleading “is‑a” relationships, API pollution via inheritance)
originate from this misuse, not from the interface itself.


Once this incorrect inheritance is explicitly excluded, only one reasonable argument remains:
**interfaces are primarily meant to define types with behavioral contracts**,
while constants usually represent implementation or configuration details that are often hidden.
This is a valid conceptual concern, but it is about design intent and conventions, not technical correctness.


On the other hand, using an interface purely as a constants namespace can offer a **simpler and clearer abstraction**:
no repetitive `public static final`, no defensive utility‑class boilerplate, and an immediately readable,
centralized definition of a stable concept. This simplicity reduces cognitive load and maintenance overhead.


The reason this topic often consumes disproportionate discussion time is that many arguments silently
assume the inheritance misuse. Once that assumption is removed, the debate becomes much simpler:
it is a **trade‑off between conceptual purity and pragmatic simplicity**.
Since simplicity is a core principle of software design, this trade‑off is legitimate and cannot be dismissed outright.


</details>

### What are tagged classes in Java?
<details><summary>Show answer</summary>

A **tagged class** is a class that represents multiple variants or behaviors by using an explicit **tag field**
(usually an `enum` or `int`) together with conditional logic (`if` / `switch`) based on that tag.


The tag determines which fields are meaningful and which logic applies at runtime.

```java
public class Shape {

  enum Kind {RECTANGLE, CIRCLE}

  private final Kind kind;

  // Rectangle fields
  private final double width;
  private final double height;

  // Circle field
  private final double radius;

  public Shape(double width, double height) {
    this.kind = Kind.RECTANGLE;
    this.width = width;
    this.height = height;
    this.radius = 0;
  }

  public Shape(double radius) {
    this.kind = Kind.CIRCLE;
    this.radius = radius;
    this.width = 0;
    this.height = 0;
  }

  public double area() {
    switch (kind) {
      case RECTANGLE:
        return width * height;
      case CIRCLE:
        return Math.PI * radius * radius;
      default:
        throw new AssertionError(kind);
    }
  }
}
```

</details>

### Why are tagged classes considered a design smell in Java, and what are the preferred alternatives?
<details><summary>Show answer</summary>

Tagged classes are considered a design smell because they **combine multiple distinct concepts into a single class**,
using a tag field (`enum`, `int`, etc.) and conditional logic to switch behavior. This often results in:
- complex `if` / `switch` logic,
- fields that are only valid for certain tags,
- violations of core OO principles such as **Single Responsibility** and **Open/Closed**.

#### Preferred alternatives

- **Class hierarchies (subclasses)**  
  Each variant is represented by its own class, using polymorphism instead of conditionals.

- **Enums with behavior**  
  When variants are fixed and well-known, enums can encapsulate variant-specific logic cleanly.

These alternatives improve readability, maintainability, and better align with object-oriented design principles.

</details>

### How can a developer prevent internal supporting types from leaking into the public API while still expressing complex behavior cleanly?
<details><summary>Show answer</summary>

A developer can prevent internal supporting types from leaking into the public API
by **encapsulating them within the scope where they are used**, keeping them invisible to external clients.


This can be achieved by using **nested classes**, which allow complex behavior to be expressed clearly
while confining auxiliary types to their natural owner, improving encapsulation, cohesion, and API clarity.


A nested class should exist only to serve its enclosing class.
If a nested class would be useful in some other context, then it should be a top-level class.

</details>

### What kinds of nested classes are supported in Java?
<details><summary>Show answer</summary>

- **Static nested classes**
    - static member classes

- **Inner classes** (non‑static nested classes)
    - non‑static member classes
    - local classes
    - anonymous classes

</details>

### What is the difference between static nested classes and inner classes?
<details><summary>Show answer</summary>

**Use a static nested class unless the class must access an enclosing instance**.

- **Static nested classes**
    - Do **not** hold an implicit reference to the enclosing instance
    - Can access only the enclosing class’s **static** members
    - Behave like top‑level classes scoped inside another class

- **Inner classes (non‑static nested classes)**
    - Hold an implicit reference to an instance of the enclosing class
    - Can access **both static and instance** members of the enclosing class
    - Are always associated with a specific enclosing object

</details>

### What is an anonymous class, and when is it useful in Java?
<details><summary>Show answer</summary>

An **anonymous class** is a class without a name that is **declared and instantiated at the point of use**.
It is not a member of the enclosing class and can appear anywhere an expression is allowed.


Anonymous classes may have an enclosing instance only when used in a non‑static context.
They cannot declare static members, except for compile‑time constant fields (`static final` primitives or strings).


They have several important limitations:
- they can be instantiated only at the point of declaration,
- they cannot be named, tested with `instanceof`, or reused,
- they can extend **one class** or implement **one interface**, but not both,
- clients can access only members of the declared supertype,
- they should be kept very small to preserve readability.


Before lambdas, anonymous classes were commonly used for small function objects and callbacks.
Today, **lambdas are generally preferred**, though anonymous classes are still useful in some cases,
such as concise implementations inside factory methods.

</details>

### What is a local class, and when is it useful in Java?
<details><summary>Show answer</summary>

A **local class** is a class declared **inside a block** (for example, within a method, constructor, or loop),
anywhere a local variable may be declared. It follows the same scoping rules as local variables.

Local classes:
- have a **name** and can be reused within their scope,
- have an enclosing instance only in a **non‑static context**,
- cannot declare static members,
- should be kept **short** to maintain readability.

They are useful when a helper class is needed **only within a small, localized scope**,
and giving it a name improves clarity compared to an anonymous class, while still avoiding exposure in the broader API.

</details>

### What are the best practices for organizing top‑level classes in Java source files, and why do they matter?
<details><summary>Show answer</summary>

A Java source file should contain **exactly one top‑level class or interface**,
and the file name should match that type.
Although the compiler allows multiple top‑level classes in one file, doing so brings no real benefits.


This practice matters because multiple top‑level classes in a single file can lead to **duplicate class definitions**,
where the version used at compile time depends on the order in which source files are compiled.
This makes builds fragile, order‑dependent, and hard to reason about.


Keeping one top‑level class per source file guarantees:
- unambiguous class definitions,
- predictable compilation and runtime behavior,
- simpler maintenance and clearer project structure.


Multiple top‑level classes per source file can silently produce
**different programs from the same code**, depending purely on compilation order.

</details>
