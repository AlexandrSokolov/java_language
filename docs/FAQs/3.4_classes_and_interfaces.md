## Classes and Interfaces - Inheritance and Composition

### Extending a hierarchy — what can go wrong?
<details><summary>Show answer</summary>

Putting something on the tree that can mix with every branch. A tree sorts objects into **one group**. If a new
thing is free to go with any branch, adding it forces a new class for every mix — a class per **pair**, so the count
grows by multiplying, not adding.

- [→ explosion on varying state](#how-does-subclassing-explode-on-varying-state)
- [→ explosion on varying behavior](#how-does-subclassing-explode-on-varying-behavior)

</details>

### How does subclassing explode on varying state?
<details><summary>Show answer</summary>

A tree splits objects into **one grouping** — every instance lands in exactly one leaf. When **state** that belongs
to every type, and combines freely with the existing leaves, is modeled as **subtypes** instead of held on the type,
each addition multiplies the leaves: a class per **combination** — the **product**, not the sum.

Example — drinks already group by kind: `Espresso`, `Tea`, `HotChocolate`. Now "milk" arrives (none / oat / soy).
Milk applies to every kind, freely. Push it into the tree and you get a class per pairing:

```java
class OatEspresso  extends Espresso { }
class SoyEspresso  extends Espresso { }
class OatTea       extends Tea { }
class SoyTea       extends Tea { }
// … kind × milk, and adding one milk adds a class to every branch
```

Milk has nothing to do with kind, yet subclassing fuses them. Add size or extras the same way and it's
kind × milk × size × extras.

</details>

### Given a hierarchy, where does new state go?
<details><summary>Show answer</summary>

Off the tree, on the object. State held as a field doesn't multiply — keep the tree for the **one** grouping you
genuinely subtype on, and put the free-varying state on the type. Which off-tree tool depends on what the state does:

- **Passive data** (a value you store and read — milk, size, a flag) → a **plain field**.
  [→ when a field fits](#when-should-a-new-concern-be-modeled-as-a-plain-field)
- **Acts and stacks** (changes behavior, layers in open combinations — coffee add-ons) → a field isn't enough; you
  need a **decorator**. [→ why a field isn't enough](#why-isnt-a-plain-field-always-enough)

</details>

### When should a new concern be modeled as a plain field?
<details><summary>Show answer</summary>

A plain field is the right tool when the new concern is **passive data** — a value you store and read, with no
behavior of its own. The markers, all pointing the same way:

- **It's data, not behavior.** Recording the value is all you need; it doesn't change how the object's methods
  compute. A flag, an enum, a number, a name.
- **It combines freely.** Every instance can carry it, independent of the other state — so it belongs on the type,
  not in a subtype tree.
- **It doesn't stack.** There's one value, not an open-ended pile of layered additions in arbitrary order.
- **You own the class.** You can add the field where it belongs (parent for all-types, branch for branch-local).

```java
class Coffee {
    Size    size;        // S / M / L — one value, pure data
    boolean decaf;       // a flag, recorded and read, no behavior
}
```

This is the default — reach for a field first. You leave it only when the concern stops being passive data: when it
carries behavior, or stacks in open combinations. Those push you toward strategy or decorator instead.
[→ when a field isn't enough](#why-isnt-a-plain-field-always-enough)

</details>

### Why isn't a plain field always enough?
<details><summary>Show answer</summary>

A field works while the concern is passive data. It breaks down the moment the concern **carries behavior** and
**stacks in open combinations** — because a field can only *record* a value, not *act*. The acting then leaks into
the type's methods as a growing pile of `if`/`else`.

Coffee add-ons are the classic case. Milk, caramel, soy each change the price and the description, and any number
combine in any order. Modeled as flags on the parent, every method has to know every add-on:

```java
class Coffee {
  boolean milk, caramel, soy;          // flags pile up — one per add-on

  double cost() {
    double c = base;
    if (milk) c += 0.5;            // behavior forced into the parent
    if (caramel) c += 0.75;
    if (soy) c += 0.4;            // every new add-on edits this method
    return c;
  }

  String description() {
    String d = "Coffee";
    if (milk) d += ", milk";       // the same branching, repeated
    if (caramel) d += ", caramel";
    return d;
  }
}
```

Three problems, all from the same root — the parent must know every add-on in advance:

- **The parent edits on every new add-on.** Adding "whip" means touching `cost()`, `description()`, and the field
  list — the class is not closed to extension.
- **Behavior is scattered into branches.** Each add-on's logic is smeared across multiple `if`s in multiple methods
  instead of living in one place.
- **Stacking is faked.** Two pumps of caramel, or order-sensitive layering, can't be expressed by booleans at all.

The field isn't wrong because the add-on combines freely — it's wrong because the add-on **does** something and
**stacks**. That's the shape a decorator handles. [→ how a decorator solves it](#when-is-decorator-the-right-tool)

</details>

### What must be considered when placing state in a hierarchy?
<details><summary>Show answer</summary>

**Gate 1 — is inheritance even safe to extend with state?** Adding a value attribute (state that counts toward
`equals`) to a subtype is sound only if **one** of these holds:

- the base is **abstract** — no base instance exists, so there's nothing to compare a subtype instance against; or
- the **ancestor** already has an **equals contract that accommodates extension**.

If **neither** holds and the state is equality-relevant, you **must not** model it with inheritance — compose
instead. (Behavior-only refinement that stays clear of `equals` is exempt; the trap is equality-relevant state on a
concrete base.) [→ equality contract and inheritance](2.1_methods_common_to_all_objects.md#equality-contract-and-inheritance)

**Gate 2 — parent or branch?** Once gate 1 passes, the **domain rules** decide, never the property's name. The same
property can sit on the parent in one system and on a branch in another:

- belongs to **every** type → **parent**;
- belongs to **one** subtype → **branch**.

</details>

### How does subclassing explode on varying behavior?
<details><summary>Show answer</summary>

Same multiplication as state, but the free concern is a **behavior**.
When a behavior applies to every type and any implementation can pair with any type,
modeling it as subtypes gives a class per **combination** — the product.

Example — messages already group by kind: `Email`, `Sms`, `Push`. Now *how it's sent* varies (sync, queued, batched),
and any kind can use any send mode. Push it into the tree and you get a class per pairing:

```java
class SyncEmail    extends Email { }
class QueuedEmail  extends Email { }
class BatchedSms   extends Sms { }
class QueuedPush   extends Push { }
// … kind × sendMode, and adding one mode adds a class to every branch
```

Kind and send mode are independent, yet subclassing fuses them.
Add a second varying behavior (say retry policy) and it's kind × sendMode × retry.

</details>

### Given a hierarchy, where does new behavior go?
<details><summary>Show answer</summary>

- **[Composition](#what-is-composition-and-what-does-it-solve) = behavior.**
  Hold behavior as a delegated part, chosen or layered at runtime:
  - **Strategy** — [one pluggable behavior, picked per instance.](#when-is-strategy-the-right-tool)
  - **Decorator** — [behavior that wraps and layers over a base.](#when-is-decorator-the-right-tool)
- **Mixin = capability.**
  [An extra interface the class implements — fixed per type at compile time.](3.6_classes_and_interfaces.md#mixin-vs-composition--whats-the-trade-off)


</details>

### When is strategy the right tool?
<details><summary>Show answer</summary>

Strategy fits when an object needs **one pluggable behavior** — a single job that can be done in different ways, where
the *which way* is held as a part the object delegates to, not baked into its class.

The markers, all pointing to strategy:

- **One slot, swappable choice.** There's a single behavior ("how it sends", "how it sorts") with several
  interchangeable implementations, and the object holds one of them at a time.
- **Chosen or changed at runtime.** The implementation can be picked when the object is built, or replaced while it
  lives — exactly what a subtype can't give you.
- **It varies independently of the tree.** Any kind of object can use any implementation, so subtyping on it would
  cross the classification and multiply (kind × strategy → explosion). Held as a delegated field, it doesn't.
- **It's behavior, not data.** A flag would only record a choice; a strategy *carries* the logic, so the object
  delegates the work to it instead of branching with `if`/`else`.

```java
interface SendStrategy { void send(Message m); }     // the one pluggable behavior

class Notification {
    private SendStrategy sender;                      // held as a part, not a subtype
    void setSender(SendStrategy s) { this.sender = s; }   // swappable at runtime
    void notify(Message m) { sender.send(m); }       // delegate, no branching
}
```

The shape in one line: **one behavior, many interchangeable implementations, chosen per instance.** If instead the
behavior *wraps and layers* over a base — many add-ons combining at once — that's decorator, not strategy.
[→ strategy vs decorator](#strategy-or-decorator--which-fits)

</details>

### When is decorator the right tool?
<details><summary>Show answer</summary>

The **Decorator** pattern — for behavior that **adds to** what's already there, rather than replacing it. You *wrap*
the object in another that implements the **same interface**, adds its own bit, and delegates the rest to the wrapped
instance. Because each wrapper returns that same interface, wrappers nest — any number combine, in any order.

This is the case strategy doesn't cover: strategy swaps *one* behavior for another; a decorator keeps the original
and layers more over it.

```java
interface Coffee {
  double cost();
  String description();
}

abstract class Addon implements Coffee {          // wraps a Coffee, holds the same interface
  protected final Coffee inner;
  protected Addon(Coffee inner) { this.inner = inner; }
}

class Milk extends Addon {
  Milk(Coffee inner) { super(inner); }

  public double cost() { return inner.cost() + 0.5; } // adds its bit, delegates the rest

  public String description() { return inner.description() + ", milk"; }
}

Coffee c = new Caramel(new Milk(new Espresso()));   // layered, any order, zero combination classes
```

Because the decorator depends only on the **interface contract** — never the wrapped object's internals — it also
sidesteps the fragile-base-class problem and preserves encapsulation.
[→ strategy or decorator](#strategy-or-decorator--which-fits)

</details>

### Strategy or decorator — which fits?
<details><summary>Show answer</summary>

Both hold behavior as a delegated part instead of subtyping, but they answer different questions — **replace one
behavior** vs **layer many behaviors**.

- **Strategy — one slot, swap the implementation.** The object has a single job done one way at a time; you pick or
  change *which* implementation. One strategy is active per slot. "How does this send / sort / price?"
- **Decorator — many layers, stack them.** Each add-on wraps the object through the same interface and adds its bit,
  so any number combine in any order. The base keeps working; wrappers nest. "What extras pile onto this?"

The deciding questions:

- **Replace or accumulate?** One behavior chosen from alternatives → strategy. Open-ended behaviors that add up → decorator.
- **One at a time, or many at once?** A single active choice → strategy. Several stacked together → decorator.
- **Same interface, nested?** If each layer must return the same type so the next can wrap it, that's the decorator
  shape; strategy doesn't nest.

```java
notification.setSender(new EmailSender());            // strategy: one sender, swapped wholesale

Coffee c = new Caramel(new Milk(new Espresso()));     // decorator: layers stacked, any order
```

In one line: **strategy swaps a single behavior; decorator stacks many.** Pick by whether the behaviors *replace*
each other or *combine*.

</details>

### Decoration vs inheritance
<details><summary>Show answer</summary>

Three ways to add behavior — inheritance, OOP Decorator, and the functional form:

**Inheritance**
- *Pro:* simple, direct; behavior fixed at compile time; one object, easy to debug.
- *Con:* a separate class per combination → **class explosion** (`CoffeeWithMilkAndSugar`, …);
  combinations fixed at compile time; couples to superclass internals.

**Decorator (OOP composition)**
- *Pro:* combine behaviors by **stacking** at runtime — `Milk` + `Sugar` with no `MilkSugarCoffee`
  class; open-ended; depends only on the interface, so encapsulation holds.
- *Con:* not free — many small wrapper objects; the wrapped object's **identity is lost** (`==`/
  `instanceof` see the decorator); debugging steps through delegation layers; order-sensitive
  decorators must be applied in the right order.

[**Functional alternative**](#functional-decoration-vs-the-oop-decorator)
A third option models decoration as a `T -> T` transform and composes the functions — best when the decorated thing is
a value, not a stateful object.


**When each wins:**
- inheritance for a small, fixed set along *one* axis;
- OOP Decorator when features combine freely and need a shared interface;
- functional composition when the decoration is a simple transform and you want minimal ceremony.

</details>

### What is composition, and what does it solve?
<details><summary>Show answer</summary>

One idea: **hold separate parts and delegate to them**, instead of inheriting a fixed implementation. 
It fixes two different inheritance failures, in two forms:

- [**Delegation** → the **encapsulation** problem](3.3_classes_and_interfaces.md#what-is-the-relationship-between-inheritance-and-encapsulation). 
  Hold an object as a field, delegate to its public contract, never touch its internals.
- [**Behavioral composition** → the **class-explosion** problem](#given-a-hierarchy-where-does-new-behavior-go). 
  Hold varying behavior as a delegated part (strategy, decorator) instead of a second subtype tree.


[Its purest form is **composing functions** — same principle, no hierarchy.]#how-does-functional-programming-reframe-strategy-and-decorator)


(A plain data field isn't composition — that's just state. Composition is delegating to a part.)

</details>

### If composition is preferable, why not avoid inheritance entirely?
<details><summary>Show answer</summary>

Because "prefer composition" is a *default*, not an absolute — and inheritance retains genuine
advantages composition can't replicate cleanly. The guidance is narrower than it sounds: it warns
against **implementation inheritance across a boundary you don't control**, not against all inheritance.

Where inheritance is still the right tool:

- **A genuine, stable is-a relationship.** When the subtype truly *is* the supertype and substitutes
  for it everywhere (LSP holds), inheritance models it directly. Forcing composition here adds
  indirection to express a relationship the language already has a keyword for.
- **Polymorphism is the goal.** If the point is to treat many subtypes uniformly through a supertype
  reference, inheritance (or interface implementation) is the mechanism that provides it. Composition
  doesn't give you subtype substitutability.
- **Designed and documented for extension.** When the superclass author has opened the class for it —
  self-use documented, hooks provided — the coupling is part of the contract, so the main objection
  (depending on undocumented internals) goes away.
- **Less boilerplate for true specialization.** Composition requires forwarding every delegated method;
  for a real subtype that extends a stable base, inheritance avoids that forwarding cost.

What composition buys that inheritance can't — and why it's the *default*:

- It avoids the encapsulation-weakening coupling entirely (you depend only on a public contract).
- It composes along **multiple independent axes** without combinatorial class explosion.
- The wrapped object can be swapped or chosen at runtime; an inherited superclass is fixed at
  compile time.

So the rule isn't "never inherit" — it's "**don't reach for inheritance by default for code reuse.**"
Use it when there's a real is-a, substitutability holds, or the class is built for extension; reach for
composition for everything else — which, in practice, is most of the time.

</details>

### How can object-oriented designs mitigate the impact of inheritance on encapsulation?
<details><summary>Show answer</summary>

- **Reach for composition when you only want reuse.** If you're extending a class just to borrow its code, holding
  it as a field and delegating to its public contract avoids the internal coupling — you depend on the contract, not
  the implementation. This isn't a verdict against inheritance; it's the better fit *when there's no real is-a*. When
  there is one, inheritance stays on the table.
  ([when inheritance is still right](#if-composition-is-preferable-why-not-avoid-inheritance-entirely))
- **Inherit only from classes built for it.** When you genuinely need inheritance, extend classes that are
  *documented for extension* or *under your own control*, so the self-use you rely on is part of the contract, not an
  accident you can trip over.
- **Program to interfaces, not implementations.** Depend on contracts rather than concrete types, so callers stay
  insulated and you keep the freedom to change what's behind the contract.

</details>

### How does functional programming reframe Strategy and Decorator?
<details><summary>Show answer</summary>

Strategy and Decorator are, at heart, just *passing a function* — once functions are first-class you don't need a
class to vary behavior:
- **Strategy → a function parameter** — pass the algorithm directly (`x -> x * x`) instead of a `Strategy` interface
  plus implementing classes.
- **Decorator → a wrapper function** — `withLogging(f)` returning an enhanced function is function composition, not a
  wrapper class.

So FP reframes a chunk of the GoF catalog: where OOP reaches for a class to vary behavior along an axis, FP often just
passes a function — same goal, far less ceremony. The patterns shrink to functions.

</details>

### Functional decoration vs the OOP Decorator?
<details><summary>Show answer</summary>

Both add behavior in layers without subclassing, but they decorate **different kinds of things**.

**The real marker: is the decorated thing a *value/transform*, or a *stateful, behavioral object*?**

- **Functional wins when you're transforming a value.** If each layer takes the thing and returns an enhanced version
  (`T -> T`), decoration is just function composition — chain the transforms with `andThen`. No interface, no abstract
  decorator, no wrapper classes. Adding a layer is one new function.
- **OOP Decorator wins when you're wrapping a behavioral object** — one with identity, state, or a multi-method
  contract you must preserve. The wrapper holds the real object and intercepts calls to it; the object graph stays
  intact.

**Where functional decoration actively loses, even though it looks simpler:**

- **Identity must be preserved.** A `T -> T` transform returns a *new* value each layer; callers holding a reference
  to the original, or relying on `==`/`instanceof`, are broken. The OOP Decorator wraps the same instance.
- **The layer carries rich state.** A decorator adding a buffer, a cache, or a connection across calls needs somewhere
  to keep that state. A pure transform is stateless; forcing state into closures gets ugly fast.
- **The contract is genuinely multi-method and you need all of it.** Collapsing a many-method type into "rebuild the
  whole thing each layer" stops being cheaper than a wrapper once the type is non-trivial — you reconstruct everything
  on every call.
- **Each layer intercepts behavior, not data.** Logging, access control, or caching *around a service* is wrapping a
  behavioral contract — the OOP Decorator's home ground. Functional fits when each layer *transforms a value*.

**The practical heuristic** (true most of the time, for the right reason): if you genuinely have a single-method
functional interface to decorate, functional almost always wins — because that situation only arises when the thing is
already a value/transform. The moment the thing is a stateful or multi-method object, reach for the OOP Decorator
despite its extra ceremony. [→ functional decoration code example](3.8_classes_and_interfaces_design.md#what-does-functional-decoration-look-like-in-code)

</details>
