## Generics 3 - Generic Methods

### What are the rules for using generic type parameters with static methods or static fields?
<details><summary>Show answer</summary>

A static member **cannot use the class-level type parameter.**

**Why:** the class-level `T` (in `class Box<T>`) is bound per *instance* — `new Box<String>()` sets `T = String`. 
But a static member belongs to the *class*, which is shared by all instances at once:
`Box<String>` and `Box<Integer>` share the same statics, so there is no single `T` a static could refer to.

**The fix:** if a static method needs to be generic, it declares its **own** type parameter, 
independent of the class — this is a **generic method**:

```java
public class Box<T> {
    // ILLEGAL — 'T' is the class parameter, not available in a static context
    // public static void printBox(Box<T> box) { ... }

    // LEGAL — the method declares its own independent parameter <E>
    public static <E> void printBox(Box<E> box) { ... }
}
```

</details>

### How do you resolve a compilation error when the compiler fails to infer the correct generic type for a method call?
<details><summary>Show answer</summary>

Supply an **explicit type argument** (a *type witness*) — put the type in angle brackets right before the method name, 
instead of letting the compiler infer it:

```java
printNames(Collections.emptyList());           // may infer List<Object> → error
printNames(Collections.<String>emptyList());   // explicit <String> → fixes it
```

**Syntax:**
* For static methods: `ClassName.<Type>methodName()`
* For instance methods: `instance.<Type>methodName()`
* For methods in the same class: `this.<Type>methodName()`

Java 8+ improved target-type inference, so this is rarely needed now — 
but it's the fallback when the compiler can't work the type out.

</details>

### Named type parameter `<E>` vs wildcard `<?>` — when should you use each, and why?
<details><summary>Show answer</summary>

Decision ladder — start at the lightest tool, drop one rung each time a requirement makes it impossible.

1. Default: plain `<?>`. The lightest tool, because the signature is simplest. It carries two separate limits:

   * **No name.** You can't refer back to the type, so you can't:
    - use it as a return type,
    - declare a local of that type to hold an instance,
    - refer to "an `E`" anywhere in the body,
    - bind two positions to the same type — every `?` is independent.
   * **Read-only.** The exact type is unknown, so the compiler won't let you write into it.

    Stay here unless a rule below forces you off. (Adding a bound — `? extends X` / `? super X`, the PECS choice — is a
   separate axis layered on *after* you've picked the wildcard, not a reason to pick it.)
   
2. Must you name the type *inside the body* — declare a local of it, return it? 
   `<?>` has no name to refer back to → `<E>`.

3. Do two positions have to be the *same* unknown type? Each `?` is a *fresh, independent* unknown — `List<?>, List<?>`
   reads as "a list of unknown-1 and a list of unknown-2", not two lists of one shared type. A name is the only way to
   say "these positions agree." → give the type a name: `<E>`.

    ```java
       // Named: one E used in both positions → both lists bound to the SAME type
       static <E> boolean sameElements(List<E> a, List<E> b) {
           for (int i = 0; i < a.size(); i++)
               if (!a.get(i).equals(b.get(i))) return false;   // a's E and b's E are the same type
           return true;
       }
    
       // Wildcard: a is List<?>, b is List<?> — two unrelated unknowns, nothing lines them up
       // static boolean sameElements(List<?> a, List<?> b) { ... }   // can't express "a and b agree"
    ```

4. Must you *put a value back* of that unknown type (swap, set, fill)? `<?>` and `<? extends X>` are read-only — the
   exact type is unknown, so the compiler rejects the write. Pick the exit by asking who the type name serves:

   * **Name it with `<E>` (public)** — when the type appears in *more than one* public position that must agree
    (e.g. `copy(List<E> dst, List<E> src)`). The name has a real public job — binding those positions — so exposing
    it is honest, not noise. Capture would buy nothing here.
   * **Keep `<?>` public, capture privately** — when the type appears in *one* public position and you need the name
     *only* to write (e.g. `swap(List<?> list, int i, int j)`). The caller never supplies or relates the type, so
     leaking `<E>` is pure noise; a private generic helper captures the wildcard and does the write out of sight.

   Rule of thumb: name it if the name serves the *caller* (relates params, or is returned); capture it if the name
   serves only the *implementation*. Both compile identically — this is an API-cleanliness call, not correctness.

Trigger that regenerates the chain: wildcard until you must name the type — to bind two positions, to use it in the
body, or to write it back — then go named, or capture it privately.

</details>

### Which signature should you prefer for a `swap(list, index1, index2)` method: one with `<E>` or one with `<?>`?
<details><summary>Show answer</summary>

Prefer the **wildcard**: `public static void swap(List<?> list, int i, int j)` — the type appears once,
so a wildcard is simpler and cleaner for callers.
([why — named vs wildcard](#named-type-parameter-e-vs-wildcard---when-should-you-use-each-and-why))

**But the wildcard breaks the implementation.** A `List<?>` is read-only — the element type is unknown,
so the compiler won't let you put anything back, *even an element you just took out of the same list*:
```java
public static void swap(List<?> list, int i, int j) {
    list.set(i, list.set(j, list.get(i)));   // COMPILE ERROR — can't write into List<?>
}
```

**The fix — wildcard capture.** Keep the clean `<?>` public API, but delegate to a private generic
helper. The helper's named `<E>` *captures* the unknown type, giving it a name so the compiler can
verify the take-out/put-back is type-safe:
```java
public static void swap(List<?> list, int i, int j) {
    swapHelper(list, i, j);                  // public: clean wildcard
}
private static <E> void swapHelper(List<E> list, int i, int j) {
    list.set(i, list.set(j, list.get(i)));   // private: E names the type → write is safe
}
```

This is the standard way to offer a simple wildcard API without losing the ability to write:
**wildcard outside, captured type parameter inside.**

</details>

### What signature should a `copy(dst, src)` method use, and why?
<details><summary>Show answer</summary>

```java
public static <T> void copy(List<? super T> dst, List<? extends T> src)
```

This case needs **both** a named type parameter **and** wildcards — here's why each is there:

- **Named `<T>` — because the type links two parameters.** The destination and source must agree on an
  element type: whatever `T` is, you read `T`s from `src` and write them into `dst`. A wildcard can't do
  this — it has no name to tie the two positions together. The type appears in *more than one position*,
  so it must be named.
  ([why — named vs wildcard](#named-type-parameter-e-vs-wildcard---when-should-you-use-each-and-why))
- **`src` is a producer → `? extends T`** — you only read out of it.
- **`dst` is a consumer → `? super T`** — you only write into it.
  (This is PECS applied to each parameter.)

So `copy` is the case where naming and wildcards work *together*: `T` names the shared element type, and
the two wildcards make each end as flexible as possible for callers.

</details>

### What signature should a `firstOf(list)` method use, and why?
<details><summary>Show answer</summary>

```java
public static <T> T firstOf(List<T> list)
```

Needs a **named `<T>`** — because the method **returns** the element type. The return value's type must
have a name the caller receives as a concrete type:
```java
String s = firstOf(listOfStrings);   // T is inferred as String — caller gets a String
```

A wildcard can't do this. `List<?> firstOf(...)` returning `?` is meaningless, and returning a wildcard
type would force the *caller* to deal with a wildcard variable — the exact "don't return wildcards"
problem. The type appears in the return position and must be named.
([why — named vs wildcard](#named-type-parameter-e-vs-wildcard---when-should-you-use-each-and-why))

Contrast with `printAll`, which only *consumes* the list and needs no name → a wildcard is enough.

</details>

### What signature should a `printAll(list)` method use, and why?
<details><summary>Show answer</summary>

```java
public static void printAll(List<?> list)
```

Needs only a **wildcard** — the simplest case. The method just *reads* each element and prints it; the
element type appears **once** and is never named or written in the body, so there's nothing to gain
from a named parameter:
```java
public static void printAll(List<?> list) {
    for (Object o : list) System.out.println(o);   // reads as Object — all you need
}
```

A named `<T>` here would be pure overhead — you'd never use the name. Wildcard keeps the API as flexible
as possible (accepts a `List` of any type) and simpler to read.
([why — named vs wildcard](#named-type-parameter-e-vs-wildcard---when-should-you-use-each-and-why))

This is the clean mirror of `firstOf`: that one *returns* the type so it must name it; this one only
*consumes*, so it shouldn't.

</details>

### How do you define a method that accepts a variable number of arguments?
<details><summary>Show answer</summary>

Use **varargs** — three dots (`...`) after the type. The caller passes zero or more arguments of that
type:

```java
public static int sum(int... numbers) {   // 'numbers' is an int[] inside the method
    int total = 0;
    for (int num : numbers) total += num;
    return total;
}

sum();          // 0  — empty array
sum(1, 2, 3);   // 6
```

**Key mechanism:** under the hood Java packs the arguments into an **array** of that type, and inside
the method you use the parameter as a normal array. (This array detail is exactly what causes trouble
when the type is generic — the next cards.)

**Rule:** a method can have only **one** varargs parameter, and it must be **last**
(`void f(String name, int... nums)`).

</details>

### How do you use varargs with generic type parameters?
<details><summary>Show answer</summary>

You declare it like normal varargs: `static <T> void process(T... args)`. But this hides a real type-safety problem.

**Why it's dangerous — the array/generics contradiction.** Varargs are implemented as an **array**, but
arrays are *reified* (carry their type at runtime) while generics are *erased* (don't). So a `T...`
parameter creates an array of a non-reifiable type — really an `Object[]` at runtime. This opens
**heap pollution**: the array can come to hold the wrong type without failing immediately, surfacing
later as a `ClassCastException` that seems to come from nowhere:
```java
static void dangerous(List<String>... stringLists) {
    Object[] objects = stringLists;        // the varargs array is really Object[]
    objects[0] = List.of(42);              // sneak in a List<Integer> — heap pollution
    String s = stringLists[0].get(0);      // CRASH: ClassCastException (Integer ≠ String)
}
```
Because of this, the compiler warns on every generic varargs method.

**If you must use generic varargs — keep it safe and mark it `@SafeVarargs`.** The method is safe only
if it **never writes into** the hidden array and **never returns/exposes** it — only reads from it.
When you guarantee that, annotate with `@SafeVarargs` to suppress the warning:
```java
@SafeVarargs
static <T> void printAll(T... elements) {
    for (T e : elements) System.out.println(e);   // safe: read-only
}
```

**The better alternative — take a `List` instead.** Generics and Lists work together cleanly 
(both checked at compile time), so replacing the varargs with a `List` removes the array entirely — no heap pollution, 
no `@SafeVarargs` needed:
```java
static void safe(List<List<String>> stringLists) {
  // Object[] objects = stringLists;   // WON'T COMPILE — a List is not an Object[]
  // no hidden array exists, so there is no heap-pollution hole to exploit
  String s = stringLists.get(0).get(0);   // always safe
}
```
Flatten example:
```java
static <T> List<T> flatten(List<List<? extends T>> lists) {
  List<T> result = new ArrayList<>();
  for (List<? extends T> list : lists) result.addAll(list);
  return result;
}

flatten(List.of(list1, list2, list3));   // List.of() keeps the call site just as clean
```

**Rule:** prefer a `List` parameter; 
use generic varargs only when you must, and only with `@SafeVarargs` on a read-only method.

</details>

### An app mixes generic and raw-type code — how do you find what's polluting a collection?
<details><summary>Show answer</summary>

**The problem.** When an application mixes generic code with old or poorly written raw-type code, 
a raw `List` reference can insert the wrong type into your `List<String>` — and because of erasure, 
nothing checks it at the point of insertion. The collection is corrupted silently; the failure surfaces *later*,
as a `ClassCastException` somewhere far away when a value is read out. 
That gap between cause and symptom makes the bug very hard to track down.


**The fix — a checked wrapper.** `Collections.checkedList(list, String.class)` (and `checkedSet`, `checkedMap`, ...) 
wrap the collection with a **type token** (`Class<T>`) and check every element *as it is added*, at runtime. 
A wrong-type insertion fails **immediately** with a `ClassCastException` — at the actual offending line, 
even through a raw reference.


So the wrapper turns a silent, delayed corruption into an instant, 
located failure — which is how you find the raw-type code polluting the collection.

</details>
