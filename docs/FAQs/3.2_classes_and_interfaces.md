## Classes and Interfaces - Mutability

### What single API design practice most improves a class's predictability and safety?
<details><summary>Show answer</summary>

**Minimize mutability** — prefer immutable classes.

They're easier to design, implement, and use than mutable classes, less error-prone, and more secure.

</details>

### What are the benefits of using immutable classes?
<details><summary>Show answer</summary>

- **Simpler reasoning** — a single, fixed state established at construction; the object never changes under you.
- **Strong invariants** — once the constructor validates them, they hold forever.
- **Inherent thread safety** — no synchronization needed. The most practical payoff in real systems.
- **Safe sharing** — freely reused and shared without defensive copies.
- **Safe in collections** — stable `equals`/`hashCode`, so keys and set members never silently "move."

</details>

### How can a class be designed to minimize mutability?
<details><summary>Show answer</summary>

**Behavioral rules — how the class presents change:**

1. **No mutators that change the object itself.** Fully eliminating "change" is impractical, so the rule is narrower: 
   no method may modify *this* instance's state in place.
2. **Single-field changes → return a modified copy (functional style).** 
   Expose change as a method that builds a new instance with one field different — the *withers* pattern.
   [→ withers](#what-is-the-withers-pattern-in-java-records)
3. **Multiple-field changes → a mutable companion (builder).** When many fields change together,
   mutate inside a builder and produce the immutable instance at the end.
   [→ mutable companion / builder](#what-techniques-can-be-used-to-reduce-allocation-overhead-in-immutable-classes)
4. **Accessors:** immutable fields may be returned directly; 
   **mutable** fields must be returned as a **defensive copy** (and copied on the way in, too).

**Enforcement rules — what locks the behavior down:**

5. **All fields `final`** — enforces immutability and guarantees safe publication across threads.
6. **All fields `private`** — without this, immutability is only *hoped for*. Minimizing mutability
   prevents state changes; private fields prevent *uncontrolled* (outside reach-in) state changes.
7. **Prevent subclassing** — so a subclass can't reintroduce mutability. (`final` class, or
   non-public constructors + static factories —
   [details](#what-options-exist-to-prevent-subclassing-in-immutable-classes-and-how-do-they-compare).)

</details>

### What is the main limitation to consider with immutable classes?
<details><summary>Show answer</summary>

The core fact: **every change requires a brand-new object** — no method modifies state in place, so any
"modified" version is a fully-constructed fresh instance.

This makes performance a **two-directional trade-off**, not a one-way cost — and the deciding factor is
the access pattern:

- **Read-heavy / shared use → immutability *improves* performance.** No defensive copies are needed (in or out), 
  and a single instance can be freely shared and reused across threads without  synchronization. 
  You pay nothing for safety you'd otherwise have to buy with copying or locking.

- **Change-heavy use → immutability *degrades* performance.** Frequent or multi-field updates
  (especially chained withers, and large object graphs) generate heavy allocation, copying, and GC
  pressure from discarded intermediates. Incremental updates also become less ergonomic — a one-field
  tweak reconstructs the whole object.

**So the real consideration is recognizing which regime you're in.** 
Immutability is close to free — even a net win — when objects are built once and read/shared many times; 
it gets expensive when objects are mutated often. 
Everything that follows (withers, builders, mutable companions) exists to manage the *change-heavy* side of this trade.

</details>

### If an immutable class must allow changes, how should that be offered?
<details><summary>Show answer</summary>

Never with in-place mutators. 
Express change as **producing a new instance** — and the right mechanism depends on *how many fields change at once*:

**1. Single-field change → wither (functional copy).**
A method that returns a new instance with one field replaced, the rest copied:
```java
User withAge(int age) { return new User(this.name, age, this.email); }
```
Clean and readable for `User u2 = u1.withAge(31);`.
[→ withers pattern](#what-is-the-withers-pattern-in-java-records)

**2. Multiple-field changes → mutable companion (builder).**
Mutate freely inside a builder, then produce one immutable instance at the end.
[→ builder for batch updates](#what-techniques-can-be-used-to-reduce-allocation-overhead-in-immutable-classes)

**Why not just chain withers for multi-field changes?**
`u.withName(n).withAge(a).withRole(r)` looks convenient but creates a throwaway instance at *every* step — 
exactly the allocation/GC cost immutability already carries, multiplied. 
Chained withers are the classic sign the class is being used as if it were mutable; that's the builder's job.

**Why not make a wither return a wrapper to block chaining?**
You *could* have `withName` return some `Pending<User>` with no further `withX` methods, forcing a terminal `build()`. 
But that just reinvents the builder through a worse door: 
you get a builder's shape (mutation steps + terminal build) plus an unexpected wrapper type, 
and you lose the wither's one real virtue — the clean single-update return. 
The idiomatic fix isn't to cripple the wither's return;
it's to provide a real builder and keep the wither for single updates only.

</details>

### What is the withers pattern in Java records?
<details><summary>Show answer</summary>

A *wither* returns a **new** record instance with **one** component changed, copying the rest from the original:

```java
public record User(String name, Integer age) {
  User withName(String name) { return new User(name, this.age); }
  User withAge(Integer age)  { return new User(this.name, age); }
}
```

Use it for **single-field updates**. 
(For multi-field changes, reach for a builder — chaining withers allocates a throwaway instance per call.)

</details>

### What should be considered when using withers in Java records?
<details><summary>Show answer</summary>

Use withers **carefully, and primarily for single-field updates** — 
a wither earns its place when you change *one* component.

The misuse to watch for is **chaining**:
```java
user.withName(n).withAge(a).withRole(r);
```
This is usually the first sign of trouble. Each call constructs a full new record, 
so the chain allocates `n-1` throwaway instances just to reach the final one — needless allocation and GC pressure,
which is exactly the cost immutability already imposes, now multiplied per step.

It's also a *design* signal, not only a performance one: 
reaching for chained withers usually means the object is being treated as if it were mutable. 
When multi-field updates are frequent, that's the cue to switch to a 
[builder](#what-techniques-can-be-used-in-java-records-to-reduce-allocation-overhead-caused-by-immutability),
which mutates internally and allocates once.

</details>

### How can allocation overhead be reduced when an immutable object needs multiple changes?
<details><summary>Show answer</summary>

Provide a **mutable companion class** — the structural answer to the multi-change case 
(chained withers allocate one throwaway per step; a companion allocates **once**).

The idea: **separate mutation from representation.**
- The companion accumulates all the changes in mutable fields — no intermediate immutable instances.
- A terminal `build()` produces a single immutable instance from the final state.
- The immutable class copies that state in and never changes it afterward.

```java
// Immutable class
final class Point {
  private final int x, y;
  Point(PointBuilder b) { this.x = b.x; this.y = b.y; }
}

// Mutable companion
class PointBuilder {
  int x, y;
  PointBuilder setX(int x) { this.x = x; return this; }
  PointBuilder setY(int y) { this.y = y; return this; }
  Point build() { return new Point(this); }
}
```

**Canonical JDK example: `String` + `StringBuilder`.** 
Building a string by repeated `+` concatenation allocates a new immutable `String` at every step. 
`StringBuilder` mutates a char buffer in place and produces the immutable `String` once, 
via `toString()` — the exact same companion pattern, and the reason string-building in loops should use it.

**Relation to withers:** wither = single change, one new instance (fine). 
Companion/builder = many changes, still one new instance (the multi-change tool). 
Same goal — never mutate the shared immutable — different allocation profile.

</details>

### What techniques can be used in Java records to reduce allocation overhead caused by immutability?
<details><summary>Show answer</summary>

The record idiom combines **both** tools on one type — a wither for single updates, 
a nested builder for batch updates — so each access pattern allocates the minimum:

- **Single-field update → wither** (`withAge`): one new record, clean call site.
- **Multi-field update → `toBuilder()` + builder**: copy the current record into a mutable builder,
  apply all changes there, build once. Mutation stays inside the builder; the record remains immutable.

The load-bearing piece is **`toBuilder()`** — it seeds a builder *from an existing instance*, 
which is what turns the builder from a from-scratch constructor into an efficient **update** mechanism.

```java
public record User(String name, int age, String email) {

  public User withAge(int age) { return new User(this.name, age, this.email); }

  public Builder toBuilder() { return new Builder(this); }

  static final class Builder {
    private String name, email;
    private int age;

    private Builder(User u) { this.name = u.name; this.age = u.age; this.email = u.email; }

    public Builder name(String name)   { this.name = name;   return this; }
    public Builder age(int age)        { this.age = age;     return this; }
    public Builder email(String email) { this.email = email; return this; }

    public User build() { return new User(name, age, email); }
  }
}
```
```java
User ageUpdated = user.withAge(10);                    // single

User updated = user.toBuilder()                        // batch
        .name("Alice").age(30).email("a@example.com")
        .build();
```

This is the [mutable-companion pattern](#how-can-allocation-overhead-be-reduced-when-an-immutable-object-needs-multiple-changes)
in its record-specific form.

</details>

### How can a Java record support both creation-from-scratch and update-of-existing via a builder?
<details><summary>Show answer</summary>

A builder has two distinct jobs, and they need two distinct entry points:

- **`builder()`** (static, no args) — create a new instance from scratch; the builder starts empty.
- **`toBuilder()`** (instance method) — update an existing instance; 
  the builder is seeded with the current values, so you change only what you want.

A single static `builder()` *cannot* express "update an existing User" — it has no starting state. 
So a record that needs both supplies both, backed by two private builder constructors (empty / from-instance):

```java
public record User(String name, int age, String email) {

  public static Builder builder() { return new Builder(); }   // create
  public Builder toBuilder()      { return new Builder(this); } // update

  public static final class Builder {
    private String name, email;
    private int age;

    private Builder() {}                                       // empty → from scratch
    private Builder(User u) {                                  // seeded → from existing
      this.name = u.name; this.age = u.age; this.email = u.email;
    }

    public Builder name(String name)   { this.name = name;   return this; }
    public Builder age(int age)        { this.age = age;     return this; }
    public Builder email(String email) { this.email = email; return this; }

    public User build() { return new User(name, age, email); }
  }
}
```
```java
User u1 = User.builder()                                       // creation
  .name("Alice").age(30).email("a@example.com")
  .build();

User u2 = u1.toBuilder()
  .age(31)
  .build();                      // batch update
```

(You can still keep a wither like `withAge` for the single-field case — the builder is for multi-field work.)

</details>

### How does subclassing interact with immutability?
<details><summary>Show answer</summary>

Immutability is a **behavioral guarantee** — "observable state never changes after construction" —
not merely a property of field modifiers. `final` fields are **necessary but not sufficient**: a class
with all-final fields is still *not* truly immutable if it can be subclassed.

If subclassing is allowed, a subclass can defeat the guarantee while still passing as an instance of
the type:

- **Add mutable state** — its own non-final fields and mutators; the object is-a parent type, but now
  it changes.
- **Override methods to return varying values** — even with the parent's fields final, overridden
  accessors (or `equals`/`hashCode`) can return different results over time, so *observable* state
  shifts.
- **Break dependent code** — any client relying on immutability (defensive-copy-free sharing,
  thread-safety, stable keys) becomes unsafe.

The reason it's dangerous is **substitutability**: the subclass flows anywhere the type is accepted,
including untrusted input. So immutability must be sealed at the **type** level, not just the field
level:

- **Effectively final** — either make the class `final`, or use non-public constructors + static
  factories.
- If subclassing genuinely cannot be prevented (e.g. consuming a type you don't control),
  **defensive copying is required** —
  [worked example](#where-in-the-jdk-is-a-class-immutable-by-design-yet-still-subclassable--and-why-is-that-dangerous).

</details>

### Where in the JDK is a class immutable by design yet still subclassable — and why is that dangerous?
<details><summary>Show answer</summary>

**The flaw:** both are *intended* to be immutable, but they are **not `final`** and allow method
overriding. This is a historical design mistake that can't be corrected without breaking backward
compatibility.

**The risk:** because they're subclassable, a malicious subclass can override methods to behave
**mutably** while still being accepted anywhere a `BigInteger`/`BigDecimal` is expected. Any code that
depends on their immutability *and* accepts instances from untrusted callers is therefore vulnerable —
the "value" it trusts can change underneath it.

**The fix — defensively copy untrusted instances** into a known-genuine instance before relying on
them:
```java
BigInteger safe = new BigInteger(untrusted.toByteArray());
```
This reconstructs a real `BigInteger` (not the untrusted subclass), guaranteeing you hold a genuinely
immutable value.

**The general lesson:** when a type's security depends on the immutability of a class you don't fully
control, don't trust the reference — copy it into an instance whose class you *do* control.

</details>

### What options exist to prevent subclassing in immutable classes, and how do they compare?
<details><summary>Show answer</summary>

Immutability must be sealed at the type level — two ways to do it:

**Option 1 — declare the class `final`.** Directly forbids subclassing.
- *Pro:* simple, explicit, easy to reason about.
- *Con:* least flexible; no inheritance even internally, limits future evolution.

**Option 2 — non-public constructors + public static factories.** All constructors `private`/
package-private; instances come from static factory methods. To outside clients the class is
effectively final.
- *Pro:* more flexible — allows multiple internal implementations and future optimizations (e.g.
  instance caching) behind the factory.
- *Con:* slightly more complex; immutability is less obvious than a plain `final`.

**Choosing:** `final` for simplicity; static factories for flexibility and long-term evolvability —
which is why libraries often prefer them.

</details>

### Can an immutable class have non-final or mutable internal fields?
<details><summary>Show answer</summary>

Immutability defined by **observable behavior** — not by field modifiers. An immutable class may hold **non-final or
internally mutable fields** and still be immutable, as long as **no externally visible state ever
changes**. The test is what observers can see, never how the class is built inside.

The canonical case is **internal caching** (lazy memoization of an expensive computation): a non-final
private field stores a value on first use. The object's *logical* state never changes — the cache only
remembers a result derived from the fixed state — so callers can never observe a difference:

```java
public final class ExpensiveValue {
  private final int input;
  private volatile Integer cachedResult;   // non-final, internal cache

  public ExpensiveValue(int input) { this.input = input; }

  public int compute() {                   // double-checked locking
    Integer result = cachedResult;
    if (result == null) {
      synchronized (this) {
        result = cachedResult;
        if (result == null) {
          result = expensiveComputation(input);
          cachedResult = result;
        }
      }
    }
    return result;
  }

  private int expensiveComputation(int value) { return value * value; }
}
```

**Key point:** immutability is a contract about what clients can *observe*. Internal mutability is fine
when it's completely hidden and never changes the object's logical state.

</details>

### How do public fields weaken or break immutability?
<details><summary>Show answer</summary>

A public field exposes the representation directly — whether that *breaks* immutability depends on two
things: is the field `final`, and is its referent mutable? Three cases:

**1. Public non-final field — immutability broken outright.**
Anyone can reassign it; the object's state changes directly.
```java
public class Point { public int x, y; }
p.x = 99;                       // state changed — not immutable at all
```

**2. Public final field, immutable referent — safe for immutability, but not free.**
The reference can't be reassigned and the referent can't change, so observable state is fixed.
```java
public final class Point { public final int x, y; ... }
```
It doesn't *break* immutability — but the field is now part of the **public contract**: you've
locked in the representation and can't change it later without breaking clients. (This is the
deliberate DTO trade-off; fine when the shape *is* the contract, costly otherwise.)

**3. Public final field, MUTABLE referent — the silent killer.**
`final` freezes the *reference*, not the object it points to. People assume `final` = safe and miss
that the contents are wide open:
```java
public final class Config {
  public final List<String> tags;          // final reference…
  public final int[] data;                 // …mutable contents
  ...
}
cfg.tags.add("x");                          // observable state changed
cfg.data[0] = 99;                           // immutability broken — despite final
```
This is exactly the [public-array hole](3.1_classes_and_interfaces.md#what-should-be-considered-about-immutability-when-using-arrays):
`final` is no protection when the referent is mutable.

**Why `private` is the rule:** making fields private removes the exposure in all three cases. The
class controls every path to its state — returning immutable referents or defensive copies — so
immutability is *enforced*, not merely *hoped for*. Public fields, even `final` ones, hand that
control to callers.

**The deeper caveat — `final` isn't even an absolute runtime guarantee.** Per **JEP 500 ("Prepare to
Make Final Mean Final")**, the JDK itself acknowledges that final fields can today be mutated via
reflection/serialization machinery — so `final` is a *compile-time* promise, not an ironclad runtime
one. The rule "private + final + immutable/defensively-copied referent" is what actually buys you
immutability; `final` alone never did.

</details>
