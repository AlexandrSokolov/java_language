## Classes and Interfaces - Immutability

### What API design practice makes classes more predictable by ensuring their invariants cannot be broken after construction?
<details><summary>Show answer</summary>

**Minimize mutability.**


Immutable classes are easier to design, implement, and use than mutable classes.
They are less prone to error and are more secure.

</details>

### What are the benefits of using immutable classes?
<details><summary>Show answer</summary>

Benefits of immutable classes:

- Simpler reasoning - an immutable object has a single, fixed state established at construction.
- Strong invariants - once validated by the constructor, invariants remain true forever.
- Inherent thread safety - no synchronization is required - the most practical payoff in real systems.
- Safe sharing without defensive copies - objects can be freely reused.
- Safe use in collections – stable equality and hashing semantics

</details>

### How can a class be designed to minimize mutability?
<details><summary>Show answer</summary>

1. Do not provide methods that modify the object’s state (no mutators).
2. Prevent subclassing to preserve immutability guarantees by
    - making the class `final`, or
    - making all constructors private (or package‑private) and providing public static factory methods instead,
      which allows controlled instance creation without permitting subclassing.
3. Make all fields `final` to clearly enforce immutability and ensure safe publication.
4. Make all fields `private` to avoid exposing internal state or future representation lock‑in.
    - **Minimizing mutability** is about preventing state changes.
    - **Making fields private** is about preventing uncontrolled state changes.
    - Without private fields, immutability cannot be enforced, only hoped for.
5. Do not expose mutable components; use defensive copies in constructors and accessors.

</details>

### How can accessor methods be designed to preserve immutability?
<details><summary>Show answer</summary>

The **functional approach** can be used to provide accessors while minimizing mutability.

In this approach, operations do not modify the object’s state.
Instead, they create and return new instances that represent the result:
```java
public Complex plus(Complex c) {
  return new Complex(re + c.re, im + c.im);
}
```

Each arithmetic operation returns a new `Complex` instance rather than mutating the existing one.
This contrasts with the **procedural (imperative) approach**, where methods modify the object’s internal state.
The **functional approach preserves immutability** while still allowing expressive, safe access to object behavior.

</details>

### What pattern is used in Java records to return modified versions of an instance?
<details><summary>Show answer</summary>

Java records commonly use the *withers* pattern to return modified versions of an instance.

Withers are methods that create and return a new record instance with one component changed,
while the remaining components are copied from the original instance:
```java
public record User(String name, Integer age) {

  User withName(String name) {
    return new User(name, this.age);
  }

  User withAge(Integer age) {
    return new User(this.name, age);
  }
}
```

</details>

### What should be considered when using withers in Java records?
<details><summary>Show answer</summary>

Withers in Java records can be overused when they are added mechanically for every component
without modeling a meaningful transformation.

Overuse typically occurs when:
- The record is small and short‑lived, where direct construction is clearer.
- Withers merely duplicate the canonical constructor without improving readability.
- Client code chains many withers, effectively simulating mutability:
  ```java
  user.withName(n).withAge(a).withRole(r);
  ```
  That may indicate that the record is too large and a builder might fit better.
- The record grows large, signalling that a builder or a different abstraction may be more appropriate.

Recommended guidelines:
- Prefer direct construction for simple, local transformations.
- Add withers only for common or semantically meaningful modifications.
- Avoid using withers to mimic setters or incremental state mutation.
- Reconsider the design if frequent chained updates are required.

Rules:
- **Used thoughtfully, withers improve clarity;**
- used excessively, they obscure intent and complicate value‑oriented design.

</details>

### What are the disadvantages of immutable classes?
<details><summary>Show answer</summary>

- **Performance and allocation overhead** – immutable objects require creating new instances for every modification,
  which can lead to increased memory usage, copying costs (especially for large object graphs).
  This also makes incremental updates less convenient, as even small changes may require reconstructing entire objects.
- **Not always a natural fit** – some domains (e.g., entities with identity or lifecycle-based state changes) are more naturally modeled as mutable.
- **API verbosity** – code may become more verbose when many “modified copies” are needed without builders or withers.

**Key takeaway**

Immutable classes simplify correctness and concurrency,
but they trade off **performance, allocation cost, and ergonomics** in certain use cases.
Choosing immutability should be a deliberate design decision based on how objects are created, used, and evolved.

</details>

### What techniques can be used to reduce allocation overhead in immutable classes?
<details><summary>Show answer</summary>

One way to address allocation overhead in immutable classes is to provide a **public mutable companion class**.

The idea is to separate mutation from representation:
- The mutable companion is used to efficiently build or modify state without creating many intermediate objects.
- Once the final state is ready, it is converted into an immutable instance, which is safe to share and reuse.

Typically, this is implemented by:
- Exposing mutating operations on the companion class.
- Providing a `build()` or similar method that creates the immutable object.
- Ensuring the immutable class copies the state and never changes it afterward.

```java
// Immutable class
final class Point {
  private final int x;
  private final int y;

  Point(PointBuilder builder) {
    this.x = builder.x;
    this.y = builder.y;
  }
}

// Mutable companion
class PointBuilder {
  int x;
  int y;

  PointBuilder setX(int x) {
    this.x = x;
    return this;
  }

  PointBuilder setY(int y) {
    this.y = y;
    return this;
  }

  Point build() {
    return new Point(this);
  }
}
```

A well-known example from the Java standard library is String and its mutable companion `StringBuilder`.
StringBuilder allows efficient in-place modification of characters and produces an immutable String via toString(),
avoiding repeated allocations.

</details>

### What techniques can be used in Java records to reduce allocation overhead caused by immutability?
<details><summary>Show answer</summary>

Java records are immutable by design, which provides strong safety guarantees.
However, immutability can lead to performance issues when multiple fields need to be updated,
because each update creates a new instance.

Goals:
- Preserve immutability and thread-safety
- Support convenient single-field updates via withers

  Withers are methods that return a new record instance with one field changed.
- Avoid excessive allocations for batch updates via builder

  Mutable Companion Builder for Batch Updates. Mutation happens only inside the builder; the record remains immutable.

```java
public record User(String name, int age, String email) {

  public Builder toBuilder() {
    return new Builder(this);
  }

  public User withAge(int age) {
    return new User(this.name, age, this.email);
  }

  static final class Builder {
    private String name;
    private int age;
    private String email;

    private Builder(User user) {
      this.name = user.name;
      this.age = user.age;
      this.email = user.email;
    }

    public Builder name(String name) {
      this.name = name;
      return this;
    }

    public Builder age(int age) {
      this.age = age;
      return this;
    }

    public Builder email(String email) {
      this.email = email;
      return this;
    }

    public User build() {
      return new User(name, age, email);
    }
  }
}
```
Usage:
```java
//single update:
User ageUpdated = user.withAge(10);
//batch updated:
User updated = user.toBuilder()
        .name("Alice")
        .age(30)
        .email("a@example.com")
        .build();
```

</details>

  public static Builder builder() {

### What options exist to prevent subclassing in immutable classes, and how do they compare?
<details><summary>Show answer</summary>

To preserve immutability, a class must prevent subclassing.
There are two main design options, each with different trade‑offs.

#### Option 1: Declare the class `final`

This directly forbids subclassing.

- **Pros:** Simple, explicit, and easy to reason about
- **Cons:** Least flexible; prevents inheritance and limits future evolution

#### Option 2: Use non‑public constructors with public static factory methods

All constructors are `private` or package‑private, and instances are created via static factories.  
To clients outside the package, the class is effectively final.

- **Pros:** More flexible; allows multiple internal implementations and future performance optimizations
  (e.g., instance caching)
- **Cons:** Slightly more complex; immutability is less obvious than with `final`

#### Comparison

Both approaches preserve immutability by preventing subclassing.  
Making the class `final` favors **simplicity**, while static factories favor **flexibility and long‑term evolvability**,
making them a common choice for library design.

</details>

### What are the consequences if a class with all final fields does not prevent subclassing?
<details><summary>Show answer</summary>

- `final` fields alone do not guarantee immutability.
- A class with all `final` fields is **not truly immutable** if it allows subclassing.

If subclassing is permitted:
- A subclass can add mutable state
- A subclass can override methods to return different values over time
- Code that relies on immutability may break or become insecure

Immutability is a behavioral guarantee, not just a property of field modifiers.


`BigInteger` and `BigDecimal` are intended to be immutable, but they are **not final** and allow method overriding.
This is a historical design issue that cannot be fixed without breaking backward compatibility.

As a result, code that depends on their immutability and accepts instances from untrusted clients may be vulnerable:
a malicious subclass could behave mutably.


If you write a class whose security depends on the immutability of such classes you must **defensively copy**:
```java
BigInteger safe = new BigInteger(untrusted.toByteArray());
```
This guarantees that you are working with a genuine, immutable instance rather than an untrusted subclass.


**When designing your own immutable classes**:
- Prevent subclassing by making the class final, or
- Use non‑public constructors with public static factory methods


**Key Takeaway**:
- final fields are **necessary but not sufficient**
- Immutable classes must be **effectively final**
- If subclassing cannot be prevented, **defensive copying is required**

</details>

### Can an immutable class legally have mutable state?
<details><summary>Show answer</summary>

Yes, an immutable class may contain **non‑final or internally mutable fields** without violating immutability,
as long as **no externally visible state changes** occur.
The true definition of immutability is behavioral: observers must never see the object change.


For performance reasons, immutable classes may use **internal caching** of expensive computations.
A non‑final private field can lazily store a computed value and return it on subsequent calls.
Because the object’s logical state never changes, the cached result is guaranteed to remain correct:
```java
public final class ExpensiveValue {
  private final int input;
  private volatile Integer cachedResult; // non-final, internal cache

  public ExpensiveValue(int input) {
    this.input = input;
  }

  //double‑checked locking
  public int compute() {
    Integer result = cachedResult;
    if (result == null) {
      synchronized (this) {
        result = cachedResult;
        if (result == null) {
          result = expensiveComputation(input);
          cachedResult = result;
        }
      }
    }
    return result;
  }

  private int expensiveComputation(int value) {
    // pretend this is very expensive
    return value * value;
  }
}

```

**Key point:** Immutability is about what clients can observe, not how the class is implemented internally.
Internal mutability is acceptable if it is completely hidden and does not affect the object

</details>
