## Lambdas and Streams - Side effects & Collectors

### What may a stream stage not do?
<details><summary>Show answer</summary>

A pipeline is a chain of transformations: each stage takes its input and produces its output, and that is all it does.
The rule: **no stage may change state that another stage, or the pipeline's result, can see.** Accumulating into a
list, incrementing a counter, mutating the element passed downstream — all forbidden.

Why the API demands it: streams promise nothing about run order, run count, or which thread runs what. That freedom
is what buys laziness, stage fusion, and parallelism. A stage that changes shared state has quietly assumed a
guarantee it never had — and the failure is silent, not an exception.

```java
// Broken: the terminal does the work, the pipeline is a disguised loop. Under parallelStream this loses elements.
List<String> result = new ArrayList<>();
words.stream().filter(w -> w.length() > 3).forEach(result::add);

// Correct: the terminal builds the result. Parallel-safe — each thread gets its own container, then they merge.
List<String> result = words.stream().filter(w -> w.length() > 3).toList();
```

`forEach` is the one place a change is allowed — its job is to consume a finished result, not to compute one. Use it
to print, send, or store what the pipeline produced.

</details>

### What counts as a side effect?
<details><summary>Show answer</summary>

A side effect is a function **changing something outside itself** — anything a later caller or a different thread
could notice.

Has one:
- `list::add`, `map.put(...)` — writes a collection someone else holds
- `counter++`, writing a field or a `static` — writes shared memory
- `user.setStatus(ACTIVE)` — mutates the argument
- `repository.save(x)`, `queue.send(x)` — writes an external system
- `log.info(...)` — writes a file; harmless, but still a change

Does not:
- `Integer.parseInt(s)` — reads its argument, returns a value
- `userService.findAddress(id)` — a query. Goes to the network, changes nothing.
- `config.getTimeout()` — a read

Reading is not a side effect, however far the read travels. The test is: **after the call, is anything different?**

</details>

### When is a function not a math function?
<details><summary>Show answer</summary>

A math function's output depends **only on its input**. Same argument, same answer, forever. `Integer.parseInt`,
`String::trim`, `Math::sqrt`.

Two separate things break it, and a function can have either alone:

- **It writes state outside itself** — [a side effect](#what-counts-as-a-side-effect). `list::add`.
- **It reads state it was not given** — an external dependency. `userService.findAddress(id)`: the DB row can change
  between calls, so the same `id` can give two answers.

Most broken functions do both, so the two words get used as one. They are not. A network query is a read only: it
depends on the outside world, but changes nothing. That gap matters — one of the two is allowed in a pipeline and
the other is not.

Note: throwing does not break it. `parseInt("x")` throws and is still a math function — the exception follows from
the input. `findAddress` failing because the network is down is different: the *environment* decided, not the
argument. That failure is a symptom of the read, not a separate defect.

</details>

### May a stage call a service?
<details><summary>Show answer</summary>

Yes. A read of an external system is not [a side effect](#what-counts-as-a-side-effect) — the stage takes its input,
asks a question, returns an answer, and changes nothing. It is [not a math function](#when-is-a-function-not-a-math-function),
but the pipeline never asked for one.

```java
users.stream()
     .map(userService::findAddress)             // reads the DB — legal
     .map(deliveryService::deliveryCompany)     // reads a REST API — legal
     .filter(deliveryService::isAvailable)
     .toList();                                 // terminal builds the result
```

What the pipeline forbids is [changing state another stage can see](#what-may-a-stream-stage-not-do) — that is the
write half, and it is absent here. The stages read state that no stage writes, so run order and run count do not
change the answer.

The prices you do pay for the read: the stage is not repeatable (the row can change between calls), and it is harder
to test. Neither is a stream problem.

One real trap: `parallelStream` on blocking I/O uses the common ForkJoinPool and starves every other user of it in
the JVM. That is a thread-pool problem, not a purity problem.

</details>

### Describe a code snippet #N
<details><summary><strong>Show details</strong></summary>

<details><summary>Show code</summary>

```java
int counter = 0;   // local

long n = list.stream()
             .peek(x -> counter++)
             .count();
```

</details>

<details><summary>Show answer</summary>

First: this does not compile. A local captured by a lambda must be effectively final, and `counter++` mutates it.

The tempting fix is a mutable holder — `int[] c = {0}` or `AtomicInteger` — so the lambda mutates the box, not the
variable. Now it compiles and runs. And `counter` may still be `0`.

`count()` asks the source how many elements it has. An `ArrayList` knows — its spliterator reports `SIZED`. If no
stage upstream can change the count (`peek` and `map` cannot; `filter` and `flatMap` can), the JDK returns the size
directly and walks nothing. `peek` never runs. Documented since Java 9.

So the effectively-final error is not a detour — it pushes you toward a workaround that compiles and hides the real
defect: a stage that writes something outside itself depends on a promise the API never made — **once per element**.
Had `peek` only read, skipping it would be invisible.

```java
long n = list.stream().filter(s -> !s.isEmpty()).peek(x -> counter++).count();   // peek runs: filter hides the size
```

Safe uses of `peek`: logging while debugging, where nothing later reads the residue. That is what the javadoc
recommends it for.

</details>

</details>

### What belongs in forEach?
<details><summary>Show answer</summary>

Only reporting the result — print it, send it, hand it to something else. Never computing it.

`forEach` is the weakest terminal: explicitly iterative, no order guarantee, and it cannot parallelize. Everything
that makes a stream worth using lives in the other terminals. If the work happens inside `forEach`, the pipeline is
decoration — you wrote a loop with more syntax.

```java
// Not streams code — a loop in disguise. All the work is in forEach, mutating a table outside the pipeline.
Map<String, Long> freq = new HashMap<>();
try (Stream<String> words = new Scanner(file).tokens()) {
  words.forEach(word -> freq.merge(word.toLowerCase(), 1L, Long::sum));
}

// The pipeline computes; nothing outside is touched. Shorter, and parallel-safe for free.
Map<String, Long> freq;
try (Stream<String> words = new Scanner(file).tokens()) {
  freq = words.collect(groupingBy(String::toLowerCase, counting()));
}
```

Why people write the first one: `forEach` looks like a for-each loop, so it feels familiar. That familiarity is the
whole reason the anti-pattern exists.

Legal exception: pushing a finished result into a collection that already exists.

</details>

### Two smells in a pipeline?
<details><summary>Show answer</summary>

- **A `forEach` doing more than presenting the result** — the computation has left the pipeline.
- **A lambda changing state outside itself** — [the one thing a stage may not do](#what-may-a-stream-stage-not-do).

They usually appear together, because the first is how the second gets in. Spot either and the question is the same:
what should the terminal have been? Almost always `collect`.

</details>

### How should Collectors be imported?
<details><summary>Show answer</summary>

Statically, all of them: `import static java.util.stream.Collectors.*;`

`collect(groupingBy(String::toLowerCase, counting()))` reads as intent; `Collectors.groupingBy(...,
Collectors.counting())` buries it in noise. The pipeline is the thing being read — the class name adds nothing.

</details>

### What governs turning a pipeline into a result?
<details><summary>Show answer</summary>

One question decides everything: **what shape is the result?**

- **A collection of the elements** → [collecting into a collection](#how-to-land-elements-in-a-chosen-collection-type)
- **A map, one entry per element** → [key + value from each element](#how-to-index-elements-by-a-derived-key)
- **A map, many elements per key** → [categorising](#how-to-bucket-elements-by-a-shared-trait)
- **A map with exactly two buckets** → [splitting by a yes/no test](#how-to-split-into-matching-and-non-matching)
- **Not a collection at all** → [a single string](#how-to-render-elements-as-one-line) or
  [a single element](#how-to-pick-the-extreme-element)

Then, for the map cases only, two follow-ups:
- keys can repeat → [collision policy](#how-to-index-when-keys-repeat)
- values should not be lists → [downstream collector](#how-to-change-what-a-bucket-holds)

Trigger: **shape first, then collisions, then what the bucket holds.**

Trap that catches everyone: many `Collectors` methods only make sense **downstream** —
[which ones you should never call directly](#which-collectors-are-pointless-at-top-level).

</details>

### How to land elements in a chosen collection type?
<details><summary>Show answer</summary>

- `toList()` — any list; **immutable** since Java 16's `Stream.toList()`, and `Collectors.toList()` returns a mutable
  `ArrayList` in practice but does not promise it
- `toSet()` — any set, duplicates gone, order not promised
- `toCollection(TreeSet::new)` — you name the type, so you get sortedness, mutability, or a specific class

Rule: use `toCollection` the moment you care *which* implementation. The other two hand you whatever the JDK picked.

```java
List<String> l = words.collect(toList());
Set<String>  s = words.collect(toSet());
TreeSet<String> t = words.collect(toCollection(TreeSet::new));   // sorted, mutable, named type
```

</details>

### How to index elements by a derived key?
<details><summary>Show answer</summary>

`toMap(keyMapper, valueMapper)` — two functions, one element becomes one entry.

```java
Map<String, Operation> byName = Stream.of(values()).collect(toMap(Object::toString, e -> e));
```

Only correct when **keys cannot repeat**. A duplicate key throws `IllegalStateException` at runtime — the pipeline
dies mid-collect, not at compile time. If repeats are possible, you need
[a collision policy](#how-to-index-when-keys-repeat).

</details>

### How to index when keys repeat?
<details><summary>Show answer</summary>

`toMap(keyMapper, valueMapper, mergeFunction)` — the third argument is a `BinaryOperator<V>` called with the existing
value and the new one. Whatever it returns is kept.

Three standard shapes:

```java
toMap(Item::sku, Item::qty, Long::sum)                            // combine: total per key
toMap(k, v, (v1, v2) -> v2)                                       // last-write-wins
toMap(Album::artist, a -> a, maxBy(comparing(Album::sales)))      // pick a winner per key
```

`maxBy` here is `BinaryOperator.maxBy` — it turns a `Comparator` into a merge function. Not `Collectors.maxBy`, which
is a different thing.

Last-write-wins is nondeterministic unless the values are interchangeable — fine when any of them is acceptable,
a silent bug when it is not.

Fourth argument: `toMap(k, v, merge, TreeMap::new)` when the map implementation matters.

</details>

### How to bucket elements by a shared trait?
<details><summary>Show answer</summary>

`groupingBy(classifier)` — the function returns the category, the category becomes the key, and values are `List`s of
everything that fell there.

```java
Map<String, List<String>> anagrams = words.collect(groupingBy(Anagrams::alphabetize));
```

The difference from `toMap`: `toMap` assumes one element per key and throws on repeats; `groupingBy` **expects**
repeats and collects them. Reaching for a merge function to keep everything is a sign you wanted `groupingBy`.

To make the values something other than lists → [downstream collector](#how-to-change-what-a-bucket-holds).

</details>

### How to change what a bucket holds?
<details><summary>Show answer</summary>

`groupingBy(classifier, downstream)` — the second collector consumes each category and produces that key's value.

```java
groupingBy(Word::first, toSet())                    // Map<Character, Set<Word>>
groupingBy(Word::first, toCollection(TreeSet::new)) // Map<Character, TreeSet<Word>>
groupingBy(String::toLowerCase, counting())         // Map<String, Long> — the frequency table
groupingBy(Emp::dept, mapping(Emp::name, toList())) // Map<String, List<String>> — transform then collect
```

Three arguments: `groupingBy(classifier, mapFactory, downstream)`. Note the order — the factory sits **before** the
downstream collector, breaking the usual telescoping convention. Gives you both containers, e.g. a `TreeMap` of
`TreeSet`s.

</details>

### How to split into matching and non-matching?
<details><summary>Show answer</summary>

`partitioningBy(predicate)` — takes a predicate instead of a classifier, returns `Map<Boolean, List<T>>`.

```java
Map<Boolean, List<Student>> passed = students.collect(partitioningBy(s -> s.score() >= 60));
passed.get(true);   // never null — both keys always present
```

The reason to prefer it over `groupingBy(s -> s.score() >= 60)`: both `true` and `false` are guaranteed present even
when a side is empty, so no null check. `groupingBy` simply omits an empty category.

Takes a downstream collector too: `partitioningBy(pred, counting())`.

</details>

### How to render elements as one line?
<details><summary>Show answer</summary>

`joining()` — only for streams of `CharSequence`.

```java
joining()                  // camesawconquered
joining(", ")              // came, saw, conquered
joining(", ", "[", "]")    // [came, saw, conquered]
```

Delimiter form is a CSV trap: any element containing the delimiter makes the output ambiguous, and nothing warns you.

</details>

### How to pick the extreme element?
<details><summary>Show answer</summary>

`Stream.max(comparator)` / `Stream.min(comparator)` — both return `Optional<T>`, empty when the stream is empty.

`Collectors.maxBy` / `minBy` exist for the same job but are only worth using **downstream** inside a `groupingBy`.
At top level `collect(maxBy(cmp))` is a longer spelling of `max(cmp)`.

Three same-named things, different types — the classic confusion:
- `Stream.max(cmp)` → `Optional<T>`, a terminal
- `Collectors.maxBy(cmp)` → a `Collector`, for downstream use
- `BinaryOperator.maxBy(cmp)` → a `BinaryOperator<T>`, for a `toMap` merge function

</details>

### Which Collectors are pointless at top level?
<summary>Show answer</summary>

Roughly sixteen of them. They duplicate `Stream` methods so that **downstream** collectors can behave like
mini-streams — inside `groupingBy` there is no pipeline to call, so the collector has to supply the operation.

Never at top level:

```java
stream.collect(counting())              // → stream.count()
stream.collect(summingInt(f))           // → stream.mapToInt(f).sum()
stream.collect(averagingInt(f))         // → stream.mapToInt(f).average()
stream.collect(maxBy(cmp))              // → stream.max(cmp)
stream.collect(mapping(f, toList()))    // → stream.map(f).toList()
stream.collect(filtering(p, toList()))  // → stream.filter(p).toList()
stream.collect(reducing(...))           // → stream.reduce(...)
```

The same names downstream are exactly right, and there is no alternative:

```java
groupingBy(Emp::dept, counting())                     // no stream to call count() on
groupingBy(Emp::dept, summingInt(Emp::salary))
groupingBy(Emp::dept, mapping(Emp::name, toSet()))
groupingBy(Emp::dept, collectingAndThen(toList(), List::copyOf))   // post-process each bucket
```

Trigger: **downstream, there is no pipeline — so the collector becomes one.**

</details>

### Which collector variants run in parallel?
<details><summary>Show answer</summary>

`toConcurrentMap` and `groupingByConcurrent` — variants of every overload of `toMap` and `groupingBy`, producing
`ConcurrentHashMap`.

They pay off only on a genuinely parallel stream: instead of each thread filling its own map and merging at the end,
all threads write one concurrent map. On a sequential stream they are pure overhead.

Cost: encounter order is lost, which `groupingBy` on a sequential stream preserves inside each bucket.

</details>
