## Lambdas and Streams - Streams - Return Types

### Which type should a public API hand back?
<details><summary>Show answer</summary>

The walk:

1. **Elements fit in memory** → return them behind the most general **interface** the caller needs, never a concrete
   class. `Collection` serves both camps (it **is** `Iterable` for for-each and declares `stream()`); narrow to
   `List` only when the caller needs index access, `Set` for set semantics. Never declare `ArrayList` / `HashSet` as
   the return type — that leaks the implementation and freezes your freedom to change it. Build with whatever
   concrete class you like inside; it stays hidden.
2. [**Cannot fit in memory, but any element can be computed on demand**](#returning-a-huge-sequence-without-building-it).
3. **Must stay lazy** (large, infinite, or produced on demand) → return `Stream` — see
   [Stream vs Iterable as a method return type?](#stream-vs-iterable-as-a-method-return-type) for why not `Iterable`.

Private or internal method with one known caller? Return whatever that caller uses. The rule protects APIs whose
users you cannot poll.

Never do this: **build a large sequence into memory just so the return type can be `Collection`.**

</details>

### Stream vs Iterable as a method return type?
<details><summary>Show answer</summary>

`Stream` does not extend `Iterable`, so each choice forces work on the other camp's callers:

- Return `Stream`, caller wants a for-each loop: must bridge through `stream.iterator()` (and the naive
  `for (X x : stream::iterator)` needs a cast to compile) — works, but ugly.
- Return `Iterable`, caller wants a pipeline: must bridge with `StreamSupport.stream(it.spliterator(), false)` —
  also works, also ugly.

Rule of thumb by result size:

- Result is bounded and cheap to hold, no need for laziness → return `Collection`. It **is** `Iterable` (real
  for-each) and gives `stream()` too, so neither caller camp has to bridge.
- Result is large, infinite, or must stay lazy (compute/fetch per element on demand) → return `Stream`. Here
  `Collection` is not an option — building it forces full evaluation and throws the laziness away.

Summary handle: 
- `Collection` when the data is already in hand; 
- `Stream` when producing it eagerly would defeat the point.

</details>

### Get an Iterable from a Stream?
<details><summary>Show answer</summary>

If laziness does not matter, materialize it — `List` is already an `Iterable`:

```java
Iterable<E> iterable = stream.toList();       // fully evaluated
```

To keep it lazy (huge or infinite stream, do not evaluate), reuse the stream's own `iterator()` as the `Iterable`'s
single abstract method:

```java
Iterable<String> iterable = stream::iterator;   // not evaluated; stream::iterator IS the Iterable
for (String s : iterable) {                     // for-each calls iterator() here; stream drained now
  System.out.println(s);
}
```

Single-use: a stream iterates once. Looping `iterable` a second time throws `IllegalStateException` (stream already
operated upon); a `toList()` result can be looped repeatedly.

Cannot inline it — `for (String s : stream::iterator)` does not compile (for-each header infers no target type). Use
the intermediate variable above, or cast: `for (String s : (Iterable<String>) stream::iterator)`.

</details>

### Get a Stream from an Iterable?
<details><summary>Show answer</summary>

`Iterable` has no `stream()`. Wrap its spliterator:

```java
Stream<String> stream = StreamSupport.stream(iterable.spliterator(), false);   // false = sequential, true = parallel
for-each source untouched until a terminal runs;                               // wrapping stays lazy
```

If the source is a `Collection`, skip all this — it already has `.stream()`. This wrap is only for an `Iterable`
that is not a `Collection`.

</details>

### Returning a huge sequence without building it?
<details><summary>Show answer</summary>

Extend `AbstractList` (or `AbstractCollection`) and compute elements on demand. Beyond `iterator()`, you owe only
`size()` and `contains()` — often both are cheap arithmetic.

Power set of an n-element set has 2^n members, so storing it is out. But index *i* in binary **is** a subset: bit k
says whether element k is in.

```java
// Returns the power set of an input set as a custom collection
public class PowerSet {
  public static final <E> Collection<Set<E>> of(Set<E> s) {
    List<E> src = new ArrayList<>(s);
    if (src.size() > 30)
      throw new IllegalArgumentException("Set too big " + s);
    return new AbstractList<Set<E>>() {
      @Override public int size() {
        return 1 << src.size();   // 2 to the power srcSize
      }
      @Override public boolean contains(Object o) {
        return o instanceof Set && src.containsAll((Set) o);
      }
      @Override public Set<E> get(int index) {
        Set<E> result = new HashSet<>();
        for (int i = 0; index != 0; i++, index >>= 1)
          if ((index & 1) == 1)
            result.add(src.get(i));
        return result;
      }
    };
  }
}
```

The 30-element cap is the catch, and it is `Collection`'s fault: `size()` returns `int`, so no collection can honestly
report more than `Integer.MAX_VALUE` elements. `Stream` and `Iterable` have no such ceiling. The spec permits
returning `2^31 - 1` for a bigger collection — a lie, tolerated.

Use this when `size()` and `contains()` are computable without walking. If the contents are not known before
iteration, it is not an option.

</details>
