## Lambdas and Streams - Streams

### What governs a sequence-returning method's type?
<details><summary>Show answer</summary>

One fact drives the whole item: **`Stream` does not extend `Iterable`**, so a stream cannot go in a for-each loop —
and `Iterable` has no `stream()`, so an iterable cannot start a pipeline. Pick either and half your callers are
stuck.

- Why either choice hurts a caller → [the gap](#why-cant-a-returned-sequence-be-looped-over)
- What a stuck caller must write → [the adapters](#bridging-between-the-two-sequence-types)
- What to return instead → [the decision](#which-type-should-a-public-api-hand-back)
- When the elements will not fit in memory → [custom collection](#returning-a-huge-sequence-without-building-it)
- When even that is not worth it → [give up and return a stream](#when-is-a-collection-not-worth-implementing)

Trigger: **`Collection` is both — it extends `Iterable` and has `stream()`. Return it unless you cannot.**

</details>

### Why can't a returned sequence be looped over?
<details><summary>Show answer</summary>

Because `Stream` does not extend `Iterable`, and for-each requires `Iterable`. Nothing else.

```java
// Your API
public Stream<Order> recentOrders() { ... }

// Caller wants a plain loop with a try/catch and a break — no pipeline form of that
for (Order o : svc.recentOrders()) { ... }   // COMPILE ERROR: for-each not applicable to Stream
```

The insult: `Stream` **has** `iterator()`, with a signature compatible with `Iterable`'s only abstract method. The
method is right there. The interface declaration just does not say `extends Iterable`, so the compiler will not look.

Reverse direction, same shape:

```java
public Iterable<Order> recentOrders() { ... }
svc.recentOrders().stream()                  // COMPILE ERROR: Iterable has no stream()
```

So the choice is not stylistic. Returning either type **removes a capability** from callers you have not met.

</details>

### Bridging between the two sequence types?
<details><summary>Show answer</summary>

The JDK ships no adapter either way. Both are one line to write — which is why the omission stings.

```java
// Stream -> Iterable
public static <E> Iterable<E> iterableOf(Stream<E> stream) {
  return stream::iterator;      // no cast needed: inference works in a return position
}

// Iterable -> Stream
public static <E> Stream<E> streamOf(Iterable<E> iterable) {
  return StreamSupport.stream(iterable.spliterator(), false);   // false = sequential
}
```

Inline, without the helper, the first one is unusable:

```java
for (Order o : svc.recentOrders()::iterator) { }                      // ERROR: method reference not expected here
for (Order o : (Iterable<Order>) svc.recentOrders()::iterator) { }    // compiles — and nobody will write this
```

The cast is needed because Java's inference does not supply a target type in a for-each header. Two real costs, not
just ugliness: every caller writes the helper themselves, and the adapted loop runs ~2.3× slower than iterating a
real collection.

Also: an adapted stream is still single-use. Loop over it twice and the second one throws.

</details>

### Which type should a public API hand back?
<details><summary>Show answer</summary>

`Collection`, or a subtype. It extends `Iterable` **and** declares `stream()` — the only common type that serves both
kinds of caller.

The walk:

1. **Elements already in a collection, or few enough to build one** → return it. `ArrayList`, `HashSet`, `List.of(...)`.
2. **Large but concisely describable** → [custom collection](#returning-a-huge-sequence-without-building-it).
3. **Neither feasible** → return `Stream` or `Iterable`, whichever fits the data better. Two methods is also allowed.

Private or internal method with one known caller? Return whatever that caller uses. The rule protects APIs whose
users you cannot poll.

Never do this: **build a large sequence into memory just so the return type can be `Collection`.**

Arrays serve both too, via `Arrays.asList` and `Stream.of`.

</details>

### Returning a huge sequence without building it?
<details><summary>Show answer</summary>

Extend `AbstractList` (or `AbstractCollection`) and compute elements on demand. Beyond `iterator()`, you owe only
`size()` and `contains()` — often both are cheap arithmetic.

Power set of an n-element set has 2^n members, so storing it is out. But index *i* in binary **is** a subset: bit k
says whether element k is in.

```java
public static <E> Collection<Set<E>> of(Set<E> s) {
  List<E> src = new ArrayList<>(s);
  if (src.size() > 30) throw new IllegalArgumentException("Set too big " + s);   // see below
  return new AbstractList<Set<E>>() {
    @Override public int size() { return 1 << src.size(); }         // 2^n, computed — nothing stored
    @Override public boolean contains(Object o) { return o instanceof Set && src.containsAll((Set) o); }
    @Override public Set<E> get(int index) {                        // element built only when asked for
      Set<E> result = new HashSet<>();
      for (int i = 0; index != 0; i++, index >>= 1)
        if ((index & 1) == 1) result.add(src.get(i));
      return result;
    }
  };
}
```

The 30-element cap is the catch, and it is `Collection`'s fault: `size()` returns `int`, so no collection can honestly
report more than `Integer.MAX_VALUE` elements. `Stream` and `Iterable` have no such ceiling. The spec permits
returning `2^31 - 1` for a bigger collection — a lie, tolerated.

Use this when `size()` and `contains()` are computable without walking. If the contents are not known before
iteration, it is not an option.

</details>

### When is a collection not worth implementing?
<details><summary>Show answer</summary>

When the custom collection is tedious enough that the stream reads better — then return a stream and accept the cost.

Sublists of a list: quadratic memory, so a standard collection is out. A custom one is doable but painful (the JDK
gives no skeletal `Iterator` to build on). The stream version needs one insight — **every sublist is a suffix of a
prefix** — and falls out:

```java
public static <E> Stream<List<E>> of(List<E> list) {
  return Stream.concat(Stream.of(List.of()),                       // concat: the empty list is a sublist too
                       prefixes(list).flatMap(SubLists::suffixes));
}
private static <E> Stream<List<E>> prefixes(List<E> list) {        // (a), (a,b), (a,b,c)
  return IntStream.rangeClosed(1, list.size()).mapToObj(end -> list.subList(0, end));
}
private static <E> Stream<List<E>> suffixes(List<E> list) {        // (a,b,c), (b,c), (c)
  return IntStream.range(0, list.size()).mapToObj(start -> list.subList(start, list.size()));
}
```

`IntStream.range` + `mapToObj` is the pipeline form of an index loop. `flatMap` merges the suffixes of every prefix
into one stream.

The honest price, measured: the custom collection is ~1.4× faster than the stream, and a caller forced through the
`Stream`→`Iterable` adapter loops ~2.3× slower. You are trading caller speed and caller convenience for your own
implementation effort — a legitimate trade, but say so.

Escape hatch the book names: if a future Java made `Stream extend Iterable`, returning a stream would cost nothing.
It has not, as of Java 25.

</details>

### Chapter router — parallel streams?
<details><summary>Show answer</summary>

Adding `.parallel()` is one call, so the cost is invisible at the call site. Everything in this chapter answers one
question: **what has to be true before that call is safe and worth it?** Four independent gates, all must pass:

1. **[Source](#which-sources-parallelize-well)** — can it be split cheaply and evenly?
2. **[Terminal op](#which-terminal-ops-suit-parallelism)** — is the finishing work parallel-friendly?
3. **[Function objects](#what-breaks-correctness-under-parallel)** — do the lambdas obey the spec?
4. **[Volume](#when-is-parallelism-worth-the-overhead)** — is there enough work to pay for the machinery?

Spine trigger: *the library splits your data and merges your results — it can only do that well if the data splits
cleanly, the merge is cheap, your lambdas don't cheat, and there's enough work to amortize the coordination.*

Failure modes if a gate fails: [liveness](#what-happens-when-parallelization-heuristics-fail),
[safety](#what-breaks-correctness-under-parallel), [ordering](#does-parallel-preserve-element-order),
[blast radius](#why-can-one-parallel-pipeline-hurt-another). And the framing that governs all of it:
[it's an optimization](#what-kind-of-change-is-parallel).

</details>

### What happens when parallelization heuristics fail?
<details><summary>Show answer</summary>

The library guesses how to divide the work. When the guess is wrong there's no error and no fallback — you get a
liveness failure: CPU pinned near full, no output, no termination in any useful time.

Concrete case from the book: an infinite `Stream.iterate` of primes, mapped to Mersenne candidates, cut off with
`limit(20)`. Sequential: ~12.5 seconds. Parallel: nothing printed, 90% CPU, still running after 30 minutes.

Why this one detonates: each element costs roughly as much as all previous elements combined. The default strategy
handles the unpredictability of `limit` by speculatively computing extra elements and throwing away what it doesn't
need — a reasonable bet when elements are cheap, ruinous when every extra element doubles the bill.

Reading: `parallel()` has no "this won't work" signal. It fails by hanging, not by complaining.

</details>

### Which sources parallelize well?
<details><summary>Show answer</summary>

Good: `ArrayList`, `HashMap`, `HashSet`, `ConcurrentHashMap`, arrays, `IntStream`/`LongStream` ranges.

Two shared properties, and they're the whole reason:

- **Cheap, accurate splitting** — size is known up front, so the range divides into equal subranges without walking
  the data. The mechanism is the *spliterator* (`spliterator()` on `Stream` and `Iterable`). Even splits mean no
  thread finishes early and idles.
- **Locality of reference** — element references sit contiguously in memory, so a thread's data arrives in cache
  instead of being waited on. Primitive arrays are best: the values themselves are contiguous, not just the
  references. Without locality, threads spend their time stalled on memory, which is exactly what parallelism was
  meant to avoid.

Bad: `Stream.iterate` — each element is defined by its predecessor, so there is no way to hand a thread "the second
half" without first computing the first half. Also `limit`, which makes the useful length unknown while work is in
flight.

Custom `Stream`/`Iterable`/`Collection`: you must override `spliterator()` and measure. Writing a good one is hard.

</details>

### Which terminal ops suit parallelism?
<details><summary>Show answer</summary>

The axis: **how much work lives in the terminal step, and can that work be done in pieces and merged cheaply?** A
terminal op that is inherently sequential and expensive caps the whole pipeline no matter how well the source split.

- **Best — reductions.** `reduce`, plus the prepackaged `min`, `max`, `count`, `sum`. Each thread reduces its chunk
  to one value; merging is combining a few values.
- **Also good — short-circuiting predicates.** `anyMatch`, `allMatch`, `noneMatch`. Any thread finding the answer
  ends it for everyone.
- **Poor — `collect` (mutable reduction).** Each thread builds its own collection and the merge means combining
  containers — copying elements, resizing, rehashing. The merge cost scales with the data instead of collapsing it.

</details>

### What breaks correctness under parallel?
<details><summary>Show answer</summary>

Safety failures — wrong results, not just slow ones — caused by lambdas that violate the `Stream` spec. The spec's
requirements on function objects (e.g. for `reduce`'s accumulator and combiner): **associative, non-interfering,
stateless**.

The trap is that sequential execution hides the violation. A stateful lambda or an accumulator whose grouping
changes the result runs fine single-threaded and gives the right answer, so nothing tells you the code is broken.
Parallelize it and it likely fails, possibly badly. Sequential correctness is not evidence the lambdas are legal.

</details>

### Does parallel preserve element order?
<details><summary>Show answer</summary>

Not with `forEach` — it makes no encounter-order promise, so a parallel run prints in whatever order threads finish.
The Mersenne example wouldn't have printed ascending primes even if it had terminated.

`forEachOrdered` guarantees traversal in encounter order for parallel streams — and pays for it, since threads must
wait to emit in sequence.

Separate from safety failures: the pipeline computed correct elements, it just delivered them in an order the
sequential version didn't have.

</details>

### When is parallelism worth the overhead?
<details><summary>Show answer</summary>

Splitting, thread coordination, and merging cost real time. That cost is fixed-ish; the benefit scales with work. So
below some threshold you pay more than you gain — even with a perfect source, a good terminal op, and legal lambdas.

Rough rule of thumb (Lea): **elements × lines of code executed per element ≥ ~100,000.**

Not a precise formula — a sanity gate. Cheap work over a small collection loses no matter how clean the pipeline is.

</details>

### Why can one parallel pipeline hurt another?
<details><summary>Show answer</summary>

All parallel stream pipelines in a program normally share one common fork-join pool. A single long-running or
misbehaving pipeline occupies that pool and degrades unrelated pipelines elsewhere in the system.

Consequence: the blast radius is bigger than the code you changed, so measurement has to happen in a realistic
system setting — a microbenchmark of one pipeline in isolation won't show the interference.

</details>

### What kind of change is `parallel()`?
<details><summary>Show answer</summary>

**Strictly a performance optimization** — which means it inherits every rule about optimization: measure before and
after, in realistic conditions, and keep it only if the numbers justify it. It is never a default or a style choice.

Sequence: (1) have reason to believe correctness survives, (2) have reason to believe it gets faster, (3) measure,
(4) only then ship it. Skipping to step 4 risks a hang or a wrong answer, and both are worse than slow.

Odds are genuinely against you: an acquaintance maintaining a multimillion-line stream-heavy codebase found only a
handful of places where parallel streams paid off.

But when the gates all pass, the payoff is real — near-linear speedup in core count from a single call. Book's
example: `LongStream.rangeClosed(2, n)` filtered by `isProbablePrime`, terminated by `count()` — splittable range,
reduction terminal, stateless filter, heavy per-element work. 31s → 9.2s, ×3.7 on four cores. (Note: not how you'd
actually compute π(n) — Lehmer's formula is far better. The pipeline is the lesson, not the algorithm.)

Domains where this tends to pay: machine learning, data processing.

</details>

### Which random source for parallel streams?
<details><summary>Show answer</summary>

`SplittableRandom` — built for this, splits into independent generators per thread, potential for linear speedup.

The other two, worst-last:
- `ThreadLocalRandom` — designed for one thread; it will adapt itself to serve as a parallel stream source, but
  won't match `SplittableRandom`.
- `Random` — synchronizes on every operation, so contention kills the parallelism you asked for. Essentially
  obsolete.

Handle: the question is whether the generator can be *split* or only *shared*. Splittable → linear. Shared with a
lock → contention.

</details>
