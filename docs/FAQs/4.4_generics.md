## Generics 4

### How can one stateless instance safely serve many different generic types?
<details><summary>Show answer</summary>

**The problem.** A stateless, immutable object — an empty set, an identity function — holds no per-type data. 
So allocating a *new* one for every type parameterization works, 
but every allocation is an identical do-nothing object — pure waste:

```java
Set<String>  a = new EmptySet<>();   // a brand-new empty set
Set<Integer> b = new EmptySet<>();   // another one — identical, holds nothing
Set<Date>    c = new EmptySet<>();   // and another... all the same empty object
```

Every one of these is the *same* behavior (an empty set that can never differ by type), 
yet each is a separate object on the heap. One shared instance would serve them all.

**But you can't simply share one.** 

Generics are invariant, so a single `Set<Object>` is *not* a `Set<String>`, 
and a `static final` field can't name a class type parameter `T`. 
So how do you hand the *same* instance to callers wanting `Set<String>`, `Set<Integer>`, `Set<Date>`?


**The solution — the Generic Singleton Factory.** 

Hold one `private static final` instance (raw or `Object`-parameterized), 
and expose it through a **generic static factory method** that casts it to the caller's `<T>`.


**Why the cast is safe:** at runtime, erasure makes all parameterizations identical 
(`Set<String>` and `Set<Integer>` are both just `Set`). 
If the object is stateless, it holds nothing type-dependent that could ever cause a `ClassCastException` — 
so casting the one instance to any `T` is structurally safe.

</details>

### What are the typical use cases for the **Generic Singleton Factory** pattern?
<details><summary>Show answer</summary>

Both share one stateless, immutable instance across all types, to avoid pointless allocation:

- [**Identity functions**](#how-do-you-implement-a-type-safe-generic-singleton-factory-for-an-identity-function-in-java) —
  a function that returns its input unchanged (`UnaryOperator<T>`, `Function<T,T>`). 
  It inspects nothing type-dependent, so one instance serves all types.
- [**Empty collections**](#how-do-you-implement-a-type-safe-generic-singleton-factory-for-an-empty-set) —
  a shared immutable empty `Set`/`List`/`Map`/`Optional`, returned instead of `null` to avoid `NPE`s in
  high-throughput code without allocating a new empty object each time.

(Both are how the JDK provides `Function.identity()`, `Collections.emptyList()`, etc.)

</details>

### How do you implement a type-safe Generic Singleton Factory for an identity function in Java?
<details><summary>Show answer</summary>

One `Object`-parameterized instance, exposed through a generic factory that casts to the caller's `<T>`:

```java
private static final UnaryOperator<Object> IDENTITY_FN = arg -> arg;

public static <T> UnaryOperator<T> identityFunction() {
    @SuppressWarnings("unchecked")                       // scoped to this one cast
    UnaryOperator<T> result = (UnaryOperator<T>) IDENTITY_FN;
    return result;
}
```

The cast is safe because an identity function returns its input untouched — it has no type-dependent
state, so the single instance works for any `T`
([why](#how-can-one-stateless-instance-safely-serve-many-different-generic-types)). 
The `@SuppressWarnings` is scoped to a local variable, not the method, to avoid masking other warnings.

</details>

### How do you implement a type-safe Generic Singleton Factory for an empty set?
<details><summary>Show answer</summary>

Same pattern as the identity function — one raw immutable instance, cast to `<T>` in a generic factory:

```java
public class MyCollections {

  @SuppressWarnings("rawtypes")
  private static final Set EMPTY_SET = Collections.unmodifiableSet(new HashSet());

  public static <T> Set<T> emptySet() {
    //100% type-safe because the raw proxy is completely empty
    // and cannot be mutated, making a ClassCastException impossible.
    @SuppressWarnings("unchecked")          // scoped to this cast
    Set<T> result = (Set<T>) EMPTY_SET;
    return result;
  }
}
```

Safe because the set is empty and immutable — it holds no elements that could ever be the wrong type,
so the cast can't cause a `ClassCastException`
([why](#how-can-one-stateless-instance-safely-serve-many-different-generic-types)). 
This is exactly how `Collections.emptySet()` works.

</details>

### How can a single container safely hold values of many different types at once?
<details><summary>Show answer</summary>

**The limitation.** A standard generic container fixes its type parameters at declaration: 
a `Map<String, Integer>` can only ever map `String→Integer`. You can't safely mix types in it. 
The moment you need *arbitrary* types in one structure — e.g. a database row: 
an `Integer` id, a `String` name, a `LocalDate` created-at — 
a `Map<String, Object>` works but **throws away compile-time safety**: 
every value comes out as `Object`, forcing casts and risking `ClassCastException`.


**The fix — a Typesafe Heterogeneous Container (THC).** 
Instead of parameterizing the *container*, parameterize the **key**. 
The key carries the type (`Class<T>`), so each entry stays individually typed,
and the container can safely hold any number of different types at once.
[→ how to implement one](#how-do-you-implement-a-typesafe-heterogeneous-container-and-how-do-you-keep-it-type-safe)

</details>

### How do you implement a typesafe heterogeneous container, and how do you keep it type-safe?
<details><summary>Show answer</summary>

**The core idea — parameterize the *key*, not the container.** The key is a `Class<T>` (a *type
token*); the backing map is `Map<Class<?>, Object>` (value `Object`, so it holds anything). Type safety
lives entirely in the API methods:

```java
public class Favorites {
  private final Map<Class<?>, Object> favorites = new HashMap<>();

  public <T> void putFavorite(Class<T> type, T instance) {
    // cast on insertion too — blocks a raw-type caller from storing a mismatched value
    favorites.put(Objects.requireNonNull(type), type.cast(instance));
  }

  public <T> T getFavorite(Class<T> type) {
    return type.cast(favorites.get(type));   // dynamic cast: Object -> T, no unchecked warning
  }
}

Favorites f = new Favorites();
f.putFavorite(String .class, "Java");
f.putFavorite(Integer .class, 42);

String s = f.getFavorite(String.class);   // typed, no cast at the call site
int n = f.getFavorite(Integer.class);
```

**Why each piece matters:**
- **`<T> void putFavorite(Class<T> type, T instance)`** — the signature forces the value to match the
  key at compile time (can't map `Integer.class` to a `String`).
- **`Class.cast()` on retrieval** — the map only knows `Object`; `type.cast(...)` dynamically casts it
  back to `T` and *returns* `T`, bridging `Object → T` with no unchecked-cast warning.
- **`type.cast(instance)` on insertion** — the safety hole: a careless/malicious caller using a **raw**
  `Class` could store a mismatched value, bypassing the compile-time check. Casting on insertion makes
  it fail immediately (`ClassCastException`) instead of corrupting the map silently.

</details>

### How do you store a `List<String>` in a typesafe heterogeneous container?
<details><summary>Show answer</summary>

You can't — not with a plain `Class<T>` key. This is the THC's main limitation.

**Why.** The key must be a `Class` object. `String.class` is fine; even `String[].class` works (arrays are reifiable). 
But **`List<String>.class` is a syntax error** — because of erasure, 
`List<String>` and `List<Integer>` share the *same* runtime class, `List.class`. 
So if you tried to store a favorite `List<String>` and a favorite `List<Integer>`, 
both would use `List.class` as the key, and the second `put` would silently overwrite the first. 
A `Class<T>` key simply can't tell parameterized types apart.


**The solution — a super type token.** Capture the full parameterized type (`List<String>`) at runtime by other means, 
and use *that* as the key instead of `Class<T>`. 
This is how Spring (`ParameterizedTypeReference`) and Guava (`TypeToken`) extend the THC to parameterized types.
[→ how a super type token captures the full type](#how-do-you-capture-a-full-parameterized-type-like-liststring-at-runtime)

</details>

### How do you restrict the types of keys allowed in a typesafe heterogeneous container?
<details><summary>Show answer</summary>

Use a **bounded type token.** A plain `Class<T>` key accepts any class; 
bounding the type parameter limits which keys are allowed:

```java
public <T extends Annotation> T getAnnotation(Class<T> annotationType);
```

Here `<T extends Annotation>` requires the key to be an annotation class — so the container 
(an annotated element, with annotation types as keys) accepts only `Class` tokens for annotations, 
rejecting anything else at compile time. This is exactly how the Java Annotations API works.

</details>

### How do you safely cast an unknown `Class<?>` object into a bounded type token?
<details><summary>Show answer</summary>

Use **`Class.asSubclass()`** — a runtime-checked cast that produces no unchecked warning.

**The situation:** a class looked up at runtime (`Class.forName(...)`) comes back as an unbounded `Class<?>`. 
To pass it to an API needing a bounded token (`Class<? extends Annotation>`), 
a plain cast `(Class<? extends Annotation>)` compiles only with an unchecked warning.

**The fix:** `asSubclass()` dynamically checks that the class really is a subclass of the given type. 
If so, it returns the cast token safely; if not, it throws `ClassCastException`:

```java
// Looks up an annotation by name and reads it off an element.
static Annotation readAnnotation(AnnotatedElement element, String annotationName) throws ClassNotFoundException {

  Class<?> unknown = Class.forName(annotationName);          // unbounded Class<?>

  // asSubclass returns a Class<? extends Annotation> — the bounded token getAnnotation needs
  Class<? extends Annotation> token = unknown.asSubclass(Annotation.class);

  return element.getAnnotation(token);                       // returns the matching Annotation
}
```

If `annotationName` names a class that is *not* an `Annotation`, 
`asSubclass` throws `ClassCastException` at that line — the check happens at runtime, where the type is known.

</details>

### What type information does erasure keep, even though it removes it from instances?
<details><summary>Show answer</summary>

Erasure removes the type arguments of **instances** at runtime — 
`new ArrayList<String>()` is just an `ArrayList` once running.


But the compiler **keeps** the generic type written in a **class or subclass declaration**, storing it in the bytecode. 
So `class StringList extends ArrayList<String>` permanently records "`String`" — 
and that recorded type can be read back later by reflection (`getGenericSuperclass()`).

This is the loophole the **super type token** exploits: you can't capture a type from an *instance*, 
but you *can* from a *subclass declaration*.

</details>

### How do you capture a full parameterized type like `List<String>` at runtime?
<details><summary>Show answer</summary>

Use a **super type token** — exploit the fact that a subclass declaration keeps its generic type
([the loophole](#what-type-information-does-erasure-keep-even-though-it-removes-it-from-instances)):

1. Define an abstract generic class to act as the token.
2. Create an **anonymous subclass** of it for the type you want — the trailing `{}`. 
   Being a subclass declaration, the compiler bakes `List<String>` into its class file.
3. Read it back by reflection, from the subclass's generic superclass.

```java
public abstract class TypeReference<T> {
    public final Type type;
    protected TypeReference() {
        // read the type argument baked into THIS subclass's declaration
        Type superclass = getClass().getGenericSuperclass();
        this.type = ((ParameterizedType) superclass).getActualTypeArguments()[0];
    }
}

// the trailing {} makes an anonymous subclass → bakes "List<String>" into bytecode
TypeReference<List<String>> token = new TypeReference<List<String>>() {};

System.out.println(token.type);   // java.util.List<java.lang.String>
```

The `{}` is the whole trick: without it you'd have an *instance* (type erased); 
with it you have a *subclass* (type kept).

</details>

### Where is the super type token pattern used in practice?
<details><summary>Show answer</summary>

It's the standard way libraries capture a generic type that erasure would otherwise lose:

- **Jackson** — `new TypeReference<List<MyObject>>() {}` tells the JSON parser the exact payload type to deserialize into.
- **Spring** — `ParameterizedTypeReference<List<MyObject>>` does the same for REST/`RestTemplate` calls.
- **Guava** — `TypeToken<List<String>>`.

It's also the fix for the [typesafe heterogeneous container's `List<String>` limitation](#how-do-you-store-a-liststring-in-a-typesafe-heterogeneous-container): 
a `Class<T>` key can't represent `List<String>`, but a super type token can, so it can serve as the key.

</details>

### What is the purpose of `Collections.checkedList` and similar wrappers?
<details><summary>Show answer</summary>

**The Purpose: Runtime Reification for Standard Collections**
Normally, a `List<String>` loses its generic type at runtime due to type erasure. If legacy code uses a
raw `List` reference to insert an `Integer` into your `List<String>`, it will corrupt the collection.

Wrappers like `Collections.checkedList` solve this by taking a `Class` object (a type token) alongside
the collection. They use this token to dynamically check every element added to the collection at runtime.
If someone tries to insert the wrong type, it immediately throws a `ClassCastException`.

This is highly useful for tracking down bugs in applications that mix generic and raw types.

</details>
