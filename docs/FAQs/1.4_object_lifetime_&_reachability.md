## Concepts – reachability, GC roots

### What is object reachability in Java?
<details><summary>Show answer</summary>

**Object reachability in Java** describes whether an object can still be accessed by a running program.

</details>

### What does it mean for an object to be garbage‑collectable?
<details><summary>Show answer</summary>

An object is **garbage‑collectable** when it is **unreachable** — 
meaning the JVM cannot reach it by following references from any GC root.  
In that case, the object becomes eligible for garbage collection.

</details>

### How does the JVM determine whether an object is reachable?
<details><summary>Show answer</summary>

An object is **reachable** if the JVM can find it by following references starting from a **GC root**, such as:
- a local variable on the call stack,
- a static field,
- a running thread.

</details>

### What is a GC root?
<details><summary>Show answer</summary>

A **GC root** is an object or reference that is always considered reachable by the JVM 
and serves as a starting point for garbage‑collection reachability analysis.


Typical GC roots include:
- local variables on the call stack, 
- static fields, 
- active threads, 
- JNI references.

</details>

### How do stack variables affect object reachability?
<details><summary>Show answer</summary>

Stack variables affect object reachability by acting as **GC roots** while their methods are executing.


If a local (stack) variable holds a reference to an object, that object is considered reachable.  
Once the method returns and the stack frame is removed, those references disappear; 
if no other GC root references the object, it becomes unreachable and eligible for garbage collection.

</details>

### How do static fields affect object reachability?
<details><summary>Show answer</summary>

Static fields affect object reachability by acting as **GC roots for as long as their class is loaded**.


If a static field references an object, that object is always considered reachable.  
Only when the class is unloaded (or the static reference is cleared) 
can the object become unreachable and eligible for garbage collection.

</details>

### What does it mean for an object to be strongly reachable?
<details><summary>Show answer</summary>

An object is **strongly reachable** if it can be reached from a GC root **only through strong references**.

As long as an object is strongly reachable, the JVM considers it alive and it **cannot be garbage‑collected**, 
regardless of memory pressure.

</details>

### How does reachability differ from object lifetime?
<details><summary>Show answer</summary>

**Reachability** describes whether an object can currently be accessed by the running program via references from GC roots.


**Object lifetime** describes how long the object actually exists in memory.


An object’s lifetime may extend beyond the moment it becomes unreachable, 
because garbage collection is **non‑deterministic**.  


Once an object is unreachable, it becomes *eligible* for garbage collection, 
but it may remain in memory until the JVM decides to reclaim it.

</details>

### Can an object still be reachable but effectively unused?
<details><summary>Show answer</summary>

Yes. An object can be **reachable but effectively unused**.


This happens when the program no longer logically needs the object, 
but a reference to it is still held by a **GC root** 
(for example, a static field, a long‑lived cache, or a forgotten listener). 
As a result, the object remains reachable and cannot be garbage‑collected, even though it is no longer used in practice.


This situation is a common cause of **memory leaks** in Java.

</details>

## Reference types – strong → weak → soft → phantom

### When we consider reference types in Java, what topics should be considered?
<details><summary>Show answer</summary>

- reachability
- garbage‑collection behavior
- intended use cases

</details>

### What reference types does Java provide and how do they differ in terms of reachability, garbage‑collection behavior, and intended use cases?
<details><summary>Show answer</summary>

Java reference types define how strongly the JVM considers an object "alive",
allowing developers to balance memory safety, performance, and lifecycle control.

Java provides four reference types that differ in how strongly they keep objects reachable, 
when the garbage collector may reclaim them, and what problems they are intended to solve.

- **Strong references**  
  The default reference type in Java. 
  As long as an object is reachable through strong references from a GC root, it is never garbage‑collected.  
  **Use case:** normal object ownership and program logic.

- **Soft references (`SoftReference`)**  
  Objects remain reachable until the JVM experiences memory pressure, at which point they may be reclaimed.  
  **Garbage‑collection behavior:** cleared at the discretion of the JVM before an `OutOfMemoryError`.  
  **Use case:** memory‑sensitive caches (though their use is generally discouraged due to unpredictability).

- **Weak references (`WeakReference`)**  
  Objects are reclaimed as soon as they are no longer strongly reachable, typically at the next GC cycle.  
  **Garbage‑collection behavior:** GC does not try to preserve them.  
  **Use case:** canonical mappings, listeners, metadata, and caches that must not prevent collection.

- **Phantom references (`PhantomReference`)**  
  The weakest reference type. The object is already unreachable; `get()` always returns `null`.  
  **Garbage‑collection behavior:** enqueued after finalization, just before memory is reclaimed.  
  **Use case:** safe, post‑mortem cleanup of native or off‑heap resources (a replacement for finalization).

**Reachability strength (strong → weak):** Strong → Soft → Weak → Phantom

- Stronger references delay garbage collection longer;
- weaker references allow earlier reclamation and finer lifecycle control.

</details>

### In which scenarios should each reference type be avoided?
<details><summary>Show answer</summary>

Each Java reference type has scenarios where it **should be avoided**, usually when its guarantees do not match the program’s needs.

- **Strong references**  
  **Avoid when:**
  - Objects may outlive their usefulness and should not control their own lifetime
  - Used unintentionally in static fields, listeners, or caches, leading to memory leaks

- **Soft references (`SoftReference`)**  
  **Avoid when:**
  - Predictable cache size or eviction behavior is required
  - Application correctness depends on cached data
  - JVM‑dependent and GC‑dependent behavior is unacceptable

- **Weak references (`WeakReference`)**  
  **Avoid when:**
  - Objects must remain available for a guaranteed period
  - Sudden reclamation would break program logic or user expectations

- **Phantom references (`PhantomReference`)**  
  **Avoid when:**
  - You need access to the referenced object
  - Simpler lifecycle management mechanisms are sufficient
  - There is no need for post‑collection cleanup coordination

**Rule of thumb:**
> Avoid weaker reference types unless you explicitly need their garbage‑collection behavior and can tolerate the loss of object availability.

</details>

### Can a strongly referenced object ever be garbage‑collected?
<details><summary>Show answer</summary>

**No, a strongly referenced object cannot be garbage‑collected.**

As long as an object is **strongly reachable** through at least one reference chain from a GC root, the JVM considers it alive and will not reclaim it.

An object becomes eligible for garbage collection **only after all strong references to it are removed** (for example, when references go out of scope or are explicitly set to `null`).


</details>

### What common memory leaks are caused by strong references?
<details><summary>Show answer</summary>

Common memory leaks in Java are often caused by **unintended strong references** that keep objects reachable even though they are no longer logically needed.

Typical examples include:

- **Static fields holding object references**  
  Objects referenced by static fields remain reachable for the lifetime of the class, preventing garbage collection even if the objects are no longer used.

- **Unremoved listeners or callbacks**  
  Registering listeners without deregistering them keeps objects alive through long‑lived event sources.

- **Long‑lived collections (maps, lists, caches)**  
  Objects added to collections and never removed stay strongly reachable indefinitely, especially in caches or registries.

- **ThreadLocal misuse**  
  Values stored in `ThreadLocal` variables may remain reachable as long as the thread lives, even after the logical work is done.

- **Object graphs rooted in application scope**  
  Retaining a single strong reference to a large object graph (for example, via a singleton) keeps all reachable objects alive.

**Key insight:**
> A memory leak in Java is not about lost memory—it’s about objects that are *still reachable but no longer useful*, usually due to unintended strong references.


</details>

## Infrastructure – reference queues, cleaners


### How do reference queues work in Java, and how are they used to observe and react to garbage‑collection of referenced objects?
<details><summary>Show answer</summary>

A **ReferenceQueue** is a mechanism that lets an application be **notified when the garbage collector clears a reference**.

When an object referenced by a **weak, soft, or phantom reference** becomes unreachable, the JVM may **enqueue the corresponding reference into a ReferenceQueue**. The application can poll or block on this queue to detect that the object has been cleared and react accordingly.

Reference queues are typically used to:
- observe when objects are reclaimed,
- clean up associated resources or metadata,
- avoid relying on finalization.

In short, a `ReferenceQueue` provides a **safe, GC‑aware notification mechanism** for tracking object lifecycle events.

</details>

### Top 5 Dangerous Mistakes with Java Reference Types

1. **Accidentally keeping a strong reference**
   A lingering strong reference (for example in a static field, cache, or listener) completely negates weak or soft references, making objects never eligible for garbage collection.

2. **Using `SoftReference` for caches**
   Soft references are cleared based on JVM heuristics and memory pressure, which makes cache behavior unpredictable and can cause sudden performance degradation or cache thrashing in production.

3. **Misusing `WeakHashMap` keys**
   Keys can be garbage‑collected at any time if no strong reference exists elsewhere, leading to entries disappearing unexpectedly and causing subtle, hard‑to‑debug logic errors.

4. **Leaking `ThreadLocal` values in long‑lived threads**
   Storing objects in `ThreadLocal` without proper removal can keep them reachable for the lifetime of the thread, a common source of memory leaks in thread pools.

5. **Failing to process `ReferenceQueue`s**
   Not draining or handling a `ReferenceQueue` allows reference objects and associated metadata to accumulate, causing memory leaks and defeating the purpose of reference‑based cleanup.
