## Classes and Interfaces - Tagged and Nested Classes

### What are tagged classes in Java?
<details><summary>Show answer</summary>

A **tagged class** represents several variants in one class via an explicit **tag field** (an `enum` or `int`)
plus conditional logic (`if`/`switch`) on that tag.
The tag decides which fields are meaningful and which logic runs:

```java
public class Shape {
  enum Kind { RECTANGLE, CIRCLE }
  private final Kind kind;

  private final double width, height;   // only for RECTANGLE
  private final double radius;          // only for CIRCLE

  // ... one constructor per kind, initializing only its fields ...

  public double area() {
    switch (kind) {
      case RECTANGLE: return width * height;
      case CIRCLE:    return Math.PI * radius * radius;
      default:        throw new AssertionError(kind);
    }
  }
}
```

The tells: a `kind` field, fields meaningful only for *some* kinds, and a `switch` on `kind`.

</details>

### Why are tagged classes considered a design smell?
<details><summary>Show answer</summary>

(For what one looks like and how to spot it, see
[what tagged classes are](#what-are-tagged-classes-in-java).)


**Root cause: it violates the Single Responsibility Principle.**
One class holds several distinct concepts at once —
one per tag — instead of one concept per class. SRP is the root of the
[component-design chain](3.1_classes_and_interfaces.md#how-do-the-core-component-design-principles-depend-on-and-reinforce-each-other),
so once it breaks, the rest cascade: the per-tag branching, the fields that apply to only some tags, and the constant
editing to add a variant are all *downstream* of that one violation — which is why counting them as separate flaws
overstates the case. There's one root problem, not a list.


**The underlying diagnosis:** a tagged class is **a subtype hierarchy emulated badly with a field**. The tag hand-rolls,
at runtime, the subtyping the language already gives you for free — so you pay for branching, dead fields, and
error-prone variant-handling to reinvent something the compiler would enforce. That's the real reason it's a smell, and
why the fix is to use real types.
[→ what to use instead of a tagged class](#what-should-you-use-instead-of-a-tagged-class)

</details>

### What should you use instead of a tagged class?
<details><summary>Show answer</summary>

Give each variant its **own type**, so subtyping replaces the tag and the `switch` disappears. Three options, by case:

**Subclass hierarchy** — the direct fix; each variant a class, polymorphism replaces the branch:
```java
abstract class Shape { abstract double area(); }

class Circle extends Shape {
  final double r;
  Circle(double r) { this.r = r; }
  double area() { return Math.PI * r * r; }
}
// class Rectangle extends Shape { ... }   // one type per variant
```

**Enum with behavior** — when the variants are a small, fixed, known set:
```java
enum Planet {
  EARTH(5.97e24), MARS(6.42e23);   // ...one constant per variant
  final double mass;
  Planet(double mass) { this.mass = mass; }
  double gravityFactor() { return mass / 5.97e24; }
}
```

**Sealed classes + pattern-matching `switch`** (Java 17+/21) — a *closed* set of subtypes with exhaustive,
compiler-checked switching (the compiler errors if you forget a case):
```java
sealed interface Shape permits Circle, Square { }
record Circle(double r) implements Shape { }
record Square(double s) implements Shape { }

double area(Shape sh) {
  return switch (sh) {
    case Circle c -> Math.PI * c.r() * c.r();
    case Square s -> s.s() * s.s();
  };   // no default needed — permits list makes it exhaustive
}
```

**Which when:**
- open-ended variants → subclass hierarchy;
- a fixed small set carrying data → enum;
- a closed set you *want* to switch over exhaustively → sealed + pattern matching.

</details>

### A class needs a supporting type used nowhere else — what should you do?
<details><summary>Show answer</summary>

Keep it hidden — make it a **nested class**, defined inside its owner, so it's confined to the scope
that uses it and invisible to external clients. This keeps internal helpers out of the public API while
still expressing the behavior cleanly (better encapsulation, cohesion, API clarity).

The rule: a nested class should exist **only to serve its enclosing class**. But single-use is what *lets* you nest it,
not on its own a reason you *should* — whether nesting is the right call depends on further considerations (is the type
intrinsically tied to the parent, how big it is, how likely it is to be reused).
[→ when to promote a nested class to top-level](#when-should-a-nested-class-be-promoted-to-top-level)

</details>

### When should a nested class be promoted to top-level?
<details><summary>Show answer</summary>

Nesting trades **findability for encapsulation**: the helper is hidden and the API stays clean, but it's invisible
from outside, hard to locate later, and when a second consumer appears the easy path is to *rewrite* it rather than
find and reuse it. So single-use only *gates* nesting; three considerations decide whether to keep it nested or promote
it:

- **Intrinsic vs incidental binding** — does the type mean anything *without* the parent? A `Node` inside a specific
  list is intrinsic — nest it. A `RetryPolicy` or `Validator` that one class happens to use today is a standalone
  concept — lean top-level even at one consumer.
- **Size** — a tiny `record` helper costs almost nothing to hide or re-derive; a large class with its own logic is a
  discoverability and duplication liability when buried. The bigger it is, the sooner it should be top-level.
- **Reuse likelihood** — if a second consumer is plausible, promoting-later is more expensive (the rewrite trap) than
  a top-level class is today.

**Promote when any of these fires:**
- **a second consumer appears** — the bright line; promote immediately, never copy;
- the nested class **grows its own substantial logic or state**;
- it starts being **described as a concept in its own right**.

**The ladder: nested → top-level → its own package.** Once it's top-level *and* it's a coherent concept with related
types (a policy with its variants, a request with its builder), group those in a **package by responsibility** — the
same cohesion/SRP logic as the rest of component design.

**Default when unsure: top-level.** A top-level class used only once is a trivial cost; a nested class needed twice is
the rewrite trap. Single-consumer lets you *consider* nesting; reserve it for types that are intrinsically bound, small,
and unlikely to be reused.

</details>

### What kinds of nested classes are supported in Java?
<details><summary>Show answer</summary>

Two families:

- **Static nested classes** — `static` member classes.
- **Inner classes** (non-static) — three kinds:
    - non-static member classes,
    - local classes,
    - anonymous classes.

</details>

### What is the difference between static nested classes and inner classes?
<details><summary>Show answer</summary>

**Use a static nested class unless it must access an enclosing instance.**
The whole difference is the implicit reference to the enclosing object:

- **Static nested** — *no* implicit enclosing reference; sees only the enclosing class's **static** members;
  behaves like a top-level class scoped inside another.
- **Inner (non-static)** — *holds* an implicit reference to an enclosing instance;
  sees **both** static and instance members; always tied to a specific enclosing object.

That implicit reference is why inner classes are a frequent **memory-leak** source:
as long as the inner instance lives, it keeps the entire enclosing instance reachable — so the enclosing object can't
be collected even when nothing else needs it. Prefer static to avoid this.

</details>

### Why define a class right where you use it, unnamed?
<details><summary>Show answer</summary>

Sometimes you need an instance of a type — implementing an interface or extending a class — at **one** spot, used
**once**. Writing a separate named top-level or nested class for that single use is pure ceremony: a name to invent, a
declaration to place, a file or block to maintain, all for something referenced in exactly one place.

The need: **define and instantiate an implementation inline, at the point of use**, without a named class. Java offers
two tools for it, differing in *what shape* of one-off they handle:

- [**Anonymous classes**](#what-is-an-anonymous-class-and-when-is-it-useful-in-java) — the general form: any one-off
  implementation, including one with state, several methods, or a class to extend.
- [**Lambdas**](#what-is-a-lambda-and-when-is-it-useful-in-java) — the compact form for the common special case: the
  one-off is a single-method interface with no state.

Which to reach for is decided by what the one-off needs, not by fashion —
[lambda vs anonymous class](#lambda-or-anonymous-class--which-fits).

</details>

### What is an anonymous class, and when is it useful in Java?
<details><summary>Show answer</summary>

An **anonymous class** is an unnamed class declared and instantiated at once, at the point of use. It can implement an
interface or extend a class, and the expression both defines the class and creates the single instance:

```java
Comparator<String> byLength = new Comparator<>() {   // unnamed, defined and instantiated inline
  @Override public int compare(String a, String b) { return a.length() - b.length(); }
};
```

It's the right tool for a **one-off implementation** that is too much for the compact alternatives — specifically when
the inline implementation needs any of:

- **multiple methods** — implementing an interface (or overriding) with more than one method;
- **instance state** — its own fields, initialized per instance;
- **to extend a class** — not just implement an interface;
- **its own `this`** — inside the body, `this` refers to the anonymous instance itself.

Keep it small: an anonymous class can't be named, reused, or matched with `instanceof`, so anything sizeable belongs in
a named class instead.

</details>

### What is a lambda, and when is it useful in Java?
<details><summary>Show answer</summary>

A **lambda** is a compact inline expression that creates an instance of a **functional interface** — an interface with
exactly one abstract method. It supplies just that method's implementation, with no class declaration:

```java
Comparator<String> byLength = (a, b) -> a.length() - b.length();   // the one method, nothing else
```

It fits the common one-off case — a single piece of behavior handed to a method or stream — and is at its best when:

- the target is a **functional interface** (one abstract method) — the only shape a lambda can take;
- the behavior is **stateless**, using only its parameters and **effectively-final** captured variables (it holds no
  instance fields of its own);
- you want **`this` to mean the enclosing object** — inside a lambda `this` is the enclosing instance, not the lambda.

This makes lambdas clean for comparators, callbacks, and stream operations, where the whole point is one short behavior
with no surrounding machinery.

</details>

### Lambda or anonymous class — which fits?
<details><summary>Show answer</summary>

Decide by **capability**, not preference — each handles a different shape of inline one-off:

- **Target type** — lambda works *only* for a functional interface (one abstract method); an anonymous class can
  implement an interface with **many** methods, or **extend a class**.
- **State** — a lambda holds no instance fields (only captures effectively-final variables); an anonymous class can
  declare **its own fields**.
- **Meaning of `this`** — in a lambda `this` is the **enclosing** instance; in an anonymous class `this` is the
  **anonymous instance** itself.

So:

- **Lambda fits** when the one-off is a single-method interface, stateless, and you want enclosing `this` — less
  boilerplate, clearer at the call site, ideal in streams and callbacks.
- **Anonymous class is required** when the one-off needs **multiple methods**, **its own state**, **to extend a
  class**, or **its own `this`** — none of which a lambda can express.

The lambda is the narrower, lighter tool for the common case; the anonymous class is the general tool for everything a
lambda can't reach. Neither replaces the other — capability decides.

</details>

### What is a local class, and when is it useful in Java?
<details><summary>Show answer</summary>

A **local class** is a named class declared **inside a block** (method, constructor, loop) —
scoped like a local variable.

The only reason to prefer it over an anonymous class:
it has a **name**, so it can be referenced/reused within that scope and reads more clearly.
Useful when you need a small helper in *one* method that's too involved for an anonymous class — rare in practice.

</details>

### What are the best practices for organizing top-level classes in Java source files, and why do they matter?
<details><summary>Show answer</summary>

**One top-level class or interface per source file**, with the file named after it.
The compiler *allows* several per file, but there's no benefit and a real hazard.

**The hazard — compile-order-dependent builds.** If the same class is defined as a secondary top-level class in two
different files, *which definition wins depends on the order the files are compiled* — so the same source can silently
produce **different programs**.

Example:
```java
// File: Main.java
class Main { public static void main(String[] a) { System.out.println(new Helper().name()); } }
class Helper { String name() { return "from Main.java"; } }   // secondary top-level class
```
```java
// File: Util.java
class Helper { String name() { return "from Util.java"; } }   // SAME class, second definition
```
Compile `Main.java` first and you get one `Helper`; compile `Util.java` first and you get the other — the program's
behavior flips with build order, and some build setups won't even warn. One class per file makes this impossible:
definitions are unambiguous and builds are order-independent.

**If a type is too small to deserve its own file**, the answer is **not** a second top-level class in the file — it's a
[nested class](#a-class-needs-a-supporting-type-used-nowhere-else--what-should-you-do), scoped to its owner. That's the
correct home for a helper that doesn't warrant a top-level file.

</details>
