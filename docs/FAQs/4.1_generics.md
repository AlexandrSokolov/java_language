### What are Java Generics, and what specific problems were they intended to address?
<details><summary>Show answer</summary>

**Generics** (Java 5) let types and methods take *types as parameters*. 
Their purpose is **compile-time type safety** — catching type mismatches before the program runs.

**The problem they solve:** before generics, collections held `Object`, so:
- every retrieval needed a **manual cast**, and
- the compiler couldn't check those casts — a wrong type slipped through and blew up at runtime as a
  `ClassCastException`.

The transformation generics bring: they move that whole class of errors **from runtime to compile time** — 
the cast is now implicit and *guaranteed* correct. 
As a bonus, the code becomes **self-documenting**: `List<String>` states its element type; a raw `List` hides it.

</details>

### What is a generic type?
<details><summary>Show answer</summary>

A class or interface whose declaration has one or more type parameters is a generic class or interface. 
These type parameters act as placeholders that are specified when the type is instantiated.

```java
public interface Box<T> { 
  void pack(T item);
  T unpack();
}
```

</details>

### Define the components of the type `List<E>`
<details><summary>Show answer</summary>

- `List<E>` - generic type
- `List` - raw type
- `E` - formal type parameter

</details>

### Define the components of the type `List<String>`
<details><summary>Show answer</summary>

- `List<String>` - parameterized type
- `String` - actual type parameter (a.k.a. type argument)

</details>

### How do raw types differ from generic types, and why is using them dangerous?
<details><summary>Show answer</summary>

A **raw type** is a generic class/interface used with no type arguments (the raw form of `List<E>` is `List`).
They exist only for backward compatibility with pre-generics code.

**The core difference:** a raw type behaves as if all generic type information is erased to `Object`.
Using one **disables the compiler's type checking** — that lost safety is the root cause, and every danger below
comes from it.

**Root cause — Loss of Compile-Time Guardrails.**
Without type parameters, the compiler cannot check what is put *into* the collection, so any object can be added
without being detected. You are then forced to cast manually when reading values out, and that is where the runtime
`ClassCastException` finally happens.

This lost checking shows up in two places — and the second is the dangerous one:

1. **Locally — corrupting your own variable.**
   You can assign a `List<String>` to a raw `List`, then assign that raw `List` to a `List<Integer>`. The compiler
   allows this, and the program fails when *you* read the wrong type back out. This is bad, but contained — the
   failure is close to its cause.

2. **Across a method boundary — corrupting another method's collection (the real hazard).**
   If you pass a `List<String>` to a method that takes a raw `List`, that method can add an `Integer` to it. This is
   called **heap pollution**: a parameterized variable now refers to a collection that holds an object of the wrong
   type. The corruption happens *inside the called method*, but the failure appears *later, in the caller*, when it
   reads a `String` out of the now-polluted list — far from where the damage was done. This is what makes raw method
   parameters especially dangerous.

**The golden rule:** never use raw types. The only required exceptions (forced by erasure) are **class literals**
(`String.class`) and the **`instanceof`** operator (`o instanceof Set`).

</details>

### Why are raw types still permitted in Java?
<details><summary>Show answer</summary>

The reason is **migration compatibility** — letting old and new code work together.

When generics were added in Java 5, millions of lines of pre-generics code already existed.
That code had to stay legal and keep working with new generic code. So the design allows:
- passing a parameterized type (e.g. `List<String>`) to an old method that takes a raw `List`, and
- the reverse — passing a raw `List` where a parameterized type is expected.

This need is also *why* generics use **type erasure**: by removing type parameters at compile time, the
compiled bytecode stays the same as pre-generics bytecode, so old and new code remain binary-compatible.

(Raw types are also *required* in a few specific places — class literals and `instanceof` — but that is
a separate question: [when raw types are mandatory](#under-what-circumstances-if-any-is-the-use-of-raw-types-still-mandatory-in-modern-java).)

</details>

### Under what circumstances, if any, is the use of raw types still mandatory in modern Java?
<details><summary>Show answer</summary>

Raw types should generally be avoided, but two cases require them — both defined by the Java Language
Specification (JLS), both caused by type erasure at runtime:

1. **Class literals.**
   You must use the raw type in a class literal, because the type information is erased at runtime.
    - **Legal:** `List.class`, `String[].class`, `int.class`
    - **Illegal:** `List<String>.class`, `List<?>.class`

2. **The `instanceof` operator.**
   Generic type information is erased at runtime, so `instanceof` cannot check a parameterized type
   (an unbounded wildcard is the only allowed form). You check with the raw type, then cast to a
   wildcard type — the cast restores type safety, so you do not fall back into raw-type usage:
```java
   if (o instanceof Set) {       // 1. check with the raw type
       Set<?> s = (Set<?>) o;    // 2. cast to wildcard — safe, no raw type from here on
       // type-safe read-only operations...
   }
```

</details>

### How does Java's implementation of generics differ from other languages?
<details><summary>Show answer</summary>

Java implements generics with **type erasure**.

The distinguishing question across languages is *does generic type information survive to runtime?*

- **Java — erased.** The compiler checks types at compile time, then removes the type parameters,
  replacing them with their bound (or `Object`). One class file serves all uses; at runtime the type
  arguments are gone.
- **C++ — separate code per type.** Templates generate a new concrete class for each type used.
- **C# — reified.** Type information is kept at runtime; the type is *reifiable*.

**Why Java chose erasure:** to stay backward-compatible with pre-generics bytecode — the compiled
output looks the same as legacy code. The cost of this choice (what you *cannot* do because types are
not reified at runtime) is the subject of the next card.

</details>

### What are the drawbacks of Java's type erasure?
<details><summary>Show answer</summary>

All the drawbacks share one root cause: **the type information is gone at runtime** (the type is not reified).

- **Runtime limitations.** Because the type parameter no longer exists at runtime, you cannot do
  anything that needs it: no `instanceof T`, no `new T()`, and no creating arrays of a generic type
  (`new T[]`, `new List<String>[]`). These are simply not expressible.
- **Performance cost.** The JVM treats elements as `Object` (or the bound), so reading values often
  needs casts, and primitives are autoboxed. This can be slower than the per-type code that languages
  like C++ generate.
- **Heap pollution.** Because the runtime cannot check type parameters, a collection can be filled with
  the wrong type through a raw-type reference, causing a `ClassCastException` later that is hard to
  trace back to its cause.

</details>

### What are the primary constructs provided by the Java Generics API for type-safe design?
<details><summary>Show answer</summary>

Four top-level constructs — recall these four, then expand each:

1. [**Generic types**](#what-is-a-generic-type) — classes or interfaces with a type parameter (`List<E>`).

2. [**Generic methods and constructors**](4.3_generics.md#what-are-the-rules-for-using-generic-type-parameters-with-static-methods-or-static-fields) —
   their own type parameter, independent of the class (`public <T> void sort(List<T> list)`).

3. **Unbounded — no constraint on the type:**
    - plain type parameter `<T>` / `<E>`,
    - [unbounded wildcard `<?>`](#what-is-the-purpose-of-the-unbounded-wildcard-type--and-when-should-it-be-used) — when the exact type does not matter.

4. [**Bounded — the type is constrained**](4.2_generics.md#what-types-of-generic-bounds-exist-in-java) **(several forms):**
    - `<T extends X>` — a type and its subtypes,
    - `? extends T` / `? super T` — bounded wildcards (PECS),
    - [recursive bound `T extends Comparable<T>`](4.2_generics.md#how-to-enforce-that-a-type-interacts-only-with-its-exact-same-type-and-when-is-this-useful),
    - multiple bounds `<T extends A & B>`,
    - simulated self-type idiom `C extends Parent<C>`.

</details>

### What are unbounded types, and what options exist?
<details><summary>Show answer</summary>

**Purpose:** express a generic type when there is **no constraint** on the type argument — 
you don't care which specific type it is.

Two options:
- **Named type parameter `<T>` / `<E>`** — used in declarations, 
  when you need to *refer to* the type by name (e.g. a method that takes a `T` and returns a `T`).
- [**Unbounded wildcard `<?>`**](#what-is-the-purpose-of-the-unbounded-wildcard-type--and-when-should-it-be-used) —
  used when the type is unknown and you don't need to name it.

(Bounded types — where the type *is* constrained — are a separate topic, covered in `4_2`.)

</details>

### Wildcard types in Java and their primary use cases?
<details><summary>Show answer</summary>

A **wildcard** is the `?` form of a type argument — used when you refer to the type without naming it.
Two kinds:

- [**Unbounded `<?>`**](#what-is-the-purpose-of-the-unbounded-wildcard-type--and-when-should-it-be-used) —
  the type is unknown and the constraint does not matter.
- **Bounded `<? extends T>` / `<? super T>`** — the type is unknown but limited to a hierarchy;
  the basis of flexible API design (PECS). Covered in `4_2`.

</details>

### What is the purpose of the unbounded wildcard type `<?>` and when should it be used?
<details><summary>Show answer</summary>

An **unbounded wildcard** `List<?>` means a list of a specific but **unknown** type.
Use it when the actual element type does not matter:

1. **The logic uses only `Object`-level methods** — `size()`, `isEmpty()`, `clear()`, iteration.
   *Example:* `public static void printSize(List<?> list) { System.out.println(list.size()); }`

2. **The code works regardless of the element type** — generic operations or reflection.
   *Example:* `Class<?> clazz = someObject.getClass();`

**Key limitation — effectively read-only.** You cannot add any element except `null`.
The reason: the compiler doesn't know the real element type, so it cannot guarantee that what you add
is compatible — a `List<?>` might actually be a `List<String>`, and allowing an `Integer` in would corrupt it. 
Reading is fine (everything is at least `Object`); writing is blocked.

</details>

### How do you represent a collection of arbitrary objects in a type-safe way?
<details><summary>Show answer</summary>

Use a parameterized type with `Object` as the argument: **`List<Object>`**.

This tells the compiler the collection is meant to hold any type of object, while keeping full
type-checking: the compiler still verifies what you add and gives you typed reads (everything comes
out as `Object`). You get maximum flexibility for mixed contents *without* turning off type safety —
which is what raw types or `<?>` would do.

</details>

### Why shouldn't you use a raw type or an unbounded wildcard to represent a collection of arbitrary objects?
<details><summary>Show answer</summary>

Both seem to "allow any object," but they fail for **opposite reasons** — 
and `List<Object>` is the balance between them.

**Raw `List` — too permissive (no safety).**
It disables the generic type system entirely. The compiler can no longer check what goes in, 
so you get silent heap pollution and delayed `ClassCastException` at runtime.

**`List<?>` — too restrictive (read-only).**
It does not mean "a list that holds any type." It means "a list of one specific but unknown type."
Since the compiler doesn't know that type, it blocks all writes (except `null`) — you cannot add
arbitrary objects. Safe, but unusable as a mutable any-object collection.

So 
- raw gives flexibility without safety; 
- `<?>` gives safety without flexibility. 
- `List<Object>` gives both — full type checking *and* the ability to add any object.

</details>

### What is the difference between `List<Object>` and the raw type `List`?
<details><summary>Show answer</summary>

Both can hold arbitrary objects, but `List<Object>` is type-safe and raw `List` is not. 
The difference is in the **subtyping rules**:

- `String` is a subtype of `Object`, but `List<String>` is **not** a subtype of `List<Object>` (generics are invariant).
- However, `List<String>` **is** a subtype of the raw `List`.

**The consequence:**
- Because `List<String>` *can* be passed where a raw `List` is expected, 
  a method taking a raw `List` can insert an incompatible object and corrupt it — type safety is lost.
- Because `List<String>` *cannot* be passed where a `List<Object>` is expected, 
  the compiler stops you from corrupting the specific list — type safety is kept.

**Analogy:**
- **Raw `List`** is like a box with the label ripped off; you can put anything in,
  but you don't know what was originally inside.
- **`List<Object>`** is a box explicitly labeled "For Any Object";
  it refuses to be confused with a box labeled "Only Strings."

```java
public class SafetyTest {
   public static void main(String[] args) {
      List<String> strings = new ArrayList<>();
      strings.add("Safe");

      unsafeAdd(strings, Integer.valueOf(42));   // raw type: allowed
      String s = strings.get(1);                 // RUNTIME ERROR: ClassCastException

      // safeAdd(strings, 42);                    // COMPILE ERROR: List<String> not a List<Object>
   }

   static void unsafeAdd(List list, Object item) {   // raw list — corrupts it
      list.add(item);
   }

   static void safeAdd(List<Object> list, Object item) {   // compiler won't let you pass a List<String> into it.
      list.add(item);
   }
}
```

</details>

### Comparison: Raw List vs. List parameterized with Object vs. Unbounded Wildcard List
<details><summary>Show answer</summary>

| Feature                     | Raw `List`        | `List<Object>`    | `List<?>`         |
|:----------------------------|:------------------|:------------------|:------------------|
| **Type Safe?**              | No                | Yes               | Yes               |
| **Can Add?**                | Yes (Any Object)  | Yes (Any Object)  | No (Only `null`)  |
| **Can Read?**               | Yes (As `Object`) | Yes (As `Object`) | Yes (As `Object`) |
| **Accepts `List<String>`?** | Yes (Unsafe)      | No                | Yes (Safe)        |

**Usage Guidelines:**

- **Use `List`:** Only for legacy code interoperability. **Never** in new code.
- **Use `List<Object>`:** When the collection must be able to hold and store **arbitrary** objects of any type.
- **Use `List<?>`:** When writing a **read-only** method
  where the specific type parameter doesn't matter (e.g., `list.size()`).

</details>

### If A is a subtype of B, is a container of A a subtype of a container of B?
<details><summary>Show answer</summary>

It depends on the container — there are two opposite behaviors:

**Arrays — covariant.** If `A` is a subtype of `B`, then `A[]` **is** a subtype of `B[]`.
- *Upside:* a method like `Arrays.sort(Object[])` works on any array.
- *Risk — fails at runtime.* You can treat a `String[]` as `Object[]` and try to store an `Integer`;
  it compiles, then throws `ArrayStoreException` at runtime.

**Generics — invariant.** If `A` is a subtype of `B`, `List<A>` has **no** subtype relationship to
`List<B>` — they are unrelated types.
- *Upside — fails at compile time.* You cannot slip an `Integer` into a `List<String>` through a  broader reference; 
  the compiler stops it.
- *Trade-off:* less flexible — you cannot pass a `List<String>` where a `List<Object>` is expected.

| Container    | Relationship | Example                                    |
|:-------------|:-------------|:-------------------------------------------|
| **Arrays**   | covariant    | `String[]` **is** an `Object[]`            |
| **Generics** | invariant    | `List<String>` is **not** a `List<Object>` |

The key contrast: arrays push the type error to **runtime**, generics catch it at **compile time** —
which is exactly why generics were made invariant.

</details>

### What is the fundamental design reason for making generic types invariant rather than covariant?
<details><summary>Show answer</summary>

To preserve **compile-time type safety** and prevent **heap pollution**. 
The reason is clearest as the disaster that covariance would allow:

1. Create a `List<String>`.
2. Treat it as a `List<Object>` — *if* generics were covariant.
3. Add an `Integer` to it — legal for a `List<Object>`.
4. Read it back through the original `List<String>` as a `String` → `ClassCastException`.

Invariance closes this at step 2: `List<String>` is simply *not* a `List<Object>`, 
so the broader reference is never allowed, and the compiler can guarantee a `List<String>` holds only strings.

The payoff: the error that arrays only catch at **runtime** (`ArrayStoreException`) 
is caught by generics at **compile time** — and compile-time errors are found and fixed immediately, 
before the code ever runs.

</details>

### Arrays or Lists: Which should be preferred when using Generics?
<details><summary>Show answer</summary>

In almost all cases, **prefer Lists**. Arrays and generics don't mix, for two reasons:

1. **Covariant vs invariant.**
    - Arrays are **covariant** (`String[]` is an `Object[]`) → type errors slip to **runtime**
      (`ArrayStoreException`).
    - Generics are **invariant** → the same errors are caught at **compile time**.

2. **Reified vs erased.**
    - Arrays are **reified** — they enforce their element type at runtime.
    - Generics are **erased** — the type exists only at compile time.

**Bottom line:** stay within the generic type system — use Lists, not arrays of generics.

</details>

### How do you create a generic array?
<details><summary>Show answer</summary>

You don't — it's a compile error. `new T[]`, `new E[n]`, and `new List<String>[]` are all illegal.

**Why — it would break type safety.** Arrays are *reified* (they check their element type at runtime),
but generics are *erased* (the type is gone at runtime). A generic array would have to enforce a type
it no longer knows, so the compiler forbids it to prevent heap pollution. If it were allowed:
```java
List<String>[] arr = new List<String>[1];   // (illegal) suppose it compiled...
Object[] objs = arr;                          // arrays are covariant
objs[0] = List.of(42);                         // store a List<Integer> — no runtime check stops it
String s = arr[0].get(0);                      // ClassCastException, far from the cause
```

**The workaround — make an `Object[]` and cast:**
```java
@SuppressWarnings("unchecked")
T[] arr = (T[]) new Object[n];
```
This is how `ArrayList` and similar collections are built internally.
It produces an unchecked warning,
which you suppress *only* after confirming the array never escapes in a way that exposes the unsafe cast.

**The lesson:** a common concrete reason to **prefer `List<T>` over `T[]`** — lists have no such
restriction.

</details>

### What is the recommended strategy for managing unchecked warnings in Java?
<details><summary>Show answer</summary>

The goal should always be to eliminate every unchecked warning to ensure **runtime type safety**.
The process follows these steps:

1. **Eliminate if possible:** Change the code to be type-safe so the warning disappears naturally.
2. **Suppress it** — only if you *can prove* the code is safe but the compiler can't, hide it with
   `@SuppressWarnings("unchecked")`. Never suppress a warning you haven't proven safe.

</details>

### What are the best practices for using `@SuppressWarnings("unchecked")` to maintain code quality?
<details><summary>Show answer</summary>

When you can't remove an unchecked warning but have proven the code safe, follow two rules:

**1. Minimize scope.** Apply the annotation to the smallest unit possible — ideally a single local variable declaration, 
never a whole method or class:
```java
@SuppressWarnings("unchecked")            // on the local, not the method
T[] result = (T[]) Arrays.copyOf(elements, size, newType);
return result;
```
Why so narrow: a method-level annotation mutes *every* unchecked warning in the method body. 
If someone later adds a genuinely unsafe cast elsewhere in that method, the compiler stays silent — 
the warning that should have caught it was muted. 
A local-variable annotation suppresses only the one line you proved safe.

*Exception:* for a one-line method (e.g. a single `return` of an unchecked cast), method-level is fine —
there's no other code for it to mute, and forcing a local variable just adds noise.

**2. Document the proof.** Always add a comment explaining *why* the cast is safe, so a future
maintainer doesn't break the hidden guarantee:
```java
// safe: the array holds only T instances added through our own typed API
@SuppressWarnings("unchecked")
T[] result = (T[]) Arrays.copyOf(elements, size, a.getClass());
```

</details>
