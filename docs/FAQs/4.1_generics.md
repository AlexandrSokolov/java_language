## Generics

### What are Java Generics, and what specific problems were they intended to address?
<details><summary>Show answer</summary>

**Generics** are a facility introduced in Java 5 that allow types to be parameters when defining types and methods. 
They provide **compile-time type safety**, catching mismatches before execution.

Before Generics, Java relied on the `Object` class for collections, requiring developers to use manual
explicit casting for every retrieval. This approach was error-prone because type mismatches could not be
detected by the compiler, often leading to `ClassCastException` failures at runtime.

</details>

### What is a generic type?
<details><summary>Show answer</summary>

A class or interface whose declaration has one or more type parameters is a generic class or interface. 
These type parameters act as placeholders that are specified when the type is instantiated.

```java
public interface Box<T> { 
  void pack(T item);
  T unpack();
}
```

</details>

### How do raw types differ from generic types?
<details><summary>Show answer</summary>

A raw type is the name of a generic class or interface used without any type arguments 
(e.g., the raw type for `List<E>` is `List`).

**Key Characteristics:**
- **Behavior:** Raw types behave as if all generic type information has been erased, 
  essentially defaulting to the `Object` type for all elements.
- **Purpose:** They exist primarily to ensure backward compatibility with legacy (pre-generics) code.
- **Risk:** Using them bypasses compile-time type safety, leading to potential `ClassCastException` errors at runtime.

You should never use raw types. If you use raw types, you lose all the safety and expressiveness benefits of generics.

</details>

### Why are raw types still permitted in Java?
<details><summary>Show answer</summary>

The inclusion of raw types was driven by the necessity of **migration compatibility**.

- **Legacy Interoperability:** It was deemed critical that millions of lines of pre-existing code
  remain legal and fully interoperate with newer code using generics.
- **Bi-directional Support:** The design allows passing instances of parameterized types to
  methods designed for raw types, and vice versa.
- **Implementation Choice:** This requirement was the primary driver behind the decision to
  implement generics using **type erasure**, ensuring the language remained backward compatible.

</details>

### How does Java's implementation of generics differ from other languages?
<details><summary>Show answer</summary>

Java implements generics using **type erasure**.

- **Mechanism:** The compiler enforces type constraints at compile-time 
  and erases type parameters during code generation, replacing them with their bounds (or `Object`).
- **Comparison:** Unlike C++ templates (which generate a new class per type) or C# (which preserves type
  information at runtime), Java's approach ensures a single class file is used for all instantiations.
- **Benefit:** This architecture was chosen specifically to maintain backward compatibility with legacy,
  pre-generic bytecode.

While type erasure ensures backward compatibility, its main drawback is **runtime type loss**.

- **Run-time Limitations:** Because type parameters are erased, you cannot perform operations that require
  type information at runtime, such as `instanceof T`, `new T()`, or creating arrays of a generic type.
- **Performance Impact:** Since the JVM treats elements as `Object` (or the bound), frequent casting and
  autoboxing of primitives may occur, which can be less efficient than the specialized code generation
  found in languages like C++.
- **Heap Pollution:** It allows for scenarios where a collection can be "polluted" with an incorrect type via
  raw type references, leading to unexpected `ClassCastException` errors that are hard to debug.

</details>


### What are the primary constructs provided by the Java Generics API for type-safe design?
<details><summary>Show answer</summary>

Java provides four primary approaches to implement and control generic behavior:

- **Generic Types:** Classes or interfaces parameterized over types (e.g., `List<E>`).
- **Generic Methods & Constructors:** Methods or constructors that introduce their own type parameters
  independently of the class (e.g., `public <T> void sort(List<T> list)`).
- **Bounded Wildcard Types:** Increase API flexibility by restricting a type to a specific hierarchy
  (e.g., `? extends T` or `? super T`).
- **Unbounded Wildcard Types:** Used when the specific type is irrelevant or only `Object` methods
  are needed (e.g., `List<?>`).
- **Recursive Type Bounds:** A type parameter bounded by its own type, often for comparison logic
  (e.g., `T extends Comparable<T>`).
- **Multiple Bounds:** Restricting a parameter to multiple types using the `&` operator
  (e.g., `<T extends Person & Serializable>`).

</details>

### How do you represent a collection of arbitrary objects in a type-safe way?
<details><summary>Show answer</summary>

The best choice is a parameterized type using `Object`, such as **`List<Object>`**.

- **Type Safety:** `List<Object>` explicitly tells the compiler that the list can hold any object. 
  You gain the benefit of generic type checking while maintaining maximum flexibility.
- **Vs. Raw List:** While a raw `List` also allows any object, it opts out of the generic type system entirely. 
  This is dangerous because it bypasses the safety checks 
  that prevent you from accidentally corrupting other parameterized collections.
- **Vs. `List<?>` :** An unbounded wildcard `List<?>` is not suitable for this purpose 
  because it is read-only (except for `null`). 
  You cannot insert arbitrary objects into a `List<?>` 
  because the compiler cannot verify the specific type the list is intended to hold.

</details>

### What is the difference between `List<Object>` and the raw type `List`?
<details><summary>Show answer</summary>

While both can hold arbitrary objects, `List<Object>` is type-safe whereas the raw type `List` is not.

- **The Sub-typing Rule:** 
  While `String` is a subtype of `Object`, `List<String>` is **not** a subtype of `List<Object>`. 
  However, `List<String>` **is** considered a subtype of the raw type `List`.
- **The Consequence of Raw Types:** Because `List<String>` can be passed to a raw `List` parameter, you lose type safety. 
  The raw type ignores generic information, allowing the insertion of incompatible objects.
- **The Safety of `List<Object>`:** Because `List<String>` cannot be passed to a `List<Object>` parameter, 
  the compiler prevents you from accidentally corrupting the specific list.

**Analogy:** 
- **Raw `List`** is like a box with the label ripped off; you can put anything in, 
  but you don't know what was originally inside.
- **`List<Object>`** is a box explicitly labeled "For Any Object"; 
  it refuses to be confused with a box labeled "Only Strings."


```java
public class SafetyTest {
   public static void main(String[] args) {
      List<String> strings = new ArrayList<>();
      strings.add("Safe");

      // 1. Using Raw Type (UNSAFE)
      unsafeAdd(strings, Integer.valueOf(42));
      String s = strings.get(1); // RUNTIME ERROR: ClassCastException

      // 2. Using List<Object> (SAFE)
      // safeAdd(strings, 42); // COMPILE-TIME ERROR: Incompatible types
   }

   // This method accepts a raw list and corrupts it
   static void unsafeAdd(List list, Object item) {
      list.add(item);
   }

   // This method would be safer, but the compiler won't even let you
   // pass a List<String> into it.
   static void safeAdd(List<Object> list, Object item) {
      list.add(item);
   }
}
```

</details>

### Wildcard types in Java and their primary use cases?
<details><summary>Show answer</summary>

Wildcard types are represented by the `?` symbol and are categorized based on how they restrict the allowable types:
- [**Unbounded Wildcards (`<?>`)**](#what-is-the-purpose-of-the-unbounded-wildcard-type--and-when-should-it-be-used) used when:
  - the code only requires functionality provided by the `Object` class 
  - the specific type parameter is irrelevant to the logic.
- **Bounded Wildcards (`<? extends T>` or `<? super T>`):** 
  Used to increase API flexibility by restricting the type to a specific hierarchy, 
  allowing for either "producer" or "consumer" behavior relative to a base type.


</details>

### What is the purpose of the unbounded wildcard type `<?>` and when should it be used?
<details><summary>Show answer</summary>

An **unbounded wildcard** (`List<?>`) represents a collection of a specific, but unknown type.
It is primarily used in two scenarios:

1. **When functionality is independent of the type parameter:**
   When your method only uses methods provided by the `Object` class (e.g., `size()`, `isEmpty()`,
   or `clear()`).
   *Example:* `public static void printSize(List<?> list) { System.out.println(list.size()); }`

2. **When the code uses a generic class that doesn't depend on the type parameter:**
   Common in logic that performs generic operations or uses reflection.
   *Example:* `Class<?> clazz = someObject.getClass();` 
   or a method that swaps elements in a list regardless of what they are.

**Crucial Limitation:** A `List<?>` is effectively read-only. 
You cannot add any element (except `null`) to it because the compiler cannot guarantee the specific type of the list.

</details>

### What is the difference between an unbounded wildcard `List<?>` and a raw type `List`?
<details><summary>Show answer</summary>

The fundamental difference lies in **type safety** and the **intent** conveyed to the compiler:

- **Type Safety:** 
  - The **raw type `List`** is considered unsafe. 
    It opts out of generic checks entirely, allowing you to insert any object, 
    which can lead to `ClassCastException` at runtime.
  - The **unbounded wildcard `List<?>`** is safe. The compiler knows it represents a specific (but unknown) type, 
    so it **prevents** you from adding any element to it (except `null`).

- **Internal Logic:**
   - `List` tells the compiler: "I don't care about generics; treat everything as Object."
   - `List<?>` tells the compiler: 
     "I am using generics, but I don't know/need to know the specific type parameter for this operation."

- **Assignment Rules:**
   - You can assign a `List<String>` to a raw `List` (unsafe).
   - You can assign a `List<String>` to a `List<?>` (safe and common for read-only utility methods).

**Summary:** Use `List<?>` when you want a truly generic, read-only collection that is enforced by the compiler. 
Avoid raw `List` as it is a legacy construct that compromises type safety.

</details>

### Comparison: Raw List vs. List<Object> vs. Unbounded Wildcard List<?>
<details><summary>Show answer</summary>

| Feature                     | Raw `List`        | `List<Object>`    | `List<?>`         |
|:----------------------------|:------------------|:------------------|:------------------|
| **Type Safe?**              | No                | Yes               | Yes               |
| **Can Add?**                | Yes (Any Object)  | Yes (Any Object)  | No (Only `null`)  |
| **Can Read?**               | Yes (As `Object`) | Yes (As `Object`) | Yes (As `Object`) |
| **Accepts `List<String>`?** | Yes (Unsafe)      | No                | Yes (Safe)        |

**Usage Guidelines:**

- **Use `List`:** Only for legacy code interoperability. **Never** in new code.
- **Use `List<Object>`:** When the collection must be able to hold and store **arbitrary** objects of any type.
- **Use `List<?>`:** When writing a **read-only** method 
  where the specific type parameter doesn't matter (e.g., `list.size()`).

</details>

### Under what circumstances, if any, is the use of raw types still mandatory in modern Java?
<details><summary>Show answer</summary>

While raw types should generally be avoided, 
there are two primary technical exceptions defined by the Java Language Specification (JLS):

1. **Class Literals:**
   You must use raw types when referring to class literals. Parameterized types are not permitted
   because type information is erased at runtime.
   - **Legal:** `List.class`, `String[].class`, `int.class`
   - **Illegal:** `List<String>.class` or `List<?>.class`

2. **The `instanceof` Operator:**
   Because generic type information is erased at runtime, it is illegal to use `instanceof` on
   parameterized types (except for unbounded wildcards). Using the raw type is the standard
   approach, but it should be followed by a cast to a wildcard type for safety.

   **Example of correct `instanceof` usage:**
   ```java
   if (o instanceof Set) {       // 1. Check using the raw type
       Set<?> s = (Set<?>) o;    // 2. Cast to wildcard type (checked cast)
       // Proceed with type-safe read-only operations...
   }
   ```
