## Creating and Destroying Objects

### How are class instances created?
<details><summary>Show answer</summary>

- [via constructor](#what-is-the-most-basic-way-to-obtain-an-instance)
- [via static factory method](#what-is-a-static-factory-method)
- [via the Builder pattern](#what-is-the-builder-pattern)

</details>

### What is the most basic way to obtain an instance?
<details><summary>Show answer</summary>

The traditional way for a class to allow a client to obtain an instance is to provide a public constructor.

</details>

### What is a static factory method?
<details><summary>Show answer</summary>

A class can provide a public static factory method, 
which is simply a static method that returns an instance of the class:
```java
public static Boolean valueOf(boolean b) {
  return b ? Boolean.TRUE : Boolean.FALSE;
}
```

</details>

### Static factory methods — trade-offs?
<details><summary>Show answer</summary>

**Advantages:**

- **Naming** — 
  [a name lets it express intent and sidestep constructor signature clashes (no parameter-order tricks)](#naming-static-factory-vs-constructor)
- **Instance control** — [needn't return a new object (a constructor must)](1.3_objects_creating_destroying.md#constructors-vs-factories-on-object-reuse), 
  which is what enables [instance-controlled classes](#what-are-instance-controlled-classes).
- [**API/implementation decoupling**](#returning-a-subtype--whats-the-payoff) 
  — a factory can return any subtype of its declared return type

**Disadvantages:**

- a class with no public/protected constructor cannot be subclassed
  (arguably a blessing — it pushes composition over inheritance, and is required for immutable types)
- factories are hard to discover — they don't stand out in API docs the way constructors do

</details>

### Naming: static factory vs constructor?
<details><summary>Show answer</summary>

Unlike constructors, static factory methods have names. 
- Carefully chosen names to highlight their differences and more clear as a result.
- Also, a class can have only a single constructor with a given signature.
  Programmers have been known to get around this restriction by providing two constructors whose parameter lists
  differ only in the order of their parameter types.
  Carefully chosen names to highlight their differences solves this issue.

</details>

### What are instance-controlled classes?
<details><summary>Show answer</summary>

Classes that maintain strict control over which instances exist at any time are said to be _instance-controlled_.

This rests on the static factory's ability to return the *same* object from repeated invocations —
something a constructor (always returning a new object) cannot do.

The category is broader than singletons — it spans the whole range of "how many instances may exist":
- **noninstantiable** — *zero* instances (utility classes)
- **singleton** — *exactly one* instance
- **enum-based / Flyweight** — *one canonical instance per logical value* (a fixed, controlled set)

</details>

### Why write instance-controlled classes?
<details><summary>Show answer</summary>

- **zero** → guarantee the class is **noninstantiable** (utility classes)
- **exactly one** → guarantee a **singleton**
- **one per logical value** → for **enum-based / interned value classes**, guarantee a fixed, canonical
  set of instances

The payoff of that last case: when there's exactly one instance per value, **no two equal instances
can exist**, so `a.equals(b)` iff `a == b`. Immutability alone does *not* give this — you can still
`new` a second object with equal data (e.g. `new BigInteger("7")`); instance control is what closes
that gap. Enum types provide it for free.

</details>

### Returning a subtype — what's the payoff?
<details><summary>Show answer</summary>

Because the choice is **invisible to clients** — they get back some subtype of the declared return
type and neither know nor care which. That insulation buys freedom to evolve implementations:

- swap one implementation for another when it stops being optimal
- add new implementations later if they prove beneficial (e.g. for performance)
- remove an implementation with no client impact

Two independent freedoms enable this:
- the returned class may **vary by input parameter** (call to call)
- the returned class may **vary from release to release**

Use cases:
- [**interface-based design**: return an interface type and hide concrete implementations](#how-can-an-api-return-objects-without-exposing-their-classes)
- pick the implementation based on input parameters — e.g. the optimal one for performance
  - **The mechanism.** It returns **any subtype** of its declared return type, selected at call time from the input.
    `EnumSet` is the canonical example — no public constructors, 
    only static factories, which pick a subclass by the enum's size:
    - **≤ 64 elements** → `RegularEnumSet`, backed by a single `long`
    - **≥ 65 elements** → `JumboEnumSet`, backed by a `long[]`

  Both subclasses are non-public; clients only ever see `EnumSet`.
- [support **service-provider frameworks**, where the impl need not exist when the API is written](3.6_classes_and_interfaces.md#what-problem-do-service-provider-frameworks-solve)

</details>

### How can an API return objects without exposing their classes?
<details><summary>Show answer</summary>

By declaring the factory's **return type as an interface** (or supertype) and returning 
a non-public implementation class. Clients program against the interface and 
never see — or depend on — the concrete type.

This works because, unlike a constructor, a factory may return **any subtype** of its declared return type.

Payoff: a compact API with hidden implementations. This is the basis of **interface-based design**
(a.k.a. interface-based frameworks), where interfaces are the natural return types for factories.

</details>

### Convenience used to define static factory methods on interfaces
<details><summary>Show answer</summary>

- Prior to Java 8, interfaces couldn’t have static methods. 
  By convention, static factory methods for an interface named `Type` were put in a _noninstantiable companion class_ 
  named `Types`.
- As of Java 8, the restriction that interfaces cannot contain static methods was eliminated, 
  so there is typically little reason to provide a _noninstantiable companion class_ for an interface. 
  Many public static members that would have been at home in such a class should instead be put in the interface itself.

</details>

### Static factory method vs the Factory Method pattern
<details><summary>Show answer</summary>

Despite the shared word "factory," these are unrelated. The dividing axis is **static vs polymorphic**:

- **Static factory method** — a *static* helper that returns an instance. No inheritance, nothing to
  override. Just a [more flexible alternative to a constructor](#static-factory-methods--trade-offs)
  (can name itself, can return an existing instance, can return a subtype). *Not* a GoF pattern.
- **Factory Method (GoF)** — a creational *design pattern*. An **instance** method that **subclasses
  override** to decide which concrete class to instantiate. The whole point is polymorphic dispatch
  through a hierarchy.

One-line tell: a static factory has **no subclass involved**; the Factory Method pattern **is**
about the subclass choosing.

</details>

### Common names for static factory methods
<details><summary>Show answer</summary>

Conventions split along two cuts: **what the result is** (value vs identity), 
and **where the factory lives** (same class vs different class).

**Describe the result by value:**
- `from` — type-conversion, one argument → one instance: `Date.from(instant)`
- `of` — aggregation, several arguments → an instance incorporating them: `EnumSet.of(JACK, QUEEN, KING)`
- `valueOf` — verbose alternative to `from` / `of`: `BigInteger.valueOf(Integer.MAX_VALUE)`

**Describe identity (same vs new instance):**
- `instance` / `getInstance` — returns *an* instance per the args; **no** same-value guarantee:
  `StackWalker.getInstance(options)`
- `create` / `newInstance` — like above, but **guarantees a fresh instance each call**:
  `Array.newInstance(classObject, arrayLen)`

**Same as the two above, but when the factory is in a *different* class** (`Type` = returned type):
- `getType` — like `getInstance`, factory elsewhere: `Files.getFileStore(path)`
- `newType` — like `newInstance`, factory elsewhere: `Files.newBufferedReader(path)`
- `type` — concise alternative to `getType` / `newType`: `Collections.list(legacyLitany)`

</details>

### How to handle many optional parameters?
<details><summary>Show answer</summary>

Both constructors and static factories share one limitation: they **don't scale to many optional parameters**. 

- ❌ [Telescoping constructor pattern](#what-is-the-telescoping-constructor-pattern) — unreadable past a few params
- ❌ [JavaBeans pattern](#what-is-the-javabeans-pattern) — allows inconsistent state, precludes immutability
- ✅ [Builder pattern](#what-is-the-builder-pattern) — the recommended option

</details>

### What is the telescoping constructor pattern?
<details><summary>Show answer</summary>

You provide a constructor with only the required params, 
then one with a single optional param, then two, and so on — each chaining to the next via `this(...)`:

```java
// Does not scale — each ctor chains to the fuller one
public NutritionFacts(int servingSize, int servings) {
  this(servingSize, servings, 0);                 // calories defaults to 0
}
public NutritionFacts(int servingSize, int servings, int calories) {
  this(servingSize, servings, calories, 0);       // fat defaults to 0
}
public NutritionFacts(int servingSize, int servings, int calories, int fat) {
  this.servingSize = servingSize;
  this.servings = servings;
  this.calories = calories;
  this.fat = fat;
}
```

It works, but client code is hard to **write** (which overload? what's that bare `0`?) and harder to **read**. 
Scales badly as optional params grow.

</details>

### What is the JavaBeans pattern?
<details><summary>Show answer</summary>

Call a no-arg constructor, then set each parameter with a setter:

```java
// Allows inconsistency, mandates mutability
NutritionFacts cocaCola = new NutritionFacts();
cocaCola.setServingSize(240);
cocaCola.setServings(8);
cocaCola.setCalories(100);   // construction spread across many calls
```

```java
public class NutritionFacts {
  private int servingSize = -1;   // required, no default
  private int calories = 0;       // optional
  public NutritionFacts() { }
  public void setServingSize(int v) { servingSize = v; }
  public void setCalories(int v)    { calories = v; }
  // ...one setter per field
}
```

Two serious disadvantages:
- **Inconsistent mid-construction state** — construction is split across calls, so the object may be
  half-built when used; the class can't enforce invariants by validating constructor args. Resulting
  failures are far from the buggy code and hard to debug.
- **Precludes immutability** — fields must stay mutable, forcing extra effort for thread safety.

</details>

### What is the Builder pattern?
<details><summary>Show answer</summary>

The client:
1. gets a **builder** from a **static `builder(...)` factory** taking the **required** params
2. calls **chaining setters** for the optional params of interest
3. calls **`build()`** to get the (typically immutable) object

```java
public class NutritionFacts {
  private final int servingSize, servings, calories, fat;

  public static Builder builder(int servingSize, int servings) {   // entry point
    return new Builder(servingSize, servings);
  }

  public static class Builder {
    private final int servingSize, servings;   // required
    private int calories = 0, fat = 0;          // optional, defaulted

    private Builder(int servingSize, int servings) {   // private; required via factory
      this.servingSize = servingSize;
      this.servings = servings;
    }
    public Builder calories(int v) { calories = v; return this; }  // chaining
    public Builder fat(int v)      { fat = v;      return this; }

    public NutritionFacts build() { return new NutritionFacts(this); }
  }

  private NutritionFacts(Builder b) {           // private; takes the builder
    servingSize = b.servingSize; servings = b.servings;
    calories = b.calories; fat = b.fat;
  }
}
```

```java
NutritionFacts cocaCola = NutritionFacts.builder(240, 8)
    .calories(100).fat(0).build();
```

> Prefer a static `builder(...)` factory over a public `Builder` constructor — it hides `new` and
> the nested type, and is a static factory in its own right. (A public `Builder` constructor still
> earns its place in *hierarchical* builders, where subclass builders call `super(...)`.)

</details>

### What language feature does the Builder pattern emulate?
<details><summary>Show answer</summary>

**Named optional parameters with defaults** — which Scala and Python have natively, and Java lacks.
The Builder is Java's workaround for that missing feature.

```scala
def connect(host: String = "localhost", port: Int = 3306, secure: Boolean = false): Unit = ...

connect(port = 5432)                          // set only what you want, by name
connect(secure = true, host = "db.example.com")
```

In Java the same call needs a Builder; here it's a plain method with named args and defaults.

</details>

### What should you consider when applying the Builder pattern to class hierarchies?
<details><summary>Show answer</summary>

A Builder hierarchy is sound exactly when the **underlying subclassing** is sound — Builder adds nothing to the safety
question, so it reduces to "is extending this base safe?", which the **`equals` contract** governs:

- **Abstract / non-instantiable base** → always safe. There's no instance of the base to compare against, so the
  equals trap can't arise — a builder hierarchy is unconditionally appropriate. (This is why the pattern is usually
  taught with an abstract base.)
- **Concrete / instantiable base** → safe *only if* the subclass adds **no value component that participates in
  `equals`**. Adding equality-relevant state to a concrete subtype breaks the contract (symmetry/transitivity) — but a
  subclass that refines behavior *without* touching equality is fine. `LinkedHashSet` extends the concrete `HashSet`
  safely because it only changes iteration order, which `equals` ignores; a builder hierarchy over such a pair would be
  equally safe.

So the rule isn't "concrete base → no hierarchy." It's: abstract base is always safe; a concrete base is safe when the
subclass adds no equals-participating component — the same condition that governs any concrete-class extension.
[→ equality contract and inheritance](2.1_methods_common_to_all_objects.md#equality-contract-and-inheritance)

</details>

### Builder pattern for class hierarchies
<details><summary>Show answer</summary>

Uses a **recursive type bound** `B extends Builder<B>` plus an abstract `self()` — the *simulated
self-type idiom* — so chaining returns the subclass type without casts.

```java
public abstract class Animal {
  private final String name;
  protected Animal(Builder<?> b) { this.name = b.name; }

  public abstract static class Builder<B extends Builder<B>> {
    private String name;
    public B name(String name) { this.name = name; return self(); }  // returns B, not Animal.Builder
    protected abstract B self();          // subclass returns `this` as its own type
    public abstract Animal build();
  }
}

public class Dog extends Animal {
  private final String breed;
  private Dog(Builder b) { super(b); this.breed = b.breed; }

  public static class Builder extends Animal.Builder<Builder> {
    private String breed;
    public Builder breed(String breed) { this.breed = breed; return self(); }
    @Override protected Builder self()  { return this; }
    @Override public Dog build()        { return new Dog(this); }   // covariant: returns Dog, not Animal
  }
}
```

Two named techniques make this work:
- **Simulated self-type idiom** — the recursive bound + `self()` lets inherited setters chain as the
  subclass type, no casts. (Java has no real self type; this emulates it.)
- **Covariant return typing** — `Dog.Builder.build()` is declared to return `Dog`, a subtype of the
  `Animal` declared in the base — so callers build without casting.

Cons: `self()` boilerplate in every subclass; awkward when build decisions must be polymorphic.

</details>

### How is Lombok used to build a single class?
<details><summary>Show answer</summary>

`@Builder` on the class — Lombok generates the static `builder()`, the chaining setters, and `build()`. 
No hand-written builder code.

```java
@Getter
@Builder
public class NutritionFacts {
  private final int servingSize;   // required and optional are not distinguished
  private final int calories;      // by @Builder — all become chaining setters
}
```

```java
NutritionFacts n = NutritionFacts.builder()
    .servingSize(240)
    .calories(100)
    .build();
```

Add `@Builder(toBuilder = true)` to also generate `toBuilder()` — copy an instance and change a few fields. 
Use plain `@Builder` whenever there's **no inheritance**: simpler, fewer generated classes.

</details>

### How is Lombok used to build a class hierarchy, and why not @Builder?
<details><summary>Show answer</summary>

Use **`@SuperBuilder`**, not `@Builder` — and it must be on **every class in the hierarchy**.

```java
@Getter
@SuperBuilder
public abstract class Animal {
  private final String name;
}

@Getter
@SuperBuilder
public class Dog extends Animal {
  private final String breed;
}
```

```java
Dog d = Dog.builder()
    .name("Rex")     // superclass field — works because of @SuperBuilder
    .breed("Shepherd")
    .build();
```

**Why plain `@Builder` fails here:** `@Builder` only sees the annotated class's own fields, 
so `Dog.builder()` would let you set `breed` but **not** the inherited `name`. 
`@SuperBuilder` generates builders that chain through the hierarchy, 
giving each subclass builder access to both its own and its superclass's fields — 
the Lombok equivalent of the hand-written simulated self-type idiom.

Constraints: `@SuperBuilder` works on **types only**, 
and **all** superclasses must also be annotated with it (mixing with `@Builder` won't compile).

</details>
