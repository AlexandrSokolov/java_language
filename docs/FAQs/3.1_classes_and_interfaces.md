## Classes and Interfaces - Encapsulation

### What qualities characterize a well-designed component?
<details><summary>Show answer</summary>

- **Clear responsibility (Single Responsibility Principle)** — one well-defined purpose; changes for
  only one reason.
- **Encapsulation (information hiding)** — internal data and implementation hidden, so the component
  can evolve without breaking clients.
- **High cohesion** — all internal parts relate closely to its one responsibility.
- **Low coupling** — depends on as few other components as possible; minimal external knowledge.
- **Clarity and simplicity via a minimal interface** — exposes only essential behavior; easy to
  understand, hard to misuse.
- **Extensibility** — new behavior added with minimal change to existing code.
- **Testability** — testable in isolation, minimal setup, predictable behavior.

</details>

### How do the core component design principles depend on and reinforce each other?
<details><summary>Show answer</summary>

They aren't independent — they form a dependency chain, each link enabling the next. 
The value of the chain: break a link early and everything downstream erodes.

**The chain:**

1. **Clear responsibility** — defines what the component is *and is not*. 
   The root; without it, everything below erodes.
2. **High cohesion** *follows from* responsibility — one purpose means the internal parts naturally relate.
3. **Encapsulation** is *enabled by* cohesion — parts aligned around one responsibility are easy to
   hide behind a necessary-only surface.
4. **Low coupling** is a *consequence of* encapsulation — when little is revealed, clients depend on
   stable abstractions, not fragile internals.
5. **Clarity / minimal interface** *emerges from* low coupling + encapsulation — depend on little,
   expose little, communicate intent, hard to misuse.

**Two outcomes hang off the end of the chain:**

- **Extensibility** — a component with the five above accepts refactoring and new behavior without breaking clients.
- **Testability** — well-defined, loosely coupled, encapsulated components test in isolation with predictable behavior.

**Break it early** — unclear responsibility or weak encapsulation — and every quality after the broken link collapses. 
That's the point of seeing it as a chain rather than a checklist.

</details>

### What is encapsulation in software design?
<details><summary>Show answer</summary>

Encapsulation is **the degree to which a component hides its internal state and implementation
details from other components**, exposing only what is necessary through a well-defined interface.

Put differently: **it is a measure of how effectively components are decoupled** — the more a
component hides, the less anything else can depend on its internals.

</details>

### Why is decoupling important in component design?
<details><summary>Show answer</summary>

The core property: decoupled components can be worked on **in isolation** — built, understood, tested,
changed, and optimized without reaching into anything else:

- **Faster development** — components built and evolved in isolation and in parallel.
- **Easier maintenance** — quicker to understand, debug, replace, or refactor, with minimal risk to others.
- **Safer performance tuning** — bottlenecks optimized without affecting the correctness of unrelated components.
- **Greater reuse** — loosely coupled components often stay useful beyond their original context.

</details>

### How does Java support information hiding in component design?
<details><summary>Show answer</summary>

The mechanisms work at widening scopes — from a single member out to whole package groups:

- **Access modifiers** (`private`, package-private, `protected`, `public`) — control visibility of
  individual classes, fields, methods, constructors. *Member-level.*
- **Packages** — group related classes and restrict access to package-internal details.
  *Class-group level.*
- **Java Modules** (`module-info.java`) — export only selected packages, keep the rest inaccessible
  even if `public`. *Package-group level — the widest boundary.*

And the type/extension mechanisms:

- **Interfaces** — separate specification from implementation; clients depend on behavior, not concrete classes.
- **Abstract classes** — expose shared behavior while hiding implementation that varies between subclasses.
- **Final classes/methods** — prevent extension where it would expose or create dependence on internals.

</details>

### Beyond hiding, what techniques protect a component's internal state from violation?
<details><summary>Show answer</summary>

Hiding controls *visibility*; 
these two protect the state that's already hidden from being *changed* out from under the component:

- **Immutability** — state can't change after construction, so there's nothing to violate.
- **Defensive copying** — hand out copies, not references, so callers can't reach in and mutate internal mutable objects.

</details>

### What principle should guide the visibility of a class or member?
<details><summary>Show answer</summary>

Make each class and member **as inaccessible as possible** — always use the lowest access level that
still lets the software function correctly.

Doing so minimizes coupling, strengthens encapsulation, and keeps internal details from becoming
accidental dependencies that you're then forced to support.

</details>

### What access levels are available for class members?
<details><summary>Show answer</summary>

From narrowest to widest:

- **private** — only within the declaring class.
- **package-private** (no modifier) — within the same package.
- **protected** — within the same package, **plus** subclasses (including those in other packages).
- **public** — from anywhere.

</details>

### Which access levels form a class's implementation, and which its exported API?
<details><summary>Show answer</summary>

- **`private` and package-private** → the class's **implementation** — free to change, replace, or
  remove without affecting clients.
- **`protected` and `public`** → its **exported API** — a commitment you must support going forward;
  changing it can break clients.

This is why widening access isn't free: every `protected`/`public` member is a promise.

</details>

### What visibility options exist for top-level classes and interfaces?
<details><summary>Show answer</summary>

Only two: **package-private** or **public** (top-level types can't be `private` or `protected`).

**If it can be package-private, make it package-private.** Then it's implementation, not API — you can
change, replace, or remove it in a later release without breaking clients. 
Make it `public` and you're committed to supporting it indefinitely.

</details>

### What should be done if a package-private top-level class or interface is used by only one class?
<details><summary>Show answer</summary>

Consider making it a **private static nested class** of its sole user.

This tightens encapsulation further: the implementation detail becomes local to the only class that
needs it, shrinks the package's exposed surface, makes the dependency explicit, colocates related
code, and prevents accidental use by other classes in the package.

</details>

### Private vs package-private members — what's the distinction?
<details><summary>Show answer</summary>

Both are **implementation**, not exported API. The distinction is **not "good vs bad"** — it's
**strong vs scoped encapsulation**:

- **private** = class-level encapsulation — the strongest. Clear ownership of state/behavior, minimal
  accidental dependencies.
- **package-private** = package-level encapsulation — lightweight; the detail is shared only within a
  trusted, cohesive package boundary.

**The naive reading of "as inaccessible as possible" is "default to private" — but that's too blunt.**
The right default depends on *what kind of class* it is:

- **Data classes** — private fields, accessed through accessors. Private is the natural default.
- **Behavior-bearing services** — package-private is often the *better* default. It makes
  collaborators observable and stubbable in tests **without** test-only setters or reflection, at no
  real cost to encapsulation — *as long as the package is cohesive* (closely related classes
  implementing one abstraction, with the test in the same package). The test seam is paid for by the
  package boundary, not by punching a hole in the public API.

**The testability rule that follows:** widening a `private` member to **package-private** for testing is legitimate. 
Widening **beyond** package-private (to `protected`/`public`) is not — that leaks implementation into the exported API, 
which clients then depend on forever.

**When package-private becomes a smell:** only when production classes lean heavily on each other's
internals with no clear ownership — that signals unclear responsibilities or an over-coupled package,
not a visibility problem per se.

</details>

### When can internal details leak beyond their intended boundary?
<details><summary>Show answer</summary>

When a class implements `Serializable`, its private fields effectively become part of the public API —
the serialized form encodes their names, types, and structure.

The consequence: once instances have been serialized and persisted or sent over the wire, you can no
longer freely rename or restructure those "private" fields — doing so breaks deserialization of
existing data. A private field has quietly become a forever-commitment.

</details>

### Which methods are overridable, and how are they resolved?
<details><summary>Show answer</summary>

Not all access levels participate in overriding:

- **`public` / `protected`** — genuinely overridable; polymorphic everywhere.
- **package-private (default)** — overridable **only within the same package** (detailed below).
- **`private`** — **not overridable at all.** The compiler treats it as belonging to its own class,
  not the hierarchy; a same-named subclass method is simply new and unrelated.

**How overridable methods are resolved (dynamic dispatch / vtable):**
They live in a per-class **virtual method table (vtable)** mapping each signature to its actual
implementation, and are bound **at runtime, not compile time**: when called, the JVM looks at the
object's *actual runtime class*, consults that class's vtable, and invokes the matching
implementation — so a superclass-typed reference still reaches the subclass's override. This is
*dynamic dispatch* (runtime polymorphism).

**The key distinction — being *in* the vtable vs *replacing* a superclass slot.** These are two
different things, and the package-private cases differ precisely here:

- **`public` / `protected`** → always in the vtable; a subclass override always **replaces** the
  superclass's slot. Dispatch through a supertype reference reaches the subclass version.

- **package-private** → also in the vtable, **but a subclass replaces the superclass's slot only
  within the same package.**
  - *Same package* → real override: the subclass method takes over the superclass's slot,
    `@Override` compiles, dynamic dispatch reaches the subclass version.
  - *Different package* → no override. The superclass's method isn't even visible, so the
    subclass's same-signature method can't replace anything — it becomes a **separate, independent
    vtable entry**. Both methods exist, unlinked; a superclass-typed call still hits the superclass
    version, and `@Override` is a compile error (nothing to override). So "it's in the vtable" and
    "no overriding" are both true — they just describe *different slots*.

- **`private`** → **never in the vtable**; resolved statically at compile time — which is exactly why
  it can't be overridden.

</details>

### What accessibility rule applies to an overriding method, and why?
<details><summary>Show answer</summary>

An overriding method **must not be more restrictive** than the method it overrides — equal or wider only:

- overridden `public` → overriding must be `public`
- overridden `protected` → overriding may be `protected` or `public`
- overridden package-private → overriding may be package-private, `protected`, or `public`
  — but only **within the same package**, since that's the only place a package-private method can be
  overridden at all. (In a different package there's no override to widen — the same-signature method
  is a separate, unrelated method, and `@Override` is a compile error.)

**Why:** the Liskov Substitution Principle — a subclass instance must be usable anywhere the
superclass is expected. If an override narrowed access, a caller holding the supertype could no longer
reach the method, breaking substitutability. The compiler enforces this and rejects the narrowing.

**Interface special case:** interface methods are implicitly `public`, so an implementing class must
declare them `public` — anything narrower is a compile error.

</details>

### What visibility rules apply to instance fields in Java?
<details><summary>Show answer</summary>

Two questions decide it, in order:

**1. Is the class visible to clients at all?**
- **No (package-private or private nested class)** → exposing fields is fine when they plainly
  represent the abstraction. Saves accessor clutter, and any dependency on the representation stays
  confined to the package — or to the enclosing class, for private nested types.


**2. If yes — is the field structure a deliberate part of the contract?**
- **No (normal public class)** → fields should **not** be public. 
  Use accessors (getters), plus mutators for mutable classes. Because:
  - public *mutable* fields stop the class enforcing invariants or reacting to changes, and make it
    generally unsafe for concurrent use;
  - even public *final* fields (immutable referent) become part of the external contract, locking
    in the internal representation.
- **Yes (DTO in a public API)** → `public final` fields are acceptable, even preferable. 
  A DTO's shape *is* the contract — it's what gets serialized across the wire — so the representation is already pinned. 
  Wrapping it in private fields + accessors adds boilerplate while buying nothing:
  the "freedom to change internals later" that accessors normally protect doesn't exist here. 
  Public fields also signal honestly: *this is data, not behavior.*

</details>

### How do public fields weaken or break encapsulation?
<details><summary>Show answer</summary>

A public field exposes the representation directly, so the field *is* the API — the type and the contract
are fused. An accessor, by contrast, is an **abstraction boundary**: it lets the representation stay an
implementation detail you can still change. A public field gives that boundary away permanently, forfeiting:

**1. Invariant enforcement** — there's no interception point on assignment.
```java
point.x =-999999;                  // public field — nobody can stop this

public void setX(int x) {           // accessor — you can enforce
  if (x < 0) throw new IllegalArgumentException("x must be non-negative");
  this.x = x;
}
```

**2. Thread-safety added later** — callers assign fields directly, so you can't make access safe without
breaking them.
```java
point.x = 10;
point.y = 20;                       // another thread can read between these → torn state

// with a setter you could synchronize, or hold an AtomicInteger internally —
// the caller never knows and never changes their code
```

**3. Side effects added later** — logging, change notifications, lazy init: impossible to add to a field
without an API break, trivial inside an accessor.
```java
public void setX(int x) {
  this.x = x;
  listeners.forEach(l -> l.onXChanged(x));   // added later, no API break
}
```

**4. Freedom to change the representation** — the field's *type* is part of the contract.
```java
// public field: caller wrote  point.x + 5  → changing int → BigDecimal breaks every caller
// behind an accessor: the type is an internal detail; only getX()'s signature is affected,
// not every arithmetic use site
```
The setter is the boundary; the field type is an implementation detail. 
With a public field the type *is* the API — change one and you change the other.

**5. Polymorphism** — fields don't participate in overriding; a subclass field only *shadows*, never overrides.
```java
class ScaledPoint extends Point {
  public int x;                   // SHADOWS Point.x, does not override — classic bug source
}
```
See [overriding & dispatch](#which-methods-are-overridable-and-how-are-they-resolved).

**The through-line:** every loss is the same loss — the representation has leaked to callers, so you can no
longer improve, constrain, or evolve that part of the implementation. (The immutability-specific case is
covered in [how public fields break immutability](3.2_classes_and_interfaces.md#how-do-public-fields-weaken-or-break-immutability).)

</details>

### Which JDK classes expose their state through public fields?
<details><summary>Show answer</summary>

`java.awt.Point`, `java.awt.Dimension`, and `java.awt.Rectangle` — they expose coordinates and sizes
directly as public fields.

These are historical exceptions, not models to emulate: they predate (or ignored) the guideline that
public classes should encapsulate their representation, and the public fields are now frozen into their APIs.

</details>

### What visibility rules apply to static fields in Java?
<details><summary>Show answer</summary>

The same rule as instance fields — don't expose them — **with one exception: constants.**

`public static final` fields are acceptable when the constant is an integral part of the class's abstraction. 
By convention they're named in `UPPER_SNAKE_CASE`.

**The critical caveat:** the field must hold a **primitive or a reference to an immutable object**. 
A `public static final` reference to a *mutable* object has the same problems as a non-final field — 
`final` freezes the reference, not the object, so callers can still mutate its contents, 
with serious unintended consequences.

</details>

### Should a `static final Logger` be named `LOGGER` (constant) or `logger`?
<details><summary>Show answer</summary>

There's no settled answer — it's a real debate, because the field sits exactly on the fault line of the constant rule.

**The tension:** the rule says `public static final` fields holding *primitives or immutable objects* 
are constants → `UPPER_SNAKE_CASE`. A logger is `static final`, but is it a *constant*? Two defensible readings:

- **`LOGGER` (constant camp):** it's `static final`, so by the literal naming convention it's a constant 
  and should be uppercase. Static-analysis tools (SonarQube S1312, PMD) historically *flag lowercase as a violation*.
- **`logger` (modern camp):** a logger is a behavioral object, not a value — 
  it does work, it isn't a `MAX_VALUE`-style constant. 
  The reference is final, but the object is not conceptually immutable in the "it's just a value" sense. 
  SLF4J/Logback's own examples use lowercase `logger`, and Google's style guidance treats it as a non-constant.

**Why it's unresolvable:** there's *no functional or performance difference* — `final` gives the optimization; 
casing is purely stylistic. And the underlying classification is genuinely ambiguous:
is a `Logger` (or a `Joiner`, a `Gson`) a "constant" because it's an immutable-ish final reference, or
a "service object" because it has behavior? Reasonable people draw that line differently.

**Practical stance:** pick one and be consistent within a team/codebase; 
let the static-analysis config enforce whichever you chose. 
Lowercase `logger` is the more common modern convention; 
uppercase `LOGGER` is equally accepted and is what tools default to flagging for. Neither is wrong.

</details>

### What should be considered about immutability when using arrays?
<details><summary>Show answer</summary>

A non-empty array is **always mutable** — and `final` freezes the *reference*, not the contents. 
So a class must never expose an array via a `public static final` field or by returning it directly from an accessor:

```java
public static final Thing[] VALUES = { ... };   // potential security hole
SomeClass.VALUES[0] = new Thing();              // any client can mutate internal state
```

The same hole opens when an accessor returns a reference to a private array — 
and some IDEs generate exactly such accessors automatically.

**Two fixes:**

1. Private array + immutable `List` view:
```java
   private static final Thing[] PRIVATE_VALUES = { ... };
   public static final List<Thing> VALUES =
       Collections.unmodifiableList(Arrays.asList(PRIVATE_VALUES));
```
2. Accessor returning a defensive copy:
```java
   private static final Thing[] PRIVATE_VALUES = { ... };
   public static Thing[] values() { return PRIVATE_VALUES.clone(); }
```

**Choosing:** immutable `List` when clients will iterate/stream/treat it as a collection; 
defensive copy when array semantics are needed, or performance is critical and the copy cost is acceptable.

</details>

### What is a Java module?
<details><summary>Show answer</summary>

A module is a grouping of packages, like a package is a grouping of classes.

</details>

### What problems do Java modules address?
<details><summary>Show answer</summary>

Two problems, both at large-system scale:

- **Encapsulation stronger than `public`.** Before modules, every `public`/`protected` type in a
  package was reachable by the whole application — you couldn't share a class *between your own
  packages* without exposing it globally. A module **exports only selected packages** and keeps the
  rest inaccessible from outside *even if their members are `public`*. Inside the module, accessibility
  is unchanged; across the boundary, unexported packages simply don't exist for clients.
- **Explicit, checked dependencies.** Dependencies are declared in `module-info.java`, so missing or
  conflicting ones are caught at compile time and startup — not at runtime.

</details>

### Why were Java modules really introduced?
<details><summary>Show answer</summary>

The driving motivation was to **strongly encapsulate the JDK itself**. Many internal JDK APIs had
historically been accessible and widely — but unofficially — used, which created long-term maintenance
and compatibility problems (every internal class people depended on became a de-facto frozen API).

With modules, unexported JDK packages are *truly* inaccessible, enforcing a clean separation between
supported APIs and internal implementation. This is where modules deliver their clearest value.

</details>

### Why is the practical benefit of Java modules limited outside the JDK?
<details><summary>Show answer</summary>

Module encapsulation only holds **if the code is actually loaded as a module** — and that's the catch:

- Whether a JAR is a real module or legacy code depends on **how it's loaded**. A modular JAR placed
  on the **classpath** (not the module path) behaves exactly as it did pre-Java-9 — the boundaries
  evaporate.
- So strong encapsulation is only enforced when **all** components run as modules; mix in classpath
  code and the guarantee weakens or disappears.

And the adoption cost is real: packages must be grouped into modules, every dependency fully declared,
source trees often reorganized, and interop with non-modular code needs special handling.

**Net:** modules solve real encapsulation/dependency problems at large scale — decisively for the JDK
— but for ordinary application code the benefits require deliberate, all-in adoption, and outside that
the cost usually outweighs the payoff.

</details>
