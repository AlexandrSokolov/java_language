### What factors distinguish a well-designed component from a poorly designed one?
<details><summary>Show answer</summary>

- **Clear responsibility (Single Responsibility Principle)**  
  The component has one well-defined purpose and changes for only one reason.

- **Encapsulation (information hiding)**  
  Internal data and implementation details are hidden, so the component can evolve without breaking its clients.

- **High cohesion**  
  All internal parts of the component are closely related to its responsibility.

- **Low coupling**  
  The component depends on as few other components as possible and minimizes external knowledge.

- **Minimal and clear interface**  
  Only necessary behavior is exposed, and the API is easy to understand and hard to misuse.

- **Testability**  
  The component can be tested in isolation with minimal setup and predictable behavior.

- **Readability and simplicity**  
  The code expresses intent clearly and avoids unnecessary complexity.

- **Extensibility**  
  New behavior can be added with minimal changes to existing code.



</details>

### How do the core component design principles depend on and reinforce each other?
<details><summary>Show answer</summary>

These design principles are not independent; they form a dependency chain where each one enables the next.

A **clear responsibility** defines what a component is responsible for and, just as importantly, what it is not. 
Without this clarity, boundaries are vague and design decisions become arbitrary.

Once responsibility is clear, the component can establish a **meaningful boundary**, 
which makes **encapsulation** possible. 
Encapsulation relies on knowing which data and decisions belong inside the component and which must be exposed.

Strong encapsulation naturally reduces **coupling**, 
because other components no longer depend on internal details but only on defined behavior. 
This allows the component to change internally without affecting its clients.

Low coupling and clear boundaries make it easier to design a **small, stable interface**, 
since only behavior that truly belongs to the component is exposed.

With explicit dependencies and a focused interface, the component becomes easier to **test in isolation**, 
because external concerns are minimized and behavior is more predictable.

As a result, the component can **evolve safely over time**, 
allowing internal refactoring or extension without breaking existing usage.

Breaking this chain early — for example, by unclear responsibility or weak encapsulation — 
undermines all subsequent qualities and leads to brittle design.

</details>

### What is encapsulation in software design?
<details><summary>Show answer</summary>

Encapsulation is **the degree to which a component hides its internal state and implementation 
details from other components**, exposing only what is necessary through a well-defined interface.


**Encapsulation is a measure of how effectively components are decoupled.**

</details>

### Why is decoupling important in component design?
<details><summary>Show answer</summary>

- Components can be developed, tested, optimized, used, understood, and modified in isolation.
- Development is faster because components can be built and evolved in parallel.
- Maintenance is easier since components are quicker to 
  understand, debug, replace, or refactor with minimal risk to others.
- Performance tuning becomes safer and more effective, 
  as identified bottlenecks can be optimized without affecting the correctness of unrelated components.
- Software reuse is increased because 
  loosely coupled components often remain useful in contexts beyond their original purpose.

</details>

### How does Java support information hiding in component design?
<details><summary>Show answer</summary>

- **Access modifiers (`private`, `protected`, package-private, `public`)**  
  Control visibility of classes, fields, methods, and constructors, allowing internal details to be hidden from clients.

- **Packages**  
  Enable grouping related classes and restricting access to package-internal implementation details.

- **Java Modules (`module-info.java`)**  
  Enforce strong encapsulation at the platform level by explicitly controlling 
  which packages are exported and which remain completely hidden from other modules.

- **Interfaces**  
  Separate specification from implementation, allowing clients to depend on behavior rather than concrete classes.

- **Abstract classes**  
  Expose shared behavior while hiding implementation details that may vary between subclasses.

- **Immutability**  
  Prevents external observation or modification of internal state changes.

- **Defensive copying**  
  Protects internal mutable objects by returning copies instead of references.

- **Final classes and methods**  
  Prevent extension where it could expose or depend on internal implementation details.

</details>

### What principle should guide how visible a class or member is to other parts of the system?
<details><summary>Show answer</summary>

The general rule of thumb is to make each class or class member as inaccessible as possible.

In practice, this means always using the lowest access level that still allows the software to function correctly. 
Doing so minimizes coupling, strengthens encapsulation, 
and reduces the risk that internal details become accidental dependencies.

</details>

### What access levels are available for members such as fields, methods, and nested types?
<details><summary>Show answer</summary>

- **private** – accessible only within the declaring class
- **package-private** (no modifier) – accessible within the same package
- **protected** – accessible within the same package and by subclasses
- **public** – accessible from anywhere

</details>

### Which access levels are considered part of a class’s implementation, and which are part of its exported API?
<details><summary>Show answer</summary>

- private and package-private members are part of a class’s implementation 
- protected and public members form its exported API.

</details>

### What visibility options are available for top-level(non-nested) classes and interfaces?
<details><summary>Show answer</summary>

For top-level (non-nested) classes and interfaces, there are only two possible access levels: 
package-private and public.


**If a top-level class or interface can be made package-private, it should be**.

- By making it package-private, you make it part of the implementation rather than the exported API, 
  and you can modify it, replace it, or eliminate it in a subsequent release without fear of harming existing clients. 
- If you make it public, you are obligated to support it forever to maintain compatibility.

</details>

### What should be done if a package‑private top‑level class or interface is used by only one class?
<details><summary>Show answer</summary>

If a package-private top-level class or interface is used by only one class, 
consider making it a **private static nested class** of that class.


This increases encapsulation by keeping the implementation detail local to its sole user, 
reduces the exposed surface of the package, and makes the dependency explicit. 
It also improves readability by colocating closely related code and 
prevents accidental use by other classes in the package.

</details>

### How do private and package‑private members differ in their role within a class design?
<details><summary>Show answer</summary>

Private and package-private members both belong to a class’s implementation and are not part of its exported API. 
Private vs package‑private is **not "good vs bad"** — it’s **"strong vs scoped encapsulation"**.

- private = class-level encapsulation
- package‑private = module-level encapsulation (lightweight)


Using **private** members provides the strongest form of encapsulation at the class level and is the natural default. 
It clearly establishes ownership of state and behavior and minimizes accidental dependencies.


Using **package-private** members is legitimate tool for encapsulating implementation details within a trusted boundary,
as long as its use is intentional and the package itself remains cohesive, such as:
- closely related classes collaborating to implement a single abstraction
- internal framework or subsystem code
- test code placed in the same package to enable effective testing without test-only setters or reflection

In these contexts, package-private access preserves encapsulation at the package boundary 
while avoiding unnecessary indirection or artificial APIs. 
**It often results in simpler, cleaner production code and more maintainable tests**.


Frequent use of package-private access becomes a concern 
only when production classes depend heavily on each other’s internals without clear ownership or boundaries. 
In that case, it may signal unclear responsibilities or an overly coupled package design.

</details>

### What are the accessibility requirements for overridden methods in Java? Why?
<details><summary>Show answer</summary>

When a method overrides a method from a superclass, 
it must not have a more restrictive access level than the overridden method.


This rule ensures that instances of the subclass can be used anywhere an instance of the superclass is expected, 
in accordance with the Liskov Substitution Principle. If this constraint is violated, the compiler reports an error.


A special case of this rule applies to interfaces: 
when a class implements an interface, 
all methods declared in the interface must be implemented as `public` in the implementing class.

</details>

### What visibility considerations arise when designing code for testability?
<details><summary>Show answer</summary>

To facilitate testing, it may be tempting to make a class, interface, or member more accessible than otherwise necessary. 
This is acceptable up to a point.


Widening a private member of a public class to package-private solely for testing purposes can be reasonable. 
However, increasing accessibility beyond package-private is not acceptable, 
as it exposes implementation details that should remain hidden from clients.

</details>

### What visibility rules apply to instance fields in Java?
<details><summary>Show answer</summary>

#### Public Classes

Instance fields of public classes should rarely be public, as doing so weakens encapsulation.

- Public mutable fields prevent a class from enforcing invariants or reacting to state changes, 
  and they generally make the class unsafe for concurrent use.
- Even public final fields referencing immutable objects become part of the class’s external contract, 
 reducing the ability to change internal representation later.

Use accessor methods (getters) and, for mutable classes, mutators (setters) for such classes.

#### DTOs used in public APIs

An important exception is **DTOs used in public APIs**. 
DTOs are simple data carriers whose structure is intentionally part of the API contract.
Changing the internal representation of DTOs later breaks the API contract and is therefore not expected.
In such cases, using public final instance fields is acceptable and can be preferable to accessors, 
clearly signaling that the class represents data, not behavior.

#### Non-public classes

**For package‑private or private nested classes, exposing fields is acceptable** 
if they clearly represent the abstraction. 
This reduces clutter compared to accessors, and any dependency on the internal representation remains safely limited 
to the package — or to the enclosing class for private nested types.

</details>

### Which Java standard library classes expose public fields despite the guideline against it?
<details><summary>Show answer</summary>

Notable examples include `java.awt.Point`, `java.awt.Dimension`, and `java.awt.Rectangle`. 
These public classes expose their state through public fields (such as coordinates and sizes), 
despite the general guideline that public classes should encapsulate their internal representation. 
These APIs are best regarded as historical exceptions rather than practices to emulate.

</details>

### What visibility rules apply to static fields in Java?
<details><summary>Show answer</summary>

The same guidance applies to static fields, with one important exception.


It is acceptable to expose constants via `public static final` fields, 
provided that those constants are an integral part of the abstraction offered by the class. 
By convention, such fields are named using capital letters with words separated by underscores.


It is essential that these fields hold either primitive values or references to immutable objects. 
A `public static final` field that refers to a mutable object 
suffers from many of the same problems as a non-final field: 
although the reference itself cannot be changed, the state of the referenced object can be modified, 
potentially leading to serious and unintended consequences.

</details>

### What should be considered about immutability when using arrays?
<details><summary>Show answer</summary>

A non-empty array is always mutable in Java. 
For this reason, a class must never expose a mutable array through a `public static final` field 
or by returning it directly from an accessor.

Declaring the array reference as `final` does **not** make the array immutable. 
Clients can still modify the array’s contents, potentially breaking invariants or introducing security issues.

```java
// Potential security hole!
public static final Thing[] VALUES = { ... };
```
Clients can modify the array contents:
```java
SomeClass.VALUES[0] = new Thing(); // Modifies internal state
```
The same issue occurs if an accessor returns a reference to a private array. 
Some IDEs generate such accessors automatically, unintentionally exposing mutable state.

Solutions:
1. Make the array private and expose an immutable List view:
    ```java
    private static final Thing[] PRIVATE_VALUES = { ... };
    public static final List<Thing> VALUES = Collections.unmodifiableList(Arrays.asList(PRIVATE_VALUES));
    ```
2. Return a defensive copy
    ```java
    private static final Thing[] PRIVATE_VALUES = { ... };
    
    public static Thing[] values() {
      return PRIVATE_VALUES.clone();
    }
    ```

Choosing between the two approaches
- Use an immutable List when clients are likely to iterate, stream, or treat the result as a collection.
- Use a defensive copy when array semantics are required or performance is critical and copying cost is acceptable.

</details>

### What problems do Java modules address?
<details><summary>Show answer</summary>

Java modules primarily address problems related to **encapsulation, 
reliable configuration, and dependency management** in large Java systems.


Before modules, Java relied on packages and access modifiers for encapsulation. 
However, all public and protected types in a package were accessible to the entire application, 
making it difficult to share classes internally without exposing them globally. 
The Java module system allows a module to **explicitly export only selected packages**, 
while keeping others completely inaccessible outside the module. 
This enables stronger encapsulation by allowing internal APIs to remain hidden 
even if their members are declared `public`.


Within a module, accessibility rules remain unchanged, 
but outside the module, public and protected members of unexported packages are not accessible. 
This makes it possible to share classes among packages 
inside a module without committing them to the module’s public API.


Modules also improve **dependency clarity and reliability**. 
By requiring dependencies to be declared explicitly in `module-info.java`, 
the module system helps detect missing or conflicting dependencies at compile time and startup, 
rather than at runtime.


A major motivation for introducing modules was the need to **strongly encapsulate the JDK itself**. 
Many internal JDK APIs had historically been accessible and widely (but unofficially) used, 
creating long-term maintenance and compatibility problems. 
With modules, unexported JDK packages are truly inaccessible, enforcing a clean separation 
between supported APIs and internal implementation details.


Outside the JDK, module-based access control is more limited in practice.
Java modules only provide **strong encapsulation if they are actually used as modules at runtime**.
Whether a JAR is treated as a module or as ordinary legacy code depends on **how it is loaded**.
If a modular JAR is placed on the classpath rather than the module path, 
its packages behave as they did before Java 9.
As a result, module encapsulation is not reliably enforced for most applications 
unless all components are loaded as modules, in which case the JVM can strictly enforce module boundaries.


Using modules also introduces additional complexity: 
packages must be grouped into modules, dependencies must be fully declared, source trees may need reorganization, 
and special handling may be required when interacting with non-modular code.


In summary, Java modules solve real problems of encapsulation and dependency management at large scale — 
most critically for the JDK itself — but their practical benefits outside of that context 
depend on deliberate and consistent adoption.

</details>

### What are the benefits of using immutable classes?
<details><summary>Show answer</summary>

Benefits of immutable classes:

- Simpler reasoning: an immutable object has a single, fixed state established at construction.
- Strong invariants: once validated by the constructor, invariants remain true forever.
- Inherent thread safety: immutable objects require no synchronization.
- Safe sharing: instances can be freely shared without risk of corruption.
- No defensive copies: immutable objects can be reused directly without cloning.
- Internal sharing: immutable objects can safely share internal data for efficiency.
- Reliable composition: immutable components simplify maintaining invariants in complex objects.
- Ideal for collections: safe to use as map keys or set elements.
- Failure atomicity: operations cannot leave objects in an inconsistent state.

</details>

### How can a class be designed to minimize mutability?
<details><summary>Show answer</summary>

1. Do not provide methods that modify the object’s state (no mutators).
2. Prevent subclassing to preserve immutability guarantees by
    - making the class `final`, or
    - making all constructors private (or package‑private) and providing public static factory methods instead, 
      which allows controlled instance creation without permitting subclassing.
3. Make all fields `final` to clearly enforce immutability and ensure safe publication.
4. Make all fields `private` to avoid exposing internal state or future representation lock‑in.
    - **Minimizing mutability** is about preventing state changes.
    - **Making fields private** is about preventing uncontrolled state changes.
    - Without private fields, immutability cannot be enforced, only hoped for.
5. Do not expose mutable components; use defensive copies in constructors and accessors.

</details>

### How can accessor methods be designed to preserve immutability?
<details><summary>Show answer</summary>

The **functional approach** can be used to provide accessors while minimizing mutability.

In this approach, operations do not modify the object’s state. 
Instead, they create and return new instances that represent the result:
```java
public Complex plus(Complex c) {
  return new Complex(re + c.re, im + c.im);
}
```

Each arithmetic operation returns a new `Complex` instance rather than mutating the existing one. 
This contrasts with the **procedural (imperative) approach**, where methods modify the object’s internal state. 
The **functional approach preserves immutability** while still allowing expressive, safe access to object behavior.

</details>

### What pattern is used in Java records to return modified versions of an instance?
<details><summary>Show answer</summary>

Java records commonly use the *withers* pattern to return modified versions of an instance.

Withers are methods that create and return a new record instance with one component changed, 
while the remaining components are copied from the original instance:
```java
public record User(String name, Integer age) {

  User withName(String name) {
    return new User(name, this.age);
  }

  User withAge(Integer age) {
    return new User(this.name, age);
  }
}
```

</details>

### What should be considered when using withers in Java records?
<details><summary>Show answer</summary>

Withers in Java records can be overused when they are added mechanically for every component 
without modeling a meaningful transformation.

Overuse typically occurs when:
- The record is small and short‑lived, where direct construction is clearer.
- Withers merely duplicate the canonical constructor without improving readability.
- Client code chains many withers, effectively simulating mutability:
  ```java
  user.withName(n).withAge(a).withRole(r);
  ```
  That may indicate that the record is too large and a builder might fit better.
- The record grows large, signalling that a builder or a different abstraction may be more appropriate.

Recommended guidelines:
- Prefer direct construction for simple, local transformations.
- Add withers only for common or semantically meaningful modifications.
- Avoid using withers to mimic setters or incremental state mutation.
- Reconsider the design if frequent chained updates are required.

Rules:
- **Used thoughtfully, withers improve clarity;** 
- used excessively, they obscure intent and complicate value‑oriented design.

</details>

### What are the disadvantages of immutable classes?
<details><summary>Show answer</summary>

- **Performance and allocation overhead** – immutable objects require creating new instances for every modification, 
  which can lead to increased memory usage, copying costs (especially for large object graphs). 
  This also makes incremental updates less convenient, as even small changes may require reconstructing entire objects.
- **Not always a natural fit** – some domains (e.g., entities with identity or lifecycle-based state changes) are more naturally modeled as mutable.
- **API verbosity** – code may become more verbose when many “modified copies” are needed without builders or withers.

**Key takeaway**

Immutable classes simplify correctness and concurrency, 
but they trade off **performance, allocation cost, and ergonomics** in certain use cases. 
Choosing immutability should be a deliberate design decision based on how objects are created, used, and evolved.

</details>

### What techniques can be used to reduce allocation overhead in immutable classes?
<details><summary>Show answer</summary>

One way to address allocation overhead in immutable classes is to provide a **public mutable companion class**.

The idea is to separate mutation from representation:
- The mutable companion is used to efficiently build or modify state without creating many intermediate objects.
- Once the final state is ready, it is converted into an immutable instance, which is safe to share and reuse.

Typically, this is implemented by:
- Exposing mutating operations on the companion class.
- Providing a `build()` or similar method that creates the immutable object.
- Ensuring the immutable class copies the state and never changes it afterward.

```java
// Immutable class
final class Point {
  private final int x;
  private final int y;

  Point(PointBuilder builder) {
    this.x = builder.x;
    this.y = builder.y;
  }
}

// Mutable companion
class PointBuilder {
  int x;
  int y;

  PointBuilder setX(int x) {
    this.x = x;
    return this;
  }

  PointBuilder setY(int y) {
    this.y = y;
    return this;
  }

  Point build() {
    return new Point(this);
  }
}
```

A well-known example from the Java standard library is String and its mutable companion `StringBuilder`.
StringBuilder allows efficient in-place modification of characters and produces an immutable String via toString(), 
avoiding repeated allocations.

</details>

### What techniques can be used in Java records to reduce allocation overhead caused by immutability?
<details><summary>Show answer</summary>

Java records are immutable by design, which provides strong safety guarantees. 
However, immutability can lead to performance issues when multiple fields need to be updated, 
because each update creates a new instance.

Goals:
- Preserve immutability and thread-safety
- Support convenient single-field updates via withers

  Withers are methods that return a new record instance with one field changed.
- Avoid excessive allocations for batch updates via builder

  Mutable Companion Builder for Batch Updates. Mutation happens only inside the builder; the record remains immutable.

```java
public record User(String name, int age, String email) {

  public Builder toBuilder() {
    return new Builder(this);
  }

  public User withAge(int age) {
    return new User(this.name, age, this.email);
  }

  static final class Builder {
    private String name;
    private int age;
    private String email;

    private Builder(User user) {
      this.name = user.name;
      this.age = user.age;
      this.email = user.email;
    }

    public Builder name(String name) {
      this.name = name;
      return this;
    }

    public Builder age(int age) {
      this.age = age;
      return this;
    }

    public Builder email(String email) {
      this.email = email;
      return this;
    }

    public User build() {
      return new User(name, age, email);
    }
  }
}
```
Usage:
```java
//single update:

//batch updated:
User updated = user.toBuilder()
        .name("Alice")
        .age(30)
        .email("a@example.com")
        .build();
```

### What options exist to prevent subclassing in immutable classes, and how do they compare?
<details><summary>Show answer</summary>

To preserve immutability, a class must prevent subclassing. 
There are two main design options, each with different trade‑offs.

#### Option 1: Declare the class `final`

This directly forbids subclassing.

- **Pros:** Simple, explicit, and easy to reason about
- **Cons:** Least flexible; prevents inheritance and limits future evolution

#### Option 2: Use non‑public constructors with public static factory methods

All constructors are `private` or package‑private, and instances are created via static factories.  
To clients outside the package, the class is effectively final.

- **Pros:** More flexible; allows multiple internal implementations and future performance optimizations 
  (e.g., instance caching)
- **Cons:** Slightly more complex; immutability is less obvious than with `final`

#### Comparison

Both approaches preserve immutability by preventing subclassing.  
Making the class `final` favors **simplicity**, while static factories favor **flexibility and long‑term evolvability**, 
making them a common choice for library design.

</details>

### What are the consequences if a class with all final fields does not prevent subclassing?
<details><summary>Show answer</summary>

- `final` fields alone do not guarantee immutability. 
- A class with all `final` fields is **not truly immutable** if it allows subclassing.  

If subclassing is permitted:
- A subclass can add mutable state
- A subclass can override methods to return different values over time
- Code that relies on immutability may break or become insecure

Immutability is a behavioral guarantee, not just a property of field modifiers.


`BigInteger` and `BigDecimal` are intended to be immutable, but they are **not final** and allow method overriding. 
This is a historical design issue that cannot be fixed without breaking backward compatibility.

As a result, code that depends on their immutability and accepts instances from untrusted clients may be vulnerable: 
a malicious subclass could behave mutably.


If you write a class whose security depends on the immutability of such classes you must **defensively copy**:
```java
BigInteger safe = new BigInteger(untrusted.toByteArray());
```
This guarantees that you are working with a genuine, immutable instance rather than an untrusted subclass.


**When designing your own immutable classes**:
- Prevent subclassing by making the class final, or
- Use non‑public constructors with public static factory methods


**Key Takeaway**:
- final fields are **necessary but not sufficient**
- Immutable classes must be **effectively final**
- If subclassing cannot be prevented, **defensive copying is required**

</details>

### Can an immutable class legally have mutable state?
<details><summary>Show answer</summary>

Yes, an immutable class may contain **non‑final or internally mutable fields** without violating immutability, 
as long as **no externally visible state changes** occur. 
The true definition of immutability is behavioral: observers must never see the object change.


For performance reasons, immutable classes may use **internal caching** of expensive computations. 
A non‑final private field can lazily store a computed value and return it on subsequent calls. 
Because the object’s logical state never changes, the cached result is guaranteed to remain correct:
```java
public final class ExpensiveValue {
  private final int input;
  private volatile Integer cachedResult; // non-final, internal cache

  public ExpensiveValue(int input) {
    this.input = input;
  }

  //double‑checked locking
  public int compute() {
    Integer result = cachedResult;
    if (result == null) {
      synchronized (this) {
        result = cachedResult;
        if (result == null) {
          result = expensiveComputation(input);
          cachedResult = result;
        }
      }
    }
    return result;
  }

  private int expensiveComputation(int value) {
    // pretend this is very expensive
    return value * value;
  }
}

```

**Key point:** Immutability is about what clients can observe, not how the class is implemented internally. 
Internal mutability is acceptable if it is completely hidden and does not affect the object
