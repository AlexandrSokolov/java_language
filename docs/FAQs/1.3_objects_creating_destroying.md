### What is a singleton?
<details><summary>Show answer</summary>

A singleton is simply a class that is instantiated exactly once.


Singletons typically represent:
- either a stateless object such as a function object
- a system component that is intrinsically unique

</details>

### What's the drawback of singletons?
<details><summary>Show answer</summary>

Making a class a singleton can make it difficult to test its clients 
because it‚Äôs impossible to substitute a mock implementation for a singleton
unless it implements an interface that serves as its type.

</details>

### How can a singleton be implemented?
<details><summary>Show answer</summary>

Group by **who enforces single-ness, and when** ‚Äî three handles, not four loose options:

- **Language / JVM enforces it** ‚Äî you can't accidentally break it
  - [Eager initialization](#eager-singleton) ‚Äî instance created at class-init time
  - [Enum singleton](#enum-singleton) ‚Äî the JVM's own instance control; also serialization-safe
- **You hand-code it** ‚Äî you own the timing *and* the thread-safety it forces
  - [Lazy initialization](#recommended-lazy-singleton) ‚Äî created on first use
- **A framework owns it** ‚Äî the container, not the class, guarantees one instance
  - [Spring / DI singleton](#java-singleton-vs-spring-singleton) ‚Äî one bean per context

</details>

### Eager singleton?
<details><summary>Show answer</summary>

Simplest and thread-safe ‚Äî the instance is a `final` static field:

```java
public final class EagerSingleton {
  private static final EagerSingleton INSTANCE = new EagerSingleton();

  private EagerSingleton() { /* init */ }

  public static EagerSingleton getInstance() {
    return INSTANCE;
  }
}
```

The private constructor runs once, at class initialization. With no public/protected constructor,
exactly one instance exists ‚Äî nothing a client does can change that.

**One caveat:** a privileged client can still call the private constructor *reflectively*
(`AccessibleObject.setAccessible`). To defend, have the constructor throw if asked to create a second instance.

</details>

### Recommended lazy singleton?
<details><summary>Show answer</summary>

The **Initialization-on-Demand Holder (IoDH)** idiom:

```java
public final class HolderSingleton {
  private HolderSingleton() { /* init */ }

  private static class Holder {
    static final HolderSingleton INSTANCE = new HolderSingleton();
  }

  public static HolderSingleton getInstance() {
    return Holder.INSTANCE;   // Holder loads only now, on first call
  }
}
```

Lazy, thread-safe, **no synchronization overhead** ‚Äî and the reason is the key insight:
the JVM initializes a class lazily and *exactly once*, with thread-safety handled by the
classloader. `Holder` isn't loaded until `getInstance` first references it, and the JVM's
class-init locking guarantees a single, safely-published instance. No `synchronized`, no
`volatile`, no double-checked locking needed.

**The recommended general-purpose lazy singleton.**

</details>

### Synchronized-accessor lazy singleton?
<details><summary>Show answer</summary>

```java
public final class LazySingleton {
  private static LazySingleton instance;

  private LazySingleton() { /* init */ }

  public static synchronized LazySingleton getInstance() {
    if (instance == null) {
      instance = new LazySingleton();
    }
    return instance;
  }
}
```

Correct and thread-safe, but **the lock is only needed once** (at creation) yet you pay it on
*every* call forever ‚Äî a bottleneck on hot paths. 
[DCL](#dcl-singleton) and [IoDH](#recommended-lazy-singleton) exist precisely to drop that per-call cost.

</details>

### DCL singleton?
<details><summary>Show answer</summary>

Double-checked locking ‚Äî lazy, with synchronization *only* on first init:

```java
public final class DclSingleton {
  private static volatile DclSingleton instance;   // volatile is mandatory ‚Äî see below

  private DclSingleton() { /* init */ }

  public static DclSingleton getInstance() {
    DclSingleton local = instance;       // read volatile once, then use local (faster)
    if (local == null) {
      synchronized (DclSingleton.class) {
        local = instance;
        if (local == null) {
          local = new DclSingleton();
          instance = local;
        }
      }
    }
    return local;
  }
}
```

**Why `volatile` is non-negotiable:** without it, the write `instance = local` can be *reordered*
so another thread sees a non-null reference to a **partially-constructed** object. `volatile`
forbids that reordering and guarantees safe publication. This is the classic broken-DCL bug.

Cons: hard to read and easy to get wrong (forget `volatile` ‚Üí subtly broken). For a lazy singleton,
prefer the [holder idiom](#recommended-lazy-singleton) ‚Äî same laziness, no DCL pitfalls.

</details>

### Enum singleton
<details><summary>Show answer</summary>

The most robust single-instance approach ‚Äî the only one **automatically safe against both
serialization and reflection** (the others need `readResolve` and still can't stop reflection):

```java
public enum EnumSingleton {
    INSTANCE;

    public void doWork() { /* ... */ }
}
```

Pros:
- single instance guaranteed by the JVM
- free serialization + reflection safety
- concise

Cons:
- can't extend a class (but enums *can* implement interfaces)
- harder to parameterize construction (though fields/constructors are allowed)

</details>

### Java singleton vs Spring singleton?
<details><summary>Show answer</summary>

- **Java singleton** ‚Äî a language-level pattern: exactly one instance *per JVM*, enforced by the class itself.
- **Spring bean (default scope)** ‚Äî *not* a language singleton. 
  Container-managed: **one instance per application context**. 
  A different context ‚Äî or another JVM ‚Äî gets its own. The class isn't a singleton; the container guarantees uniqueness.

When to use which:
- **In a Spring app, use a bean** ‚Äî the container is already your instance-control mechanism;
  hand-rolling a singleton fights it (harder to inject, test, and manage lifecycle).
- **Use a Java singleton only** outside a DI container, or for very low-level utilities.

</details>

### Why write a class with only static members?
<details><summary>Show answer</summary>

A *utility class* (only static members, not meant to be instantiated). It has a bad reputation ‚Äî
sometimes misused instead of proper OO design ‚Äî but has three legitimate uses, all variants of
*"behavior with no natural object to live on"*:

- **No object exists** ‚Äî primitives and native arrays aren't objects, so they can't have instance
  methods; a static class is the only way to associate behavior with them. (`Math`, `Arrays`)
- **Too many unrelated objects** ‚Äî an interface (e.g. `Collection`) is implemented by many unrelated
  classes across libraries, and you can't retrofit new behavior onto all of them; shared operations
  go in a companion. (`Collections`)
- **The object is sealed** ‚Äî a `final` class can't be subclassed, so extra operations live in a
  utility class alongside it. `String` is `final`; companions like `StringUtils` (Apache Commons)
  add operations you can't put on `String` itself.

</details>

### How should you design an all-static helper class well?
<details><summary>Show answer</summary>

- keep methods small and focused ‚Äî avoid letting it grow into a *"god class"*
- organize by domain (`MathUtils`, `StringUtils`, `IOUtils`) rather than dumping everything in one place

</details>

### What's the inherent limitation of an all-static helper class?
<details><summary>Show answer</summary>

Static dispatch has **no instance to substitute**, so it's harder to test in isolation than an
injected dependency. (Same root issue as a singleton without an interface ‚Äî global access defeats
substitution.)

It's a *friction*, not a wall: since **Mockito 3.4.0** (2020), static methods can be mocked via
`mockStatic` ‚Äî `mockito-inline` on 3.4‚Äì4.x, built into `mockito-core` from 5.0. The mock is
*scoped*, so it's used inside a try-with-resources block and auto-closed (the same pattern this
file's cleanup section is about). Before 3.4.0 you needed PowerMock or a hand-written wrapper.

Caveats: still debated as a code-smell / mostly-for-legacy escape hatch; 
and some classes can't be mocked anyway (JVM intrinsics, certain standard-library classes).

</details>

### Utility class ‚Äî implementation options?
<details><summary>Show answer</summary>

- [Final utility class with a private constructor](#class-based-utility)
- [Utility interface with static methods](#interface-based-utility)

</details>

### Class-based utility?
<details><summary>Show answer</summary>

The classic, recommended form ‚Äî `final` class, private constructor that throws:

```java
public final class MathUtils {
  private MathUtils() {
    throw new AssertionError("No instances");   // guards against internal + reflective calls
  }
  public static int clamp(int value, int min, int max) {
    return Math.max(min, Math.min(max, value));
  }
  public static boolean isEven(int n) {
    return (n & 1) == 0;
  }
}
```

The private constructor already blocks outside instantiation; the `throw` additionally stops the
class's *own* code from calling it by mistake, and defends against reflective access. (`final` is
strictly optional here ‚Äî a private constructor already prevents subclassing ‚Äî but it documents
intent.)

</details>


### Interface-based utility?
<details><summary>Show answer</summary>

```java
public interface Hashing {
  static int murmur332(byte[] data) {
    // implementation here
    return 0;
  }
}
```
Pros
- No instantiation; clear namespacing.
- Useful when grouping helper functions conceptually tied to an API.

Cons
- Static methods on interfaces are not polymorphic and not inherited by implementors.
- Less common for general-purpose utilities than classes.

Use when
- You want namespacing for helper methods related to an interface contract.

</details>

### Utility class vs utility interface ‚Äî which, and when?
<details><summary>Show answer</summary>

Both hold only static helpers. Start with the case **against** the interface form ‚Äî and it's smaller
than it first appears, because the objections collapse into just two:

1. **The one real, intrinsic objection** ‚Äî an interface is **implementable by anyone, and the
   language won't stop them**. A `final` class with a private constructor is *sealed*: cannot be
   extended, cannot be instantiated ‚Äî the misuse is **impossible**. A utility interface offers no
   such guarantee; "don't implement this" holds **by convention only**. This isn't aesthetics
   ("interfaces should represent types") ‚Äî it's an *enforceability hole the language leaves open*.

2. **Everything else is one root cause: misuse.** The classic horror stories about utility
   interfaces are almost all the *same* mistake ‚Äî a client doing `implements Hashing`. That's
   pointless (static methods aren't inherited, so you gain nothing) and easily avoided: you **call**
   the statics (`Hashing.murmur332(...)`), you never implement the interface. So those "many cons"
   aren't intrinsic flaws ‚Äî they're all downstream of one wrong usage, and counting them separately
   inflates the case dishonestly.

So there's really **one** legitimate objection (#1), not a long list.

Now the actual trade-off. The real axis is **enforcement vs convention**:

- **Class** (`final` + private throwing constructor) ‚Äî *more ceremony*, but the language **seals it**.
  Safe by construction.
- **Interface** (only static methods) ‚Äî *less code, more readable*: no `final`, no private ctor, no
  `throw`; obviously nothing-but-utilities at a glance. But the constraint is **by convention only**.

Put plainly: **safe-by-construction (class) vs simpler-to-read (interface)**. And note this is *not*
"a philosophical preference for types (class) vs a **KISS** principle (interface)" ‚Äî that framing
undersells the class. It's "the language *enforces* the constraint (class) vs you *trust everyone* to
honor it (interface)." Enforcement is the heavier consideration; **KISS** (keep it simple) only
outweighs it when the enforcement isn't actually buying you anything.

Which means the deciding factor is **who consumes it**:

- **Reused library / public API / many unknown consumers** ‚Üí **class**, and this is why it's the
  *default*. "They'll know not to implement it" is unsafe across a large or external audience; the
  language guarantee earns its three lines.
- **Internal to your project / your team / contained blast radius** ‚Üí the **interface** is
  defensible. The enforceability hole is theoretical (nobody's going to implement your helper), and
  the readability/simplicity win is real ‚Äî this is **KISS** earning out. It applies precisely when
  you control who calls it, so the class's guarantee isn't buying you anything.

</details>

### Examples of unnecessary object creation?
<details><summary>Show answer</summary>

Prefer reusing one object over repeatedly creating functionally-equivalent instances ‚Äî more
efficient, and any **immutable** object is safe to reuse. Common pitfall sites:

- [Autoboxing](#autoboxing) ‚Äî accidental boxed-primitive creation in loops
- [String creation](#string-instance-creation---common-pitfalls) ‚Äî `new String(...)` vs literals
- [Boolean from String](#what-should-be-kept-in-mind-when-creating-a-boolean-from-a-string)
- [Regular expressions](#reusing-expensive-to-build-objects--what-to-watch-for) ‚Äî recompiling `Pattern`

</details>

### How to avoid unnecessary object creation?
<details><summary>Show answer</summary>

Two sides ‚Äî *who* avoids the waste:

**Designer side** ‚Äî you build the type so reuse is automatic; the caller does nothing special:
- expose **reusable constants** as fields ‚Äî `BigInteger.ZERO`, `BigInteger.ONE`, `Boolean.TRUE`
- make the type **immutable** ‚Äî then any instance is safe to share
- hand out instances via a [**static factory**](#constructors-vs-factories-on-object-reuse),
  which needn't return a new object ‚Äî by [preconstructing or caching](#constructors-vs-factories-on-object-reuse)

**Caller side** ‚Äî the type doesn't help, so you avoid waste at the call site:
- use **string literals**, not `new String(...)` (literals are interned and shared)
- **hoist a constant out of a loop/method** ‚Äî build once, reuse (e.g. a precompiled `Pattern`, a `DateTimeFormatter`)
- keep **your own cache** ‚Äî a single reused instance, or a map of bounded distinct values (flyweight)

Either side, last resort only: [**object pools**](#when-are-object-pools-justified)
‚Äî justified **only** when objects are genuinely heavyweight or expensive to *acquire* 
(DB connections, cached auth tokens ‚Äî a network round-trip / handshake), 
**not** when they're merely expensive to `new`. 
Most objects are cheap to construct; pool those and you add complexity for nothing.

</details>

### Constructors vs factories on object reuse?
<details><summary>Show answer</summary>

A constructor **must** return a new object every call; 
a static factory **needn't** ‚Äî it can hand back an existing instance instead, by one of two mechanisms:

- **preconstruct** ‚Äî build the instances up front, return them (e.g. `Boolean.TRUE`)
- **cache as constructed** ‚Äî build on first request, store, return the stored one thereafter

That's what lets factories avoid unnecessary duplicates.

</details>

### Autoboxing
<details><summary>Show answer</summary>

Autoboxing auto-converts between primitives and their boxed types ‚Äî convenient, 
but it **silently creates a boxed object** on every operation, and the primitive/boxed distinction never goes away.

```java
// Hideously slow ‚Äî spot the object creation
private static long sum() {
  Long sum = 0L;                        // boxed accumulator ‚Äî the bug
  for (long i = 0; i <= Integer.MAX_VALUE; i++)
    sum += i;                           // sum = Long.valueOf(sum.longValue() + i);
  return sum;
}
```

`sum` is `Long`, so each `sum += i` unboxes, adds, and **boxes a new `Long`** ‚Äî ~2¬≥¬π throwaway objects, 
massive GC pressure, 10‚Äì100√ó slower than the primitive version. 
One wrong type declaration turns a simple loop into a performance bug with no visible warning in the source.

The fix: declare the accumulator `long`, not `Long`.

</details>

### Primitive vs boxed ‚Äî which to prefer, and the trap?
<details><summary>Show answer</summary>

The autoboxing lesson ‚Äî "prefer primitives, they're faster and avoid the boxing bug" ‚Äî 
is **narrow, and over-generalizing it is a classic, costly mistake.** 
It applies *only* to counters and numeric accumulators in hot paths. 
Outside that, the choice isn't about performance at all ‚Äî it's about *representation*, 
and forcing the choice either way walks into three bugs, worst first:

**1. Floating-point is mathematically wrong for decimals (the blocking one).**
`float`/`double` use binary IEEE 754, which *cannot represent* most decimal fractions exactly ‚Äî
and **boxing doesn't help: `Double` is just as broken as `double`.**
```java
0.1 + 0.2          // 0.30000000000000004
1.03 - 0.42        // 0.6100000000000001
```
For money or any exact-decimal math, `float`/`double`/`Float`/`Double` are disqualified outright.
Use **`BigDecimal`** (or scaled integers ‚Äî `long` cents). This isn't an edge case; it's the normal arithmetic path.

**2. `null`-collapse (NPE).** A boxed type can be absent (`null`); a primitive cannot. 
Assigning a `null` boxed value to a primitive throws on unboxing, often far from the cause ‚Äî map lookups
(`int n = map.get(k)` ‚Üí NPE if absent), nullable DB columns, optional fields.

Worse, and separately: 
an absent value leaves the primitive at 0 with no error at all, so "no value" and a real 0 become indistinguishable.
For instance, a JSON payload without the field leaves an int at 0; an Integer would be null.

**3. `==` identity.** On boxed types `==` compares references, not values, but caching (default -128..127) 
makes it *look* correct for small numbers and fail for large ones:
`Integer a = 1000, b = 1000; a == b` ‚Üí `false`. Works in tests, breaks in production.

**The principle: correctness decides representation first; performance second.**
Ask ‚Äî must this be exact-decimal? (‚Üí `BigDecimal`) can it be absent? is it identity or value? 
Often the answer is *neither* primitive nor boxed. 
*Only* in counters/accumulators does "prefer primitive for speed" apply. 
Performance never compensates for incorrect representation.

</details>

### String instance creation - common pitfalls
<details><summary>Show answer</summary>

```java
String s = new String("bikini");
```

**This statement creates a new String instance every time it is executed, 
and none of these object creations is necessary.**

The argument passed to the `String` constructor `("bikini")` is already a `String` instance, 
functionally identical to the newly created objects.
If such code appears in a loop or a frequently invoked method, 
it can result in the needless creation of millions of String instances, 
wasting memory and increasing garbage‚Äëcollection pressure.

The improved version is simply the following:
```java
String s = "bikini";
```
This version uses a single String instance, rather than creating a new one each time it is executed.


**Create strings using string literals or existing String values, and avoid using `new String(...)`**

</details>

### What should be kept in mind when creating a Boolean from a String?
<details><summary>Show answer</summary>

- `Boolean(String)` constructor creates a new object each time it‚Äôs called.
- the factory method `Boolean.valueOf(String)` is preferable, it reuses existing immutable object

Note: `Boolean(String)`  constructor was deprecated in Java 9.

</details>

### A method rebuilds the same costly object on every call ‚Äî what's the issue and the fix?
<details><summary>Show answer</summary>

Some objects are **costly to construct but immutable and reusable** ‚Äî `Pattern`, `DateTimeFormatter`,
`ObjectMapper`, `DocumentBuilderFactory`. The trap: a convenience path silently *rebuilds* them on
every call instead of reusing one.

The classic case ‚Äî `String.matches` looks like a cheap string op, but is really
`Pattern.compile(re).matcher(s).matches()`, compiling a throwaway `Pattern` (a finite-state machine) every invocation:

```java
static boolean isRomanNumeral(String s) {
  return s.matches("^(?=.)M*(C[MD]|D?C{0,3})(X[CL]|L?X{0,3})(I[XV]|V?I{0,3})$");
}
```

Fix ‚Äî compile/construct **once** into a `static final`, reuse it:

```java
public class RomanNumerals {
  private static final Pattern ROMAN = Pattern.compile(
      "^(?=.)M*(C[MD]|D?C{0,3})(X[CL]|L?X{0,3})(I[XV]|V?I{0,3})$");

  static boolean isRomanNumeral(String s) {
    return ROMAN.matcher(s).matches();
  }
}
```

Same rule applies to `DateTimeFormatter` and friends: build it once as a constant, not per call.
(These types are immutable / thread-safe precisely so they *can* be shared.)

</details>

### Reusing one instance to avoid costly creation ‚Äî what to consider?
<details><summary>Show answer</summary>

Reuse is safe only if the shared object is **immutable / thread-safe**. Caching one instance in a `static final` field
means *every thread* touches that one object at once ‚Äî fine if it has no mutable state, a data race if it does.

- **Safe to share** ‚Äî immutable, thread-safe types: `Pattern`, `DateTimeFormatter`, `ObjectMapper` (once configured).
  Build once as a constant, reuse everywhere. (The
  [costly-object reuse](#a-method-rebuilds-the-same-costly-object-on-every-call--whats-the-issue-and-the-fix) fix
  assumes you're in this case.)

- **The trap ‚Äî a mutable, non-thread-safe object shared as a constant.** The textbook case is `SimpleDateFormat`: it
  keeps an internal `Calendar` and mutates it on every `format`/`parse`. Share one as a `static final` and concurrent
  calls corrupt each other's state ‚Äî wrong dates, exceptions, intermittently and unreproducibly. The very "build once,
  reuse" optimization becomes a concurrency bug.

Handling the mutable case, best first:
- **Use the immutable replacement** ‚Äî `DateTimeFormatter` instead of `SimpleDateFormat`; then sharing a constant is
  safe and you're back in the first case.
- **A `ThreadLocal` instance** ‚Äî one per thread, so no sharing across threads, when you must keep a mutable formatter.
- **Construct per call** ‚Äî simplest, but gives back the cost the reuse was meant to save; fine when calls are rare.

So before caching an expensive object as a constant: confirm it's immutable/thread-safe. If it isn't, don't share one
instance ‚Äî replace it, confine it per-thread, or rebuild it.

</details>

### When is lazy initialization discouraged?
<details><summary>Show answer</summary>

Default to **eager** initialization; reach for lazy only when init is *both* 
- genuinely expensive *and* 
- frequently skipped (many instances never touch the field). 
Otherwise it loses.

Why it usually loses:
- it **complicates the code** (and, for instance fields, drags in thread-safety concerns) for **no
  measurable gain** when the field is cheap or nearly always used anyway
- eager is already safe and simple ‚Äî a `static final` 
  (or the [holder idiom](#recommended-lazy-singleton)) is thread-safe for free

E.g. the eager `static final ROMAN` pattern in [the regex card](#reusing-expensive-to-build-objects--what-to-watch-for): 
you *could* compile it lazily on first `isRomanNumeral` call, but that's pure complication ‚Äî compile it eagerly.

</details>


### When are object pools justified?
<details><summary>Show answer</summary>

Maintaining your own object pool is a **bad idea by default** ‚Äî it clutters code, raises the memory
footprint, and often *harms* performance (and modern GCs make short-lived allocation very cheap).

Justified **only** when objects are either:
- **heavyweight to construct** ‚Äî genuinely costly to build, or
- **expensive to *acquire*** ‚Äî the cost is a resource handshake, not the `new`: a network round-trip,
  authentication, a connection setup

A **database connection** is the classic case ‚Äî and it's the *acquire* kind: cheap to represent, but
costly to establish. Cached **auth tokens** are similar. Do **not** pool objects that are merely
expensive to `new` ‚Äî most aren't, and pooling them adds complexity for nothing.

</details>

### When must you *not* reuse an object?
<details><summary>Show answer</summary>

When **defensive copying** is required ‚Äî then you *must* create a new object, not reuse an existing one.

The asymmetry is the point: 
**reusing when you should have copied is far more dangerous than copying when you didn't need to.**
- failing to copy ‚Üí shared mutable state, broken invariants, security holes ‚Äî subtle and severe
- copying needlessly ‚Üí only wasted performance / style ‚Äî minor and harmless

So this run's "avoid unnecessary creation" advice is the *low-stakes* side; 
defensive copying is the *high-stakes* side, and it wins whenever the two conflict.

</details>

### What are the options for object cleanup?
<details><summary>Show answer</summary>

- **try-with-resources** ‚Äî the right answer: deterministic, runs at scope exit (not tied to GC at all)
- **cleaners** ‚Äî Java 9 replacement for finalizers; still **non-deterministic** (may run late, or never)
- **finalizers** ‚Äî **deprecated and unsafe**; never use

</details>

### Java cleanup actions vs C++ destructors
<details><summary>Show answer</summary>

Same idea ‚Äî run actions on an object before its memory is reclaimed ‚Äî but opposite guarantees:

- **C++ destructor** ‚Äî runs **deterministically**, at the exact moment the object leaves scope.
  Timing is defined and under the programmer's control.
- **Java finalizers/cleaners** ‚Äî run **non-deterministically**. The JVM can't guarantee *when*, or
  even *whether*, they run at all.

This is why C++ habits don't transfer: in Java, "cleanup on destruction" isn't a thing you can rely
on ‚Äî you must release explicitly (try-with-resources).

</details>

### What resources need cleanup?
<details><summary>Show answer</summary>

Anything the **GC doesn't manage** ‚Äî four categories (recall the category; examples are just illustrations):

- **OS / native handles** ‚Äî file descriptors, sockets, pipes, memory-mapped files, native memory (JNI / Panama). 
  The OS owns these, not the Java heap.
- **Network connections** ‚Äî HTTP, WebSocket, DB connections, message-queue channels (Kafka, AMQP).
  Remote and finite ‚Äî must be closed.
- **Expensive runtime resources** ‚Äî thread pools, executors, schedulers, caches, GPU contexts.
  Heavyweight; won't release themselves.
- **Buffered data** ‚Äî buffered writers, loggers, compression streams (GZIP/ZIP), DB batch writers.
  *Different in kind:* the risk isn't a leak but **silent data loss** if not flushed.

</details>

### `FileInputStream` not closed ‚Äî consequences and how to spot it?
<details><summary>Show answer</summary>

A `FileInputStream` holds an OS **file descriptor** ‚Äî a finite, native resource the GC doesn't manage. 
Leak them (rely on GC to close) and the OS budget runs out.

**Consequences:**
- file-descriptor exhaustion ‚Üí new file/socket opens start failing
- files stay locked (especially on Windows)
- because release waits on GC, exhaustion can hit even while the objects are already unreachable

**How it shows up (the diagnostic signature):**
- the error `java.io.IOException: Too many open files` ‚Äî no new descriptors can be opened
- the open-descriptor count **grows monotonically under steady load** (a leak, vs a fixed high baseline)
- inspect with `lsof -p <PID>` on Linux, Resource Monitor on Windows
- note the limit is **shared** across files + sockets + pipes, so a file leak can starve sockets too

Fix ‚Äî deterministic release via try-with-resources:
```java
try (FileInputStream fis = new FileInputStream(path)) {
    // use the stream
}
```

</details>

### Network connections not closed ‚Äî consequences and how to spot it?
<details><summary>Show answer</summary>

Open sockets hold OS resources (descriptors + ports) that aren't released promptly.

**Consequences:**
- hanging / half-open connections
- resource (descriptor + port) leaks
- server-side timeouts as connections pile up

**How it shows up (the diagnostic signature):**
- same `Too many open files` exhaustion (sockets draw on the **same** descriptor budget as files)
- a steadily climbing socket count ‚Äî `lsof -p <PID>` or `netstat` showing many sockets stuck in
  states like `CLOSE_WAIT` (the local side never closed) or `ESTABLISHED` long past their use
- connection-pool-exhausted errors / rising latency before outright failure

Fix ‚Äî close explicitly via try-with-resources.

</details>

### Buffered data not flushed ‚Äî consequences and how to spot it?
<details><summary>Show answer</summary>

**Different failure mode from the others ‚Äî loss, not exhaustion.** 
A buffered writer/stream holds accumulated data in memory; 
if it dies before flushing, that data is **silently lost** ‚Äî no exception, no error.

**How it shows up (the diagnostic signature):**
- **truncated or missing tail** ‚Äî the file/output ends short, the last records simply absent
- no error at all ‚Äî this is what makes it dangerous; the leak announces itself, this stays silent
- often only the *last* buffer's worth is gone (everything already flushed survived), so the loss
  looks like "the end got cut off"

Fix ‚Äî try-with-resources guarantees `close()`, and `close()` flushes.

</details>

### Problems with finalizers and cleaners?
<details><summary>Show answer</summary>

Four kinds of problem ‚Äî and which apply tells you why cleaners are the *lesser* evil:

- **Timing ‚Äî can't rely on *when* or *whether*** (both finalizers and cleaners)
  - arbitrary delay between unreachability and execution
  - no guarantee they run *at all*
- **Failure-handling ‚Äî silent failure** (finalizers only)
  - an uncaught exception in a finalizer is *swallowed*: no stack trace, finalization just stops ‚Äî
    failures become invisible. Cleaners avoid this (the library controls the executing thread).
- **Performance** (both) ‚Äî objects with cleanup logic need special GC handling; significant penalty.
- **Security** (finalizers only) ‚Äî finalizer attacks can resurrect/exploit a partially-constructed
  object.

**Conclusion ‚Äî derived from which problems hit which:**
- **finalizers** fail on all four ‚Üí *unpredictable, dangerous, generally unnecessary*
- **cleaners** fix only failure-handling (and partly security) ‚Üí *less dangerous, but still
  unpredictable, slow, and generally unnecessary*
- **never** put time-critical actions in either; relying on them for resource management is a grave
  error (e.g. closing files this way ‚Üí descriptor exhaustion ‚Äî see the
  [`FileInputStream` card](#fileinputstream-not-closed--consequences-and-how-to-spot-it))

</details>

### Does forcing GC rescue finalizers/cleaners?
<details><summary>Show answer</summary>

No ‚Äî this is the common bad argument ("I'll just call `System.gc()` to make cleanup run"), and it fails three ways:

- **No guarantee, only odds.** `System.gc()` is a *hint* the JVM may ignore; `System.runFinalization()`
  raises the chance finalizers/cleaners run but never ensures it.
- **The methods that *did* promise it are long dead.** `System.runFinalizersOnExit` and its twin
  `Runtime.runFinalizersOnExit` were deprecated for decades ‚Äî they could run finalizers on objects
  *still in use* by other threads, risking corruption and deadlock. "Fatally flawed" by design.
- **Even if it worked, it's backwards.** You'd be triggering a full, expensive, stop-the-world GC
  just to release a resource that **try-with-resources frees instantly and for free**. Trading a
  cheap deterministic mechanism for an expensive non-deterministic one.

So `System.gc()` is not a reason to keep using finalizers/cleaners ‚Äî it's confirmation they can't be made reliable.

</details>

### Designing a resource-owning class?
<details><summary>Show answer</summary>

Expose the cleanup obligation so callers can discharge it:

- **implement `AutoCloseable`** ‚Äî a single `close()` method that releases the resource; 
  signals the type owns something needing release, and enables try-with-resources on the caller's side
- **track a closed-state flag** ‚Äî `close()` marks the object unusable; every subsequent method call
  checks it and throws `IllegalStateException` (fail fast, don't operate on a released resource)

(The designer *exposes* the duty; the caller *triggers* it via try-with-resources)

</details>

### When are cleaners/finalizers legitimate?
<details><summary>Show answer</summary>

Only for **non-critical** cleanup, and **never** as a replacement for `AutoCloseable` + try-with-resources. 

Two cases:

- **Safety net** ‚Äî a last-resort backstop if the client forgets to call `close()`. No guarantee it
  runs (promptly or ever), but releasing late beats never. It's a *backup* to explicit close, not
  the primary mechanism.
- **Native peers** ‚Äî a *native peer* is a non-Java object a Java object delegates to via native
  methods; the GC can't see it, so it won't be reclaimed automatically. A cleaner can release it ‚Äî
  **but only if** the performance hit is acceptable *and* the native resource isn't time-critical.
  If timeliness matters, use an explicit `close()` instead.

</details>

### Pitfalls of try-finally for resources?
<details><summary>Show answer</summary>

- **Verbosity compounds.** One resource is already awkward; a second forces *nested* try-finally:

```java
static void copy(String src, String dst) throws IOException {
  InputStream in = new FileInputStream(src);
  try {
    OutputStream out = new FileOutputStream(dst);
    try {
      byte[] buf = new byte[BUFFER_SIZE];
      int n;
      while ((n = in.read(buf)) >= 0) out.write(buf, 0, n);
    } finally { out.close(); }
  } finally { in.close(); }
}
```

- **Exception masking (the real bug).** Both the `try` body *and* `close()` can throw ‚Äî and if both do, 
  the `close()` exception **obliterates the first**, with no trace of it. 
  But the *first* is usually the one you need to diagnose the failure. 
  You *could* write code to suppress the second in favor of the first, but it's so verbose virtually no one did.

try-with-resources fixes both.

</details>

### Recommended way to release resources?
<details><summary>Show answer</summary>

**try-with-resources** ‚Äî the one reliable mechanism:

- deterministic ‚Äî `close()` runs at scope exit (not whenever GC eventually runs), even on exception or early return
- handles exception masking ‚Äî a `close()` exception is *suppressed* in favor of the original, 
  not allowed to hide it (the try-finally bug, fixed)

The resource just has to implement **`AutoCloseable`** (a single `void close()`) ‚Äî 
that's the contract that makes a type usable in try-with-resources.

</details>

### try-with-resources with multiple resources?
<details><summary>Show answer</summary>

Declare each resource in the `try` header ‚Äî they close in reverse order, automatically:

```java
static void copy(String src, String dst) throws IOException {
  try (InputStream in = new FileInputStream(src);
       OutputStream out = new FileOutputStream(dst)) {
    byte[] buf = new byte[BUFFER_SIZE];
    int n;
    while ((n = in.read(buf)) >= 0)
      out.write(buf, 0, n);
  }
}
```

Versus the [nested try-finally version](#pitfalls-of-try-finally-for-resources):

- **shorter** ‚Äî no nesting, no explicit `close()`
- **better diagnostics via suppression** ‚Äî if both `read` and the invisible `close()` throw, the
  `close()` exception is *suppressed in favor of* the original (the one you actually want).
  Suppressed exceptions aren't discarded ‚Äî they're printed in the stack trace marked "suppressed,"
  and reachable programmatically via `Throwable.getSuppressed()` (added in Java 7).

</details>
