### Eager Initialization singleton
<details><summary>Show answer</summary>

Simplest, thread-safe, the member is a final field:

```java
public final class EagerSingleton {
  private static final EagerSingleton INSTANCE = new EagerSingleton();

  private EagerSingleton() { /* init */ }

  public static EagerSingleton getInstance() {
    return INSTANCE;
  }
```
The private constructor is called only once, to initialize the public static final field `EagerSingleton.INSTANCE`.
The lack of a public or protected constructor guarantees a single universe:
exactly one `EagerSingleton` instance will exist once the `EagerSingleton` class is initialized -
no more, no less.
Nothing that a client does can change this, with one caveat:
a privileged client can invoke the private constructor reflectively
with the aid of the `AccessibleObject.setAccessible` method.
If you need to defend against this attack, modify the constructor to make it throw an exception
if it’s asked to create a second instance.

</details>

### Lazy Initialization singleton, recommended approach
<details><summary>Show answer</summary>

Using Initialization-on-Demand Holder (IoDH)

```java
public final class HolderSingleton {
  private HolderSingleton() { /* init */ }

  private static class Holder {
    static final HolderSingleton INSTANCE = new HolderSingleton();
  }

  public static HolderSingleton getInstance() {
    return Holder.INSTANCE;
  }
}
```
Lazy, thread-safe, no synchronization overhead.
**This is recommended general-purpose lazy singleton.**

</details>

### Lazy Initialization singleton with synchronized accessor:
<details><summary>Show answer</summary>

```java
public final class LazySingleton {
  private static LazySingleton instance;

  private LazySingleton() { /* init */ }

  public static synchronized LazySingleton getInstance() {
    if (instance == null) {
      instance = new LazySingleton();
    }
    return instance;
  }
}
```
`synchronized` on every call can be a bottleneck in hot paths

</details>

### Lazy Initialization singleton using Double-Checked Locking (DCL) with volatile:
<details><summary>Show answer</summary>

Solution without `synchronized` accessor using:
```java
public final class DclSingleton {
  private static volatile DclSingleton instance;

  private DclSingleton() { /* init */ }

  public static DclSingleton getInstance() {
    DclSingleton local = instance; // local var improves performance
    if (local == null) {
      synchronized (DclSingleton.class) {
        local = instance;
        if (local == null) {
          local = new DclSingleton();
          instance = local;
        }
      }
    }
    return local;
  }
}
```
Lazy, with minimal synchronization after init.
Cons:
- More complex to read and get right.
- Still some synchronization on first initialization.

</details>

### What are the reasons for creating a class that contains only static methods and static fields?
<details><summary>Show answer</summary>

Such classes usually called - _utility_ classes.

They are not intended to be instantiated, since creating an instance would have no semantic meaning.


Such classes have gained a negative reputation because they are sometimes misused as a substitute for proper 
object-oriented design; however, they also serve legitimate and well-defined purposes.


They can:
- be used to group related operations on primitive values or native arrays, 
  which are not objects in Java and thus cannot have instance methods, 
  making a static utility class the only practical way to associate behavior with them 
  (e.g., java.lang.Math, java.util.Arrays).
- also group static methods, including factories, that operate on objects implementing a common interface. 
  Because interfaces such as `Collection` are implemented by many unrelated classes, 
  often defined in different libraries, and cannot be extended in a way that forces existing implementations 
  to inherit new behavior, this functionality is instead provided via utility classes like `java.util.Collections`.
- be used to group additional operations for a final class, since its behavior cannot be extended through subclassing.

</details>

### What are the design tips and limitations of utility classes?
<details><summary>Show answer</summary>

Design tips:
- Keep utility methods small and focused; avoid turning them into _"god classes"_.
- Organize by domain (e.g., `MathUtils`, `StringUtils`, `IOUtils`) rather than dumping everything in one place.

Cons:
- Static utilities are inherently harder to mock.

</details>

### Utility class vs utility interface
<details><summary>Show answer</summary>

**Use an interface with static methods to define companion methods tied to an interface contract.**

These methods conceptually belong to the interface itself, not to any particular implementation.


Such methods:
- are defined in terms of the interface
- operate on any implementation of that interface
- help use, create, validate, transform, or adapt instances of the interface
- are part of the interface’s API contract, even though they are not instance methods


Because they are static, these methods cannot participate in polymorphism and therefore 
cannot be expressed as instance behavior of the interface.
Nevertheless, they remain logically associated with the interface and its contract.


**In other cases—where functionality is not clearly tied to an interface contract — 
prefer an implementation based on a final utility class**.

| Aspect                          | Final Utility Class + Private Constructor                        | Interface with Static Methods                                          |
|---------------------------------|------------------------------------------------------------------|------------------------------------------------------------------------|
| **Use Case Fit**                | General-purpose utility libraries                                | Companion methods tied to an interface contract                        |
| **Instantiation**               | Impossible (private constructor)                                 | Impossible (interfaces cannot be instantiated)                         |
| **Encapsulation**               | ✅ Can have private static fields, caches, helpers                | ❌ All fields are `public static final`; no private state               |
| **Static Initialization Block** | ✅ Supported                                                      | ❌ Not supported                                                        |
| **Semantic Intent**             | Clear: a namespace of utilities                                  | Ambiguous: interfaces imply a contract                                 |
| **Inheritance**                 | ❌ Cannot be subclassed (final)                                   | N/A (interfaces cannot be extended for implementation)                 |
| **Future Evolution**            | ✅ Can add nested classes, factories, even instance methods later | ❌ Adding non-static methods changes semantics (implies implementation) |
| **Tooling & Conventions**       | Widely recognized pattern                                        | Less common for general-purpose utilities                              |
| **Private Helpers**             | ✅ Possible                                                       | ❌ Not possible                                                         |
| **Static Import Ergonomics**    | ✅ Same (`import static ...`)                                     | ✅ Same                                                                 |

</details>

### Autoboxing
<details><summary>Show answer</summary>

Autoboxing - creates unnecessary objects, which allows the programmer to mix primitive and boxed primitive types,
boxing and unboxing automatically as needed. Autoboxing reduces syntactic friction but does not eliminate 
the fundamental distinction between primitive types and their boxed counterparts.

```java
// Hideously slow! Can you spot the object creation?
private static long sum() {
  Long sum = 0L;
  for (long i = 0; i <= Integer.MAX_VALUE; i++)
    sum += i;
  return sum;
}
```

This line: `sum += i;` conceptually expands to something like: `sum = Long.valueOf(sum.longValue() + i);`

That means for every loop iteration:
1. sum is unboxed (Long → long)
2. The addition is performed using primitives
3. The result is boxed again into a new Long object
4. The previous Long instance becomes garbage

That implies:
- Billions of temporary Long objects
- Massive allocation pressure on the heap
- Constant garbage collection
- Orders of magnitude worse performance

In practice, this version can be 10–100× slower than the primitive equivalent.

Autoboxing can turn a simple arithmetic loop into a serious performance bug without any obvious warning 
in the source code.

</details>

### When autoboxing is undesirable, should primitive types or boxed primitive types be preferred?
<details><summary>Show answer</summary>

- Prefer primitive types for counters and numeric accumulators.
- Do not confuse this recommendation with numeric correctness. 
  For many domains, neither primitives nor boxed primitives are appropriate; 
  `BigDecimal` or domain‑specific types should be used instead.

**Even in performance‑critical paths, primitives are only the right recommendation 
when the operation is numerically appropriate — primarily for counters and numeric accumulations**.


**Performance considerations do not justify and cannot compensate for numerically incorrect semantics.**

</details>

### String instance creation - common pitfalls
<details><summary>Show answer</summary>

```java
String s = new String("bikini");
```

**This statement creates a new String instance every time it is executed, 
and none of these object creations is necessary.**

The argument passed to the `String` constructor `("bikini")` is already a `String` instance, 
functionally identical to the newly created objects.
If such code appears in a loop or a frequently invoked method, 
it can result in the needless creation of millions of String instances, 
wasting memory and increasing garbage‑collection pressure.

The improved version is simply the following:
```java
String s = "bikini";
```
This version uses a single String instance, rather than creating a new one each time it is executed.


**Create strings using string literals or existing String values, and avoid using `new String(...)`**

</details>

### What are the pitfalls of unnecessary object creation when using regular expressions?
<details><summary>Show answer</summary>

```java
static boolean isRomanNumeral(String s) {
  return s.matches("^(?=.)M*(C[MD]|D?C{0,3})" + "(X[CL]|L?X{0,3})(I[XV]|V?I{0,3})$");
}
```
- It’s not suitable for repeated use in performance-critical situations.
- The problem is that it internally creates a `Pattern` instance for the regular expression and uses it only once,
  after which it becomes eligible for garbage collection.
- Creating a `Pattern` instance is expensive
  because it requires compiling the regular expression into a finite state machine.

Solution to improve the performance:
- explicitly compile the regular expression into a Pattern instance (which is immutable) as part of class initialization,
- cache it
- reuse the same instance for every invocation

```java
public class RomanNumerals {
  private static final Pattern ROMAN = Pattern.compile(
          "^(?=.)M*(C[MD]|D?C{0,3})"
                  + "(X[CL]|L?X{0,3})(I[XV]|V?I{0,3})$");

  static boolean isRomanNumeral(String s) {
    return ROMAN.matcher(s).matches();
  }
}
```

</details>

### In which situations is it necessary to create a new object instead of reusing an existing one?
<details><summary>Show answer</summary>

When **defensive copying** is required, you must create a **new object** and avoid reusing an existing one.
- The **cost of reusing an object when defensive copying is necessary is far greater** 
  than the cost of creating an unnecessary duplicate.
- Failing to make defensive copies where required can lead to **subtle bugs, broken invariants, 
  and serious security vulnerabilities**.
- In contrast, creating objects unnecessarily usually affects **only performance or code style**, 
  which is typically a much smaller and more manageable concern.

</details>

### What Java features can be used to execute cleanup actions when an object becomes unreachable, prior to garbage collection?

### Java cleanup actions vs C++ destructors
<details><summary>Show answer</summary>

Conceptually - they do the same thing - **execute actions on object before its memory being reclaimed**.


However, there are **important differences in behavior and guarantees**:
- In C++, a destructor runs deterministically at the exact moment an object goes out of scope.
  The timing is well-defined and under the programmer’s control.
- **In Java**, both **finalizers (legacy)** and **cleaners (the modern replacement)** run **non‑deterministically**.
  Java cannot guarantee when or even whether this code will execute.

</details>

### What are the key problems associated with finalizers and cleaners, and what conclusions follow from these issues?
<details><summary>Show answer</summary>

- There may be an **arbitrarily long delay** between the moment an object becomes unreachable and 
  the execution of its finalizer or cleaner.
- There is **no guarantee** that a finalizer or cleaner will ever run at all.
- If a finalizer throws an **uncaught exception**, the exception is silently ignored and 
  finalization for that object terminates.
    - Normally, an uncaught exception terminates the thread and prints a stack trace
    - In a finalizer, however, **no warning is printed**, making failures extremely hard to detect.
    - Cleaners do not have this problem because a library using a cleaner **controls the executing thread**.
- Both finalizers and cleaners impose a **significant performance penalty**, 
  as objects with cleanup logic require special GC handling.
- **Finalizers introduce serious security risks**, including finalizer attacks, 
  which can allow partially constructed objects to be used maliciously.


Recommendations:
- **Finalizers are unpredictable, often dangerous, and generally unnecessary.**
- **Cleaners are less dangerous than finalizers, but still unpredictable, slow, and generally unnecessary.**
- **Never perform time‑critical actions in a finalizer or cleaner**


Relying on finalizers or cleaners for critical resource management is a **grave error**.

For example, depending on a finalizer or cleaner to close files can quickly exhaust system resources:
- File descriptors are limited.
- If finalizers or cleaners run too slowly — or not at all — files may remain open.
- The program can fail simply because it can no longer open new files.

</details>

### What are the implications of using `System.gc()` and `System.runFinalization()` to try to force cleaners to run?
<details><summary>Show answer</summary>

Two methods once claimed to make this guarantee:
- `System.runFinalizersOnExit` and its evil twin,
- `Runtime.runFinalizersOnExit`.

These methods are fatally flawed and have been deprecated for decades.

### What is the correct way to design a class that owns resources requiring explicit cleanup?
<details><summary>Show answer</summary>

- Have the class implement `AutoCloseable` and require clients to invoke `close()` when the instance is no longer needed, 
  typically using _try‑with‑resources_ to ensure deterministic cleanup even in the presence of exceptions.
- The class should maintain a closed‑state flag. The `close()` method records that the object is no longer usable, 
  and all subsequent method invocations must detect this state and throw an `IllegalStateException`.

</details>

### What legitimate use cases, if any, exist for cleaners and finalizers?
<details><summary>Show answer</summary>

**Key Takeaway**:
Cleaners and finalizers should be used **only for non‑critical cleanup**, 
as a fallback or for managing non‑Java resources that the GC cannot see. 
They must **never replace explicit resource management** via `AutoCloseable` and _try‑with‑resources_.

1. **As a safety net for resource cleanup**

   Cleaners and finalizers can serve as a **last‑resort safety net** when the owner of a resource 
   neglects to call its `close()` method. While there is no guarantee that a cleaner or finalizer will run promptly — 
   or even run at all — it is still preferable to release a resource late rather than never if a client fails to do so.

2. **Cleaning up objects with native peers**

   Cleaners and finalizers may also be appropriate for objects that manage **native peers**.

   A _native peer_ is a native (non‑Java) object to which a Java object delegates via native methods. 
   Since native peers are **not managed by the JVM**, the garbage collector is unaware of them and cannot reclaim 
   their resources automatically when the Java object becomes unreachable.

   In such cases, a cleaner or finalizer can be used to release the native resource, provided that:
   - the performance impact is acceptable, and
   - the native peer does not hold critical resources that must be released promptly.

   If either condition is not met — particularly if timely cleanup is essential — the class should instead provide 
   an explicit `close()` method and rely on deterministic resource management.


</details>

### Describe a code snippet with `Cleaner`
<details><summary><strong>Show details</strong></summary>

<details><summary>Show code</summary>

```java
public class Room implements AutoCloseable {
  private static final Cleaner cleaner = Cleaner.create();

  // Resource that requires cleaning. Must not refer to Room!
  private static class State implements Runnable {
    int numJunkPiles; // Number of junk piles in this room

    State(int numJunkPiles) {
      this.numJunkPiles = numJunkPiles;
    }

    // Invoked by close method or cleaner
    @Override
    public void run() {
      System.out.println("Cleaning room");
      numJunkPiles = 0;
    }
  }

  // The state of this room, shared with our cleanable
  private final State state;
  // Our cleanable. Cleans the room when it’s eligible for gc
  private final Cleaner.Cleanable cleanable;

  public Room(int numJunkPiles) {
    state = new State(numJunkPiles);
    cleanable = cleaner.register(this, state);
  }

  @Override
  public void close() {
    cleanable.clean();
  }
}
```

</details>

<details><summary>Show answer</summary>

- The static nested `State` class holds the resources that are required by the cleaner to clean the room.
- In this case, it is simply the `numJunkPiles` field, which represents the amount of mess in the room.
  More realistically, it might be a final long that contains a pointer to a native peer.
- State implements `Runnable`, and its `run` method is called at most once, by the `Cleanable`.
- The call to the `run` method will be triggered by one of two things:
    - Usually it is triggered by a call to Room’s `close` method calling Cleanable’s `clean` method.
    - If the client fails to call the close method by the time a Room instance is eligible for garbage collection,
      the cleaner will (hopefully) call State’s run method.
- It is critical that a `State` instance does not refer to its `Room` instance.
  If it did, it would create a circularity that would prevent the Room instance from becoming eligible
  for garbage collection (and from being automatically cleaned).
- Therefore, `State` must be a static nested class because nonstatic nested classes contain references
  to their enclosing instances (Item 24).
- It is similarly inadvisable to use a lambda because they can easily capture references to enclosing objects.

</details>

</details>

### What pitfalls should you consider when managing resources with try-finally?
<details><summary>Show answer</summary>

Historically, a try-finally statement was the best way to guarantee that a resource would be closed properly,
even in the face of an exception or return:

```java
static String firstLineOfFile(String path) throws IOException {
  BufferedReader br = new BufferedReader(new FileReader(path));
  try {
    return br.readLine();
  } finally {
    br.close();
  }
}
```

This may not look bad, but it gets worse when you add a second resource:

```java
static void copy(String src, String dst) throws IOException {
  InputStream in = new FileInputStream(src);
  try {
    OutputStream out = new FileOutputStream(dst);
    try {
      byte[] buf = new byte[BUFFER_SIZE];
      int n;
      while ((n = in.read(buf)) >= 0)
        out.write(buf, 0, n);
    } finally {
      out.close();
    }
  } finally {
    in.close();
  }
}
```
- The code in both the `try` block and the `finally` block is capable of throwing exceptions.
- For example, in the `firstLineOfFile` method, the call to `readLine` could throw an exception
  due to a failure in the underlying physical device, and the call to `close` could then fail for the same reason.
- Under these circumstances, the second exception completely obliterates the first one.
  There is no record of the first exception in the exception stack trace,
  which can greatly complicate debugging in real systems - usually it’s the first exception that you want to see
  in order to diagnose the problem.
- While it is possible to write code to suppress the second exception in favor of the first,
  virtually no one did because it’s just too verbose.

All of these problems were solved with the try-with-resources.

</details>

### What is the only reliable and recommended way to clean up resources?
<details><summary>Show answer</summary>

- Use the try-with-resources statement.
- A resource must implement the `AutoCloseable` interface, which consists of a single `void close()` method.

</details>

### Provide an example of using try-with-resources with multiple resources
<details><summary>Show answer</summary>

```java
static void copy(String src, String dst) throws IOException {
  try (InputStream in = new FileInputStream(src);
       OutputStream out = new FileOutputStream(dst)) {
    byte[] buf = new byte[BUFFER_SIZE];
    int n;
    while ((n = in.read(buf)) >= 0)
      out.write(buf, 0, n);
  }
}
```

- It is shorter
- It provides far better diagnostics.
    - If exceptions are thrown by both the `read` call and the (invisible) `close`,
      the latter exception is suppressed in favor of the former.
    - In fact, multiple exceptions may be suppressed in order to preserve the exception that you actually want to see.
    - These suppressed exceptions are not merely discarded;
      they are printed in the stack trace with a notation saying that they were suppressed.
    - You can also access them programmatically with the `getSuppressed` method, which was added to `Throwable` in Java 7.
