### What are the requirements of the `hashCode()` contract?
<details><summary>Show answer</summary>

The `hashCode()` method in Java follows these rules:

- **Consistency**  
  When `hashCode()` is invoked repeatedly on the same object during a single execution of an application, 
  it must consistently return the same integer value, 
  provided that no information used in `equals()` comparisons has been modified.  
  The returned value does **not** need to remain consistent across different executions of the application.

- **Equality requirement**  
  If two objects are considered equal according to the `equals(Object)` method, 
  calling `hashCode()` on each of them **must** produce the same integer result.

- **Inequality flexibility**  
  If two objects are not equal according to the `equals(Object)` method, 
  it is **not required** that they return different hash codes.  
  However, generating distinct hash codes for unequal objects can 
- significantly improve the performance of hash-based collections.

</details>

### What rules should a `hashCode()` method follow?
<details><summary>Show answer</summary>

When defining a `hashCode()` method, follow these rules:

- **Include all fields used in equality checks**  
  Every field that participates in `equals()` comparisons **must** also be included in the `hashCode()` computation.

- **Exclude fields not used in equality checks**  
  Fields that are not used by `equals()` **must not** be included in `hashCode()`.  
  Including such fields can violate the second provision of the `hashCode` contract, 
  causing equal objects to produce different hash codes.

- **Derived fields may be omitted**  
  You may exclude derived (computed) fields from the hash code calculation.  
  Any field whose value can be derived entirely from fields already included does not need to be incorporated separately.

</details>

### What can be used to implement `hashCode()`?
<details><summary>Show answer</summary>

#### 1. Using `Objects.hash()` to Implement `hashCode()`

The `java.util.Objects` class provides a static `hash()` method that accepts 
a variable number of arguments and computes a hash code based on them.

```java
// One-line hashCode method - mediocre performance
@Override public int hashCode() {
  return Objects.hash(lineNum, prefix, areaCode);
}
```

This utility allows you to write concise, one-line `hashCode()` implementations 
whose quality is comparable to manually written methods that follow recommended practices.

However, this convenience comes with a performance cost:

- The method creates an array to hold the variable arguments.
- Primitive values passed to the method are automatically boxed and later unboxed.

As a result, `Objects.hash()` is slower than a hand-crafted implementation 
and is best suited for situations where performance is not critical.

#### 2. Using Generated Implementations (IDE / Tools)

A `hashCode()` method can also be automatically generated using development tools, such as:

- IDEs (e.g. IntelliJ IDEA, Eclipse, NetBeans)
- Lombok (e.g. `@EqualsAndHashCode`)
- **Google AutoValue** generates immutable value classes with correct `equals` and `hashCode` implementations.

These tools generate efficient and correct implementations based on the fields you select, 
helping reduce boilerplate code and minimize the risk of violating the `equals`/`hashCode` contract.

</details>

### What must be considered when implementing hashCode() for immutable classes?
<details><summary>Show answer</summary>

- **Caching the hash code**  
  Since immutable objects never change after creation, 
  you may cache the hash code in a field instead of recomputing it every time `hashCode()` is called.  
  This is especially beneficial if computing the hash code is expensive.

- **Eager computation**  
  If you expect most instances of the class to be used as keys in hash-based collections, 
  it is often best to compute the hash code eagerly during object construction.

- **Lazy initialization**  
  If hash-based usage is less common, you may choose to compute the hash code lazily, 
  initializing it the first time `hashCode()` is invoked.  
  This avoids unnecessary computation for objects that are never used in hashed collections.

- **Java records handle this automatically**  
  When using Java records, the compiler automatically generates a correct and efficient `hashCode()` implementation, 
  making manual handling unnecessary.

</details>

### What should be considered before overriding clone()?
<details><summary>Show answer</summary>

Overriding `clone()` introduces several risks and design challenges:

- **Broken encapsulation**  
  `clone()` bypasses constructors, which may violate class invariants and skip validation logic.

- **Shallow vs deep copy issues**  
  The default cloning behavior performs a shallow copy, often leading to shared mutable state and subtle bugs.

- **Error-prone inheritance**  
  Implementing `clone()` correctly in class hierarchies is difficult and fragile, 
  especially when subclasses add mutable fields.

- **Unclear contract and expectations**  
  The `Cloneable` interface does not define a clear contract for cloning behavior, leaving semantics ambiguous.

- **Exception handling complexity**  
  `clone()` relies on checked exceptions (`CloneNotSupportedException`), complicating usage and API design.

- **Maintenance and evolution difficulties**  
  Changes to class structure require careful updates to `clone()` to avoid bugs, increasing long-term maintenance cost.

Overriding `clone()` is risky and often unnecessary; 
**copy constructors** or **factory methods** usually provide safer, clearer alternatives.

</details>

### How do `Cloneable` and `Object.clone()` relate to each other?
<details><summary>Show answer</summary>

- **`Cloneable` is a marker interface** that signals a class permits cloning, 
  but it does **not** declare a `clone()` method.

- **`Object.clone()` provides the cloning logic**, performing a field-by-field copy. 
  It checks at runtime whether the object implements `Cloneable`.

- If a class does **not** implement `Cloneable`, `Object.clone()` throws `CloneNotSupportedException`.

- **Design issues**:
    - `clone()` is `protected` in `Object`
    - Implementing `Cloneable` does not guarantee an accessible `clone()` method
    - You cannot safely invoke `clone()` based on the interface alone

**Summary:**  
In practice, a class implementing `Cloneable` is expected to provide a properly functioning public `clone()` method.

`Cloneable` acts as a permission flag, while `Object.clone()` performs the actual cloning, 
but the separation between them makes Java’s cloning mechanism awkward and error-prone.

</details>

It also introduces an alternative object-creation mechanism that bypasses constructors or factories,

</details>

### What should you do if objects of a class need a natural ordering?
<details><summary>Show answer</summary>

If instances of a class have a **natural ordering**, 
implementing `Comparable` is the best choice to express this ordering directly in the class definition.

Implementing `Comparable`:
- **Defines a natural order** for objects that makes their sorting behavior clear and consistent across the application.
- **Enables seamless integration with the Java standard library**, 
  allowing objects to be sorted or used naturally in APIs 
  such as `Collections.sort`, `Arrays.sort`, `TreeSet`, and `TreeMap`.
- **Improves readability and correctness**, as the ordering logic is centralized and tied to the class itself 
  rather than duplicated across multiple comparators.
- **Reduces boilerplate**, avoiding the need to repeatedly supply external `Comparator` 
  implementations for the most common ordering.

In short, implementing `Comparable` makes ordering an inherent property of the class, 
leading to clearer APIs, safer usage, and better interoperability with Java collections.

</details>

### What is the contract of `compareTo()`?
<details><summary>Show answer</summary>

### Contract of `compareTo()`

The `compareTo()` method defines a **natural ordering** for objects and 
follows a contract that is conceptually similar to the one defined by `Object.equals()`.

- **Consistency**  
  The result of `compareTo()` must be consistent: 
  repeated comparisons of the same two objects should return the same value as long as neither object is modified.

- **Sign semantics**  
  For any non-null references `x` and `y`:
    - `x.compareTo(y) < 0` means `x` is less than `y`
    - `x.compareTo(y) == 0` means `x` is equal to `y` in terms of ordering
    - `x.compareTo(y) > 0` means `x` is greater than `y`

- **Transitivity**  
  If `x.compareTo(y) > 0` and `y.compareTo(z) > 0`, then `x.compareTo(z) > 0`.

- **Symmetry**  
  The sign of `x.compareTo(y)` must be the opposite of `y.compareTo(x)`.

- **Consistency with `equals()` (strongly recommended)**  
  If `x.compareTo(y) == 0`, then `x.equals(y)` should return `true`.  
  While not strictly required, violating this rule can lead to incorrect behavior in sorted collections such as `TreeSet` and `TreeMap`.

**Summary:**  
Like `equals()`, `compareTo()` defines a fundamental object relationship. 
Its contract ensures consistent, predictable ordering 
and safe interoperability with Java’s sorting and sorted-collection APIs.

</details>
