### What must accompany an equals override, and why?
<details><summary>Show answer</summary>

**Rule:** always override `hashCode` whenever you override `equals`.

**Why:** the `hashCode` contract requires equal objects to have equal hash codes. If you override
`equals` but leave the inherited `hashCode`, two objects that are logically equal will (almost always)
have *different* hash codes — so in hash-based collections (`HashMap`, `HashSet`) they land in
different buckets. The object you stored can no longer be reliably found or retrieved.

</details>

### What are the requirements of the hashCode contract?
<details><summary>Show answer</summary>

Three clauses:

- **Consistency** — repeated calls on the same object during one execution must return the same
  integer, as long as no field used in `equals` changes. The value need *not* stay the same across
  different runs of the application.

- **Equality requirement** — if two objects are equal per `equals(Object)`, their `hashCode` values
  **must** be equal.

- **Inequality flexibility** — unequal objects are *not required* to have different hash codes.
  But producing distinct codes for unequal objects improves hash-collection performance (fewer
  collisions).

</details>

### What are the field-selection rules for implementing hashCode?
<details><summary>Show answer</summary>

The guiding principle: `hashCode` must use **exactly the fields `equals` uses** — no more, no less.

- **Include every field used in `equals`.** Each field that participates in equality comparison must
  also feed into the hash computation.

- **Exclude every field *not* used in `equals`.** Adding such a field can make two equal objects
  produce different hash codes — violating the contract (equal objects → equal hash codes).

- **Derived fields may be omitted.** A field whose value is fully computable from fields already
  included adds nothing and can be skipped.

</details>

### What are the existing approaches to implement hashCode?
<details><summary>Show answer</summary>

- **Hand-written** — combine the significant fields manually. Best performance; most verbose. Worth it
  for performance-critical, frequently-hashed objects.

- **`java.util.Objects.hash(...)`** — varargs JDK utility; a one-line implementation whose quality
  matches a correct hand-written one:
  ```java
    @Override public int hashCode() {
      return Objects.hash(lineNum, prefix, areaCode);
    }
  ```
  Trade-off: slower — it allocates an array for the varargs and autoboxes primitives (then unboxes).
  Use where conciseness matters more than performance.

- **Java records** — the compiler generates a correct, efficient `hashCode` automatically; 
  no manual implementation needed.

- **Tooling** — Lombok (`@EqualsAndHashCode`), AutoValue, and IDE generation produce `hashCode`
  together with `equals` (same mechanism, different methods). See:
  [What tools can generate a correct equals implementation?](2.1_methods_common_to_all_objects.md#what-tools-can-generate-a-correct-equals-implementation)

</details>

### What must be considered when implementing hashCode for immutable classes?
<details><summary>Show answer</summary>

Because an immutable object's fields never change, its hash code never changes either — so it can be
**computed once and cached** instead of recomputed on every call. Worthwhile when the computation is expensive.

Two caching strategies:

- **Eager** — compute in the constructor. Best when most instances will be used as keys in hash-based collections.
- **Lazy** — compute on the first `hashCode()` call and store it. 
  Best when hash usage is uncommon — avoids the cost for instances never hashed.

**Records:** the compiler generates a correct `hashCode` for you, so you rarely implement this by
hand. (If caching specifically matters, you'd still need a custom class.)

</details>

### What method is recommended to always be overridden and why?
<details><summary>Show answer</summary>

`toString()`. Unlike `equals`, `hashCode`, or `clone` — which you override only when the context calls
for it — `toString` is worth overriding on essentially every class.

The default `Object.toString` returns just the class name + hash (`Foo@1b6d3586`) — useless. 
Override it to return a meaningful representation of the object's state. 
A good `toString` significantly simplifies debugging, logging, and diagnostics, 
because the object prints usefully wherever it appears — 
log lines, exception messages, debugger views, assertion failures.

</details>

### What are the pros and cons of using JSON as a toString representation?
<details><summary>Show answer</summary>

**Pros:**
- Structured, unambiguous, familiar — anyone reading a log recognises it instantly.
- Easy to read, log, inspect, and paste into tooling.
- Matches how the object is often represented elsewhere (APIs, diagnostics), so mental models line up.

**Cons:**
- **Dependency coupling** — pulls a serialization library (e.g. Jackson) into the domain object, 
  and couples that object to a specific format and its versioning/maintenance.
- **Cost on an implicitly-called method** — `toString` runs in places you never wrote it: 
  every log line, exception message, debugger view. 
  A heavy implementation has a cost that's invisible at the call site and only surfaces under load — 
  typically batch jobs doing lots of logging.
- **The cost is often self-inflicted** — the real killer is constructing a `new ObjectMapper()` per call. 
  A reused, shared `ObjectMapper`/`ObjectWriter` (immutable, thread-safe) removes most of the overhead. 
  JSON-in-`toString` is "expensive" mainly when done naively.
- **Security / leakage** — a serialize-everything approach will happily dump fields you'd never hand-write 
  into `toString`: passwords, tokens, PII. If logs are exposed, that's a real breach.
  A hand-written `toString` forces you to choose fields; JSON tempts you to skip that choice.


The cleaner default is to keep `toString` lightweight and dependency-free, 
and expose JSON **explicitly** via a separate method (`toJson()`) or an outside serializer — so the domain object
isn't coupled to a format, and serialization happens only when you actually ask for it, not every
time something logs the object.

</details>

### How should objects be copied?
<details><summary>Show answer</summary>

**Don't use `Cloneable` / `Object.clone()`.** It's broadly considered a design mistake and 
has been treated as bad practice for years — `clone()` bypasses constructors (skipping validation and invariants), 
does a shallow copy by default (shared mutable state), is fragile across inheritance, 
carries an ambiguous contract and a checked exception (`CloneNotSupportedException`), among other drawbacks.

**Use a copy constructor or copy factory instead:**
```java
public Foo(Foo other) { ... }          // copy constructor
public static Foo newInstance(Foo f)   // copy factory
```
They go through normal construction (invariants enforced), give you explicit control over shallow vs deep copy, 
behave correctly under inheritance, throw no checked exceptions, and don't tie the class to the `Cloneable` mechanism. 
This is the accepted best practice.

</details>

### Copying immutable classes — how is it done?
<details><summary>Show answer</summary>

It isn't — and shouldn't be. Immutable objects are safe to share freely, so copying them serves no purpose: 
preserving correctness never requires a distinct copy.

Adding `clone()` (or even a copy constructor) to an immutable class is misleading: 
it suggests copying is meaningful when it isn't, and introduces an extra object-creation path 
that bypasses the normal constructor/factory — added API complexity for no benefit.

</details>

### How do you make a class's instances sortable by a built-in order?
<details><summary>Show answer</summary>

Implement `Comparable<T>` and define `compareTo`. 
This expresses the ordering as an intrinsic property of the class itself.

Why this is the right choice:
- The class gains a single, consistent default order, defined in one place rather than duplicated
  across external comparators.
- It plugs directly into the standard library — `Collections.sort`, `Arrays.sort`, `TreeSet`,
  `TreeMap`, and anything that relies on natural order — for free.
- For *alternative* orderings, you still supply a `Comparator`; `Comparable` is for the one canonical order.

</details>

### What is the contract of compareTo?
<details><summary>Show answer</summary>

`compareTo` defines the natural ordering, with a contract conceptually similar to `equals`. For
non-null `x`, `y`, `z`:

- **Sign semantics** — `x.compareTo(y)` returns negative / zero / positive when `x` is less than /
  equal to / greater than `y`.
- **Symmetry (of sign)** — `sgn(x.compareTo(y))` must equal `-sgn(y.compareTo(x))`.
- **Transitivity** — if `x.compareTo(y) > 0` and `y.compareTo(z) > 0`, then `x.compareTo(z) > 0`.
- **Consistency** — repeated comparisons of unchanged objects return the same result.
- **Consistency with `equals` (strongly recommended, not required)** — `x.compareTo(y) == 0` should imply `x.equals(y)`. 
  Violating it leads to surprising behavior in sorted collections (`TreeSet`, `TreeMap`), 
  which use `compareTo`, not `equals`, to decide equality.

</details>

### How should compareTo be implemented?
<details><summary>Show answer</summary>

- **Comparator construction methods (preferred for most cases)** — build the comparison fluently 
  and let it generate `compareTo`, comparing fields in priority order:
  ```java
    private static final Comparator<PhoneNumber> COMP =
        Comparator.comparingInt((PhoneNumber p) -> p.areaCode)
                  .thenComparingInt(p -> p.prefix)
                  .thenComparingInt(p -> p.lineNum);
  
    @Override public int compareTo(PhoneNumber pn) { return COMP.compare(this, pn); }
  ```
  Readable and hard to get wrong. Minor performance cost vs hand-writing.

- **Hand-written** — compare fields manually in priority order, returning at the first difference.
  Use `Integer.compare`, `Double.compare`, etc. — **never** subtraction (`a - b`), which can overflow
  and break the sign contract. Most control and best performance; more error-prone.

- **IDE generation** — IDEs can scaffold a correct `compareTo`; 
  useful to avoid mistakes, but the output is verbose and needs maintenance as fields change.

- **Records / tooling** — note that `Comparable` is *not* auto-implemented by records, 
  Lombok, or AutoValue the way `equals`/`hashCode` are — ordering is domain-specific, so you define it yourself.

</details>

### Describe a code snippet
<details><summary><strong>Show details</strong></summary>

<details><summary>Show code</summary>

```java
public final class Priority implements Comparable<Priority> {
  private final int rank; 

  public Priority(int rank) { this.rank = rank; }

  @Override public int compareTo(Priority other) {
    return this.rank - other.rank;   // order by rank
  }
}
```

</details>

<details><summary>Show answer</summary>

**The bug: integer subtraction can overflow, flipping the sign of the result.**

`compareTo` must return a value whose *sign* encodes the ordering — negative / zero / positive.
The subtraction trick `this.rank - other.rank` works only while the difference fits in an `int`.
When the operands span a wide range, the subtraction overflows and **wraps around**, 
producing a result with the wrong sign.

Concrete failure: `this.rank = Integer.MIN_VALUE` (`-2147483648`), `other.rank = 1`.
The true ordering is `this < other`, so `compareTo` must return a negative number.
But `-2147483648 - 1` overflows `int` and wraps to `+2147483647` — **positive**. 
The method now reports `this > other`. 
The sign (and symmetry) contract is broken, and sorts / `TreeSet` / `TreeMap` misbehave silently.

**The fix: use the type's `compare` method, which is overflow-safe.**
```java
@Override public int compareTo(Priority other) {
  return Integer.compare(this.rank, other.rank);
}
```
Same idea for the other primitives: `Long.compare`, `Double.compare`, `Float.compare`, etc.

**Rule:** in `compareTo`, never derive the result by subtracting one value from another. 
Always compare with `Integer.compare` / `Double.compare` / a `Comparator` method — they encode the ordering
directly and cannot overflow.

(Note: the subtraction shortcut is *safe* only when you can prove the operands can never produce an
out-of-range difference — e.g. both are known non-negative. Even then, `Integer.compare` is clearer
and carries no risk, so prefer it unconditionally.)

</details>
