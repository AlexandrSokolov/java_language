## Classes and Interfaces - Inheritance

### What is inheritance, and what benefits does it provide?
<details><summary>Show answer</summary>

**Inheritance** lets a class (subclass) reuse and extend the behavior of another (superclass).

**Benefits:**

- **Code reuse** — common behavior defined once in the base class, reused by subclasses; 
  less duplication, easier maintenance.
- **Abstraction & hierarchy modeling** — expresses *is-a* relationships; 
  this is about how you **structure** your types.
- **Polymorphism** — follows *from* that hierarchy: it's what you can do **at runtime** because of the
  structure — treating related objects uniformly through a common supertype.

The last two are a pair: hierarchy is the compile-time structure, polymorphism is the runtime payoff it enables.

</details>

### What is polymorphism?
<details><summary>Show answer</summary>

**Polymorphism** ("many forms") — the ability of a language to bind one name to multiple types or behaviors.

It allows:
- a value or variable to have more than one type, and
- an operation to apply to values of more than one type.

Either side can vary:
- the data (one variable, several possible types)
  ```java
  List<String> list = new ArrayList<>(); // declared List, actually an ArrayList
  CharSequence cs = "hello";             // declared CharSequence, actually a String
  ```
- the operation (one operation, many types it accepts)
  ```java
  void print(int i)    { ... }
  void print(String s) { ... } // same name "print", different argument types
  ```

</details>

### What forms of polymorphism exist?
<details><summary>Show answer</summary>

Each takes one name across many types — the difference is whether the **dispatched behavior** is the same for all
or chosen per type:

- **Parametric** — same name, different types, **one shared body** (type not named); generics, `List<T>`; 
  resolved at **compile time**.
- **Subtype (inclusion)** — same name, different subtypes, **dispatch picks each subclass's own body**; 
  caller still treats them uniformly; also known as inheritance; resolved at **runtime**.
- **Ad-hoc overloading** — same name, different parameter types, **a separate body per type**; 
  resolved at **compile time**.

</details>

### How do inheritance and polymorphism relate?
<details><summary>Show answer</summary>

Inheritance provides **subtype polymorphism** specifically — the *runtime* kind. 
A subclass instance can be used wherever its supertype is expected, and a call to an overridden method runs the
**subclass's** implementation, chosen by the object's actual runtime type:

```java
Shape s = new Circle();   // supertype reference, subtype object
s.draw();                 // runs Circle.draw(), not Shape.draw()
```

**Mechanism — dynamic dispatch:** overridable methods live in a per-class virtual table; 
the call is bound at runtime via the object's actual class.

Hierarchy is the *compile-time* structure (is-a); subtype polymorphism is the *runtime* payoff — override plus
dynamic dispatch let you treat related objects uniformly through a common supertype.

</details>

### When should you inherit — and when not?
<details><summary>Show answer</summary>

- [**True is-a relationship**](#how-do-you-know-its-a-real-is-a-relationship) — 
  the subtype really *is* a kind of the base, not just shared code.

  *Fails when:* you use `extends` only to reuse code — that's has-a.

- [**Substitutability (LSP)**](#how-does-a-subtype-violate-substitutability-lsp) — 
  a subclass stands in for the base everywhere.

  *Fails when:* a subclass changes base behavior, or callers must downcast to reach behavior the base doesn't expose.
  **Classic warning sign:** *"this class extends another just to override half of its behavior."*

- [**Designed polymorphism**](#how-do-inheritance-and-polymorphism-relate) — 
  using subtypes through the supertype is a deliberate goal.

  *Fails when:* you reach for `extends` with no intent to use the objects through the supertype.


**Standing cost**:

- [**It always weakens encapsulation.**](#what-is-the-relationship-between-inheritance-and-encapsulation)
  Implementation inheritance couples a subclass to the base's internals, so inheriting is never free.


</details>

### How do you know it's a real is-a relationship?
<details><summary>Show answer</summary>

Test it by **substitution, not by shared fields**: the subtype is a real is-a only if it can stand in for the parent
*everywhere the parent is used*, through a parent reference, without callers noticing or breaking. If code must know
the concrete subtype to use it correctly, it isn't a true is-a — it's reuse wearing an is-a costume.

</details>

### How does a subtype violate substitutability (LSP)?
<details><summary>Show answer</summary>

A subtype breaks substitutability when it can't truly stand in for the base everywhere the base is expected. Two forms:

- **Altering base behavior** — the subtype must override, disable, or significantly change what the base does, so code
  written against the base behaves wrongly when handed the subtype.
- **Hidden behavior the base doesn't expose** — callers must downcast to a concrete subtype to reach a method the base
  doesn't declare (`((Dog) animal).bark()` where `bark` is only on `Dog`). The supertype reference can't actually be
  used on its own; callers must know the real type, which defeats the abstraction.

The downcast form usually shows up as an `instanceof` check followed by a cast — the classic sign that the supertype is
the wrong abstraction (incomplete, or chosen only for reuse). You're manually re-dispatching on the concrete type, which
is exactly what polymorphism was meant to remove. The fix is normally to put the behavior on the supertype (so it
dispatches polymorphically) or to admit the subtype isn't really substitutable for that base.

</details>

### When is it safe to subclass a class, and when does subclassing introduce significant risk?
<details><summary>Show answer</summary>

**Safe:**
- **Package-private superclass** — *structurally* safe: 
  it can't be subclassed from outside the package at all (compiler-enforced), so no external subclass can ever exist.
- **Designed for inheritance** — the superclass is explicitly documented and structured to be extended safely 
  (self-use documented, hooks provided).

**Dangerous:**
- **Across package boundaries, on an ordinary concrete class** — such a class was *not* designed for subclassing. 

</details>

### What is the main problem of using inheritance?
<details><summary>Show answer</summary>

Inheritance [**weakens encapsulation**](#what-is-the-relationship-between-inheritance-and-encapsulation). 
A subclass depends on the *implementation details* of its superclass, not just its public contract.

</details>

### What is the relationship between inheritance and encapsulation?
<details><summary>Show answer</summary>

**Inheritance weakens encapsulation.** A subclass depends not just on the base's public contract but on its
*internal implementation* — most often the superclass's **self-use**: which overridable methods the base calls, and when.
That self-use isn't part of the public API, yet a subclass that overrides those methods is silently coupled to it.


The consequence: a change to the superclass's internal implementation can break a subclass **even
though the subclass's code is unchanged and the superclass's public API is unchanged**. The coupling is
strong but *implicit*


So inheritance is safe only when the base is **designed and documented for extension** — self-use spelled out,
overridable methods and their guarantees made part of the contract — and its dependents can absorb its changes.

</details>

### A JDK class where inheritance violated the subclass's own invariants?
<details><summary>Show answer</summary>

`java.util.Properties extends Hashtable` — a classic mistake.

The root error: a `Properties` object isn't a general hashtable, it *has* one — so the is-a was false;
it should have been composition. Because it inherited instead:

- **Leaked, conflicting methods** — inherited `get(key)` behaves differently from `Properties`'
  own `getProperty(key)`, giving two confusingly different access paths.
- **Broken invariants** — `Properties` is meant to hold only string keys and values, but inherited
  `Hashtable` methods (`put(Object, Object)`) let clients insert arbitrary objects, violating that
  guarantee.

And once clients depended on the inherited behavior, the flaw was frozen — unfixable without breaking
backward compatibility. Composition (`Properties` *holds* a map, exposes only string operations) would
have avoided all of it.

</details>

### What types of inheritance exist?
<details><summary>Show answer</summary>

Two kinds:

- **Implementation inheritance** — a class extends another class and inherits its *implementation and
  behavior* (`extends`).
- **Interface inheritance** — a class implements an interface, or an interface extends another;
  *only the method contracts* are inherited, no implementation.

Why the distinction matters: the encapsulation-weakening problem is almost entirely a property of **implementation** 
inheritance — there's an implementation to depend on, so the subclass can couple to internals. 
**Interface** inheritance carries no implementation, so there's nothing internal to couple to — 
which is a large part of why interfaces are the safer extension mechanism.

</details>

### What are the consequences of extending a class by adding new methods while leaving existing methods unchanged?
<details><summary>Show answer</summary>

Even purely *additive* subclassing — only adding new methods, never overriding — is not risk-free. 
The danger comes from a **future change to the superclass** that retroactively collides with your added
method, breaking code that was correct when written:

- **Superclass later adds a method with the same signature but a different return type** — your
  subclass no longer compiles (signature conflict).
- **Superclass later adds a method with the same signature and return type** — your method silently
  *overrides* it, and almost certainly fails to honor the new method's contract — a contract that
  didn't exist when you wrote the method.

The deeper point: you can be broken by a change you didn't make and couldn't have anticipated. 
Adding methods feels safe but still couples you to the superclass's *future* evolution.

</details>
