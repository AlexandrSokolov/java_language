## Classes and Interfaces - Inheritance

### What is inheritance, and what benefits does it provide?
<details><summary>Show answer</summary>

**Inheritance** is an object‑oriented mechanism that allows a class (subclass) to
reuse and extend the behavior of another class (superclass).


**Benefits of inheritance**:

- **Code reuse (reduced duplication)**  
  Common behavior is defined once in a base class and reused by subclasses, 
  minimizing duplicated logic and improving maintainability.

- **Abstraction and hierarchy modeling**  
  Helps represent *is‑a* relationships in a domain model.
  It is about how you structure your types

- **Polymorphism** - is achieved through abstraction and hierarchy.
  It is about what you can do at runtime because of that hierarchical structure.
  Enables treating related objects uniformly through a common superclass or interface.

</details>

### When is inheritance appropriate, and when should it be used cautiously or avoided?
<details><summary>Show answer</summary>

Should inheritance be used? (concept & semantics)

**Inheritance is appropriate when**:
- there is a true and stable **is‑a relationship**/semantic, 
- **stability** - the base class represents a reliable abstraction, 
- **substitutability** - subclasses fully satisfy substitutability (LSP), 
- **polymorphic** use is an intentional design goal.


Inheritance should be used cautiously or avoided 
when the subclass relationship violates **one or more core inheritance constraints** — 
semantic correctness, stability, substitutability, or the need for polymorphism.

When inheritance should be used cautiously or avoided:
- **Semantic misuse (not a real is‑a)**
  When inheritance is used only for code sharing rather than modeling a true conceptual relationship, 
  it leads to accidental inheritance.

- **Fragile or unstable hierarchies**
  Deep or rigid hierarchies tightly couple subclasses to base classes, 
  so changes in the base class can break large parts of the system.

- **Violation of substitutability (LSP)**
  If a subclass must override, disable, or significantly alter base behavior, 
  it can no longer be safely used in place of the base type.

- **Misaligned or forced polymorphism**
  When behavior needs to change along different axes, inheritance doesn’t scale; composition provides better flexibility.

Classic warning sign:

> "This class extends another just to override half of its behavior".

</details>

### When is it safe to subclass a class, and when does subclassing introduce significant risk?
<details><summary>Show answer</summary>

Is it safe to subclass this class? (encapsulation & ownership)

#### When inheritance is considered safe

- **Within the same package**  
  The superclass and subclass are controlled by the same team, allowing coordinated changes.

- **Classes designed for inheritance**  
  The superclass is explicitly documented and structured to be safely extended.

#### When inheritance is dangerous

- **Across package boundaries**  
  Ordinary concrete classes are not designed for subclassing and may change in ways that break subclasses.

</details>

### What are the main problems of using inheritance? What structural costs does inheritance introduce?
<details><summary>Show answer</summary>

What structural costs does inheritance introduce? (evolution & rigidity)

- **Fragile base class and [**broken encapsulation**](#what-is-the-relationship-between-inheritance-and-encapsulation-in-objectoriented-systems)
  (class-level problem)** - problem at the class contract and encapsulation level.
  Subclasses depend on superclass implementation details rather than only its public contract, which breaks encapsulation.  
  **This exists even with a small hierarchy**:
  even a single subclass can be broken by internal changes to the superclass, even when its public API remains unchanged.

- **Tight coupling and limited evolvability (system-level problem)** -
  Problem at the architecture and long-term evolution level.
  As inheritance hierarchies grow, many subclasses become structurally dependent on a central base class.  
  **This emerges as the system grows**: the more widely a superclass is subclassed (across packages, modules, or teams),
  the riskier and more expensive it becomes to refactor or evolve the design.

</details>

### What types of inheritance exist?
<details><summary>Show answer</summary>

Inheritance can be divided into two main types:

- **Implementation inheritance**  
  A class extends another class and **inherits its implementation and behavior**.

- **Interface inheritance**  
  Either a class implements an interface, or one interface extends another interface; 
  **only method contracts are inherited, not implementation**.

</details>

### What is the relationship between inheritance and encapsulation in object‑oriented systems?
<details><summary>Show answer</summary>

**Inheritance weakens encapsulation in object‑oriented systems.**  
A subclass depends not only on the public contract of its superclass, but also on its internal implementation details.


As a result, changes to the superclass implementation can break subclasses even when their code has not changed, 
creating strong and often implicit coupling. Subclasses are therefore forced to evolve in tandem with their superclasses, 
making inheritance brittle and limiting independent evolution.


**Inheritance is safe only when the superclass is explicitly designed and documented for extension**, 
clearly defining overridable methods and the guarantees preserved for subclasses.

</details>

### What are the consequences of extending a class by adding new methods while leaving existing methods unchanged?
<details><summary>Show answer</summary>

Even if a subclass only adds new methods and does not override existing ones,
this form of extension is still not risk‑free.

- **Superclass adds a new method with the same signature but a different return type**  
  The subclass no longer compiles due to a method signature conflict.

- **Superclass adds a new method with the same signature and return type**  
  The subclass unintentionally overrides it and is unlikely to satisfy the new
  method’s contract, since that contract did not exist when the subclass was written.

</details>

### How can object‑oriented designs mitigate the impact of inheritance on encapsulation?
<details><summary>Show answer</summary>

- **Favoring composition over inheritance**  
  Depend on object composition and delegation instead of inheriting implementation details.

- **Using inheritance only when explicitly designed for it**  
  Extend only classes that are documented for inheritance or are under the same control.

- **Programming to interfaces, not implementations**  
  Depend on contracts rather than concrete implementations to preserve encapsulation.

</details>

### Which design pattern allows adding behavior to objects without subclassing, while keeping the same interface?
<details><summary>Show answer</summary>

The **Decorator** pattern replaces inheritance with composition by wrapping an object and implementing the same interface.
Because it depends only on the interface contract, it preserves encapsulation and avoids fragile base class problems.

</details>

### Can you give an example of the Decorator pattern?
<details><summary>Show answer</summary>

Common interface:
```java
public interface Coffee {
  double cost();
  String description();
}
```
Base implementation:
```java
public class SimpleCoffee implements Coffee {

  @Override
  public double cost() {
    return 2.0;
  }

  @Override
  public String description() {
    return "Simple coffee";
  }
}
```
Abstract decorator:
```java
public abstract class CoffeeDecorator implements Coffee {
  protected final Coffee coffee;

  protected CoffeeDecorator(Coffee coffee) {
    this.coffee = coffee;
  }
}
```
Concrete decorators:
```java
public class MilkDecorator extends CoffeeDecorator {

  public MilkDecorator(Coffee coffee) {
    super(coffee);
  }

  @Override
  public double cost() {
    return coffee.cost() + 0.5;
  }

  @Override
  public String description() {
    return coffee.description() + ", milk";
  }
}
public class SugarDecorator extends CoffeeDecorator {

  public SugarDecorator(Coffee coffee) {
    super(coffee);
  }

  @Override
  public double cost() {
    return coffee.cost() + 0.2;
  }

  @Override
  public String description() {
    return coffee.description() + ", sugar";
  }
}
```
Usage:
```java
Coffee coffee = new SimpleCoffee();
coffee = new MilkDecorator(coffee);
coffee = new SugarDecorator(coffee);

System.out.println(coffee.description()); // Simple coffee, milk, sugar
System.out.println(coffee.cost());        // 2.7

//or alternatively to wrap the coffee multiple time:
coffee = new SugarDecorator(
  new MilkDecorator(
    new SimpleCoffee()));

System.out.println(coffee.description()); // Simple coffee, milk, sugar
System.out.println(coffee.cost());        // 2.7
```

</details>

### Compare decoration with inheritance
<details><summary>Show answer</summary>

Without decorators, you’d need classes like:
- CoffeeWithMilk
- CoffeeWithSugar
- CoffeeWithMilkAndSugar
- CoffeeWithMilkSugarAndCinnamon

Which leads to class explosion. Decorator avoids this completely.

With the Decorator pattern, you combine behaviors by stacking decorators,
allowing multiple features (like milk and sugar) to be added dynamically without subclassing.

</details>

### Can you show an example where using inheritance instead of composition leads to design problems?
<details><summary>Show answer</summary>

A classic example is Java’s `Properties` class inheriting from `Hashtable`.


Because of inheritance, `Properties` exposes `Hashtable` methods that break its own assumptions:
- `get(key)` behaves differently from `getProperty(key)`, leading to confusing semantics.
- Although `Properties` is meant to store only string keys and values, 
  inherited methods allow inserting arbitrary objects, breaking its invariants.


Once clients relied on this inherited behavior, the design flaw could not be fixed without breaking compatibility.  
This problem would have been avoided if `Properties` had used composition instead of inheritance.

</details>

### What design considerations are required to make a class appropriate for inheritance?
<details><summary>Show answer</summary>

A class is fit for inheritance only if it is intentionally designed, carefully documented, 
and robust against being extended.

- **Document inheritance contracts explicitly**  
  Clearly specify which methods may be overridden and what invariants subclasses must preserve.

- [**Document self-use of overridable methods**](#how-can-a-class-properly-document-its-internal-use-of-overridable-methods-for-subclasses)  
  Clearly describe all places where the class internally calls overridable methods, so
  subclasses understand how and when their overrides will be invoked.

- [**Provide protected hooks, not fragile internals**](#what-fundamental-tension-does-inheritance-introduce-between-extensibility-and-encapsulation)  
  Expose well-defined extension points instead of implementation details.

- [**Design and test for subclassing**](#how-to-test-a-class-designed-for-inheritance)  
  Ensure the class behaves correctly when methods are overridden and state is extended.

- [**Avoid calling overridable methods from constructors**](#what-happens-if-a-constructor-invokes-an-overridable-method)  
  Subclasses may not be fully initialized when such methods are invoked.

- **Preserve and enforce invariants**  
  Subclasses must not be able to break the class’s internal consistency.

- **Keep the API minimal and stable**  
  Any public or protected method becomes part of the subclass API forever.

</details>

### How can a class properly document its internal use of overridable methods for subclasses?
<details><summary>Show answer</summary>

If a method invokes one or more overridable methods, this behavior must be **explicitly documented**. 
The documentation should describe *when* and *how* these invocations occur, 
so subclasses can safely override the methods.

This information is placed in a special Javadoc section called **Implementation Requirements**, 
generated using the `@implSpec` tag.  
Unlike the general specification, this section describes the **internal workings** of the method.

```java

/**
 * Removes a single instance of the specified element from this collection,
 * if it is present.
 *
 * @implSpec
 * This implementation iterates over the collection looking for the specified
 * element. If it finds the element, it removes it using the iterator's
 * {@code remove} method.
 *
 * @throws UnsupportedOperationException if the iterator does not support
 *         {@code remove}
 */
public boolean remove(Object o) { ... }
```

- It makes it explicit that remove depends on the behavior of iterator()
- It clearly explains how overriding iterator() can affect remove
- Subclass authors know exactly which methods influence each other

</details>

### How does documenting implementation details for inheritance fit within general API documentation principles?
<details><summary>Show answer</summary>

Good API documentation normally describes **what** a method does, not **how** it does it. 
Documenting implementation details therefore *does* violate this best practice.

However, this violation is unavoidable when designing a class for inheritance.
Because inheritance breaks encapsulation, subclasses depend on certain internal behaviors of the superclass. 
To allow safe subclassing, these implementation details must be explicitly documented, 
even though they would otherwise remain unspecified.

</details>

### What fundamental tension does inheritance introduce between extensibility and encapsulation?
<details><summary>Show answer</summary>

Inheritance introduces a core tension between two competing needs in object‑oriented design:

- **Subclasses need a way to extend or customize behavior**
- **Superclasses must protect their internal consistency and their ability to evolve**


If subclasses are given too much access, they become dependent on implementation details, making the hierarchy fragile. 

If they are given too little access, inheritance becomes unusable.  

This tension is resolved by providing **controlled extension points (hooks)**, 
which allow limited customization while preserving encapsulation and stability.

</details>

### What are hooks in object‑oriented inheritance, and why are they important?
<details><summary>Show answer</summary>

**Protected hooks** exist to structurally limit what subclasses can influence.
They answer a crucial design question:

> *Where exactly is customization allowed — and nowhere else?*

The core idea is simple:
- **Final or non‑overridable methods** define the algorithm and enforce invariants
- **Protected hooks** allow only controlled and intentional variation

This approach is stronger than documentation alone because it is **enforced by the type system**, not just by convention.

Crucially, a protected hook is part of the **inheritance contract**,
not an accidental exposure of implementation details.


Protected hooks solve several fundamental inheritance problems:

1. **They provide a safe place to extend behavior**  
   Subclass authors no longer need to guess which methods to override or reverse‑engineer superclass behavior.
   The intended customization point is explicit.

2. **They decouple extension from implementation structure**  
   Subclasses depend on documented behavior rather than internal algorithms.
   This allows superclasses to evolve internally without breaking subclasses,
   restoring encapsulation despite the use of inheritance.

3. **They prevent accidental overriding of critical logic**  
   Without hooks, subclasses often override important methods simply because no alternative exists,
   leading to broken invariants and subtle bugs.
   Hooks keep core logic protected while isolating customization in safe override points.

</details>

### Can you provide an example that illustrates the use of protected hooks?
<details><summary>Show answer</summary>

We want:
- A superclass with non‑trivial core logic
- Subclasses must not replace that core logic
- Subclasses may customize behavior before and/or after
- No reliance on fragile internal details
- Clear, enforceable extension points

Fragile approach:
```java
public class RequestProcessor {

    public void process(Request request) {
        authenticate(request);
        execute(request);
        audit(request);
    }

    protected void execute(Request request) {
        // complex core logic
    }
}
```
Problem: Subclasses override execute
They might:
- skip important steps
- forget to call super.execute()
- depend on execution order

Correct approach: core logic + pre/post hooks:
```java
public abstract class RequestProcessor {

    public final void process(Request request) {
        authenticate(request);

        beforeExecute(request);
        executeCore(request);
        afterExecute(request);

        audit(request);
    }

    protected void authenticate(Request request) {
        // security checks
    }

    protected void audit(Request request) {
        // auditing and logging
    }

    /**
     * Core logic that must never be replaced.
     */
    protected final void executeCore(Request request) {
        // complex, invariant-sensitive logic
    }

    /**
     * Hook: called immediately before core execution.
     * Default implementation does nothing.
     */
    protected void beforeExecute(Request request) {
        // optional extension point
    }

    /**
     * Hook: called immediately after core execution.
     * Default implementation does nothing.
     */
    protected void afterExecute(Request request) {
        // optional extension point
    }
}
```
Subclass customization (safe and intentional):
```java
public class MetricsRequestProcessor extends RequestProcessor {

    @Override
    protected void beforeExecute(Request request) {
        startTimer();
    }

    @Override
    protected void afterExecute(Request request) {
        stopTimerAndRecordMetrics();
    }
}
```
What the subclass cannot do
- Cannot bypass authentication
- Cannot alter execution order
- Cannot replace or partially execute core logic
- Cannot depend on internal structure

All of this is enforced structurally, not “by convention”.

</details>

### Documentation using vs protected hooks
<details><summary>Show answer</summary>

- Documentation answers: “What happens if you override this?”
- Protected hooks answer: “This is the only place you are allowed to customize behavior.”

</details>

### Which design pattern underlies the use of protected hooks in inheritance?
<details><summary>Show answer</summary>

The design pattern that underlies the use of **protected hooks** in inheritance is the **Template Method** pattern.


In the Template Method pattern, the superclass defines the overall structure of an algorithm
in a **final or non‑overridable method**,
while delegating specific steps to **protected methods (hooks)** that subclasses may override.


This pattern allows subclasses to customize parts of the algorithm without
changing its structure, preserving invariants and encapsulation while enabling
controlled extensibility.

</details>

### What problems must be considered when using hooks to customize behavior in inheritance?
<details><summary>Show answer</summary>

- **Increased subclass coupling**  
  Subclasses become tightly coupled to the superclass’s hook contracts and execution order, reducing flexibility.

- **Complex reasoning about behavior**  
  Understanding runtime behavior may require knowing both superclass logic and subclass overrides.

</details>

### How does a functional approach provide an alternative to the Template Method pattern?
<details><summary>Show answer</summary>

> A functional approach replaces Template Method by making customization explicit
> and declarative, allowing clients to supply behavior directly without needing
> to understand or depend on the internal structure of the algorithm.

A **functional approach** provides an alternative to the **Template Method** pattern
by allowing behavior to be customized through **functions or callbacks** rather than inheritance and overridable hooks.


Instead of subclassing and overriding specific steps in an algorithm, the client **passes behavior explicitly**
(for example, a function, lambda, or callback) to a method that performs the overall process.

The algorithm remains internal and opaque, while the variable behavior is supplied as data.


**Key benefits over Template Method:**

- **Intent‑driven customization**  
  Clients express *what behavior they want*, not *where to hook into execution*.

- **No inheritance required**  
  Eliminates tight coupling to superclass structure and avoids fragile hierarchies.

- **Reduced cognitive load**  
  Clients do not need to understand internal control flow, hook timing, or invariants.

- **Stronger encapsulation**  
  Implementation details remain hidden; only the customization contract is exposed.

- **Better composability**
  Small functions can be combined, reused, or replaced independently.


| Template Method                            | Functional approach               |
|--------------------------------------------|-----------------------------------|
| Customization via overriding hooks         | Customization via passed behavior |
| Inheritance‑based                          | Composition‑based                 |
| Control‑flow‑centric                       | Intent‑centric                    |
| Client must understand algorithm structure | Client only supplies behavior     |
| Hard to evolve safely                      | Easier to evolve and refactor     |


</details>

### How to test a class designed for inheritance?
<details><summary>Show answer</summary>

The only way to test a class designed for inheritance is to write subclasses.
One or more of these subclasses should be written by someone other than the superclass author.

</details>
