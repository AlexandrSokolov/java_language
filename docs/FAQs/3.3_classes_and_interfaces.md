### What is inheritance, and what benefits does it provide?
<details><summary>Show answer</summary>

**Inheritance** is an object‑oriented mechanism that allows a class (subclass) to
reuse and extend the behavior of another class (superclass).


**Benefits of inheritance**:

- **Code reuse**  
  Common behavior can be defined once and reused across related classes.

- **Abstraction and hierarchy modeling**  
  Helps represent *is‑a* relationships in a domain model.

- **Polymorphism**  
  Enables treating related objects uniformly through a common superclass or interface.

- **Reduced duplication**  
  Shared logic is centralized, making code easier to maintain.

</details>

### In which cases is inheritance considered safe to use, and in which cases is it not?
<details><summary>Show answer</summary>

#### When inheritance is considered safe

- **Within the same package**  
  The superclass and subclass are controlled by the same team, allowing coordinated changes.

- **Classes designed for inheritance**  
  The superclass is explicitly documented and structured to be safely extended.

#### When inheritance is dangerous

- **Across package boundaries**  
  Ordinary concrete classes are not designed for subclassing and may change in ways that break subclasses.

</details>

### What types of inheritance exist?
<details><summary>Show answer</summary>

Inheritance can be divided into two main types:

- **Implementation inheritance**  
  A class extends another class and **inherits its implementation and behavior**.

- **Interface inheritance**  
  Either a class implements an interface, or one interface extends another interface; 
  **only method contracts are inherited, not implementation**.

</details>

### What are the main problems of using inheritance?
<details><summary>Show answer</summary>

- **Fragile base class problem**  
  Changes in a superclass implementation can break subclasses, even if the public API remains unchanged.

- [**Breaks encapsulation**](#what-is-the-relationship-between-inheritance-and-encapsulation-in-objectoriented-systems)  
  Subclasses often depend on superclass implementation details, not just its contract.

- **Tight coupling**  
  Subclasses are strongly bound to their superclasses, reducing flexibility and reuse.

- **Violates the "is-a" relationship**  
  Inheritance is often used where composition would better model the domain.

- **Limited evolvability**  
  Superclasses become hard to change once widely subclassed.

- **Inheritance of unwanted behavior**  
  Subclasses may inherit methods or state they do not need or should not expose.

- **Poor substitutability (LSP violations)**  
  Subclasses may not correctly preserve superclass behavior expectations.

- **Language-specific limitations (Java)**  
  Single class inheritance and final methods restrict design and extensibility.

</details>

### What is the relationship between inheritance and encapsulation in object‑oriented systems?
<details><summary>Show answer</summary>

Unlike method invocation, inheritance violates encapsulation. 
A subclass depends not only on the public contract of its superclass, 
but also on its internal implementation details for correct behavior.

As a result, changes in the superclass implementation — introduced, for example, in a new library release — 
can break subclasses even if their code remains unchanged. 
This creates a strong and often implicit coupling between superclass and subclass.

Consequently, subclasses are forced to evolve in tandem with their superclasses.
This dependency makes inheritance brittle and limits the ability to independently modify 
or evolve superclass implementations.

**Inheritance is therefore safe only when the superclass is explicitly designed and documented for extension, 
clearly defining which methods may be overridden and what guarantees are maintained for subclasses**.

</details>

### What are the consequences of extending a class by adding new methods while leaving existing methods unchanged?
<details><summary>Show answer</summary>

Even if a subclass only adds new methods and does not override existing ones,
this form of extension is still not risk‑free.

- **Superclass adds a new method with the same signature but a different return type**  
  The subclass no longer compiles due to a method signature conflict.

- **Superclass adds a new method with the same signature and return type**  
  The subclass unintentionally overrides it and is unlikely to satisfy the new
  method’s contract, since that contract did not exist when the subclass was written.

</details>

### How can object‑oriented designs mitigate the impact of inheritance on encapsulation?
<details><summary>Show answer</summary>

- **Favoring composition over inheritance**  
  Depend on object composition and delegation instead of inheriting implementation details.

- **Using inheritance only when explicitly designed for it**  
  Extend only classes that are documented for inheritance or are under the same control.

- **Programming to interfaces**  
  Depend on contracts rather than concrete implementations to preserve encapsulation.

</details>

### Which design pattern replaces inheritance with composition to extend behavior while preserving the original interface?
<details><summary>Show answer</summary>

The **Decorator** pattern replaces inheritance with composition to extend 
behavior while preserving the original interface.


The Decorator pattern works by **wrapping** an existing object rather than subclassing it. The wrapper:
- Implements the **same interface** as the wrapped object
- Delegates all operations to the wrapped object
- Adds behavior **before or after delegation**

Because the decorator depends only on the interface contract — not on implementation details —
it **preserves encapsulation** and avoids the fragility associated with inheritance.

Changes in the original class implementation do not affect the decorator, as long as the interface remains stable.

**Key benefits over inheritance**
- Avoids reliance on superclass implementation details
- Preserves substitutability (objects remain interchangeable)
- Prevents fragile base class problems
- Allows behavior to be added dynamically and selectively

</details>

### Can you give an example of the Decorator pattern?
<details><summary>Show answer</summary>

Component interface:
```java
public interface Set<E> {
  boolean add(E e);
  boolean addAll(Collection<? extends E> c);
}
```
Implementation:
```java
public class SimpleSet<E> implements Set<E> {
  private final java.util.Set<E> delegate = new java.util.HashSet<>();

  public boolean add(E e) {
    return delegate.add(e);
  }

  public boolean addAll(Collection<? extends E> c) {
    return delegate.addAll(c);
  }
}
```
Decorator (Wrapper):
```java
public class InstrumentedSet<E> implements Set<E> {
  private final Set<E> wrapped;
  private int addCount = 0;

  public InstrumentedSet(Set<E> wrapped) {
    this.wrapped = wrapped;
  }

  @Override
  public boolean add(E e) {
    addCount++;
    return wrapped.add(e);
  }

  @Override
  public boolean addAll(Collection<? extends E> c) {
    addCount += c.size();
    return wrapped.addAll(c);
  }

  public int getAddCount() {
    return addCount;
  }
}
```
Usage:
```java
Set<String> set = new InstrumentedSet<>(new SimpleSet<>());

set.add("A");
set.addAll(List.of("B", "C"));

System.out.println(((InstrumentedSet<?>) set).getAddCount()); // 3
```

</details>

### In which cases is it safe to use inheritance?
<details><summary>Show answer</summary>

- **When the superclass is explicitly designed and documented for inheritance**  
  Such classes define clear contracts for overriding behavior.

- **Within the same package or under the same control**  
  The superclass and subclass are maintained by the same team, allowing coordinated evolution.

- **When a true "is‑a" relationship exists between subclass and superclass**  
  Inheritance is appropriate only if the subclass is genuinely a subtype of the superclass.
  A class `B` should extend class `A` only if every `B` is truly an `A`.  
  If this cannot be truthfully stated, inheritance is inappropriate; 
  in such cases, composition should be preferred, with `B` containing an instance of `A` rather than extending it.

</details>

### What happens if inheritance is used where composition would be more appropriate?
<details><summary>Show answer</summary>

Using inheritance where composition is appropriate can expose implementation details that should remain hidden. 
This unnecessarily ties the API to a specific superclass implementation, limiting future evolution and optimization.

Exposing superclass internals allows clients to interact with them directly, often leading to confusing semantics. 
For example, in `Properties`, calling `getProperty(key)` behaves differently from `get(key)` because the latter,
inherited from `Hashtable`, ignores default values.

More seriously, direct access to superclass behavior can violate subclass invariants. 
Although `Properties` is intended to store only string keys and values, 
access to the underlying `Hashtable` allows non‑string entries. 
Once this invariant is broken, parts of the `Properties` API (such as `load` and `store`) no longer work correctly. 
By the time this flaw was discovered, it could not be fixed without breaking existing clients.

</details>
