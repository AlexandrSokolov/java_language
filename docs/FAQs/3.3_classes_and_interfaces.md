## Classes and Interfaces - Inheritance

### What is inheritance, and what benefits does it provide?
<details><summary>Show answer</summary>

**Inheritance** lets a class (subclass) reuse and extend the behavior of another (superclass).

**Benefits:**

- **Code reuse** — common behavior defined once in the base class, reused by subclasses; 
  less duplication, easier maintenance.
- **Abstraction & hierarchy modeling** — expresses *is-a* relationships; 
  this is about how you **structure** your types.
- **Polymorphism** — follows *from* that hierarchy: it's what you can do **at runtime** because of the
  structure — treating related objects uniformly through a common supertype.

The last two are a pair: hierarchy is the compile-time structure, polymorphism is the runtime payoff it enables.

</details>

### What is polymorphism?
<details><summary>Show answer</summary>

**Polymorphism** ("many forms") — a single name, operation, or interface that works across multiple types, 
where the behavior depends on the actual type involved.

The point: write code once against a general form, and have it do the right thing for many concrete types — 
without the calling code knowing which type it's dealing with.

</details>

### What forms of polymorphism exist?
<details><summary>Show answer</summary>

Four kinds, in two families:

**Universal** — works uniformly over an open-ended range of types:
- **Subtype (inclusion)** — a subclass instance stands in for its supertype; resolved at **runtime**.
  The kind inheritance provides — [details](#how-do-inheritance-and-polymorphism-relate).
- **Parametric** — one piece of code written without naming a concrete type, usable across many (generics, `List<T>`); 
  resolved at **compile time**.

**Ad-hoc** — works over a fixed, finite set of types, each handled specifically:
- **Overloading** — same method name, different parameter types; resolved at **compile time**.
- **Coercion** — implicit type conversion (e.g. `int` → `double` in mixed arithmetic).

</details>

### How do inheritance and polymorphism relate?
<details><summary>Show answer</summary>

Inheritance provides **subtype polymorphism** specifically — the *runtime* kind. 
A subclass instance can be used wherever its supertype is expected, and a call to an overridden method runs the
**subclass's** implementation, chosen by the object's actual runtime type:

```java
Shape s = new Circle();   // supertype reference, subtype object
s.draw();                 // runs Circle.draw(), not Shape.draw()
```

**The mechanism — dynamic dispatch via the vtable.** Overridable methods live in a per-class virtual method table; 
the call is bound at runtime by looking up the object's actual class and invoking the matching entry. 
This is why the supertype reference still reaches the subclass's override. 
(Full mechanism, and which access levels participate:
[which methods are overridable, and how they're resolved](3.1_classes_and_interfaces.md#which-methods-are-overridable-and-how-are-they-resolved).)

**The connection:** the hierarchy is the *compile-time structure* (the is-a relationship); 
subtype polymorphism is the *runtime payoff* that structure enables — 
overriding + dynamic dispatch let you treat related objects uniformly through a common supertype.

</details>

### When is inheritance appropriate, and when should it be used cautiously or avoided?
<details><summary>Show answer</summary>

Four criteria decide it — inheritance is appropriate when **all** hold, and suspect when **any** is
violated:

- **True is-a relationship** — a real conceptual *is-a*, not just shared code.
  *Violation:* using inheritance only for code reuse → accidental, semantically wrong inheritance.
- **Stability** — the base class is a reliable, settled abstraction.
  *Violation:* deep/rigid hierarchies tightly couple subclasses to the base, so base changes break large
  parts of the system.
- **Substitutability (LSP)** — subclasses can stand in for the base everywhere.
  *Violation:* if a subclass must override, disable, or significantly alter base behavior, it's no
  longer safely substitutable.
- **Intentional polymorphism** — polymorphic use is a deliberate design goal.
  *Violation:* behavior varies along several independent axes — inheritance forces a combinatorial tree
  and doesn't scale. [→ modeling multiple axes with inheritance](#why-does-inheritance-fail-when-an-objects-behavior-varies-along-several-independent-axes)

**Classic warning sign:** *"this class extends another just to override half of its behavior."*

</details>

### Why does inheritance fail when an object's behavior varies along several independent axes?
<details><summary>Show answer</summary>

An **axis** is one dimension along which objects differ. 
Inheritance can only express **one** is-a chain, so it forces you to pick a single primary axis for the hierarchy. 
When behavior actually varies along several *independent* axes at once, 
every axis multiplies the others → **class explosion**.

Example — a coffee drink varies along four independent axes:
- base: Espresso / Tea / HotChocolate
- milk: none / whole / oat / soy
- size: small / medium / large
- extras: caramel / whip / extra shot

Modeled with inheritance you'd need a class per *combination*: `LargeOatEspressoWithCaramel`,
`SmallSoyTeaWithWhip`, … — 3 × 4 × 3 × … classes. Add one new milk and a whole branch of the tree doubles.

The root problem: the axes are **independent** (milk has nothing to do with size), 
but inheritance binds them into one rigid tree — so changes that should be local ripple across the hierarchy. 
Inheritance is the right tool for *one* is-a axis; the moment there are two or more independent ones, it doesn't scale.

</details>

### How is the multiple-axes modeling problem solved?
<details><summary>Show answer</summary>

By **composition** — assemble the object from independent pieces (one per axis) instead of one rigid inheritance tree. 
Each axis becomes a field you vary freely, so adding an option is one new value, not a new branch.

Three named patterns are specialized forms of this:
- **Strategy** — a pluggable behavior chosen at runtime (the "how it behaves" axis).
- **Decorator** — wraps an object to stack behavior on a base, same interface (the "extras" axis).
- **Template Method** — fixes the skeleton, lets subclasses fill steps (the inheritance-based cousin).

**The functional lens.** Functional programming exposes that several of these patterns are, at heart,
*just passing a function* — once functions are first-class, you don't need a class hierarchy to vary behavior:
- **Strategy → a function parameter.** Instead of a `Strategy` interface + implementing classes, 
  pass the algorithm directly: `input -> input * input`. The interface collapses into a function type.
- **Decorator → a wrapper function.** A decorator that takes a function and returns an enhanced one 
  (e.g. add logging around it) is function composition — `withLogging(f)` rather than a wrapper class.
- **Template Method → a higher-order function** that takes the varying steps as function arguments.

So functional programming reframes a chunk of the GoF catalog: 
where OOP reaches for a class to vary behavior along an axis, 
FP often just passes a function for that axis — same goal (compose independent variations), far less ceremony. 
The patterns don't disappear; they shrink to functions.

</details>

### When is it safe to subclass a class, and when does subclassing introduce significant risk?
<details><summary>Show answer</summary>

This is a question of **control**, not semantics: 
safety depends on whether the superclass author intended and supports extension. 
The rule: subclass only across a boundary you **control** or that has been **explicitly opened** for it.

**Safe:**
- **Package-private superclass** — *structurally* safe: 
  it can't be subclassed from outside the package at all (compiler-enforced), so no external subclass can ever exist.
- **Public superclass, subclassed within the same package** — safe by *organizational* control: 
  it *could* be extended from other packages, but your same-package subclass is coordinated because one
  team owns both files and changes them together. (The base/subclass coupling is managed, not prevented.)
- **Designed for inheritance** — the superclass is explicitly documented and structured to be extended safely 
  (self-use documented, hooks provided).

**Dangerous:**
- **Across package boundaries, on an ordinary concrete class** — such a class was *not* designed for subclassing. 
  Its author owes you no stability guarantee about internal behavior, 
  so a later release can change self-use and silently break your subclass.

This is why the guidance is: *design and document a class for inheritance, or else prohibit it*
(make it `final`, or use non-public constructors).

</details>

### What are the main problems of using inheritance? What structural costs does inheritance introduce?
<details><summary>Show answer</summary>

One root problem: a subclass depends on the *implementation details* of its superclass, not just its public contract. 
This [**weakens encapsulation**](#what-is-the-relationship-between-inheritance-and-encapsulation)
and couples the two through internals — so an internal change to the superclass can break the subclass
**even when its public API is unchanged**, and the whole hierarchy becomes fragile to change.

It needs no large hierarchy to appear — a *single* subclass already exhibits it. 
Scale doesn't change the defect, only its cost: 
the more widely a superclass is subclassed (across packages, modules, teams),
the more code is pinned to its internals, and the riskier the base becomes to evolve.

</details>

### What is the relationship between inheritance and encapsulation?
<details><summary>Show answer</summary>

**Inheritance weakens encapsulation.** A subclass depends not only on the *public contract* of its
superclass but on its *internal implementation* — most often the superclass's **self-use**: which of
its own overridable methods it calls, and when. That self-use isn't part of the public API, yet a
subclass that overrides those methods is silently coupled to it.

The consequence: a change to the superclass's internal implementation can break a subclass **even
though the subclass's code is unchanged and the superclass's public API is unchanged**. The coupling is
strong but *implicit* — invisible at both the public contract and the subclass source. Subclasses are
forced to evolve in lockstep with superclass internals.

This is why inheritance is safe **only** when the superclass is explicitly *designed and documented for
extension* — its self-use spelled out, its overridable methods and the guarantees they preserve made
part of the contract. Absent that, the subclass is reaching through a seam the language left open but
the author never promised to keep stable.

</details>

### What types of inheritance exist?
<details><summary>Show answer</summary>

Two kinds:

- **Implementation inheritance** — a class extends another class and inherits its *implementation and
  behavior* (`extends`).
- **Interface inheritance** — a class implements an interface, or an interface extends another;
  *only the method contracts* are inherited, no implementation.

Why the distinction matters: the encapsulation-weakening problem is almost entirely a property of **implementation** 
inheritance — there's an implementation to depend on, so the subclass can couple to internals. 
**Interface** inheritance carries no implementation, so there's nothing internal to couple to — 
which is a large part of why interfaces are the safer extension mechanism.

</details>

### What are the consequences of extending a class by adding new methods while leaving existing methods unchanged?
<details><summary>Show answer</summary>

Even purely *additive* subclassing — only adding new methods, never overriding — is not risk-free. 
The danger comes from a **future change to the superclass** that retroactively collides with your added
method, breaking code that was correct when written:

- **Superclass later adds a method with the same signature but a different return type** — your
  subclass no longer compiles (signature conflict).
- **Superclass later adds a method with the same signature and return type** — your method silently
  *overrides* it, and almost certainly fails to honor the new method's contract — a contract that
  didn't exist when you wrote the method.

The deeper point: you can be broken by a change you didn't make and couldn't have anticipated. 
Adding methods feels safe but still couples you to the superclass's *future* evolution.

</details>

### How can object-oriented designs mitigate the impact of inheritance on encapsulation?
<details><summary>Show answer</summary>

In priority order — the first is the default, the others are for when you must inherit or want to
reinforce the boundary:

1. **Favor composition over inheritance.** The primary answer: hold the other object as a field and
   delegate to it, instead of inheriting its implementation. You depend only on its public contract,
   so its internals can't couple to you — the encapsulation-weakening problem simply doesn't arise.
2. **Inherit only when the class is designed for it.** When you genuinely need inheritance, extend only
   classes that are *documented for extension* or *under your own control* — so the self-use you depend
   on is part of the contract, not an accident.
3. **Program to interfaces, not implementations.** Depend on contracts rather than concrete types, so
   clients are insulated from implementation and you retain freedom to change it.

The thread: #1 *avoids* the coupling, #2 *contains* it to a boundary the author supports, #3 *limits*
how far any implementation dependency can spread.

</details>

### If composition is preferable, why not avoid inheritance entirely?
<details><summary>Show answer</summary>

Because "prefer composition" is a *default*, not an absolute — and inheritance retains genuine
advantages composition can't replicate cleanly. The guidance is narrower than it sounds: it warns
against **implementation inheritance across a boundary you don't control**, not against all inheritance.

Where inheritance is still the right tool:

- **A genuine, stable is-a relationship.** When the subtype truly *is* the supertype and substitutes
  for it everywhere (LSP holds), inheritance models it directly. Forcing composition here adds
  indirection to express a relationship the language already has a keyword for.
- **Polymorphism is the goal.** If the point is to treat many subtypes uniformly through a supertype
  reference, inheritance (or interface implementation) is the mechanism that provides it. Composition
  doesn't give you subtype substitutability.
- **Designed and documented for extension.** When the superclass author has opened the class for it —
  self-use documented, hooks provided — the coupling is part of the contract, so the main objection
  (depending on undocumented internals) goes away.
- **Less boilerplate for true specialization.** Composition requires forwarding every delegated method;
  for a real subtype that extends a stable base, inheritance avoids that forwarding cost.

What composition buys that inheritance can't — and why it's the *default*:

- It avoids the encapsulation-weakening coupling entirely (you depend only on a public contract).
- It composes along **multiple independent axes** without combinatorial class explosion.
- The wrapped object can be swapped or chosen at runtime; an inherited superclass is fixed at
  compile time.

So the rule isn't "never inherit" — it's "**don't reach for inheritance by default for code reuse.**"
Use it when there's a real is-a, substitutability holds, or the class is built for extension; reach for
composition for everything else — which, in practice, is most of the time.

</details>

### How can behavior be added to an object without extending its class?
<details><summary>Show answer</summary>

The **Decorator** pattern. Instead of subclassing to add behavior, you *wrap* the object in another
that implements the **same interface**, adds its own behavior, and delegates the rest to the wrapped
instance.

Because the decorator depends only on the **interface contract** — never the wrapped object's
internals — it sidesteps the fragile-base-class problem entirely and preserves encapsulation. It's the
composition answer to "add behavior" that the previous cards' guidance points toward.

</details>

### Can you give an example of the Decorator pattern?
<details><summary>Show answer</summary>

```java
// Common interface
public interface Coffee {
  double cost();
  String description();
}

// Base implementation
public class SimpleCoffee implements Coffee {
  @Override public double cost()        { return 2.0; }
  @Override public String description() { return "Simple coffee"; }
}

// Abstract decorator — holds and implements the same interface
public abstract class CoffeeDecorator implements Coffee {
  protected final Coffee coffee;
  protected CoffeeDecorator(Coffee coffee) { this.coffee = coffee; }
}

// Concrete decorators — add behavior, delegate the rest
public class MilkDecorator extends CoffeeDecorator {
  public MilkDecorator(Coffee coffee) { super(coffee); }
  @Override public double cost()        { return coffee.cost() + 0.5; }
  @Override public String description() { return coffee.description() + ", milk"; }
}

public class SugarDecorator extends CoffeeDecorator {
  public SugarDecorator(Coffee coffee) { super(coffee); }
  @Override public double cost()        { return coffee.cost() + 0.2; }
  @Override public String description() { return coffee.description() + ", sugar"; }
}
```

Stacking — wrap a base object in successive decorators; each adds its bit and delegates the rest:
```java
Coffee coffee = new SimpleCoffee();
coffee = new MilkDecorator(coffee);
coffee = new SugarDecorator(coffee);

coffee.description();  // "Simple coffee, milk, sugar"
coffee.cost();         // 2.7

// equivalently, nested in one expression:
coffee = new SugarDecorator(new MilkDecorator(new SimpleCoffee()));
```

`Milk` and `Sugar` combine freely without a `MilkSugarCoffee` class — the "extras" axis composing
without class explosion. Each decorator depends only on the `Coffee` interface, never on another
decorator's internals, so encapsulation holds.

</details>

### Compare decoration with inheritance
<details><summary>Show answer</summary>

Three ways to add behavior — inheritance, OOP Decorator, and the functional form:

**Inheritance**
- *Pro:* simple, direct; behavior fixed at compile time; one object, easy to debug.
- *Con:* a separate class per combination → **class explosion** (`CoffeeWithMilkAndSugar`, …);
  combinations fixed at compile time; couples to superclass internals.

**Decorator (OOP composition)**
- *Pro:* combine behaviors by **stacking** at runtime — `Milk` + `Sugar` with no `MilkSugarCoffee`
  class; open-ended; depends only on the interface, so encapsulation holds.
- *Con:* not free — many small wrapper objects; the wrapped object's **identity is lost** (`==`/
  `instanceof` see the decorator); debugging steps through delegation layers; order-sensitive
  decorators must be applied in the right order.

**Functional alternative**
A decorator is conceptually a function that takes the thing and returns an enhanced version of it. 
When the decorated type is data, model each decoration as a `Coffee -> Coffee` transform and **compose** them — 
no class hierarchy at all:
```java
UnaryOperator<Coffee> milk  = c -> new Coffee(c.cost() + 0.5, c.description() + ", milk");
UnaryOperator<Coffee> sugar = c -> new Coffee(c.cost() + 0.2, c.description() + ", sugar");

Coffee result = milk.andThen(sugar).apply(new Coffee(2.0, "Simple coffee"));
```
Stacking decorators becomes `andThen` composition; adding a feature is a new function, not a new class.
- *Pro:* least ceremony — no interface, no abstract decorator, no concrete classes; composition is
  built into the function type.
- *Con:* fits best when the decorated thing is a value/function; if decorators need rich state or must
  expose a full multi-method interface, the OOP Decorator is clearer.

**When each wins:** 
- inheritance for a small, fixed set along *one* axis; 
- OOP Decorator when features combine freely and need a shared interface; 
- functional composition when the decoration is a simple transform and you want minimal ceremony.

</details>

### A JDK class where inheritance violated the subclass's own invariants?
<details><summary>Show answer</summary>

`java.util.Properties extends Hashtable` — a classic mistake.

The root error: a `Properties` object isn't a general hashtable, it *has* one — so the is-a was false;
it should have been composition. Because it inherited instead:

- **Leaked, conflicting methods** — inherited `get(key)` behaves differently from `Properties`'
  own `getProperty(key)`, giving two confusingly different access paths.
- **Broken invariants** — `Properties` is meant to hold only string keys and values, but inherited
  `Hashtable` methods (`put(Object, Object)`) let clients insert arbitrary objects, violating that
  guarantee.

And once clients depended on the inherited behavior, the flaw was frozen — unfixable without breaking
backward compatibility. Composition (`Properties` *holds* a map, exposes only string operations) would
have avoided all of it.

</details>

### What design considerations are required to make a class appropriate for inheritance?
<details><summary>Show answer</summary>

A class is fit for inheritance only if it's *intentionally* designed, *documented*, and *robust* under extension. 
The duties group into three kinds:

**Document the extension contract**
- **Specify what may be overridden and what invariants subclasses must preserve** — the extension contract itself.
- [**Document self-use of overridable methods**](#how-does-a-class-document-its-self-use-of-overridable-methods)
  — every place the class calls its own overridable methods, so subclasses know when their overrides fire.

**Design safe extension points**
- [**Provide `protected` hooks, not exposed internals**](3.4_classes_and_interfaces.md#how-is-the-tension-between-extensibility-and-encapsulation-resolved-in-inheritance)
  — well-defined extension points instead of leaking implementation.
- **Keep the API minimal and stable** — every `public`/`protected` member becomes part of the subclass API forever.
- [**Never call an overridable method from a constructor**](#what-happens-if-a-constructor-invokes-an-overridable-method)
  — the subclass's override runs before the subclass is initialized.

**Protect and verify**
- **Preserve and enforce invariants** — subclasses must not be able to break internal consistency.
- [**Design and test for subclassing**](#how-to-test-a-class-designed-for-inheritance) — 
  confirm correct behavior when methods are overridden and state extended.

</details>

### How does a class document its self-use of overridable methods?
<details><summary>Show answer</summary>

Why it's needed: self-use is the *hidden* coupling that makes inheritance fragile. 
Documenting it converts that hidden dependency into an **explicit part of the contract** — 
which is the whole point of "designing for inheritance."

The mechanism: if a method calls one or more overridable methods, document *when* and *how*, 
so subclass authors know exactly when their overrides will fire. This goes in a dedicated Javadoc section,
**Implementation Requirements**, via the `@implSpec` tag — unlike the normal spec (what the method does), 
this describes the method's **internal workings**.

```java
/**
 * Removes a single instance of the specified element from this collection,
 * if it is present.
 *
 * @implSpec
 * This implementation iterates over the collection looking for the specified
 * element. If it finds the element, it removes it using the iterator's
 * {@code remove} method.
 *
 * @throws UnsupportedOperationException if the iterator does not support
 *         {@code remove}
 */
public boolean remove(Object o) { ... }
```

This makes explicit that `remove` depends on `iterator()`, shows how overriding `iterator()` affects
`remove`, and tells subclass authors exactly which methods influence each other.

</details>

### A JDK class whose undocumented self-use breaks subclasses?
<details><summary>Show answer</summary>

`HashSet`. Its `addAll` internally calls `add`, but this isn't documented — 
so a subclass overriding both to count additions double-counts: `addAll` of 3 elements reports 6.

</details>

### How does documenting implementation details for inheritance fit general API doc principles?
<details><summary>Show answer</summary>

It breaks the usual rule. Good docs describe **what** a method does, not **how** — 
but designing for inheritance forces you to document the **how** (self-use), 
because inheritance weakens encapsulation and subclasses depend on those internals. 
The violation is unavoidable, and it's the price of safe subclassing.

</details>

### How to test a class designed for inheritance?
<details><summary>Show answer</summary>

Write subclasses — there's no other way to exercise the extension points.
Critically, have *someone other than the superclass author* write at least one:
an outsider surfaces missing hooks and undocumented assumptions the author can't see in their own design.

</details>

### What happens if a constructor invokes an overridable method?
<details><summary>Show answer</summary>

The superclass constructor runs **before** the subclass constructor, so an overridable method called
from the superclass constructor executes the subclass's override **on a partially constructed object** —
before the subclass's fields are initialized.

The override then sees uninitialized state and misbehaves. The sharp version: a subclass field, *even a
`final` one*, is still `null`/zero when the override reads it, because the subclass's field initializers
and constructor body haven't run yet.

**Rule: constructors must not invoke overridable methods, directly or indirectly.**

</details>

### In which situations — besides constructors — should overridable methods not be invoked?
<details><summary>Show answer</summary>

The constructor rule generalizes: **any method that acts like a constructor** — running on a
not-yet-fully-initialized object — must not invoke overridable methods.
In a class designed for inheritance, that means:

- **`clone`** (from `Cloneable`) — runs before the clone's state is restored.
- **`readObject`** (from `Serializable`) — runs before the deserialized object's state is set.

Both can execute on a **partially initialized object**, so calling an overridable method from them fires
subclass code too early — the same broken-invariant trap as constructors.

</details>
