### Which `Object` methods should typically be overridden, and what problems can arise if they are not?
<details><summary>Show answer</summary>

In practice, `equals` and `hashCode` are the most critical overrides, 
as violating their contracts leads to subtle and hard-to-diagnose bugs in collections, 
while `toString` improves observability, and `compareTo` establishes consistent ordering semantics.

- `equals(Object)` - determines logical equality between objects.

  **If not overridden**: equality falls back to reference comparison (==), 
  which often breaks logical comparisons and causes collections like Set or Map to behave incorrectly 
  (e.g. duplicate logical objects in a Set)
- `hashCode()` - returns a hash value consistent with equals.

  **If not overridden (when equals is overridden)**: objects that are logically equal may end up 
  in different hash buckets, breaking hash-based collections such as `HashMap` and `HashSet`  
- `toString()` - provides a human-readable string representation of the object.

  **If not overridden**: the default implementation prints the class name and a hash code, 
  which is usually useless for logging, debugging, or diagnostics.  
- `clone()` - creates and returns a copy of the object.

  **If not overridden (but used)**: cloning may not work as intended, may result in shallow copies, 
  or may not be available at all (`CloneNotSupportedException`), leading to broken object copying semantics.
- `finalize()` - **deprecated in modern Java** - allows cleanup before garbage collection.

  **overriding `finalize` is discouraged**; modern Java recommends try-with-resources and explicit cleanup instead.
- `Comparable.compareTo` - not an `Object` method, but similar in nature - defines a natural ordering for objects
  **If not implemented**: objects cannot be sorted naturally, and usage in sorted collections (TreeSet, TreeMap) 
  will fail or behave unexpectedly.

</details>

### When is it appropriate **not** to override equals?
<details><summary>Show answer</summary>

There are several situations in which overriding equals is unnecessary 
and the default identity-based behavior from Object is appropriate:

-  **Each instance is inherently unique**. 

  This applies to classes that represent _active entities_ rather than values.
  For example, `Thread` instances represent independent executions and are not interchangeable. 
  In such cases, two objects should only be considered equal if they are the same instance.
  The `equals` implementation inherited from Object provides exactly the correct semantics here.

- **The class does not need logical equality**. 

  Some classes conceptually represent values but do not benefit from defining logical equality.
  For example, `java.util.regex.Pattern` could have overridden `equals` to compare the underlying regular expressions, 
  but its designers chose not to. Clients typically do not need or expect such comparisons, 
  and identity-based equality is sufficient.

- **A superclass already provides a suitable equals implementation**

  If a superclass has overridden `equals` and its behavior is correct for the subclass, 
  there is no need to override it again.
  For example, most `Set` implementations inherit equals from `AbstractSet`, `List` implementations from `AbstractList`, 
  and `Map` implementations from `AbstractMap`.

- **The class is private or package-private and equality is never used**

  If a class is not exposed outside its package (or enclosing class) and its instances are never compared for equality, 
  overriding equals is unnecessary.

</details>

### When is it appropriate to override `equals`?
<details><summary>Show answer</summary>

It is when a class has a notion of logical equality that differs from mere object identity 
and a superclass has not already overridden equals. 
This is generally (but not always) the case for _value classes_.

Don’t override the equals method unless you have to!

### What is the difference between active‑entity classes and value classes? Give examples.
<details><summary>Show answer</summary>

- active-entity classes - represents an **active process**, **identity matters**, use identity equality.

  - Examples that represents a distinct execution unit: 
    `Thread`, `Runnable` (when used as a task instance), `ExecutorService`
  - Examples that represent stateful coordination mechanisms, not values:
    `Lock`, `Semaphore`, `CountDownLatch`, `Condition`
  - System / resource handles: `ClassLoader`, `Process`, `Socket`
- value classes - represents a **value**, content matters, use logical (value) equality 

</details>

### What is the contract of the equals() method?
<details><summary>Show answer</summary>

The `equals` method implements an equivalence relation. It must satisfy the following properties:
- **Reflexive**: For any non-null reference value x, `x.equals(x)` must return true.
- **Symmetric**: For any non-null reference values x and y,
  `x.equals(y)` must return true if and only if `y.equals(x)` returns true.
- **Transitive**: For any non-null reference values x, y, z,
  if `x.equals(y)` returns true and `y.equals(z)` returns true, then `x.equals(z)` must return true.
- **Consistent**: For any non-null reference values x and y,
  multiple invocations of `x.equals(y)` must consistently return true or consistently return false,
  provided no information used in `equals` comparisons is modified.
- For any non-null reference value x, x.equals(null) must return false.

</details>

### What considerations should be taken into account when implementing equals in value classes?
<details><summary>Show answer</summary>

1. **Value classes represent a value**: their content matters, 
  and equality is typically based on **logical (value‑based) comparison** rather than object identity. 
  In most cases, such classes should override equals to compare their significant fields.
2. **There are important exceptions**.
   One category of value classes does **not** require overriding equals: 
   classes that use **instance control** to guarantee that at most one instance exists for each logical value. 
   Enum types are the canonical example of this approach.

   For such classes, **logical equality coincides with object identity**. 
   Consequently, the `equals` method inherited from `Object` already provides correct logical equality, 
   and overriding it is unnecessary.

3. [**Adding a value component through inheritance is problematic**.](#what-should-you-consider-regarding-the-equality-contract-when-creating-subclasses)

   It is not possible to extend a **concrete, instantiable class** and **add a new value component** 
   while fully preserving all requirements of the equals contract (in particular, symmetry and transitivity). 
   
   In such cases, you should [favor composition over inheritance](#how-should-valuesubclass-equality-problems-be-handled).

</details>

### What should you consider regarding the equality contract when creating subclasses?
<details><summary>Show answer</summary>

This is a fundamental limitation of equivalence relations in object‑oriented languages.

An instantiable class cannot be safely extended with an additional value component while preserving the equals contract, unless one is willing to abandon key benefits of object‑oriented abstraction.

Assume there is a class `Point` and a concrete, instantiable subclass `ColorPoint`:
```java
public class Point {
  private final int x;
  private final int y;
  ...
  @Override public boolean equals(Object o) {
    if (!(o instanceof Point))
      return false;
    Point p = (Point)o;
    return p.x == x && p.y == y;
  }
}
public class ColorPoint extends Point {
  private final Color color;
  ...
  @Override public boolean equals(Object o) {
    if (!(o instanceof ColorPoint))
      return false;
    return super.equals(o) && ((ColorPoint) o).color == color;
  }
}
```
`ColorPoint.equal` violates symmetry.

You can try to fix it as:
```java
  @Override public boolean equals(Object o) {
    if (!(o instanceof Point))
      return false;
    // If o is a normal Point, do a color-blind comparison
    if (!(o instanceof ColorPoint))
      return o.equals(this);
    // o is a ColorPoint; do a full comparison
    return super.equals(o) && ((ColorPoint) o).color == color;
  }
```
While this approach preserves the symmetry requirement, it violates transitivity.


Note that it is possible to add a value component to a subclass **without violating the equals contract**, 
provided the superclass is **abstract and non‑instantiable**.


For this reason, **class hierarchies are preferable to tagged classes**. 
For example, you might define an abstract class `Shape` with **no value components**, 
and concrete subclasses such as `Circle`, which adds a `radius`, and `Rectangle`, which adds `length` and `width`.


</details>

### How can the transitivity requirement of `equals` be violated?
<details><summary>Show answer</summary>

The transitivity requirement of equals can be violated when 
**a concrete, instantiable subclass adds a new value component** that affects equality comparisons.

</details>

### How should value‑subclass equality problems be handled?
<details><summary>Show answer</summary>

Value‑adding subclasses should use composition, not inheritance, to preserve the equals contract.

Instead of having `ColorPoint` extend `Point`, define `ColorPoint` to contain a private `Point` field, 
and provide a public view method that returns the point corresponding to the same position as this color point:
```java
// Adds a value component without violating the equals contract
public class ColorPoint {
  private final Point point;
  private final Color color;
  public ColorPoint(int x, int y, Color color) {
    point = new Point(x, y);
    this.color = Objects.requireNonNull(color);
  }
  /**
   * Returns the point-view of this color point.
   */
  public Point asPoint() {
    return point;
  }
  @Override public boolean equals(Object o) {
    if (!(o instanceof ColorPoint))
      return false;
    ColorPoint cp = (ColorPoint) o;
    return cp.point.equals(point) && cp.color.equals(color);
  }
}
```

Note: You can no longer compare `Point` and `ColorPoint`.
Even though they represent the same geometric location.
This is intentional. This is not a bug, but a trade‑off.

</details>

### Give an example from the Java standard library where inheritance was a poor design choice.
<details><summary>Show answer</summary>

A well-known example in the Java standard library is `java.sql.Timestamp`, 
which extends `java.util.Date` and adds a `nanoseconds` field. 
Its `equals` implementation violates the **symmetry** requirement of the `equals` contract: 
a Timestamp may be equal to a Date, while the reverse comparison is false.


This can lead to unpredictable behavior when Date and Timestamp instances are mixed in collections. 
Although the documentation warns against such usage, nothing prevents it at compile time, 
making errors subtle and hard to debug. 
This design is widely regarded as a mistake and illustrates why extending instantiable value classes should be avoided.

</details>


### How can the consistency property of `equals` be violated, and what should be considered to avoid it?
<details><summary>Show answer</summary>

**Consistency** means that if two objects are equal, repeated calls to `equals` must continue 
to return the same result unless one of the objects is modified.


To satisfy this requirement, an `equals` method must not depend on **mutable or unreliable external resources**. 


For example, `java.net.URL.equals` relies on hostname resolution, 
which may require network access and can yield different results over time. 
As a result, URL equality may change without the objects themselves changing, violating consistency. 

This design is considered a mistake and should not be emulated.
In general, equals should rely only on **deterministic, in‑memory data**.

</details>

Non‑nullity means that an object must never be considered equal to null.

### What are the key implementation steps for writing a high‑quality `equals` method?
<details><summary>Show answer</summary>

1. **Check for reference equality (`==`).**  
   Use the `==` operator to check whether the argument is a reference to **this object**.  
   If so, return `true`. This is a performance optimization that is worthwhile when the
   equality comparison may be expensive.

2. **Check the argument’s type.**  
   Use the **`instanceof` operator** to verify that the argument has the **correct type**.
   If not, return `false`.  
   Typically, the correct type is the **class** in which `equals` is defined. Occasionally,
   it may be an **interface** that refines the `equals` contract to permit comparisons across
   implementations (for example, `Set`, `List`, `Map`, or `Map.Entry`).

3. **Cast to the correct type.**  
   Cast the argument to the correct type. Because this cast is preceded by an
   **`instanceof` check**, it is **guaranteed to succeed**.

4. **Compare all significant fields.**  
   For each **significant field** in the class, compare the corresponding field of the
   argument with this object’s field

   If any comparison fails, return `false`; otherwise, return `true`.  
   If the type checked is an **interface**, access fields via **interface methods**.
   If it is a **class**, fields may be accessed directly, depending on their accessibility.

5. **Handle `null` safely.**  
   Some object reference fields may legitimately contain `null`. To avoid
   `NullPointerException`, compare such fields using  
   **`Objects.equals(Object, Object)`**.

6. **Remain consistent with `hashCode`.**  
   If `equals` is overridden, **`hashCode` must also be overridden**, ensuring that equal
   objects produce the same hash code.

</details>

### How should objects of different types be compared in an equals method?
<details><summary>Show answer</summary>

- For **primitive fields** (except `float` and `double`), use the `==` operator.
- For **object reference fields**, compare them recursively using `equals`.
- For **`float` fields**, use `Float.compare(float, float)`.
- For **`double` fields**, use `Double.compare(double, double)`.

  Although `Float.equals` and `Double.equals` could be used, they require
  **autoboxing**, which incurs unnecessary performance overhead.

- For **array fields**, apply these rules to each element.
  If all elements are significant, use the appropriate
  `Arrays.equals` method.

</details>

### What are the performance considerations when implementing equals?
<details><summary>Show answer</summary>

- Compare fields in an order that improves performance:
    - fields that are **cheap to compare** or **likely to differ** should be checked first.

- Do **not** compare fields that are **not part of the logical state**, such as synchronization locks.

- **Derived fields** do not need to be compared, but comparing them first can improve performance.
  If a derived field summarizes the object’s state, it can allow early inequality detection.

  **Example:**  
  If a `Polygon` caches its area, unequal areas immediately imply inequality,
  avoiding expensive comparisons of edges or vertices.

</details>


### What steps should be taken after completing an equals method implementation?
<details><summary>Show answer</summary>

1. **Verify the contract.**  
   Ask yourself:
    - Is it **symmetric**?
    - Is it **transitive**?
    - Is it **consistent**?

2. **Write tests.**  
   Don’t rely on reasoning alone — write **unit tests** to verify the contract.
   An exception is when `equals` is generated by a proven tool such as **AutoValue**,
   in which case additional tests are usually unnecessary.

3. **Keep it simple.**  
   Avoid clever or overly aggressive equality definitions.
   Comparing the object’s significant fields for equality is usually sufficient.
   Attempts to account for aliasing or external equivalence can easily lead to
   contract violations.

   **Example:**  
   `File.equals` should not attempt to treat symbolic links to the same file as equal.
   Fortunately, it does not.

</details>

### What tools can be used to generate a correct equals implementation?
<details><summary>Show answer</summary>

Several tools and language features can be used to generate correct and reliable
implementations of `equals` (and `hashCode`), avoiding common mistakes.

- **Java records**  
  Records automatically generate `equals`, `hashCode`, and `toString` based on
  their components. They are ideal for immutable value classes with a fixed set
  of fields and provide the safest and most concise solution.

- **Lombok (`@EqualsAndHashCode`)**  
  Lombok can generate `equals` and `hashCode` at compile time using annotations.
  While convenient, it introduces a build-time dependency and relies on annotation
  processing, which may not be acceptable in all projects.

- **IDE code generation**  
  Most IDEs can generate `equals` and `hashCode` methods automatically.
  This is generally preferable to writing them by hand, as IDEs do not make
  careless mistakes. However, the generated code is often verbose, does not adapt
  automatically to future field changes, and therefore requires maintenance and
  testing.

- **Google AutoValue**  
  AutoValue generates immutable value classes with correct `equals` and `hashCode`
  implementations. The generated code is concise, reliable, and automatically
  updated when the class definition changes. It is widely regarded as one of the
  best options for complex value classes.

> **Note:** Automatically generated implementations are usually safer than
> handwritten ones, as they reduce the risk of violating the `equals` contract.


</details>
