### Object methods — which to override, and what breaks if you don't?
<details><summary>Show answer</summary>

`equals` and `hashCode` are the most critical overrides — violating their contracts causes subtle,
hard-to-diagnose bugs in collections. `toString` improves observability. `compareTo` establishes
consistent ordering semantics.

- **`equals(Object)`** — determines logical equality.

  [When not to override](#when-should-equals-not-be-overridden) |
  [When to override](#when-is-it-appropriate-to-override-equals) |
  [Contract](#what-is-the-contract-of-equals) |
  [Implementation](#what-are-the-key-implementation-steps-for-writing-a-high-quality-equals-method)

  **If not overridden**: 
  - falls back to reference comparison (`==`); 
  - logical duplicates appear in `Set`, 
  - wrong keys retrieved from `Map`.

- **`hashCode()`** — hash value, must be consistent with `equals`.

  [Contract + implementation](2.2_methods_common_to_all_objects.md#what-are-the-requirements-of-the-hashcode-contract)

  **If not overridden when `equals` is**: logically equal objects land in different buckets —
  `HashMap`/`HashSet` silently broken.

- **`toString()`** — human-readable representation.

  [Details](2.2_methods_common_to_all_objects.md#what-method-is-recommended-to-always-be-overridden-and-why)

  **If not overridden**: prints class name + hash code — useless for logging and diagnostics.

- **`clone()`** — copies the object.

  [Risks](2.2_methods_common_to_all_objects.md#how-should-objects-be-copied)

  **Overriding is risky** — bypasses constructors, shallow-copy traps, fragile in hierarchies.
  Prefer copy constructors or factories.

- **`finalize()`** — **deprecated**. Do not override. Use try-with-resources and explicit cleanup.

- **`Comparable.compareTo`** — natural ordering (not an `Object` method, but same family).

  [Contract](2.2_methods_common_to_all_objects.md#what-is-the-contract-of-compareto)

  **If not implemented**: no natural sort; `TreeSet`/`TreeMap` fail or behave unexpectedly.

</details>

### When should equals not be overridden?
<details><summary>Show answer</summary>

When the identity-based behavior inherited from `Object` is correct — i.e. when there is no meaningful
notion of logical equality distinct from object identity:

- **Active-entity classes** — objects represent ongoing processes or independent identities, not values.
  Two `Thread` instances are never interchangeable, even with identical state.

- **Value-like classes that intentionally rely on identity** — some value-like classes are designed
  so that identity equality is sufficient and expected.
  `java.util.regex.Pattern` is the canonical example — its designers chose not to override `equals`
  for regex comparison; clients don't need it.
  Another case: nodes in a cyclic graph tracked in a *visited* set — value-based equality would
  collapse distinct nodes with identical content, breaking traversal.

- **Instance-controlled classes** — when at most one instance per logical value is guaranteed,
  logical equality and identity coincide; `Object.equals` is already correct.

- **Superclass already provides a correct implementation** — `Set` implementations inherit from
  `AbstractSet`, `List` from `AbstractList`, `Map` from `AbstractMap`.

- **Private or package-private class where equality is never used** — no need to override.

</details>

### How to guard against accidental equals calls in a private or package-private class?
<details><summary>Show answer</summary>

Override `equals` to throw unconditionally — signals clearly that the method must never be called:

```java
@Override public boolean equals(Object o) {
  throw new AssertionError(); // must never be called
}
```

In practice this is rarely required, but it is the most explicit defensive option.

</details>

### When is it appropriate to override `equals`?
<details><summary>Show answer</summary>

Don't override `equals` unless you have to.

When a class has a notion of logical equality that differs from mere object identity,
and no superclass has already provided a correct `equals` implementation.
This is generally the case for value classes — classes that represent a value, not an entity.

The practical trigger: whenever instances of the class will be used as keys in hash-based collections
(`HashMap`, `HashSet`) or compared for logical equality in application code.

</details>

### What distinguishes active-entity classes from value classes?
<details><summary>Show answer</summary>

- **Active-entity classes** — represent an active process or independent identity; **identity matters**,
  use identity equality.
  Even when stateful, comparing by content is meaningless — two `Semaphore` instances with the same
  count are not interchangeable; what matters is *which* semaphore, not *what* it holds.
    - Distinct execution units: `Thread`, `ExecutorService`
    - Stateful coordination mechanisms: `Lock`, `Semaphore`, `CountDownLatch`, `Condition`
    - System/resource handles: `ClassLoader`, `Process`, `Socket`

- **Value classes** — represent a **value**; content matters, use logical (value) equality.
    - Primitives wrappers: `Integer`, `Long`, `Double`
    - Domain values: `Money`, `Temperature`, `PhoneNumber`
    - JDK value types: `String`, `LocalDate`, `BigDecimal`

</details>

### What is the contract of equals?
<details><summary>Show answer</summary>

`equals` implements an equivalence relation. Five required properties:

- **Reflexive** — `x.equals(x)` must be `true`.
- **Symmetric** — `x.equals(y)` must be `true` if and only if `y.equals(x)` is `true`.
- **Transitive** — if `x.equals(y)` and `y.equals(z)`, then `x.equals(z)` must be `true`.
- **Consistent** — repeated calls return the same result as long as neither object is modified.
- **Non-nullity** — `x.equals(null)` must return `false`.

</details>

### Equality contract and inheritance
<details><summary>Show answer</summary>

A concrete, instantiable class cannot be safely extended with an additional value component while 
preserving the `equals` contract. This is a fundamental limitation of equivalence relations in OO languages.

The naive attempt (strict subtype check) violates **symmetry** — 
a `Point` and a `ColorPoint` at the same location are not symmetric:
```java
public class ColorPoint extends Point {
  private final Color color;

  @Override public boolean equals(Object o) {
    if (!(o instanceof ColorPoint)) return false;         // breaks symmetry
    return super.equals(o) && ((ColorPoint) o).color == color;
  }
}
```

Attempting to fix symmetry (mixed comparison) violates **transitivity**:
```java
  @Override public boolean equals(Object o) {
    if (!(o instanceof Point)) return false;
    if (!(o instanceof ColorPoint)) return o.equals(this); // color-blind fallback
    return super.equals(o) && ((ColorPoint) o).color == color;
  }
```

There is no way to fix both simultaneously when extending a concrete class.
See: [How should value-subclass equality problems be handled?](#how-should-value-subclass-equality-problems-be-handled)

</details>

### How can the transitivity requirement of `equals` be violated?
<details><summary>Show answer</summary>

When a concrete, instantiable subclass adds a new value component to equality comparisons —
the contract cannot be preserved regardless of how `equals` is written.
Fixing symmetry breaks transitivity; fixing transitivity breaks symmetry.

See: [Equality contract and inheritance](#equality-contract-and-inheritance)

</details>

### How should value-subclass equality problems be handled?
<details><summary>Show answer</summary>

**1. Abstract, non-instantiable superclass — safe to add value components.**
If the parent cannot be instantiated directly, no mixed comparisons are possible — there is no bare
`Shape` instance that could be compared against a `Circle`.
The abstract class may have its own value fields (e.g. `area`); 
the key requirement is non-instantiability, not field absence.
This is why class hierarchies are preferable to tagged classes.

**2. Superclass already provides a correct `equals` — safe to add fields.**
If an ancestor in the hierarchy has already overridden `equals` and its implementation is correct
for the subclass (e.g. `AbstractSet`, `AbstractList`, `AbstractMap`), adding new fields that are
*not part of equality* is safe. The inherited `equals` continues to work correctly.

**3. Neither applies — use composition, not inheritance.**
Instead of extending `Point`, `ColorPoint` contains a private `Point` field and exposes it via a view method:
```java
public class ColorPoint {
  private final Point point;   // composition
  private final Color color;

  public Point asPoint() { return point; }  // view method

  @Override public boolean equals(Object o) {
    if (!(o instanceof ColorPoint)) return false;
    ColorPoint cp = (ColorPoint) o;
    return cp.point.equals(point) && cp.color.equals(color);
  }
}
```
Trade-off: `Point` and `ColorPoint` can no longer be compared directly — even at the same location.
This is intentional, not a bug.

</details>

### A poor inheritance design choice in the JDK itself?
<details><summary>Show answer</summary>

`java.sql.Timestamp` extends `java.util.Date` and adds a `nanoseconds` field — a value component
added to a concrete superclass. 
Its `equals` violates **symmetry**: a `Timestamp` may be equal to a `Date`, while the reverse is `false`.

Mixing the two in collections produces unpredictable behavior. 
The documentation warns against it, but nothing prevents it at compile time, so errors are subtle and hard to debug.

</details>

### How can the consistency property of equals be violated?
<details><summary>Show answer</summary>

**Consistency** means that if two objects are equal, repeated calls to `equals` must keep returning
the same result — unless one of the objects is modified.

To satisfy this, `equals` must not depend on **mutable or unreliable external resources**. 
It should rely only on **deterministic, in-memory data**.

The classic violation is `java.net.URL.equals`. It does **not** compare URL strings — it resolves each
hostname to an IP address and compares the IPs. This breaks consistency in several ways:

- **Online vs offline** — when resolution succeeds, it compares IP addresses; when it fails (e.g. no
  network), it silently falls back to comparing the hostname strings. The comparison *algorithm itself*
  changes with connectivity, so the same two URLs can be equal in one state and unequal in another —
  with no change to the objects.
- **DNS changes over time** — the domain owner controls what IP a hostname maps to; remapping it flips
  the equality result.
- **Round-robin DNS** — one hostname can resolve to multiple IPs for load balancing, so the result
  depends on which IP is returned first — nondeterministic.

(As a side effect, `URL.equals` also performs blocking network I/O — a second reason not to put `URL`
in hash-based collections.)

</details>

### Non-nullity in the equals contract — how is it satisfied?
<details><summary>Show answer</summary>

**The requirement:** an object must never be considered equal to `null` — `x.equals(null)` must return `false`.

**The common mistake:** adding an explicit null check:
```java
@Override public boolean equals(Object o) {
  if (o == null) return false;   // redundant
  ...
}
```

This is unnecessary. The type check you already need for correctness handles it: `null instanceof AnyType` 
evaluates to `false`, so a standard `instanceof` test rejects `null` automatically:
```java
@Override public boolean equals(Object o) {
  if (!(o instanceof MyType)) return false;  // also rejects null
  ...
}
```

So non-nullity is satisfied for free by the `instanceof` check — no explicit null test required.

</details>

### What are the key implementation steps for writing a high-quality `equals` method?
<details><summary>Show answer</summary>

A four-step sequential algorithm:

1. **Reference-equality check (`==`).**
   Check whether the argument is a reference to this object; if so, return `true`.
   A performance optimization, worthwhile when the comparison is expensive.

2. **Type check (`instanceof`).**
   Verify the argument has the correct type; if not, return `false`.
   Usually the correct type is the class itself. Occasionally it's an *interface* that refines the
   `equals` contract to permit cross-implementation comparison (`Set`, `List`, `Map`, `Map.Entry`).

3. **Cast.**
   Cast the argument to the correct type. Guaranteed to succeed because it follows the `instanceof`
   check.

4. **Compare significant fields.**
   For each significant field, compare the argument's field with this object's.
   If all match, return `true`; otherwise `false`.
   If the type checked is an interface, access fields via interface methods; if a class, directly
   (subject to accessibility).

After writing `equals`, you must also override `hashCode`.

</details>

### How should different field types be compared in equals?
<details><summary>Show answer</summary>

- **Primitive fields** (except `float`/`double`) — use `==`.
- **`float` fields** — use `Float.compare(float, float)`.
- **`double` fields** — use `Double.compare(double, double)`.
  (`Float.equals`/`Double.equals` would work but require autoboxing — unnecessary overhead.)
- **Object reference fields** — compare recursively with `equals`. 
  If a field may legitimately be `null`, use `Objects.equals(a, b)` to avoid a `NullPointerException`.
- **Array fields** — apply these rules per element; if all elements are significant, use the appropriate `Arrays.equals`.

</details>

### Performance considerations when implementing equals?
<details><summary>Show answer</summary>

- **Field order matters** — compare fields that are cheap to compare, or likely to differ, first.
  This short-circuits unequal objects faster.

- **Skip fields not part of logical state** — e.g. synchronization locks. 
  (Comparing them would also be incorrect, not just slow.)

- **Derived fields** — need not be compared, but comparing one *first* can speed up inequality
  detection if it summarizes the object's state.
  Example: a `Polygon` caching its `area` — unequal areas immediately imply inequality, 
  skipping the expensive edge/vertex comparison.

</details>

### What to do after writing an equals method?
<details><summary>Show answer</summary>

1. **Verify the contract.** Re-check the three properties that real bugs come from:
   **symmetric? transitive? consistent?**
   (Reflexivity and non-nullity are almost never violated accidentally with the standard recipe.)

2. **Write tests.** Don't rely on reasoning alone — unit-test the contract.
   Exception: when `equals` is generated by a proven tool (e.g. AutoValue), extra tests are usually unnecessary.

3. **Keep it simple.** Compare significant fields as they are — no more. 
   Don't write a "clever" `equals` that tries to treat two *different* field values as equivalent 
   because they refer to the same external thing (**aliasing**). 
   Doing so usually drags in external, mutable state and breaks consistency — the same `URL.equals` trap.

   **Example — `File.equals`:** a `File` wraps a *pathname string*. On disk, two different paths can
   point to the same physical file (a symbolic link `link.txt` → `data.txt`). A "helpful" `equals`
   might resolve the link and call them equal — but that would (a) require querying the filesystem,
   so the result could change if the link is repointed or deleted between calls, and (b) behave
   differently across operating systems. `File.equals` correctly avoids this: it compares path strings
   only. Two `File`s are equal *iff their pathnames match*, even if they point to the same file.
   (Need the other semantics? Use `Files.isSameFile(a, b)` explicitly — never hidden inside `equals`.)


Note: "iff" - if and only if

</details>

### What tools can generate a correct equals implementation?
<details><summary>Show answer</summary>

Generated implementations are usually safer than handwritten ones — they reduce the risk of violating the contract. 
Four options, each with a trade-off:

- **Java records** — auto-generate `equals`, `hashCode`, and `toString` from components.
  Safest and most concise; ideal for immutable value classes with a fixed field set.
- **Lombok (`@EqualsAndHashCode`)** — generates both at compile time via annotation.
  Convenient, but adds a build-time dependency and relies on annotation processing.
- **IDE code generation** — no careless mistakes, but the output is verbose, 
  doesn't adapt to future field changes, and so needs maintenance and testing.
- **Google AutoValue** — concise, reliable, and auto-updates when the class changes.
  One of the best options for complex value classes.

</details>

### Lombok vs Google AutoValue — generating equals/hashCode?
<details><summary>Show answer</summary>

**What they do (the goal):** both eliminate the boilerplate of hand-writing correct,
contract-compliant `equals` and `hashCode` for value classes — you declare intent, the tool produces the implementations.

**Where they are similar (mechanism level):** both register a standard `javax.annotation.processing.Processor`, 
discovered by `javac` at compile time. At the surface, both are annotation processors hooked into the compiler.

**Where they differ:**

- **Effect on your class** — the decisive difference.
    - *AutoValue* requires your class to be **abstract**; it generates a concrete subclass
      (`AutoValue_Foo extends Foo`). Your type is now shaped to serve the tool, not your domain — an
      infrastructure concern leaking into your design.
    - *Lombok* annotates a **concrete class as-is** and injects the methods into it. No design distortion.
- **Mechanism under the hood**
    - *AutoValue* — conventional "source from source"; emits real, inspectable generated source.
    - *Lombok* — edits the compiler's internal AST to modify the existing class (a non-standard use of
      the processor API). The methods land in the `.class` with no readable source.
- **Ecosystem fit**
    - *Lombok* — first-class in the mainstream toolchain: a Spring-Boot-managed dependency, offered by Spring Initializr, 
      and supported by IntelliJ (needs annotation processing enabled — a one-time step). 
      Occasional friction exists but is fixable and not recurring.
    - *AutoValue* — fewer moving parts in the IDE, but the abstract-class constraint is always present.

**The trade-off:**
- **Lombok** — no design distortion, concise, ecosystem-blessed. 
  Cost: a non-standard AST-rewrite mechanism (rarely visible in practice).
- **AutoValue** — standard mechanism, inspectable generated output. 
  Cost: forces your class to be abstract, letting the tool dictate your type design.

</details>
