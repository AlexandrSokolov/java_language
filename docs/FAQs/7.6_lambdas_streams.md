## Lambdas and Streams - Parallel Streams

### What governs whether parallel() helps?
<details><summary>Show answer</summary>

Five things must hold, and the sixth decides.

1. [**The source splits cheaply**](#how-does-source-type-affect-parallel-streams) — a chain source cannot be cut, so
   one thread does all the work.
2. [**The lambdas are legal**](#which-lambda-rules-break-under-parallel) — stateful or interfering ones give a wrong
   answer, and sequential runs hide it.
3. [**The terminal op merges cheaply**](#which-terminal-ops-suit-parallelism) — a finish that cannot be split caps
   the whole pipeline.
4. [**Order is not needed**](#does-parallel-preserve-element-order) — asking for encounter order back puts the
   sequence back too.
5. [**The pool is not a problem**](#how-do-parallel-streams-share-threads) — one shared pool for the whole program,
   so blocking work starves everyone else.
6. [**It measures faster**](#parallel--when-do-you-add-it) — the only thing that actually answers the question.

Spine: it has to split, the lambdas have to be legal, the finish has to merge, order has to be affordable, the
threads are shared — and then you measure.

</details>

### How does source type affect parallel streams?
<details><summary>Show answer</summary>

Splitting is cheap when the source has known size and a structure that can be cut without walking it:

- arrays, `ArrayList`, `IntStream.range` — random access by index, so the splitterator hands out index ranges
  instantly.
- `HashMap`, `HashSet` — no index, but the splitterator cuts the internal bucket table into ranges of buckets.

It fails when the source is a **chain** — element N is produced from element N-1, so reaching N means walking 1..N-1.
No cut point exists.

</details>

### Which sources split badly in parallel?
<details><summary>Show answer</summary>

A chain source: element N is produced from element N-1, so reaching N means walking 1..N-1. No cut point exists.

- `Stream.iterate(seed, f)` — each element is `f` applied to the one before, so #500000 can't be produced without
  first producing all 499999 ahead of it.

    ```java
    Stream.iterate(BigInteger.ONE, n -> n.multiply(TWO))  // #3 needs #2, #2 needs #1
          .parallel()                                     // no thread can start at #500000
          .limit(500_000)
          .forEach(System.out::println);                  // one thread does all the producing
    ```
- `BufferedReader.lines()` — reads forward through the file; size is unknown and you can't jump to line N without
  reading every line before it.
- `Iterator`-based streams — the only operation is "give me the next one"; there's no size and no random access to
  hand a chunk to another thread.

    ```java
    Iterable<Order> orders = repository.streamAll();      // Iterator only: hasNext() / next()
    StreamSupport.stream(orders.spliterator(), true)      // true = parallel
                 .forEach(this::handle);                  // spliterator has no size, no index
                                                          // it can only buffer next() results in batches
    ```

- Cursor-paginated APIs — page N's request needs the cursor returned by page N-1, so pages can only be fetched one
  after another.

</details>

### Which random source for parallel streams?
<details><summary>Show answer</summary>

`SplittableRandom` — built for this: it splits into independent generators, one per thread, so the source parallelizes
cleanly and can scale near-linearly.

The other two, worst-last:
- `ThreadLocalRandom` — made for a single thread; used as a parallel source it works but won't match
  `SplittableRandom`.
- `Random` — synchronizes on every call, so all threads contend on one lock and the parallelism collapses. (Fine
  single-threaded; only bad as a parallel source.)

Handle: can the generator be *split* or only *shared*? Splittable → scales. Shared behind a lock → contention.

</details>

### Which terminal ops suit parallelism?
<details><summary>Show answer</summary>

The axis: **how much work lives in the terminal step, and can that work be split and merged cheaply?** A terminal
op that is inherently sequential caps the whole pipeline no matter how well the source split.

- **Best — reductions.** `reduce`, plus the prepackaged `min`, `max`, `count`, `sum`. Each thread reduces its chunk
  to one value; merging is combining a few values.
- **Also good — short-circuiting predicates.** `anyMatch`, `allMatch`, `noneMatch`. Any thread finding the answer
  ends it for everyone.
- **Fine — `collect` with a plain collector.** `toList`, `groupingBy`, `toMap`. Each thread builds its own container
  and the merge combines them. Merge cost scales with the number of keys, not the number of elements, so a few
  buckets merge in nothing. This is the normal choice. See
  [Reduce vs collect](#reduce-vs-collect--whats-the-difference).
- **Optimization — concurrent collectors.** `groupingByConcurrent`, `toConcurrentMap`. All threads write into one
  shared container, so there is no per-thread build and no merge step. The coordination is not removed, it moves
  into the container and is paid on every write. Pays only when the merge would dominate — many keys, many workers.
- **Worst — ordered terminal ops.** `forEachOrdered`, and `findFirst` where `findAny` would do. Threads must wait
  for the elements before theirs, which puts the sequence back.

</details>

### Reduce vs collect — what's the difference?
<details><summary>Show answer</summary>

`reduce` folds elements into a single immutable value by repeatedly applying a combine function — each step takes the
running result and the next element and returns a new value. Nothing is mutated. Fits `sum`, `min`, `max`, `count`.

`collect` accumulates elements into a **mutable container** — each element is added into the same growing structure
(list, map, `StringBuilder`). This is "mutable reduction": the result is a container being filled, not a value being
folded.

Consequence for parallel: `reduce` merges by combining a few final values — cheap. `collect` (non-concurrent
collector) merges by combining whole containers — copying, resizing, rehashing — a cost that scales with the data.

</details>

### Reduce into a collection?
<details><summary>Show answer</summary>

It compiles — `reduce` takes any function, so folding into a collection is legal. It breaks the contract two ways.

**The identity must be a value, not an instance.** `reduce(identity, acc, comb)` requires `comb(identity, x)` to
equal `x`. That holds only if the identity is empty. One container handed to every chunk is not empty after the
first write.

**The accumulator must not mutate its input.** It has to return a *new* value holding the running result plus one
element. Writing into the container and returning the same reference breaks it.

```java
List<Integer> numbers = List.of(1, 2, 3, 4, 5);
ConcurrentMap<Boolean, Integer> shared = new ConcurrentHashMap<>();  // one instance used as the identity

ConcurrentMap<Boolean, Integer> result =
    numbers.parallelStream()
           .reduce(shared,
                   (acc, n) -> { acc.merge(n % 2 == 0, n, Integer::sum); return acc; },
                   (a, b) -> { b.forEach((k, v) -> a.merge(k, v, Integer::sum)); return a; });
```

The container is the best one available: concurrent writes lose nothing, merging is safe, no copying. It is not the
problem. Every chunk starts from `shared`, so `a` and `b` are the same object in every combiner call, and
`combine(m, m)` sums every value with itself. The number of combiner calls depends on how the source split, so the
totals change between runs.

**Sequential hides it.** One thread, no combiner call, so it prints the right answer with both contract breaks still
in place. `shared` lives outside the pipeline, so a second run accumulates on top of the first; and the returned map
is the same object as `shared`, so the caller cannot tell result from input. Anyone who later adds `parallel()` gets
the doubling.

To obey the contract the accumulator must build a new container each step: step 2 copies 1 entry, step 3 copies 2 —
O(N²) copies for N elements. `reduce` is for values: `sum`, `min`, `max`, `count`.

</details>

### What lets other functional languages reduce into collections?
<details><summary>Show answer</summary>

Their collections are **persistent**: adding an element returns a new version that shares structure with the old
one instead of copying it. Both versions stay valid, so `reduce`'s no-mutation contract costs almost nothing.

In Scala, for instance:

```scala
val a = List(2, 3)
val b = 1 :: a        // one new cell pointing at a — a is untouched, nothing copied, O(1)

val v = Vector(1, 2, 3)
val w = v :+ 4        // trie with wide nodes — copies one path, shares the rest, O(log N)

(1 to 5).foldLeft(List.empty[Int])((acc, n) => n :: acc)   // each step O(1), total O(N)
```

Clojure, Haskell and F# have the same thing under different names.

The language does not force this; the immutability does. A language where mutation is not the normal tool has to
make immutable collections cheap, or nothing would be usable.

The JDK never made that choice. `List.of` cannot add an element at all, and every `List` copy constructor copies
the whole array, so the same reduction is O(N²). The difference is the collection library, not the operation.

</details>

### Parallel work merged into one collection?
<details><summary>Show answer</summary>

`collect` — mutable reduction. The collector carries three functions plus a set of characteristics:

- **supplier** — makes a fresh container. `reduce` has one identity value instead, and that is the whole difference.
- **accumulator** — writes into that container. Allowed to mutate, because the container belongs to one worker.
- **combiner** — merges two containers the framework knows are separate objects.

Plain `groupingBy` or `partitioningBy` on a parallel stream is correct and is the normal way. The supplier gives
every worker its own map, each worker fills only its own, and the combiner merges those maps pairwise at the end.
No two threads ever touch the same container. `reduce` cannot do this: with one identity value and no supplier,
every worker starts from the same object.

```java
Map<Boolean, Integer> result =
    numbers.parallelStream()
           .collect(Collectors.partitioningBy(n -> n % 2 == 0,
                                              Collectors.summingInt(n -> n)));
```

`groupingByConcurrent` and `toConcurrentMap` declare the `CONCURRENT` characteristic. That is an optimization, not
a safer option: it skips the per-worker maps and lets every worker write into one shared map, so there is no merge
step. It requires a concurrent container, and the collector must also be `UNORDERED` — within a key the elements
land in whatever order the workers wrote them.

```java
ConcurrentMap<Boolean, List<Integer>> byParity =
    numbers.parallelStream()
           .collect(Collectors.groupingByConcurrent(n -> n % 2 == 0));
```

It pays when merging would dominate — many workers and many keys. It does not remove the coordination cost, it
moves it into the shared map, paid on every write. Merge cost scales with the number of keys, not the number of
elements, so a two-bucket result has nothing to merge and plain `groupingBy` wins. Default to the plain collector.

</details>

### Which lambda rules break under parallel?
<details><summary>Show answer</summary>

Safety failures — wrong results, not just slow ones — from lambdas that violate the `Stream` spec. Three requirements
on the function objects (e.g. `reduce`'s accumulator and combiner): **stateless, non-interfering, associative**.

- [**Stateless**](#why-does-a-stateful-lambda-pass-sequential-but-fail-parallel) — the result for an element mustn't
  depend on other elements or on a running count of them.
- [**Non-interfering**](#what-happens-if-a-lambda-modifies-the-streams-source) — doesn't modify the source while the
  stream runs. Broken by `list.stream().peek(x -> list.add(...))`.
- [**Associative**](#when-does-reduce-give-different-results-in-parallel) — regrouping the operands mustn't change the
  result; non-associative ops break. Subtraction: `reduce(0, (a, b) -> a - b)` gives -36 sequential, 0 parallel.

The trap: sequential execution hides **stateless** and **associative** — one thread means no race and no regrouping,
so they run right and nothing flags the bug; parallelize and they fail. (Non-interference throws in both modes, so it
isn't hidden.) Sequential correctness is not evidence the lambdas are legal.

</details>

### Why does a stateful lambda pass sequential but fail parallel?
<details><summary>Show answer</summary>

A stateful lambda carries a value from one element to the next, so an element's result depends on **how many elements
were processed before it**. Sequential runs on one thread in encounter order, so "before" is fixed and the result is
predictable. Parallel splits the elements across threads, so "before" is no longer fixed — the count each element
sees depends on thread timing.

```java
Set<String> names = new TreeSet<>(List.of("Anna", "Ben", "Carl"));   // sorted: Anna, Ben, Carl
AtomicInteger seq = new AtomicInteger();
List<Row> rows = names.parallelStream()
  .map(name -> new Row(seq.getAndIncrement(), name))                 // id = how many ran before this one
  .collect(Collectors.toList());
```

Expected: Anna→0, Ben→1, Carl→2. Actual under parallel: the **list order** is still Anna, Ben, Carl (`collect` keeps
encounter order), but the **ids are scrambled** — e.g. Anna→5, Ben→0, Carl→1 — and can differ from run to run. The
number no longer matches the element it was meant to label. The `AtomicInteger` prevents duplicate ids, not the
mismatch.

Handle: The spec says *stateless* because carried state depends on order, and parallel doesn't promise order.

</details>

### What happens if a lambda modifies the stream's source?
<details><summary>Show answer</summary>

**Non-interference:** no lambda in the pipeline may change the source collection while the stream runs. Unlike the
stateful trap, this one usually fails **loudly and in both modes** — not silently under parallel.

```java
List<Integer> list = new ArrayList<>(List.of(1, 2, 3, 4, 5));
list.stream()
  .peek(x -> { if (x == 3) list.add(99); })   // modifies the source mid-stream
  .collect(Collectors.toList());              // throws ConcurrentModificationException
```

Fail-fast sources (`ArrayList`, `HashMap`) detect the structural change and throw `ConcurrentModificationException` —
sequential *and* parallel alike. The stream captures the source's state at the terminal operation; changing it
underneath is the violation.

Caveat: non-fail-fast sources (`CopyOnWriteArrayList`, concurrent collections) don't throw — the change is silently
ignored or seen inconsistently, which is worse. The rule holds regardless: don't write to the source you're reading.

Handle:
- interference breaks the source you're iterating;
- statefulness breaks the result you're building.

</details>

### When does reduce give different results in parallel?
<details><summary>Show answer</summary>

When the combine function is **not associative** — when regrouping the operands changes the result. Sequential folds
strictly left to right, so one grouping is used and the answer is stable. Parallel splits the elements into chunks,
reduces each chunk, then combines the partial results — a *different* grouping. If the operation is sensitive to
grouping, the two paths disagree.

```java
List<Integer> nums = List.of(1, 2, 3, 4, 5, 6, 7, 8);
nums.stream().reduce(0, (a, b) -> a - b);          // sequential:  -36
nums.parallelStream().reduce(0, (a, b) -> a - b);  // parallel:      0
```

Subtraction breaks because `(a - b) - c ≠ a - (b - c)`. Addition, multiplication, `min`, `max`, string concatenation
are associative — grouping doesn't matter, so they give the same answer either way.

Handle: parallel reduces in chunks and combines, so the grouping differs from left-to-right. Only associative
operations survive that regrouping.

</details>

### Does parallel preserve element order?
<details><summary>Show answer</summary>

Not with `forEach` — it makes no encounter-order promise, so a parallel run emits in whatever order threads finish.

`forEachOrdered` restores encounter order — but only at the **emit** step. All elements are still computed fully in
parallel; the stream buffers finished results and hands them to your consumer in order. If a later element finishes
first, its result is held until the earlier ones have been emitted. The heavy work stays parallel; only delivery is
serialized, and that holding-back is the cost.

Separate from safety failures: the pipeline computed correct elements, it just delivered them in an order the
sequential version didn't have.

</details>

### Terminal ops — which thread runs them?
<details><summary>Show answer</summary>

The workers do it. The terminal op is not a separate step where one thread gathers what the others produced — its
per-element work splits across the workers like every other stage, and what is left is a cheap merge of a few
partial results.

- `forEach` — the action runs on the worker holding the element, no merge at all.
- `reduce`, `min`, `max`, `count`, `sum` — each worker folds its chunk, the combiner merges partials.
- `anyMatch`, `allMatch`, `noneMatch` — the predicate runs on the workers, the first answer ends the run.
- `collect` — each worker fills its own container, merged as the split tree unwinds.

`forEachOrdered` is the exception. The action still runs on the workers, but each must wait for the elements before
it, so the pipeline is parallel in name only.

The caller thread blocks until the pipeline finishes; on the common pool it usually takes a share of the work too.

</details>

### How do parallel streams share threads?
<details><summary>Show answer</summary>

Every parallel stream in the whole program runs on **one shared pool** — the common `ForkJoinPool`, sized by default
to the machine's core count minus one. You don't create it or pass it in; `parallelStream()` and `.parallel()` use it
automatically.

Because it's shared program-wide, one heavy or slow pipeline occupies those workers and starves every other parallel
pipeline running at the same time — unrelated code slows down because it's waiting for the same threads.

Consequence: the blast radius is larger than the pipeline you wrote, so it must be measured in a realistic running
system. A benchmark of one pipeline alone never shows the interference, because alone it has the whole pool.

</details>

### How do you keep a parallel stream off the common pool?
<details><summary>Show answer</summary>

Run the stream as a task submitted to your own `ForkJoinPool`. A parallel stream started while executing inside a
`ForkJoinPool` task uses *that* pool's workers instead of the common pool:

```java
ForkJoinPool pool = new ForkJoinPool(4);
try {
  List<Row> rows = pool.submit(() ->
    items.parallelStream().map(this::heavy).collect(Collectors.toList())
  ).get();
} finally {
  pool.shutdown();
}
```

Caveat — this is **undocumented behavior**, not a contract. It works reliably across JDK versions but the spec never
promises it, so a reviewer who knows the internals will flag it. The officially-supported way to control the pool is
not `parallelStream()` at all — use `CompletableFuture` with an explicit `Executor`, or submit your own recursive
task, where the executor is part of the API.

Handle: wrapping in a custom pool isolates the threads but rides on an implementation detail; `CompletableFuture` +
`Executor` is the contract-backed alternative.

</details>

### parallel() — when do you add it?
<details><summary>Show answer</summary>

**Strictly a performance optimization** — so it inherits every rule about optimization: measure before and after in
realistic conditions, and keep it only if the numbers justify it. It is never a default or a style choice.

The order:

1. Have reason to believe correctness survives — legal lambdas, no order dependence.
2. Have reason to believe it gets faster — splittable source, heavy per-element work.
3. Measure in a realistic setting.
4. Only then ship it.

Step 2 is a filter, not an answer. `elements × lines of code ≥ ~100,000` is only the cheap-CPU-work case — it treats
every line as equal cost, so it misses blocking work. A few elements doing seconds of I/O each (file generation,
uploads, network calls) clear the bar that a million trivial elements never would. Heavy per-element work wins even
on a small collection; cheap work loses even on a large one. Count of elements is a poor proxy.

Only step 3 answers whether parallel helps. Skipping to step 4 risks a hang or a wrong answer, both worse than slow.

Skipping to step 4 risks a hang or a wrong answer, both worse than slow. The odds are against you: in large
stream-heavy codebases, the places where parallel actually pays off are rare.

</details>
