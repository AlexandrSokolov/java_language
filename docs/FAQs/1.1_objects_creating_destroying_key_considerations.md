### How do you avoid needless allocation at runtime?
<details><summary>Show answer</summary>

- Avoid unnecessary object creation (autoboxing, `new String(...)`, recompiled regex `Pattern`s, etc.)
- Reuse shared instances where appropriate:
  - preferably immutable objects (always safe to share)
  - mutable objects only when sharing is safe and well-defined

</details>

### At design time, how do you control how many instances can exist?
<details><summary>Show answer</summary>

- **One, or a fixed few** → make the class *instance-controlled* (singletons, enum-based classes)
- **Zero** → when instances are meaningless, make it a *final utility class* with a private constructor

</details>

### When designing a class's destruction and lifetime, what must be considered, and why?
<details><summary>Show answer</summary>

Read it as one causal chain — each link forces the next, so none can be silently dropped:

1. **Lifetime** — *when* should this object end? The root question; everything below follows from it.
2. **Memory and reachability** — a still-reachable object cannot be garbage-collected, so if it outlives its
   purpose it leaks memory. *(the mechanism — how leaks happen, and how to prevent them)*
3. **Resource management** — objects hold more than memory (file handles, sockets, connections); these leak too,
   and worse, because they are finite and fail fast. *(what else breaks beyond memory)*

One-sentence trigger to recall the whole chain:
> An object has a lifetime; if it outlives its purpose it leaks memory; and if it held resources, it leaks those
> too — which is worse.

</details>

### What should a class designer consider about an object's destruction?
<details><summary>Show answer</summary>

The core issue is that **garbage collection is not a destruction mechanism** — for two independent reasons:

- **Timing** — GC does reclaim memory, but *when* it runs is non-deterministic. 
  You cannot rely on it to act at any particular moment. 
  This is true for *every* object, even one that holds nothing but memory.
- **Scope** — GC reclaims *memory only*. 
  Anything that is not memory — file handles, sockets, connections, native allocations — it never reclaims at all.

So the designer's two decisions:

1. **Is end-of-life timing significant for this object?** If anything must happen *promptly* when
   the object is done, GC's non-deterministic timing already disqualifies it as the plan — before
   any resource is even involved.
2. **Does the object own anything outside the heap?** If so, GC is not merely late, it is absent;
   release must be explicit.

Where both are true (non-memory *and* must-be-prompt) is the painful case — and *why* timely,
explicit release is required, with finalizers/cleaners only a last-resort safety net.

Note: _timely_ = deterministic, immediate, predictable — precisely what GC does not provide.

This is about **obligation** — what the *class* owes at end-of-life.

</details>

### What specific topics belong to *Memory and Reference Management*?
<details><summary>Show answer</summary>

One idea underneath all of it: **a memory leak is a reference that outlives its object's usefulness.** 
GC collects only what's unreachable, so a forgotten reference *is* the leak.

Three things hang off that:

- **Mechanism — how the GC sees a reference**
  Reachability strength: strong / soft / weak / phantom. 
  This is the lever — weakening a reference is how you let the GC reclaim something you can't manually clear.

- **The mistake — what kind of unwanted reference you're holding**
  - *obsolete* — object no longer needed, you simply forgot to clear the reference
  - *lingering* — reference still valid, but outlives its intended scope
  - *unintended* — reference you never meant to create (closure / inner-class capture)

- **Where it happens — long-lived containers that quietly retain**
  - caches
  - listeners & observers (forgotten unregistration = retention = leak)
  - callbacks / event handlers (closure-capture leaks)
  - thread-local storage

</details>

### Obsolete references vs lingering
<details><summary>Show answer</summary>

Notes:
- _obsolete_ references - “The object is dead, but I forgot to throw away the body.”
    - You no longer need the object.
    - You simply forgot to clear the reference.
    - The reference still exists, but has no logical purpose.
    - This is a usefulness problem, object usefulness ended.
    - The programmer knows it's useless, but forgot to release it.
    - Example in one line: “We already processed the request; why are we still holding its DTO?”
- _lingering_ reference - “The object is still theoretically needed, but it lives much longer than planned.”
    - The reference **is still logically valid**.
    - But it outlives its intended scope, often by accident:
        - stored in a static collection
        - in a cache that never evicts
        - an event listener you forgot to unsubscribe
        - a closure holding context longer than the function call
    - This is a lifetime mismatch problem, object lifetime exceeded expectations.
    - The programmer did not want the reference forever — but the system kept it.
    - Example in one line: “This listener stayed registered 5 hours after the UI closed.”

</details>

### Memory leak vs accidental retention
<details><summary>Show answer</summary>

Same phenomenon in Java — but the two words name different ends of it:

- **Accidental retention** — the **cause** (what you did): 
  you are still holding a reference to an object you no longer need.
- **Memory leak** — the **effect** (what you observe): 
  memory that should have been reclaimed stays occupied, and usage grows over time.

So in Java a memory leak is *always* produced by accidental retention — retention is the action,
the leak is the result. The words are not synonyms; they sit at opposite ends of one cause→effect line.

> Retention is *why*. The leak is *what you see*.

</details>

### Listeners vs Observers vs Event Handlers vs Callbacks
<details><summary>Show answer</summary>

All four are *"store a reference to call later"* — which is what makes them leak sources when registration is forgotten. 
They differ by **convention and generality, not mechanism.** 
Place them on one axis — specific/registered → general/transient — and all four fall out in order:

| Term              | What it is                                         | Generality                                                 |
|-------------------|----------------------------------------------------|------------------------------------------------------------|
| **Listener**      | a concrete object you *register* with a source     | most specific; classic event model (Swing, AWT)            |
| **Observer**      | the *same idea*, named as an architectural pattern | same mechanism, higher altitude (Observer pattern, RxJava) |
| **Event handler** | the *logic that runs* when an event fires          | orthogonal — can *be* a listener, lambda, or method ref    |
| **Callback**      | any "call this function later"                     | most general; often passed inline, no event system needed  |

Two relationships the axis makes visible:

- **Listener ⊂ Observer** in spirit — listener is the concrete API object, observer is the pattern-level name for it.
- **Callback is the superset** — listeners, observers, and handlers are all special cases of "call this later." 
  Handler is the odd one out: it names the *logic*, not the registration.

Examples:
- `source.addListener(l)` 
- `observable.subscribe(o)`
- `button.setOnClick(e -> ...)`
- `runAsync(() -> ...)`

</details>

### Resource management — what's the class designer's responsibility?
<details><summary>Show answer</summary>

To **expose the cleanup obligation** so callers can meet it:

- Implement `AutoCloseable` (a single `close()` method), so the type advertises that it owns
  something requiring release.
- Make `close()` idempotent and have post-close method calls fail fast (e.g. `IllegalStateException`).

</details>

### Resource management — what's the caller's responsibility?
<details><summary>Show answer</summary>

To **discharge the obligation** the type exposes:

- Wrap the instance in **try-with-resources**, which guarantees `close()` runs even on exception or early return.
- This delivers **deterministic release** — cleanup at a known point, not whenever GC eventually runs.

The designer made the type closeable; 
the caller is the one who must actually **trigger** the close. 
Forgetting to is precisely how resource leaks happen.

</details>

### Construction & destruction — what makes an object correct?
<details><summary>Show answer</summary>

Two separate correctness questions:

- **Is the object valid once built?**
  - all required fields initialized before use (no half-built object escapes)
  - immutability where appropriate (decided *at* construction)
  - defensive copies of mutable inputs, so outsiders can't mutate internal state

- **Who is allowed to build it?** — constructor visibility. Don't recall this as a rule; recall it as its cases:
  - *private constructor* → utility class (no instances) or singleton/instance-controlled (limited instances)
  - *package-private constructor + static factory* → blocks outside subclassing while allowing it inside
  - *public constructor* → the default, only when unrestricted creation is fine

</details>

### Construction & destruction — is the object worth its cost?
<details><summary>Show answer</summary>

The **performance** axis, kept realistic (not premature):

- weigh the cost of object creation honestly — but don't contort the design to avoid cheap allocations
- avoid maintaining your own object pools **unless** the objects are genuinely heavyweight
  (DB connections) or expensive to *acquire* (e.g. cached auth tokens), not merely expensive to `new`

</details>
