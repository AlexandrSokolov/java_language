## Generics Bounded Types

### What is the main limitation of `<T>` and its solution?
<details><summary>Show answer</summary>

A plain `<T>` has **two** separate limitations — different problems, different tools:

- **No capability** — inside a generic body, `<T>` exposes only `Object` methods.
  *Fixed by a bounded type parameter* `<T extends X>`.
  [→ bounded type parameter](#t-extends-x--the-bounded-type-parameter)

- **No API flexibility** — a plain `<T>`/`<E>` in a signature is invariant, 
  so callers holding a subtype or supertype collection often cannot pass it in.
  *Fixed by a bounded wildcard* `<? extends T>` / `<? super T>`.
  [→ API flexibility challenges](#what-concrete-api-flexibility-challenges-occur-when-using-strict-unbounded-type-parameters-like-e-or-t)

**Key separation:** 
- bounded *type parameter* → capability for the implementer; 
- bounded *wildcard* → flexibility for the caller. 

Different problems, not interchangeable.

</details>

### What concrete API flexibility challenges occur when using strict, unbounded type parameters like `<E>` or `<T>`?
<details><summary>Show answer</summary>

The root cause is **invariance**: even though `Integer` is a subtype of `Number`, 
`Iterable<Integer>` is *not* a subtype of `Iterable<Number>`. They are unrelated types. 
This makes an API built on a plain `<E>` rigid whenever it interacts with an inheritance hierarchy — 
in two opposite situations:

**1. Producer — reading data *in* (a source).**
A method that takes data from a caller's collection rejects valid subtypes:
```java
public class Stack<E> {
   public void pushAll(Iterable<E> src) { ... }
}

Stack<Number> numberStack = new Stack<>();
List<Integer> integers = List.of(1, 2, 3);

numberStack.pushAll(integers);   // COMPILE ERROR
// Integer is a Number, but Iterable<Integer> is not an Iterable<Number>
```
The signature demands an exact `Iterable<Number>`, 
so you cannot push a list of integers into a stack of numbers — even though it is perfectly safe.

**2. Consumer — writing data *out* (a destination).**
The opposite: a method that puts data into a caller's collection rejects valid supertypes:
```java
public class Stack<E> {
    public void popAll(Collection<E> dst) { ... }
}

Stack<Integer> integerStack = new Stack<>();
Collection<Object> objects = new ArrayList<>();

integerStack.popAll(objects);    // COMPILE ERROR
// a Collection<Object> can safely hold Integers, but it is not a Collection<Integer>
```
The signature demands an exact `Collection<Integer>`, 
so you cannot pop the integers into a broader `Collection<Number>` or `Collection<Object>`.

**The fix:** plain `<E>` cannot express either case — you need **bounded wildcards** (`? extends E` for
producers, `? super E` for consumers),
transforming the methods into `pushAll(Iterable<? extends E> src)` and `popAll(Collection<? super E> dst)`,
governed by the **PECS** rule.
[→ PECS: when to use which](#when-should-you-use--extends-t-versus--super-t-in-generic-apis)

</details>

### What types of generic bounds exist in Java?
<details><summary>Show answer</summary>

Bounds restrict which types a parameter accepts. They come in two forms — **named parameter** (`T`) and
**wildcard** (`?`) — each with an upper variant; only wildcards have a lower variant:

**Named type-parameter bounds** (`T` — gives the implementer capability):
1. [**Upper bound `<T extends X>`**](#what-is-a-bounded-upper-bounded-type-parameter-t-extends-x) — `T` is `X` or a
   subtype; lets you call `X`'s methods on `T`.
2. [**Multiple bounds `<T extends A & B>`**](#multiple-bounds-t-extends-a--b) — `T` must satisfy all listed types
   (at most one class, listed first; the rest interfaces).
3. [**Recursive bound `<T extends Comparable<T>>`**](#how-to-enforce-that-a-type-interacts-only-with-its-exact-same-type-and-when-is-this-useful) —
   `T` bounded by an expression involving itself; e.g. "a type comparable to itself."
4. [**Self-type idiom `<T extends Base<T>>`**](#self-type-idiom) — a recursive-bound application where
   a base type carries its concrete subtype; used for fluent chaining across a hierarchy.

**Wildcard bounds** (`?` — gives the caller API flexibility):
5. [**Upper-bound wildcard `<? extends T>`**](#how-do-you-make-a-generic-api-accept-a-collection-of-a-specific-type-or-any-of-its-subtypes) —
   unknown type that is `T` or a subtype (producer).
6. [**Lower-bound wildcard `<? super T>`**](#how-do-you-make-a-generic-api-accept-a-collection-of-a-specific-type-or-any-of-its-supertypes) —
   unknown type that is `T` or a supertype (consumer). *(Lower bounds exist only as wildcards, never
   as named parameters.)*

The split mirrors the two limitations from card 1: 
**named bounds → implementer capability; wildcard bounds → caller flexibility.**

</details>

### What is a bounded (upper-bounded) type parameter `<T extends X>`?
<details><summary>Show answer</summary>

This is an **upper bound** — `<T extends X>` makes `X` the *ceiling*: 
`T` may be `X` or any **subtype** of `X`, but nothing above it. 
(Picture the hierarchy with `Object` at the top: `extends X` says "X is the highest you may go," 
so `X` is the upper limit. `<T extends Number>` allows `Number`, `Integer`, `Double` — never `Object`.)

**The limitation it fixes — no information about `T`.**
With a plain `<T>`, the compiler knows nothing about `T` and assumes `Object`. Inside the generic body
you can only call `Object` methods (`toString()`, `equals()`, ...) — no math, no sorting, no custom
method, because the compiler can't prove `T` has them.

**The bound — a compile-time contract.**
`<T extends X>` promises the compiler that `T` is always at least an `X`. In return, the compiler lets
you call any `X` method directly on a `T`:

```java
// Without a bound — cannot call doubleValue(); T is only Object
static <T> double sumPlain(List<T> list) {
    double sum = 0;
    for (T n : list) sum += n.doubleValue();   // COMPILE ERROR
    return sum;
}

// With an upper bound — T is guaranteed to be a Number, so Number's methods are available
static <T extends Number> double sum(List<T> list) {
    double sum = 0;
    for (T n : list) sum += n.doubleValue();   // OK
    return sum;
}
```

So an upper bound trades *some* flexibility (`T` is no longer "anything") for **capability** 
(you can use the bound's methods). That trade is the whole motivation for a bounded type parameter.

(`extends` here means "is or is a subtype of" — it works for both classes and interfaces. 
And note the contrast coming later: `? super X` is a **lower** bound — `X` as the *floor*, allowing supertypes.)

</details>

### Multiple bounds `<T extends A & B>`
<details><summary>Show answer</summary>

A type parameter can require **several bounds at once**, joined with `&`. 
`T` must satisfy *all* of them — so you can call methods from every bound:

```java
static <T extends Number & Comparable<T>> T max(List<T> list) { ... }
// T is both a Number (doubleValue, ...) and Comparable (compareTo, ...)
```

Two rules:
- **At most one class**, and it must be listed **first**; the rest must be interfaces.
- Order matters only for that "class first" rule.

</details>

### How do you make a generic API accept a collection of a specific type or any of its subtypes?
<details><summary>Show answer</summary>

Use an **upper-bounded wildcard**: `<? extends Type>` — "this type, or any subtype of it."

This fixes the invariance rigidity (a `List<Integer>` is not a `List<Number>`, 
so a plain `Iterable<E>` parameter rejects it — 
see [the flexibility problem](#what-concrete-api-flexibility-challenges-occur-when-using-strict-unbounded-type-parameters-like-e-or-t)).

**When to use it — the collection is a *producer* (you only read *out* of it).**
The upper bound guarantees every element is at least the base type, so reading them as the base type is always safe. 
(This is the **PE** in PECS — *Producer → extends*.)

```java
public class Stack<E> {
  // rigid:    public void pushAll(Iterable<E> src)          // rejects Iterable<Integer>
  public void pushAll(Iterable<? extends E> src) {           // flexible
    for (E e : src) push(e);    // safe: every e is at least an E
  }
}

Stack<Number> numberStack = new Stack<>();
numberStack.pushAll(List.of(1, 2, 3));   // OK: Iterable<Integer> matches Iterable<? extends Number>
```

**Why you can only read:** since the exact subtype is unknown, 
the compiler won't let you *add* to a `? extends E` collection (you can't prove what you add fits) — 
reading out as `E` is the only safe direction. 
That read-only nature is exactly why it suits a producer.

</details>

### How do you make a generic API accept a collection of a specific type or any of its supertypes?
<details><summary>Show answer</summary>

Use a **lower-bounded wildcard**: `<? super Type>` — "this type, or any supertype of it."

This fixes the opposite invariance rigidity: a method that writes `E`s into a caller's collection, 
but requires exactly `Collection<E>`, rejects a perfectly valid `Collection<Object>`
([the flexibility problem](#what-concrete-api-flexibility-challenges-occur-when-using-strict-unbounded-type-parameters-like-e-or-t)).

**When to use it — the collection is a *consumer* (you only write *into* it).**
A collection of any supertype of `E` can safely hold an `E`, so writing is always safe. 
(This is the **CS** in PECS — *Consumer → super*.)

```java
public class Stack<E> {
  // rigid:    public void popAll(Collection<E> dst)          // rejects Collection<Object>
  public void popAll(Collection<? super E> dst) {             // flexible
    while (!isEmpty()) dst.add(pop());   // safe: dst can hold any E
  }
}

Stack<Number> numberStack = new Stack<>();
numberStack.popAll(new ArrayList<Object>());   // OK: Collection<Object> matches Collection<? super Number>
```

**Why you can only write:** the exact supertype is unknown, so reading back out gives only `Object`
(that's all the compiler can guarantee). Writing an `E` in is safe; reading a specific type out is not.
That write-only nature is exactly why it suits a consumer — the mirror image of `? extends`, which is read-only.

</details>

### When should you use `<? extends T>` versus `<? super T>` in generic APIs?
<details><summary>Show answer</summary>

Apply **PECS — Producer-Extends, Consumer-Super.** Decide by what the API does with the collection:

- **Only reads *out* of it** → it's a **producer** → `<? extends T>`.
- **Only writes *into* it** → it's a **consumer** → `<? super T>`.
- **Both reads and writes** → no wildcard; require the exact type `<T>` (only an exact match is safe both ways). 
  *(If that exact type happens to be `Object` — a collection of truly anything — this is just `List<Object>`; 
  same principle, read and write the same declared type.)*

(Naftalin & Wadler call this the **Get and Put Principle** — *get → extends, put → super*.)

The reasoning: 
- an upper bound guarantees everything you *read* is at least a `T` (safe to get); 
- a lower bound guarantees the target can *hold* a `T` (safe to put). 


Full mechanics in the 
[`? extends`](#how-do-you-make-a-generic-api-accept-a-collection-of-a-specific-type-or-any-of-its-subtypes)
and [`? super`](#how-do-you-make-a-generic-api-accept-a-collection-of-a-specific-type-or-any-of-its-supertypes) cards.

</details>

### `T extends Type` vs `? extends Type` (vs `? super Type`) — what's the difference?
<details><summary>Show answer</summary>

All three are upper/lower bounds, but they split on one axis: **can you name the type?**

- **`<T extends X>` — named parameter.** The type has a name `T`, so you can *refer to it again*:
  return it, declare variables of it, or force two arguments to be the *same* type.
  → gives the **implementer capability**.
- **`<? extends X>` — upper-bounded wildcard.** The type is unknown and *cannot be named* — only known
  to be some subtype of `X`. Read-only (producer). → gives the **caller flexibility**.
- **`<? super X>` — lower-bounded wildcard.** Unknown, unnameable, known to be some supertype of `X`.
  Write-only (consumer). → caller flexibility, the other direction.

**The deciding rule — how many times does the type appear in the signature?**
- Appears **once** → a wildcard is enough, and cleaner.
- Appears **more than once** (the occurrences must be the *same* type) → you need a named `T`.

```java
<T extends Number> T firstOf(List<T> list)          // T returned → must name it
void printAll(List<? extends Number> list)          // only consumed → wildcard
<T> void copy(List<? super T> dst, List<? extends T> src)  // T links the two params → named + wildcards
```

The last line shows all three together: `T` names the shared element type, 
`? extends T` is the producer source, `? super T` is the consumer destination — PECS in one signature.

</details>

### How to enforce that a type interacts only with its exact same type, and when is this useful?
<details><summary>Show answer</summary>

Use a **recursive type bound** — bound `T` by an expression that mentions `T` itself: `<T extends Comparable<T>>`.
It reads: "`T` must be comparable **to its own type**" — a peer-to-peer guarantee, 
so a `T` interacts only with another `T`.

**When it's useful:**
- **Mutual comparison** — `max`/`min`/sort utilities, where a `T` must compare against another `T`.
- **Merging state** — an interface like `Mergeable<T>` where a `UserAccount` merges only with another `UserAccount`.

**Example — finding a maximum:**
```java
public static <T extends Comparable<T>> T findMax(List<T> list) {
   T max = list.get(0);
   for (T current : list) {
      if (current.compareTo(max) > 0) max = current;   // safe: T compares to T
   }
   return max;
}
```
The bound is what makes `current.compareTo(max)` legal: it guarantees `T` has a `compareTo` that takes a `T`.

</details>

### What should you consider with `<T extends Comparable<T>>`?
<details><summary>Show answer</summary>

It is the **strict** form — it demands that comparability be declared on the *exact* type `T`, 
which **inheritance breaks**.

**The problem, concretely.** Suppose a base class declares the ordering:
```java
class Animal implements Comparable<Animal> {
    public int compareTo(Animal other) { ... }   // animals compare as animals
}
class Dog extends Animal { }                       // Dog adds nothing
```
Is `Dog` comparable? Yes — but it implements `Comparable<Animal>` (inherited), **not** `Comparable<Dog>`. 
So `Dog` does *not* satisfy `Dog extends Comparable<Dog>` — there is no `Comparable<Dog>` anywhere. 
A method bounded `<T extends Comparable<T>>` therefore **rejects `Dog`**, even though dogs are perfectly comparable.

**The fix — accept comparability from `T` or any supertype:**
```java
<T extends Comparable<? super T>>
```
This reads: "`T` is comparable to itself **or to one of its supertypes**." 
For `Dog`, `Comparable<Animal>` now satisfies the bound, because `Animal` is `? super Dog`. 
The method accepts `Dog` again.

**Why `? super` specifically — it's just PECS.** A `Comparable<T>` is a **consumer** of `T`: 
its `compareTo(T)` *takes in* a `T`. PECS says *consumer → super*. 
So whenever you see `Comparable` (a consumer) in a bound, `? super` is the mechanical choice — not a special incantation.
`Comparable<? super T>` is simply PECS applied to the recursive bound.

**Takeaway:** `Comparable<T>` is the strict version; 
`Comparable<? super T>` is the inheritance-aware version that honors comparability declared on a parent. 
Library code prefers the latter (e.g. `Collections.max`).

</details>

### How can a base class method return the concrete subtype rather than the base type?
<details><summary>Show answer</summary>

Use the **self-type idiom** — a recursive bound where the base type carries its own concrete subtype as a parameter: 
`class Parent<C extends Parent<C>>`.

**The problem it solves.** In a fluent builder with inheritance, a method declared on the parent returns `Parent`, 
which breaks the chain on a subclass:
```java
class Builder {
    Builder name(String n) { ...; return this; }   // returns Builder
}
class DogBuilder extends Builder {
    DogBuilder breed(String b) { ...; return this; }
}

new DogBuilder().name("Rex").breed("Lab");   // COMPILE ERROR
// name() returns Builder, which has no breed()
```

**The fix — parameterize the base on its own subtype:**
```java
class Builder<C extends Builder<C>> {
    @SuppressWarnings("unchecked")
    C self() { return (C) this; }
    C name(String n) { ...; return self(); }        // returns C, the concrete subtype
}
class DogBuilder extends Builder<DogBuilder> {
    DogBuilder breed(String b) { ...; return self(); }
}

new DogBuilder().name("Rex").breed("Lab");   // OK — name() returns DogBuilder
```

`C` stands for "the concrete subclass," so inherited methods return that subclass, and the chain keeps
working. Java has no native *self-type*, so this idiom simulates one. 
(It needs one unchecked cast in `self()`, which is safe by construction.)

</details>

### How should wildcard types be used in the return types of generic methods, and why?
<details><summary>Show answer</summary>

**The rule: never return a bounded wildcard.** Accept wildcards as *inputs* (PECS), 
but always return an **exact** parameterized type (`List<T>`, `Set<Number>`), not `List<? extends T>`.

**Why — wildcard complexity must stay inside your API.** Wildcards exist to make a method *accept*
broader inputs; that complexity is meant to be hidden. If you *return* a wildcard, you force the client
to write wildcards in their own variables to hold the result, spreading generic complexity into their
code:
```java
// BAD — leaks the wildcard to the caller
public Set<? extends Number> getNumbers() { ... }
Set<? extends Number> mine = api.getNumbers();   // client now polluted

// GOOD — exact return type, clean call site
public Set<Number> getNumbers() { ... }
Set<Number> mine = api.getNumbers();
```

**The diagnostic (a code smell):** *if a client must think about or write wildcard types just to use
your API, the API is flawed* — almost always because it returns a wildcard. Good generic APIs are
nearly invisible: flexible wildcard *inputs*, exact concrete *outputs*.

</details>
