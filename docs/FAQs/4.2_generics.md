## Generics

### What are the rules for using generic type parameters with static methods or static fields?
<details><summary>Show answer</summary>

Static members cannot use the class-level type parameters.

**The Reason:**
Class-level type parameters (e.g., the `T` in `public class Box<T>`) are tied to a specific **instance** of the class.
Because static methods and fields belong to the class itself and are evaluated before any instances are created,
the compiler has no way to know what `T` is supposed to be.

**The Solution:**
If a static method needs to be generic, it must declare its own independent type parameters,
completely separate from the class. This is known as a **Generic Method**.

**Example:**
```java
public class Box<T> {
    // ILLEGAL: Cannot use 'T' in a static context
    // public static void printBox(Box<T> box) { ... }

    // LEGAL: The method declares its own independent type parameter '<E>'
    public static <E> void printBox(Box<E> box) { ... }
}
```
</details>

### Suppose you have a stateless, immutable object (like an empty set). How can you share a single instance of this object across multiple different generic types in a type-safe way?
<details><summary>Show answer</summary>

To achieve this, you must use a pattern called the **Generic Singleton Factory**.
This pattern leverages the fact that generic types are erased at runtime,
allowing you to use a single instance for any type parameterization.

Here is how it works, broken down into three logical steps:

**1. The Problem:**
Because generics are invariant, a `Set<Object>` cannot be assigned to a `Set<String>`.
Therefore, you cannot simply declare a single `public static final Set<T>`
because the compiler doesn't know what `T` is in a static context.
Creating `new EmptySet<T>()` every time wastes memory.

**2. The Mechanism (Why it works):**
At runtime, due to **Type Erasure**, a `Set<String>` and a `Set<Integer>` look exactly the same—they are just a raw `Set`.
If the object has no state (no data that can cause a `ClassCastException`),
it is logically safe to cast it to any generic type.

**3. The Implementation Pattern:**
You create one private instance using the raw type (or `Object`),
and then expose it through a **generic static factory method** that casts it to the caller's requested type.

**Code Example (Identity Function):**
```java
// 1. The Singleton: A single instance created once, using Object.
private static final UnaryOperator<Object> IDENTITY_FN = (arg) -> arg;

// 2. The Factory Method: Exposes the instance safely to the requested type <T>.
public static <T> UnaryOperator<T> identityFunction() {
    
    // 3. The Strict Cast: We introduce a local variable to strictly scope the warning suppression.
    // We know this cast is safe because the function does not modify or depend on the type's state.
    @SuppressWarnings("unchecked")
    UnaryOperator<T> result = (UnaryOperator<T>) IDENTITY_FN;
    
    return result;
}
```
</details>

### What is the main limitation of `<T>` and its solution?
<details><summary>Show answer</summary>

**The Limitation: Lack of Type Information**
When you declare a basic generic type parameter like `<T>`, the compiler has no information about what specific
type `T` will be at runtime. It must assume the safest, most general type possible: `java.lang.Object`.
As a result, inside your generic class or method, you are strictly limited to calling methods defined on `Object`
(such as `.toString()` or `.equals()`). You cannot perform math, sort the objects, or call custom business logic.

**The Solution: Applying Type Bounds**
To unlock specific functionality, you must restrict the generic parameter by applying a **bound** (e.g.,
`<T extends SomeType>`).

This creates a compile-time contract. You guarantee to the compiler that `T` will always be at least a `SomeType`.
The compiler then safely grants you access to invoke any methods defined on `SomeType` directly on instances of `T`.

**Advanced Applications:**
Once this basic contract is understood, it serves as the foundation for complex generic idioms:
1. **Wildcard Bounds (`<? extends T>` / `<? super T>`):** Used to restore subtyping flexibility for collections (covariance and contravariance).
2. **Recursive Type Bounds (`<T extends Comparable<T>>`):** Used when an object needs to be compared to or interact with objects of its exact same type.
3. **The Self-Type Idiom (`<T extends Builder<T>>`):** Used in fluent APIs (like Builders) to ensure method chaining works correctly across class inheritance hierarchies.

</details>


### What types of generic bounds exist in Java?
<details><summary>Show answer</summary>

In Java, bounded types restrict the types that can be used as arguments. The main categories include:

1. **Upper Bound (`<T extends Type>`):** Restricts the parameter to a specific class and its subclasses
   (or interface implementers).
2. **Multiple Bounds (`<T extends ClassA & InterfaceB>`):** Forces the type to be a subtype of all listed types.
   (Note: Only one class is allowed, and it must be listed first).
3. **Lower Bound Wildcard (`<? super Type>`):** Restricts the unknown type to be a specific class or its
   superclasses. (Note: Lower bounds can only be used with wildcards `?`, not with named parameters like `T`).
4. **Recursive Type Bound (`<T extends Comparable<T>>`):** A type parameter is bounded by an expression involving
   itself. Commonly used to allow objects to compare themselves.
5. **Self-Type Idiom / F-Bound (`<T extends Base<T>>`):** An advanced pattern where a base class takes a type
   parameter representing its concrete subclass, enabling fluent method chaining across inheritance hierarchies.

</details>

### How do you make a generic API accept a collection of a specific type or any of its subtypes?
<details><summary>Show answer</summary>

**The Problem: API Rigidity due to Invariance**
Because generics are invariant, a `List<Integer>` is strictly NOT a `List<Number>`.
If you write an API that accepts an `Iterable<Number>` to read data from, the compiler will reject a
`List<Integer>`, even though every Integer is a Number and it is logically perfectly safe.
This strictness makes your API rigid and frustrating for clients to use.

**The Solution: Upper Bounded Wildcards**
You solve this by using an upper bound wildcard: `<? extends Type>`.
This tells the compiler: "Accept a collection parameterized by this exact type, or any type that
extends (subclasses) it."

**When is this useful? (The "Producer" Rule)**
This is strictly used when the generic collection acts as a **Producer**—meaning your API is only
*reading* data out of it. Because the upper bound guarantees every element is at least the base type,
reading them as the base type is 100% safe.
(This is the "PE" in the famous PECS mnemonic: Producer-Extends).



**Code Example (A flexible pushAll method):**
Imagine a custom `Stack<E>`. We want a method to push multiple elements onto it at once.

```java
public class Stack<E> {
  // ... basic stack methods ...

  // RIGID API: Only accepts exactly Iterable<E>
  // public void pushAll(Iterable<E> src) { ... }

  // FLEXIBLE API: Accepts Iterable of E or any subtype of E
  public void pushAll(Iterable<? extends E> src) {
    for (E e : src) {
      push(e); // Safe: we know 'e' is at least an 'E'
    }
  }
}

// Client Usage:
Stack<Number> numberStack = new Stack<>();
List<Integer> integers = List.of(1, 2, 3);

// This compiles smoothly because Iterable<Integer> matches Iterable<? extends Number>
numberStack.

pushAll(integers);
```
</details>

### How do you make a generic API accept a collection of a specific type or any of its supertypes?
<details><summary>Show answer</summary>

**The Problem: API Rigidity when Writing Data**
Because generics are invariant, a `List<Object>` is strictly NOT a `List<Number>`.
If you write an API that pops `Number` elements off a stack and dumps them into a user-provided collection,
requiring exactly a `Collection<Number>` rejects a perfectly valid `Collection<Object>`.
This makes the API rigid and needlessly restricts where clients can store their results.

**The Solution: Lower Bounded Wildcards**
You solve this by using a lower bound wildcard: `<? super Type>`.
This tells the compiler: "Accept a collection parameterized by this exact type, or any of its superclasses."



**When is this useful? (The "Consumer" Rule)**
This is strictly used when the generic collection acts as a **Consumer**—meaning your API is only
*writing* data into it. Because a collection of `Object` can safely hold a `Number`, writing into a
supertype's collection is 100% safe.
(This is the "CS" in the famous PECS mnemonic: Consumer-Super).

**Code Example (A flexible popAll method):**
Imagine our custom `Stack<E>`. We want a method to pop all elements and dump them into a target collection.

```java
public class Stack<E> {
    // ... basic stack methods ...

    // RIGID API: Only accepts exactly Collection<E>
    // public void popAll(Collection<E> dst) { ... }

    // FLEXIBLE API: Accepts Collection of E or any supertype of E
    public void popAll(Collection<? super E> dst) {
        while (!isEmpty()) {
            // Safe: dst is guaranteed to be a collection that can hold an E
            dst.add(pop()); 
        }
    }
}

// Client Usage:
Stack<Number> numberStack = new Stack<>();
// Assume stack is populated...
Collection<Object> objects = new ArrayList<>();

// This compiles smoothly because Collection<Object> matches Collection<? super Number>
numberStack.popAll(objects);
```
</details>

### When should you use `<? extends T>` versus `<? super T>` in generic APIs?
<details><summary>Show answer</summary>

To decide which wildcard bound to use, apply the **PECS** rule: **Producer-Extends, Consumer-Super**.
This rule dictates how to handle collections based on whether they are providing data or receiving data.
Naftalin and Wadler call it **the Get and Put Principle**.

**1. Producer Extends (`<? extends T>`)**
If your API only reads data *out of* the collection, the collection acts as a Producer.
Use `extends`. This allows clients to pass a collection of `T` or any of its subtypes, ensuring maximum
flexibility while guaranteeing that everything you read is safely treated as at least a `T`.

**2. Consumer Super (`<? super T>`)**
If your API only writes data *into* the collection, the collection acts as a Consumer.
Use `super`. This allows clients to pass a collection of `T` or any of its supertypes, guaranteeing
that the target collection is broad enough to safely accept elements of type `T`.

**What if the collection does both?**
If your API needs to both read from and write to the same collection, do not use wildcard bounds at all.
You must require an exact type match (e.g., `List<T>`) to ensure complete type safety.

**Summary Checklist:**
* Only reading from the collection? -> It's a Producer -> Use `<? extends T>`
* Only writing to the collection? -> It's a Consumer -> Use `<? super T>`
* Doing both? -> Use the exact type -> `<T>`

</details>

### How to enforce that a type interacts only with its exact same type, and when is this useful?
<details><summary>Show answer</summary>

**The Problem: The Risk of Cross-Type Interactions**
When you declare that a type simply "can be compared" or "can be merged" using a raw or broad bound
(like `<T extends Comparable>`), you leave a loophole. For example, the compiler might allow an `Apple`
to be compared to an `Orange` if both inherit from a common `Comparable<Fruit>` interface.

To prevent this, we need a strict peer-to-peer guarantee: the compiler must prove that an object is
interacting *exclusively* with its exact same concrete type.

**When is this strict peer-to-peer guarantee useful?**
1. **Mutual Comparison:** Utility methods that sort or find the max/min. You need to guarantee that
   an `Apple` can only ever be compared to another `Apple`.
2. **Merging or Updating State:** If you have an interface `Mergeable<T>`, you want to ensure that a
   `UserAccount` can only merge data from another `UserAccount`, not from an `AdminAccount`.
3. **Fluent Chaining (Builders):** In inheritance hierarchies, ensuring that calling a method on a
   subclass builder returns the subclass type (not the parent type) so you can keep chaining methods.

**The Solution: Recursive Bounds**
You solve this by bounding the type parameter with an expression containing the type parameter itself.
The standard syntax is: `<T extends SomeInterface<T>>`.

This establishes a strict compile-time rule: "Type `T` must implement `SomeInterface`, and that interface
must be specifically parameterized with type `T` itself."



**Code Example (Finding a Maximum):**
```java
// <T extends Comparable<T>> guarantees an Apple only compares to an Apple.
public static <T extends Comparable<T>> T findMax(List<T> list) {
    if (list.isEmpty()) {
        throw new IllegalArgumentException("List is empty");
    }
    
    T max = list.get(0);
    for (int i = 1; i < list.size(); i++) {
        T current = list.get(i);
        
        // We can safely invoke compareTo(T o) because the recursive bound 
        // guarantees it is a strictly peer-to-peer interaction.
        if (current.compareTo(max) > 0) {
            max = current;
        }
    }
    return max;
}
```
</details>

### How should wildcard types be used in the return types of generic methods, and why?
<details><summary>Show answer</summary>

**The Rule: Avoid Wildcards in Return Types**
You should never use bounded wildcard types (like `<? extends T>` or `<? super T>`) as the return type of a method.
Instead, you should always return an exact, specific parameterized type (like `List<T>`).

**The Problem: Leaking Complexity**
Wildcard types are designed to make APIs flexible by accepting broader inputs, but their complexity should
remain hidden inside the API itself.

If you return a bounded wildcard, you force the client to use wildcards in their own local variables to
handle the result. This spreads generic complexity throughout the client's codebase. As a general rule:
if the user of your class has to think about wildcard types, there is a flaw in your API's design.

**Code Example:**
```java
// BAD: Forces the client to deal with the wildcard complexity
public Set<? extends Number> getNumbers() { ... }

// Client code becomes polluted:
Set<? extends Number> myNumbers = api.getNumbers(); 

// GOOD: Returns an exact type, keeping the client code clean and simple
public Set<Number> getNumbers() { ... }

// Client code is clean:
Set<Number> myNumbers = api.getNumbers();
```
</details>

### What code smell in client code indicates a poorly designed generic API?
<details><summary>Show answer</summary>

**The Indicator:**
If the user of a class has to think about or explicitly write wildcard types (like `<? extends T>`)
in their own client code just to use your API, there is likely a fundamental flaw in its design.

**Why is this a code smell?**
Wildcards are an advanced feature designed strictly to make APIs more flexible in what they accept
(as method parameters). Their complexity is supposed to be completely hidden from the user.

When a client is forced to use wildcards—almost always because the API improperly returns a wildcard type—the
complexity of the API has "leaked" into the client's codebase. This forces every user to deal with
generic boundaries just to store or process the results.

**The Golden Rule:**
Good generic APIs are virtually invisible. They should accept flexible wildcard inputs (PECS), but always
return exact, concrete generic types so the client can use them cleanly and without mental overhead.

</details>

### How do you resolve a compilation error when the compiler fails to infer the correct generic type for a method call?
<details><summary>Show answer</summary>

**The Problem: Type Inference Failure**
Java's compiler uses type inference to guess the type parameters of a generic method based on the arguments
and the target variable. However, in complex expressions (like passing a generic method's result directly
into another method), the compiler might fail to infer the correct type, resulting in a compilation error.

**The Solution: Explicit Type Arguments (Type Witness)**
You can bypass the compiler's inference engine by explicitly telling it which type to use. You do this by
placing the type parameter in angle brackets directly before the method name.

**Syntax:**
* For static methods: `ClassName.<Type>methodName()`
* For instance methods: `instance.<Type>methodName()`
* For methods in the same class: `this.<Type>methodName()`

**Code Example:**
Imagine a method `void printNames(List<String> names)`. In older Java versions, inference might fail here:

```java
// The compiler might infer List<Object> instead of List<String>, causing an error:
printNames(Collections.emptyList()); 

// The fix: Provide an explicit type argument <String> to guide the compiler:
// Explicit type parameter - required prior to Java 8
printNames(Collections.<String>emptyList());
```

Note: Java 8 vastly improved target type inference, making explicit type arguments rarely necessary today,
but they remain the ultimate fallback when the compiler gets confused

</details>

### Which signature should you prefer for a `swap` method: one with `<E>` or one with `<?>`?
<details><summary>Show answer</summary>

As a rule, if a type parameter appears only once in a method declaration,
replace it with a wildcard. If it’s an unbounded type parameter, replace it with
an unbounded wildcard; if it’s a bounded type parameter, replace it with a
bounded wildcard.



**The Preferred API: Unbounded Wildcard (`<?>`)**
Between `public static <E> void swap(List<E> list)` and `public static void swap(List<?> list)`, you
should prefer the wildcard version (`<?>`).

**The Rule:** If a type parameter appears only once in a method declaration, replace it with a wildcard. It makes the
API simpler and cleaner for clients, as they don't have to think about generic type parameters at all.

**The Implementation Problem:**
If you implement `swap` using `List<?>`, you will hit a compilation error. Because the list's type is
unknown (`?`), the compiler prohibits putting any element (other than `null`) into it. You cannot even
put an element back that you just took out of the exact same list!

```java
public static void swap(List<?> list, int i, int j) {
    // ERROR: The compiler won't allow inserting into a List<?>
    // list.set(i, list.set(j, list.get(i))); 
}
```

The Solution: Wildcard Capture
To fix this, you keep the public API simple with <?>, but you delegate the actual work to a private
helper method that uses <E>. The helper method "captures" the unknown type, giving it a name (E)
so the compiler can verify that taking an element out and putting it back is perfectly type-safe.

```java
// 1. The Public API: Clean and simple for the client
public static void swap(List<?> list, int i, int j) {
    swapHelper(list, i, j);
}

// 2. The Private Helper: Captures the type 'E' for safe insertion
private static <E> void swapHelper(List<E> list, int i, int j) {
    list.set(i, list.set(j, list.get(i)));
}
```

</details>

### How do you define a method that accepts a variable number of arguments?
<details><summary>Show answer</summary>

**The Solution: Varargs (`...`)**
In Java, you can pass a variable number of arguments of the same type to a method using the "varargs" syntax.
You define this by placing three dots (`...`) after the type name in the method parameter list.

**How it works:**
Under the hood, Java automatically packs the provided arguments into an array of that specific type. Inside
the method, you simply treat the parameter as a standard array.

**Code Example:**
```java
// The parameter 'numbers' is treated as an int[] inside the method
public static int sum(int... numbers) {
  int total = 0;
  for (int num : numbers) {
    total += num;
  }
  return total;
}

// Client Code: You can pass zero or more arguments
sum();           // Returns 0 (passes an empty array)
sum(1);          // Returns 1
sum(1,2,3,4); // Returns 10
```

**The Core Rule**:
A method can only have one varargs parameter, and it must be the very last parameter in the method signature
(e.g., `public void doSomething(String name, int... numbers)`).

</details>

### How do you use varargs with generic type parameters?
<details><summary>Show answer</summary>

**The Implementation:**
You define it just like standard varargs, using the generic type parameter: `public static <T> void process(T... args)`.

**The Problem: The "Generic Array" Contradiction**
While the syntax is simple, it hides a dangerous contradiction in Java's type system:
1. Varargs are implemented by creating an array under the hood to hold the arguments.
2. Arrays are *reified* (they enforce their type information at runtime).
3. Generics use *type erasure* (their type information is removed at runtime).

Because of this, declaring `T... args` forces the JVM to create an array of a non-reifiable type
(essentially an `Object[]` at runtime). This creates a vulnerability known as **Heap Pollution**, where
the array might hold objects of the wrong type without failing immediately.

If you modify this hidden array or return it to the caller, you can cause `ClassCastException`s in client
code that seem to come out of nowhere. Because of this danger, the compiler issues a strong warning.

**The Solution: `@SafeVarargs`**
If you guarantee that your method is type-safe—meaning it only reads from the array and never exposes
or returns the array to the client—you should annotate the method with `@SafeVarargs`. This tells the
compiler to suppress the warning because you have architecturally guaranteed its safety.

**Code Example:**
```java
// @SafeVarargs promises we won't do anything dangerous with the hidden generic array.
@SafeVarargs
public static <T> void printAll(T... elements) {
  for (T element : elements) {
    System.out.println(element); // Safe: We are strictly reading data
  }
}
```

</details>

### How can combining generic types and varargs cause a runtime crash?
<details><summary>Show answer</summary>

**The Problem: Heap Pollution**
Because varargs create arrays under the hood, and generic types are erased at runtime, a generic varargs
parameter like `List<String>...` actually becomes a raw array (behaving like an `Object[]`) at runtime.

If you assign this array to an `Object[]` reference, you can sneak a completely different type of list
into it. The compiler will not stop you, but the code will crash later when an invisible, auto-generated
cast fails. This contamination of the type system is called **Heap Pollution**.

**Code Example (The Danger):**
```java
public static void dangerousMethod(List<String>... stringLists) {
  // 1. The varargs array is secretly an Object[] at runtime.
  Object[] objects = stringLists;

  // 2. We create a list of Integers.
  List<Integer> intList = List.of(42);

  // 3. We put the Integer list into the array.
  // HEAP POLLUTION: The array now holds the wrong generic type.
  objects[0] = intList;

  // 4. We try to read a String from the first list.
  // The compiler secretly inserted a cast to String here.
  // CRASH: Throws ClassCastException (Integer cannot be cast to String).
  String s = stringLists[0].get(0);
}
```

**The Golden Rule** to prevent this:
- never store a value into a generic varargs array inside your method, and
- never return the hidden varargs array to the client.

If a method does either of these, it is critically unsafe.

</details>

### What is the type-safe alternative to using a generic varargs parameter?
<details><summary>Show answer</summary>

**The Alternative: Use a `List`**
Instead of using generic varargs (e.g., `List<? extends T>... lists`), which creates hidden arrays and
risks heap pollution, you should replace the varargs parameter with a `List` (e.g., `List<List<? extends T>>`).

**Why is this better?**
Generics and arrays have fundamentally different type rules (erased vs. reified). Generics and Lists,
however, are designed to work perfectly together. By demanding a `List`, you rely entirely on the compiler's
strict type checking.

This completely eliminates the risk of heap pollution. You do not have to worry about the complex safety
rules of generic arrays, and you completely eliminate the need for the `@SafeVarargs` annotation.

**Code Example (Flattening Lists):**
Here is how you safely merge multiple lists into one, avoiding varargs entirely:

```java
// SAFE: Using a List of Lists instead of generic varargs
static <T> List<T> flatten(List<List<? extends T>> lists) {
    List<T> result = new ArrayList<>();
    for (List<? extends T> list : lists) {
        result.addAll(list);
    }
    return result;
}
```

**Client Usage:**
Thanks to modern Java convenience methods like `List.of()`, calling this type-safe method is just as
clean and concise for the client as calling a varargs method:

```java
// The client simply wraps their arguments in List.of()
List<String> combined = flatten(List.of(list1, list2, list3));
```

</details>

### What is the primary limitation of standard generic containers, and what concept solves this for dynamic data?
<details><summary>Show answer</summary>

**The Limitation: Homogeneous Containers**
Standard generic containers (like `Set<T>` or `Map<K, V>`) are strictly homogeneous. They have a fixed
number of type parameters defined at declaration. Once a `Map<String, Integer>` is created, it can only
ever hold Strings mapped to Integers. You cannot safely mix different types in the same container.

**When is this limitation a problem?**
This rigidness fails when you need to store arbitrarily many different types in a single structure.
A classic example is representing a database row: column 1 might be an `Integer` ID, column 2 a `String`
name, and column 3 a `LocalDate` timestamp.

If you try to store this dynamic data in a standard `Map<String, Object>`, you completely lose compile-time
type safety. The compiler cannot help you, and you are forced to cast every value you retrieve, risking
`ClassCastException`s.

**The Solution Concept:**
To solve this, you use a **Typesafe Heterogeneous Container**. Rather than parameterizing the container
itself, you parameterize the *key* used to insert and retrieve values. This shifts the type-safety burden
to the key, allowing the container to safely hold an arbitrary number of different types.

</details>

### How do you implement a typesafe heterogeneous container to store arbitrarily many different types?
<details><summary>Show answer</summary>

**The Implementation Pattern:**
To build a typesafe heterogeneous container, you parameterize the *key* instead of the container itself.
You use a `Class<T>` object as the key (often called a "type token").

Inside the container, you use a private backing `Map<Class<?>, Object>`. Because the map's value type
is `Object`, it can hold absolutely anything. The type safety is enforced entirely by your public API methods.



**How it guarantees type safety:**
1. **Insertion (`put`):** The method signature `<T> void putFavorite(Class<T> type, T instance)` forces the
   compiler to ensure the value perfectly matches the type token. You cannot map `Integer.class` to a `String`.
2. **Retrieval (`get`):** The method signature `<T> T getFavorite(Class<T> type)` pulls the raw `Object` from
   the map and uses the `Class.cast()` method to dynamically and safely cast it back to `T`.

**Code Example (The `Favorites` Class):**
```java
public class Favorites {
    // The internal map uses an unbounded wildcard for the key, and Object for the value.
    private Map<Class<?>, Object> favorites = new HashMap<>();

    // The method signature guarantees the value type 'T' matches the Class<T> key.
    public <T> void putFavorite(Class<T> type, T instance) {
        // We can optionally use type.cast(instance) here to achieve runtime type safety 
        // in case a raw type is maliciously passed in, but the signature handles compile-time.
        favorites.put(Objects.requireNonNull(type), instance);
    }

    // Returns exactly the type requested without the client needing to cast.
    public <T> T getFavorite(Class<T> type) {
        Object untypedValue = favorites.get(type);
        // type.cast() dynamically casts the Object back to T. It is 100% safe here.
        return type.cast(untypedValue);
    }
}

// Client Usage:
Favorites f = new Favorites();
f.putFavorite(String.class, "Java");
f.putFavorite(Integer.class, 42);
f.putFavorite(Class.class, Favorites.class);

// The client gets perfect type safety and no ugly casts!
String favoriteString = f.getFavorite(String.class);
int favoriteInt = f.getFavorite(Integer.class);
```

</details>

### Why can a Typesafe Heterogeneous Container store a `String`, but fail to safely store a `List<String>`?
<details><summary>Show answer</summary>

**The Trick: Non-Reifiable Types**
While a container backed by `Map<Class<?>, Object>` can hold almost anything, it has one major limitation:
it cannot be used to safely store and retrieve distinct **parameterized types** (like `List<String>`).

**Why is this impossible?**
To store a value in the container, you must provide its `Class` object as the key. You can easily pass
`String.class` or even an array type like `String[].class` (because arrays are reifiable).

However, `List<String>.class` is a syntax error in Java. Because of type erasure, both `List<String>`
and `List<Integer>` share the exact same runtime class object: `List.class`.

**The Consequence:**
If you try to store a "favorite `List<String>`" and a "favorite `List<Integer>`", you are forced to use
`List.class` as the key for both. The second insertion will silently overwrite the first one in the map.
The container simply cannot distinguish between different parameterizations of the same generic base class.

*(Note: Frameworks like Spring and Guava solve this limitation using an advanced workaround known as
"Super Type Tokens" or `TypeToken`, but a standard `Class<?>` key cannot handle it).*

</details>

### How does a typesafe heterogeneous container safely cast an `Object` back to its original type upon retrieval?
<details><summary>Show answer</summary>

**The Problem:**
The backing `Map` in a typesafe heterogeneous container stores values as `Object`. When retrieving a value,
the compiler only knows it is an `Object`, but the client expects the specific type `T`.

**The Solution: `Class.cast()`**
The `get` method uses the dynamic `cast()` method provided by the `Class<T>` object.

This method checks if the object is an instance of the class. If it is, it returns the object cast to `T`.
Because the signature of `Class.cast` returns `T`, it perfectly bridges the gap between the raw `Object`
in the map and the generic type required by the client, avoiding unchecked cast warnings.

**Code Example:**
```java
public <T> T getFavorite(Class<T> type) {
    Object untypedValue = favorites.get(type);
    return type.cast(untypedValue); // Safely casts Object to T
}
```

</details>

### How can a client corrupt a typesafe heterogeneous container, and how do you prevent it?
<details><summary>Show answer</summary>

**The Vulnerability: Raw Types**
A malicious or careless client could use a `Class` object in its raw form (without generics) to bypass
the compile-time checks. This would allow them to map `Integer.class` to a `String` value, completely
breaking the type safety of the container.

**The Solution: Dynamic Casting on Insertion**
To achieve bulletproof runtime type safety, you can use the `Class.cast()` method during the *insertion* process as well. This guarantees the instance perfectly matches the type token before it enters the map.

**Code Example:**
```java
// Achieving runtime type safety with a dynamic cast
public <T> void putFavorite(Class<T> type, T instance) {
    // type.cast() will throw ClassCastException immediately if types don't match
    favorites.put(type, type.cast(instance));
}
```

### What is the purpose of `Collections.checkedList` and similar wrappers?
<details><summary>Show answer</summary>

**The Purpose: Runtime Reification for Standard Collections**
Normally, a `List<String>` loses its generic type at runtime due to type erasure. If legacy code uses a
raw `List` reference to insert an `Integer` into your `List<String>`, it will corrupt the collection.

Wrappers like `Collections.checkedList` solve this by taking a `Class` object (a type token) alongside
the collection. They use this token to dynamically check every element added to the collection at runtime.
If someone tries to insert the wrong type, it immediately throws a `ClassCastException`.

This is highly useful for tracking down bugs in applications that mix generic and raw types.

</details>

### How do you restrict the types of keys allowed in a typesafe heterogeneous container?
<details><summary>Show answer</summary>

**The Solution: Bounded Type Tokens**
By default, a type token `Class<T>` is unbounded (it accepts any class). If you want to limit the
container to only accept certain types of keys, you use a bounded type parameter in the method signature.

**Code Example (The Annotations API):**
The Java Annotations API uses this perfectly. To ensure a method only accepts classes that represent
annotations, it bounds the type token using `<T extends Annotation>`:

```java
// The bounded token Class<T> strictly requires T to be a subclass of Annotation
public <T extends Annotation> T getAnnotation(Class<T> annotationType);
```
In this scenario, the annotated element itself acts as a typesafe heterogeneous container where the keys
are exclusively annotation types.

### How do you safely cast an unknown `Class<?>` object into a bounded type token?
<details><summary>Show answer</summary>

**The Problem:**
If you look up a class at runtime (e.g., using `Class.forName()`), you get an unbounded `Class<?>`.
If you need to pass this into an API that requires a bounded token (like `Class<? extends Annotation>`),
a standard cast like `(Class<? extends Annotation>)` will generate an unchecked compiler warning.

**The Solution: `Class.asSubclass()`**
You use the `asSubclass()` instance method. This method dynamically checks if the class object on which
it is called is a valid subclass of the argument provided. If successful, it safely returns the casted
class object; otherwise, it throws a `ClassCastException`.

**Code Example:**
```java
// Safely cast an unknown Class<?> to a bounded Class<? extends Annotation>
Class<?> unknownType = Class.forName("com.example.MyAnnotation");

// asSubclass guarantees type safety without unchecked compiler warnings
return element.getAnnotation(unknownType.asSubclass(Annotation.class));
```

### How do you capture and retain full parameterized type information at runtime, bypassing type erasure?
<details><summary>Show answer</summary>

**The Problem: Type Erasure of Parameterized Types**
As we discussed, you cannot use `List<String>.class` because type erasure reduces it to `List.class`.
This is a massive hurdle for libraries like Jackson (JSON deserialization) or Spring (dependency injection)
that absolutely must know the exact generic payload type to reconstruct objects at runtime.

**The Solution: Super Type Tokens**
To bypass type erasure, you use an ingenious reflection trick known as the **Super Type Token** pattern.
It relies on a specific JVM loophole: while the JVM erases the type arguments of instantiated objects, it
strictly retains the generic type definitions of *classes and subclasses* in the compiled bytecode.

To exploit this, you create an abstract generic base class. When you need to capture a complex type,
you create an anonymous inner subclass of that base class. Because it is a subclass declaration, the
compiler permanently bakes the generic type parameters directly into the class file.

**Code Example:**
```java
// 1. The framework provides an abstract class to act as the token.
public abstract class TypeReference<T> {
    public final Type type;
    
    protected TypeReference() {
        // Reflection magic: retrieves the generic type from the subclass declaration itself
        Type superclass = getClass().getGenericSuperclass();
        // Extracts the actual type argument (e.g., List<String>)
        this.type = ((ParameterizedType) superclass).getActualTypeArguments()[0];
    }
}

// 2. The Client creates an anonymous subclass using empty braces {}.
// By subclassing, the compiler permanently bakes "List<String>" into the bytecode.
TypeReference<List<String>> token = new TypeReference<List<String>>() {};

// The framework can now read the exact, fully parameterized type at runtime!
System.out.println(token.type); // Prints: java.util.List<java.lang.String>
```

*(Note: This is exactly how Jackson's `new TypeReference<List<MyObject>>() {}` and Spring's
`ParameterizedTypeReference` operate under the hood).*

</details>
