## Classes and Interfaces - Interfaces and Abstract Classes

### How can Java define a type that allows multiple implementations?
<details><summary>Show answer</summary>

Two mechanisms:

- **Interfaces** — define a contract that multiple, unrelated classes can implement.
- **Abstract classes** — define a partially implemented type that subclasses extend and complete.

The decisive difference: a class can implement **many** interfaces but extend **only one** abstract class —
which is the root of the guidance to
[**prefer interfaces over abstract classes**](#interfaces-or-abstract-classes-how-should-a-type-with-multiple-implementations-be-defined).

</details>

### Interfaces or abstract classes: how should a type with multiple implementations be defined?
<details><summary>Show answer</summary>

**Prefer interfaces.** A class can extend only **one** abstract class (Java's single inheritance),
so an abstract class as a type definition forces itself into the subclass's one inheritance slot.
An interface can be implemented by **any** class regardless of its hierarchy position —
so it composes freely where abstract classes compete for the single slot.

</details>

### What is a mixin, and how is it used in Java?
<details><summary>Show answer</summary>

A **mixin** is a type a class implements *in addition to* its primary type, to declare some **optional behavior**. 
It doesn't define what an object **is** — that's the primary type's job — but what it **can do**.

Example: `Comparable` — implementing it declares instances *can be ordered*, a capability mixed in alongside the
class's real purpose.

**What it's for: adding an axis without overloading inheritance.** A class has only one is-a slot, so inheritance
carries exactly one axis safely. A mixin adds an *independent capability* as a separate axis — and because a class can
implement **many** interfaces, each in its own slot, you stack as many such axes as you need with no competition for
the inheritance slot and no diamond. This is the design-time way to escape the single-axis limit.
[→ how the multiple-axes problem is solved](3.4_classes_and_interfaces.md#how-is-the-multiple-axes-modeling-problem-solved)

Abstract classes can't serve as mixins — single inheritance leaves no slot to mix one in.

</details>

### Mixin vs composition — what's the trade-off?
<details><summary>Show answer</summary>

- A mixin binds its axis to the **type** at compile time: every instance of the class has the capability, and you cannot
  swap it at runtime.
- Composition binds per **instance** at runtime — you choose or replace the piece freely.


So a mixin is the right tool when the extra axis is a fixed trait of the type
(e.g. `Comparable` declaring "can be ordered"), and composition when the axis must vary or be swapped per object.

</details>

### How many classes can a Java class extend, and why?
<details><summary>Show answer</summary>

To avoid the **diamond problem** — the ambiguity multiple class inheritance creates.

If a class could extend two classes that both define (or both inherit) a method with the same signature,
a call to that method would be ambiguous: the compiler/runtime couldn't decide *which* inherited implementation to run.
With state, it's worse — the class could inherit two copies of the same ancestor's fields.
Restricting a class to **one** superclass removes the ambiguity at the source.

**Why interfaces are exempt:** traditionally they carried no implementation and no state — only method signatures —
so implementing many created no conflict to resolve. (Default methods changed that —
[→ when two interfaces collide](#how-do-interfaces-handle-multiple-inheritance-conflicts).)

</details>

### How do interfaces handle multiple inheritance conflicts?
<details><summary>Show answer</summary>

Default methods gave interfaces real implementation, so a class implementing two interfaces can inherit **two default
methods with the same signature** — interfaces reaching their own version of the diamond conflict.

Java does **not** guess or pick one. It refuses to compile until the programmer resolves it explicitly: the class must
override the method and choose, calling the one it wants with `Interface.super.method()`:

```java
interface A { default String greet() { return "A"; } }
interface B { default String greet() { return "B"; } }

class C implements A, B {
  @Override public String greet() {
    return A.super.greet();   // pick A's default explicitly — or B's, or something new
  }
}
```

**Why this stays safe where class inheritance didn't:** the conflict is *method-only*. Interfaces still carry no
instance state, so there's no two-copies-of-the-fields problem — only a behavior clash, and the language forces the
programmer to settle it rather than resolving it silently. That's why many interfaces remain allowed while multiple
class inheritance does not.

</details>

### How can implementation code be reused across many implementers of a type?
<details><summary>Show answer</summary>

Two mechanisms, with different sweet spots:

**1. Interface default methods.** The interface itself supplies reusable behavior, so every implementer
inherits it without an abstract class — and it composes with multiple-interface inheritance.
*Limits:* a default method **cannot hold state** (no instance fields) and there are no constructors —
so it can only provide behavior derivable from other interface methods.
Best for small, stateless, broadly-useful helpers.

**2. Abstract skeletal implementation (`AbstractInterface`).** An abstract class implements the
interface and provides most methods on top of a few **primitive** methods the subclass must supply —
the Template Method pattern. It *can* hold state and define constructors, so it handles the reuse that
default methods can't. The cost: it consumes the subclass's single inheritance slot.
By convention named `AbstractXxx` — e.g. `AbstractCollection`, `AbstractSet`, `AbstractList`.

**Choosing:** default methods for stateless behavior expressible via the interface's own methods;
a skeletal class when reuse needs state, constructors, or a larger Template-Method skeleton.
They combine well — an interface with defaults, *plus* an `AbstractXxx` for implementers who want the fuller scaffold.

</details>

### What should be considered before adding a default method to an interface?
<details><summary>Show answer</summary>

**The risk: it silently breaks existing implementers.** A default method is injected into *every* class
already implementing the interface — including ones you don't control and can't see. If such a class
has invariants the default method doesn't respect (e.g. synchronization, or a representation the
default ignores), it can misbehave at runtime **while still compiling cleanly**. This actually happened
when `removeIf`/`stream` were added to `Collection` — some third-party implementations broke.

So:
- **Existing interfaces — avoid unless critical.** If unavoidable, audit whether any implementer could
  be affected (and you often can't, for external ones).
- **New interfaces — the right use.** Defaults shine when defined up front: standard behavior, less
  implementation effort, no backward-compatibility risk.

</details>

### What is the proper purpose of an interface in Java?
<details><summary>Show answer</summary>

**An interface should define a *type*, and nothing else.**
Implementing it tells clients **what operations they can perform** on instances — a contract of capability
(the mixin lens: what an object *can do*).

The test: if implementing an interface doesn't make a client able to *do* something with instances,
it isn't serving as a type. Two common non-type uses are widely discouraged —
but each has a legitimate, narrow scope rather than being flatly forbidden:

- **Exposing constants** — the constant-interface pattern is discouraged for public APIs, though it
  has a defensible internal use.
  [→ exposing constants: options & trade-offs](#what-are-the-design-options-for-exposing-constants-in-java)
- **Grouping static utility members** — a utility *interface* (only static methods) is usually a *class*'s job,
  but is defensible in a contained/internal scope;
  the real axis is enforcement (sealed `final` class) vs convention (interface).
  [→ utility class vs utility interface](1.3_objects_creating_destroying.md#utility-class-vs-utility-interface--which-and-when)

So the rule is firm — *interfaces define types* — but the two non-type uses aren't absolute taboos:
each is acceptable under specific, contained circumstances, covered in its own card.

</details>

### What are the design options for exposing constants in Java?
<details><summary>Show answer</summary>

The deciding question is *how strongly the constants belong to a type*:

- **On an existing class/interface** — when the constants are an integral part of a specific abstraction,
  define them there. *Example:* `Integer.MIN_VALUE`, `Integer.MAX_VALUE`.
- **As an `enum`** — when they form a fixed set of related values; gives type safety and expressiveness.
- **In a non-instantiable utility class** — when they belong to no particular type:
  a `final` class with a private constructor and `public static final` fields.
- **In a constant interface** — conventionally discouraged for public APIs
  (it can leak constants into implementers' types), but the objection is about misuse and design intent,
  not correctness; in a contained internal scope its simplicity can be a legitimate win —
  the same enforcement-vs-convention / KISS trade-off as the utility-interface choice.
  [→ full argument](#what-are-the-drawbacks-of-using-interfaces-to-expose-constants)

</details>

### What are the drawbacks of using interfaces to expose constants?
<details><summary>Show answer</summary>

This is the **Constant Interface Pattern**, conventionally considered a poor use of interfaces.
But — exactly as with the 
[utility class vs utility interface](1.3_objects_creating_destroying.md#utility-class-vs-utility-interface--which-and-when)
debate — the case against it is smaller than it first looks, and collapses into two parts:


**1. Most of the "drawbacks" are really one misuse: implementing the interface.**
The classic criticisms — type abuse, a misleading *is-a* relationship, API pollution via inheritance —
all stem from a class doing `implements Constants` merely to access the constants. That's the error:
a constant interface is neither a marker nor a behavioral contract, so it should never be implemented or extended —
you reference the constants as `Constants.FOO`. Once that misuse is excluded, every one of those criticisms disappears.
Counting them as separate intrinsic flaws inflates the case dishonestly.

**2. The one real, intrinsic objection — enforceability.**
Same objection as the utility-interface case: an interface is **implementable by anyone, and the language won't stop
them**. "Don't implement this constants holder" holds **by convention only** — there's no `final` + private
constructor to seal it, so the misuse path stays open. That's an *enforceability hole the language leaves*, a checkable
technical fact, not a matter of taste.

Note the argument people *usually* reach for here is the **weak** one: "interfaces are meant to define types, so this
is a conceptual mismatch." That's belief, not mechanism — it appeals to what interfaces are *for*, which you can simply
disagree with. Don't lean on it. The real objection is enforceability, above.
(A few constant interfaces exist even in the JDK, e.g. `java.io.ObjectStreamConstants` — treated as anomalies, not models.)


**The trade-off (once misuse is off the table): enforcement vs pragmatic simplicity.**
A constant interface as a pure namespace offers real simplicity — no repeated `public static final`,
no private-constructor utility-class boilerplate, an immediately readable centralized definition.
That cuts cognitive load and maintenance.


So the deciding factor is **scope** — identical to the utility-interface verdict:
- **Public API / many unknown consumers** → avoid it; you can't prevent someone implementing it,
  and the enforcement guarantee earns its keep at scale.
- **Contained / internal scope you control** → defensible; the misuse won't happen,
  and the simplicity (KISS) earns out precisely when the enforcement guarantee isn't buying you anything.


The reason this topic consumes disproportionate debate:
most arguments silently assume the implementation-misuse.
Remove that assumption and it reduces to a legitimate enforcement-vs-simplicity trade-off —
and since simplicity is a core design principle, it can't be dismissed outright.

</details>

## Service-Provider Frameworks

### What problem do service-provider frameworks solve?
<details><summary>Show answer</summary>

They **decouple clients from implementations**: providers implement a service, the system makes
those implementations available to clients, and clients never depend on the concrete classes.

The enabling property (and the link back to static factories): the implementation class **need not
exist when the API is written**. New providers can appear long after the framework ships, and
clients pick them up without changing code.

</details>

### What are the components of a service-provider framework?
<details><summary>Show answer</summary>

Don't memorize four flat names — recall **two axes**, and the components fall out of the grid:

- **who faces it?** → the **client**, or the **provider**
- **what does it carry?** → the **product** (what clients use), or the **factory** (what makes products)

|                     | carries the **product**                 | carries the **factory**                        |
|---------------------|-----------------------------------------|------------------------------------------------|
| **client-facing**   | **Service Interface** — type of product | **Service Access API** — client gets product   |
| **provider-facing** | *(provider implements the svc iface)*   | **SPI** + **Registration API** — register/make |

The four components:
1. **Service Interface** — the *type of the product* clients use
2. **Service Access API** — *clients* call it to *obtain* a product (the flexible static factory at
   the heart of the framework)
3. **Provider Registration API** — *providers* call it to *register* their implementation
4. **Service Provider Interface (SPI)** *(optional)* — the *type of the factory* that makes products

</details>

### Service Access API vs Provider Registration API?
<details><summary>Show answer</summary>

Two **doors** on the same registry — opposite actors, opposite directions:

|            | **Registration API**              | **Access API**                     |
|------------|-----------------------------------|------------------------------------|
| whose door | the **provider's**                | the **client's**                   |
| direction  | implementations flow **in**       | products flow **out**              |
| when       | **once**, at startup              | **many times**, at runtime         |
| JDBC       | `DriverManager.registerDriver(d)` | `DriverManager.getConnection(url)` |

They feel similar only because JDBC happens to put both on `DriverManager` — but they're unrelated
methods doing opposite jobs. The Access API may also let clients specify selection criteria; absent
criteria, it returns a default or lets them cycle through all registered implementations.

</details>

### Why is a Service Provider Interface needed — and why optional?
<details><summary>Show answer</summary>

**Needed** because two different things exist, and the client wants only one of them:

- the **product** — what the client uses (`Connection`), typed by the *Service Interface*
- the **factory** — the thing that *manufactures* products on demand (`Driver`), typed by the *SPI*

A provider can't register a single product — one `Connection` is useless, clients need a fresh one
per call. So the provider registers a **factory** (`Driver`), and the framework calls *that* every
time a client asks. The SPI is the **type of that factory**.

> client ← Connection ← *[Driver makes it]* ← provider registered the Driver

**Optional** because the framework can instead create implementations **reflectively** (given a class
name, instantiate it) — reflection acts as a generic factory, removing the need for a typed SPI.

</details>

### JDBC as a service-provider framework — map the roles
<details><summary>Show answer</summary>

Plug the four roles into the grid:

- **Service Interface** → `Connection` *(the product)*
- **Service Access API** → `DriverManager.getConnection` *(client gets product)*
- **Provider Registration API** → `DriverManager.registerDriver` *(provider registers)*
- **SPI** → `Driver` *(the factory that makes `Connection`s)*

Note both APIs sit on `DriverManager` — same class, two opposite-direction methods.

</details>

### Modern logging as a service-provider framework — map the roles
<details><summary>Show answer</summary>

Modern Java logging is built on **API/implementation separation**: code depends only on a logging
*API*, and the *implementation* is chosen by the end application — even bridged across ecosystems.
This is a textbook service-provider framework, and a *better SPI example than JDBC* because the SPI
type is explicit and the backend is **discovered**, not registered.

Two current, clean exemplars run this design:

- **SLF4J** — API artifact `slf4j-api` (type `org.slf4j.Logger`); backends ship an implementation of
  the SPI `org.slf4j.spi.SLF4JServiceProvider`, found via Java's `ServiceLoader`. Reference backend:
  Logback (`logback-classic`).
- **Log4j2** — API artifact `log4j-api` (façade) is separate from `log4j-core` (the implementation).
  You compile against `log4j-api`; consumers who already run Logback don't need `log4j-core`.

**Map to the four roles** (identical shape for both):

- **Service Interface** *(product clients use)* — `org.slf4j.Logger` / `org.apache.logging.log4j.Logger`
- **Service Access API** *(client gets product)* — `LoggerFactory.getLogger()` / `LogManager.getLogger()`
- **SPI** *(factory type a backend supplies)* — `SLF4JServiceProvider` (Log4j2 backends implement this
  too, via `log4j-slf4j2-impl`)
- **Provider discovery, not registration** — the backend is found by being **on the classpath**
  (`ServiceLoader` / `META-INF/services`), i.e. *the consumer chooses the implementation*

**The decoupling in practice — API and implementation are independent, joined by bridges:**

- A library/shared component depends **only on an API**, never on a provider (a compile-time
  dependency on a provider would impose it on the end-user and defeat the purpose).
- API and implementation can come from *different* ecosystems, adapted by bridges:
  - **SLF4J API → Log4j2** via `log4j-slf4j2-impl`
  - **Log4j API → SLF4J/Logback** via `log4j-to-slf4j`
  - **`java.util.logging` → SLF4J** via `jul-to-slf4j`

**Why it fits perfectly:** the client codes against a product type (`Logger`), never naming a backend;
backends implement a factory type (the SPI) and are swapped purely by classpath choice — "decouple
clients from implementations, and let the implementation be chosen even after the calling code is
written."

**Contrast with JDBC — same role, older mechanism:** JDBC uses an explicit
`DriverManager.registerDriver` **registration** call; modern logging uses classpath **discovery**.
Two variants of the provider-availability role: JDBC *registers*, logging *discovers*.

</details>
