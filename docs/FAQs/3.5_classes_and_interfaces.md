## Classes and Interfaces - Inheritance and Extension

### What design considerations are required to make a class appropriate for inheritance?
<details><summary>Show answer</summary>

A class is fit for inheritance only if it's *intentionally* designed, *documented*, and *robust* under extension.
The duties group into three kinds:

**Document the extension contract**
- **Specify what may be overridden and what invariants subclasses must preserve** — the extension contract itself.
- [**Document self-use of overridable methods**](#how-does-a-class-document-its-self-use-of-overridable-methods)
  — every place the class calls its own overridable methods, so subclasses know when their overrides fire.

**Design safe extension points**
- [**Provide `protected` hooks, not exposed internals**](3.5_classes_and_interfaces.md#how-is-the-tension-between-extensibility-and-encapsulation-resolved-in-inheritance)
  — well-defined extension points instead of leaking implementation.
- **Keep the API minimal and stable** — every `public`/`protected` member becomes part of the subclass API forever.
- [**Never call an overridable method from a constructor**](#what-happens-if-a-constructor-invokes-an-overridable-method)
  — the subclass's override runs before the subclass is initialized.

**Protect and verify**
- **Preserve and enforce invariants** — subclasses must not be able to break internal consistency.
- [**Design and test for subclassing**](#how-to-test-a-class-designed-for-inheritance) —
  confirm correct behavior when methods are overridden and state extended.

</details>

### How does a class document its self-use of overridable methods?
<details><summary>Show answer</summary>

Why it's needed: self-use is the *hidden* coupling that makes inheritance fragile.
Documenting it converts that hidden dependency into an **explicit part of the contract** —
which is the whole point of "designing for inheritance."

The mechanism: if a method calls one or more overridable methods, document *when* and *how*,
so subclass authors know exactly when their overrides will fire. This goes in a dedicated Javadoc section,
**Implementation Requirements**, via the `@implSpec` tag — unlike the normal spec (what the method does),
this describes the method's **internal workings**.

```java
/**
 * Removes a single instance of the specified element from this collection,
 * if it is present.
 *
 * @implSpec
 * This implementation iterates over the collection looking for the specified
 * element. If it finds the element, it removes it using the iterator's
 * {@code remove} method.
 *
 * @throws UnsupportedOperationException if the iterator does not support
 *         {@code remove}
 */
public boolean remove(Object o) { ... }
```

This makes explicit that `remove` depends on `iterator()`, shows how overriding `iterator()` affects
`remove`, and tells subclass authors exactly which methods influence each other.

</details>

### A JDK class whose undocumented self-use breaks subclasses?
<details><summary>Show answer</summary>

`HashSet`. Its `addAll` internally calls `add`, but this isn't documented —
so a subclass overriding both to count additions double-counts: `addAll` of 3 elements reports 6.

</details>

### How does documenting implementation details for inheritance fit general API doc principles?
<details><summary>Show answer</summary>

It breaks the usual rule. Good docs describe **what** a method does, not **how** —
but designing for inheritance forces you to document the **how** (self-use),
because inheritance weakens encapsulation and subclasses depend on those internals.
The violation is unavoidable, and it's the price of safe subclassing.

</details>

### How to test a class designed for inheritance?
<details><summary>Show answer</summary>

Write subclasses — there's no other way to exercise the extension points.
Critically, have *someone other than the superclass author* write at least one:
an outsider surfaces missing hooks and undocumented assumptions the author can't see in their own design.

</details>

### What happens if a constructor invokes an overridable method?
<details><summary>Show answer</summary>

The superclass constructor runs **before** the subclass constructor, so an overridable method called
from the superclass constructor executes the subclass's override **on a partially constructed object** —
before the subclass's fields are initialized.

The override then sees uninitialized state and misbehaves. The sharp version: a subclass field, *even a
`final` one*, is still `null`/zero when the override reads it, because the subclass's field initializers
and constructor body haven't run yet.

**Rule: constructors must not invoke overridable methods, directly or indirectly.**

</details>

### In which situations — besides constructors — should overridable methods not be invoked?
<details><summary>Show answer</summary>

The constructor rule generalizes: **any method that acts like a constructor** — running on a
not-yet-fully-initialized object — must not invoke overridable methods.
In a class designed for inheritance, that means:

- **`clone`** (from `Cloneable`) — runs before the clone's state is restored.
- **`readObject`** (from `Serializable`) — runs before the deserialized object's state is set.

Both can execute on a **partially initialized object**, so calling an overridable method from them fires
subclass code too early — the same broken-invariant trap as constructors.

</details>

### How is the tension between extensibility and encapsulation resolved in inheritance?
<details><summary>Show answer</summary>

**The tension** — two opposing needs:
- subclasses need access to *extend or customize* behavior;
- the superclass needs to *protect its internals* and stay free to evolve.

Give subclasses **too much** access → they depend on implementation details → fragile hierarchy.
Give them **too little** → inheritance is unusable.

**The resolution: hooks** — deliberately designed, limited extension points
(typically `protected` methods the superclass calls at well-defined moments).
They grant subclasses exactly the customization they need *and nothing more*, so the superclass exposes a stable,
intentional seam instead of leaking its internals.
This is essentially the *only* clean resolution: it's how you make extension a contract rather than an accident.

</details>

### Why are hooks important — what problems do they solve?
<details><summary>Show answer</summary>

One purpose: allow extension in a **limited, safe form** that keeps encapsulation intact. Two sides to it:

- **Open the safe points** — declare exactly where subclasses may customize, so they extend at intended hooks instead
  of guessing or overriding internals.
- **Close everything else** — keep core logic in `final`/non-overridable methods, so subclasses *can't* reach in and
  break invariants. This is *how* the limit is enforced, and it's why encapsulation survives.

So hooks both grant and restrict: variation is confined to the declared points, the rest is protected.

</details>

### What problems must be considered when using hooks to customize behavior in inheritance?
<details><summary>Show answer</summary>

Hooks make extension *safer* than overriding internals, but two difficulties remain — both fall on the human, because
the language can't express what's really going on.

- **Telling the real hooks apart from merely-overridable methods.** "Hook" is a *design role*, not a language
  construct. `protected` is only an access level: a class can have `protected` methods that are internal helpers in the
  core algorithm, not extension points — so `protected` never positively means "override this." In languages without
  access control (JavaScript, Python), *every* method is overridable and there's no narrowing at all. Either way,
  separating "methods I'm *meant* to override (hooks)" from "methods I merely *can* override" is a human inference from
  documentation and intent — the code can't just tell you.

- **Even the right hook needs the whole algorithm in your head.** Knowing *which* method is a hook isn't enough. Its
  safe behavior depends on *when* the superclass calls it — what runs before and after, what state exists at that
  point — and none of that shows in the hook's signature. So you end up reading the full algorithm to override
  correctly. The dependency on internals isn't gone; it's turned from a *code* coupling into a *mental* one.

So hooks narrow the risk but don't remove the reasoning burden: you infer which methods are extension points, then
reconstruct the control flow to use them safely.

</details>

### Which design pattern underlies the use of protected hooks in inheritance?
<details><summary>Show answer</summary>

**Template Method.** The superclass fixes the algorithm's skeleton in a `final` method and delegates
specific steps to `protected` hooks that subclasses override — customizing parts without changing the
structure, preserving invariants and encapsulation.

</details>

### Can you provide an example that illustrates the use of protected hooks?
<details><summary>Show answer</summary>

**Fragile — subclasses override the core method directly:**
```java
public class RequestProcessor {
  public void process(Request r) { authenticate(r); execute(r); audit(r); }
  protected void execute(Request r) { /* complex core logic */ }
}
```
Subclasses overriding `execute` may skip steps, forget `super.execute()`, or depend on order.

**Correct — `final` skeleton + `final` core + empty pre/post hooks:**
```java
public abstract class RequestProcessor {

  public final void process(Request r) {   // skeleton: fixed, can't be altered
    authenticate(r);
    beforeExecute(r);
    executeCore(r);                          // core: can't be replaced
    afterExecute(r);
    audit(r);
  }

  protected final void executeCore(Request r) { /* invariant-sensitive logic */ }

  protected void beforeExecute(Request r) { }  // hook: empty default
  protected void afterExecute(Request r)  { }  // hook: empty default
}
```
```java
public class MetricsRequestProcessor extends RequestProcessor {
  @Override protected void beforeExecute(Request r) { startTimer(); }
  @Override protected void afterExecute(Request r)  { stopTimerAndRecordMetrics(); }
}
```

The subclass **cannot** bypass authentication, alter order, or replace core logic — enforced structurally (`final`),
not by convention. It customizes only at the declared hook points.

</details>

### Functional alternative to Template Method — benefits and limits?
<details><summary>Show answer</summary>

Instead of subclassing and overriding hooks, the algorithm **takes its variable steps as functions** — the client
supplies *what* each step does, never touching the fixed structure:

```java
public final class RequestProcessor {
  public void process(Request r, Consumer<Request> before, Consumer<Request> after) {
    authenticate(r);
    before.accept(r);     // was: beforeExecute hook
    executeCore(r);
    after.accept(r);      // was: afterExecute hook
    audit(r);
  }
}
```
```java
processor.process(req, r -> startTimer(), r -> recordMetrics());
```

The hooks become parameters; the subclass disappears.

**Benefits**
- **Composition, not inheritance** — no hierarchy, no fragile-base coupling, nothing to subclass.
- **The client passes behavior, not an override** — it supplies *what* to do and needn't know call order or what runs
  around it. (A hook override, by contrast, must reconstruct that timing to be safe.)
- **The steps are ordinary functions** — composed, reused, and tested on their own, independent of the algorithm.

**Limits** — the functional form breaks down as the steps stop being few and independent:
- **Many hook points → parameter bloat.** Two or three functions read cleanly; a dozen steps become an unreadable
  parameter list. Subclassing scales to many hooks better.
- **Steps that must share state.** Subclass hooks coordinate through instance fields; loose functions can't — if
  `before` must hand something to `after`, you're pushed back into an object to hold that state, and the subclass form
  may be simpler.
- **Steps needing the algorithm's internals.** A subclass hook can call other `protected` members; a passed-in
  function is external and can only use what it's given. If a step needs privileged access, functions don't fit.

So the functional form wins for a few independent steps; subclassing wins when the hooks are many, must coordinate, or
need inside access.

</details>

### Self-use documentation vs protected hooks?
<details><summary>Show answer</summary>

Two points on the convention-vs-enforcement axis:

- **Documentation** (`@implSpec`) answers *"what happens if you override this?"* — it **informs**,
  but you can still ignore it and override anything.
- **Protected hooks** answer *"this is the only place you may customize"* —
  they **enforce**, confining variation structurally.

Documentation describes the seam; hooks *are* the seam.
The first relies on the subclass author's discipline; the second removes the choice.

</details>

### In Java, how can inheritance be restricted for concrete classes that are not designed for extension?
<details><summary>Show answer</summary>

This is the "**or else prohibit it**" half of the rule *design and document for inheritance, or prohibit
it*. Two accepted ways:

- **Declare the class `final`** — simplest; the class cannot be subclassed at all.
- **Non-public constructors + static factories** — all constructors `private`/package-private,
  instances via static factory methods. Blocks subclassing from outside the package while still
  permitting it internally — more flexible than `final`.

(Same two options, with the full trade-off comparison, as
[preventing subclassing in immutable classes](3.2_classes_and_interfaces.md#what-options-exist-to-prevent-subclassing-in-immutable-classes-and-how-do-they-compare).)

</details>
