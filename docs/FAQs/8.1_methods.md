
### Skipping parameter validation — what breaks?
<details><summary>Show answer</summary>

A checked parameter fails fast and cleanly. Unchecked, the failure degrades through three worsening stages:

1. **Confusing exception mid-work** — throws deep inside, far from the cause.
2. **Wrong result, no exception** — caller silently trusts a bad value.
3. **Corrupted object** — breaks later, somewhere unrelated (a failure-atomicity violation).

Ordered by traceability: an immediate throw is easy to find, delayed corruption nearly impossible.

</details>

### Documenting parameter restrictions?
<details><summary>Show answer</summary>

For **public and protected** methods, use the Javadoc `@throws` tag to document the exception thrown when a
parameter restriction is violated. Then enforce it at method entry. Typical exceptions: `IllegalArgumentException`,
`IndexOutOfBoundsException`, `NullPointerException`.

```java
/**
 * Returns a BigInteger whose value is (this mod m).
 * @param m the modulus, which must be positive
 * @throws ArithmeticException if m is less than or equal to 0
 */
public BigInteger mod(BigInteger m) {
  if (m.signum() <= 0)
    throw new ArithmeticException("Modulus <= 0: " + m);
  ... // Do the computation
}
```

</details>

### Why document restrictions only on these methods?
<details><summary>Show answer</summary>

Because only **public and protected** methods face unknown callers you can't control, so their restrictions must be
part of the published contract. A non-exported (private / package-private) method is called only by code *you* own, so
you can guarantee valid arguments at the call sites instead of documenting them for a client that doesn't exist.

Such internal invariants can be stated with `assert`:

```java
private static void sort(long a[], int offset, int length) {
  assert a != null;
  assert offset >= 0 && offset <= a.length;
  assert length >= 0 && length <= a.length - offset;
  ... // Do the computation
}
```

`assert` differs from a normal `if`/throw check on two axes:
- Throws **`AssertionError`** (an `Error` — "this should be impossible"), not a client-facing exception. A failure
  means *your own* code broke the invariant, not that a caller passed bad input.
- **Disabled by default** — zero cost unless you run with `-ea`. That's the point in a hot path like a recursive
  sort: you want the check during dev/test, not on every recursive call in production.

Caveat — why it's rare in practice: because it's off by default, the check isn't protecting production at all. Many
teams consider that **worse than useless — a false sense of safety** — and prefer an always-on `if`/throw or a
library precondition (e.g. Guava `checkArgument`) instead. Reserve `assert` for internal invariants that are
expensive enough to want out of production yet worth checking under test.

</details>

### Handling null parameters — what governs?
<details><summary>Show answer</summary>

Trigger: **null is a bug, not a case — so enforce it where the context lives, not everywhere.**

Four sub-questions, each its own card:

1. [Should every method null-check its parameters?](#should-every-method-null-check-its-parameters) — no; pass-through
   checks are noise.
2. [When does an explicit null check earn its place?](#when-does-an-explicit-null-check-earn-its-place) — stored
   parameters.
3. [If null is a valid input?](#if-null-is-a-valid-input) — overload, don't accept it.
4. [Where should absence actually be caught?](#where-should-absence-actually-be-caught) — upstream, at the lookup.

</details>

### Should every method null-check its parameters?
<details><summary>Show answer</summary>

No. For a **pass-through** parameter — dereferenced immediately in the body, then discarded — an explicit check is
noise: the next line throws an NPE at the right place anyway. Guarding every public method drowns the code in
boilerplate for a failure that already fails cleanly on its own.

The real target isn't "every parameter"; it's the one whose failure is [*displaced*](#when-does-an-explicit-null-check-earn-its-place).

</details>

### When does an explicit null check earn its place?
<details><summary>Show answer</summary>

When the parameter is **stored, not used** — a constructor, setter, or builder stashing the reference in a field.
Then a null doesn't fail here; it fails on first use, arbitrarily later, with an NPE that names nothing and points at
innocent code. `Objects.requireNonNull(user, "user")` converts that delayed, misattributed failure into an immediate
throw at the store, naming the parameter. That's a **failure-atomicity** win the implicit check can't give — which is
why "just let it NPE later" is wrong precisely here.

```java
// stored → check pays off
this.strategy = Objects.requireNonNull(strategy, "strategy");
```

</details>

### NPE from a Spring-injected dependency?
<details><summary>Show answer</summary>

Doesn't happen under defaults. A required dependency Spring can't resolve fails the **context at startup** with
`NoSuchBeanDefinitionException` — not a later NPE at first use. `@Autowired` defaults to `required=true`, so this
holds for **field, setter, and constructor** injection alike; the style changes nothing on this axis.

Conclusion: you never NPE on an injected field, and `Objects.requireNonNull` on injected values is never needed —
independently of injection style.

The only way back to a null is to opt out explicitly — `@Autowired(required=false)`, `@Nullable`, or `Optional<T>` —
which *declares* the dependency genuinely optional. Then null is a valid state you chose, not a gap in the default,
and handling its absence is on you.

</details>

### If null is a valid input?
<details><summary>Show answer</summary>

Don't accept it and branch on it. Add an **overload with fewer parameters** — one signature that takes the value, one
that doesn't. A null that means "use the default" overloads the parameter with two jobs and pushes a null check into
every caller. Two methods say the same thing in the type system, with no null anywhere.

</details>

### Where should absence actually be caught?
<details><summary>Show answer</summary>

Upstream, at the point of lookup — not inside the method you eventually pass the value to. When you resolve a user by
id, absence (bad id, deleted row) is a real outcome: return `Optional` and fail *there*, where you still hold the
inputs. Then the exception names the id you searched for — context the downstream method has already lost.

```java
User user = repo.findById(id)
  .orElseThrow(() -> new NoSuchUserException("no user for id=" + id));
process(user); // guaranteed non-null; no check needed here
```

This doesn't remove the check — it **relocates it to where the context lives**. Downstream methods may then assume
non-null, because the boundary already guaranteed it.

</details>

### Auto-enforce null parameters without hand-written checks?
<details><summary>Show answer</summary>

Yes — you can drop the `if (x == null) throw` boilerplate, but **not** the annotation: every tool needs a
`@NonNull` / JSpecify marker to know *what* to enforce. Zero-annotation-and-zero-code doesn't exist. Two mechanisms:

- **Runtime weaving** — a Java agent instruments bytecode at class-load, adding the check on the fly (e.g.
  *notnullcheckweaver*; also **IntelliJ IDEA**'s built-in "add runtime assertions for @NotNull" compiler option).
- **Compile-time insertion** — the check is baked into bytecode during compilation (e.g. *Cabe*, JSpecify-based;
  the *eclipselabs nullness* compile participant, which also honors *inherited* annotations).

Failure looks like: `Argument for non-null parameter aParam at index 1 of MyType#myMethod must not be null`.

So the annotation is the contract; the tool supplies the enforcement code. This matches the design-by-contract split:
`@NonNull` on a parameter makes null the **caller's** responsibility, and the method body may then dereference
without checking.

</details>

### Range-checking indices without a manual if?
<details><summary>Show answer</summary>

`java.util.Objects` (Java 9+) has three range-check helpers for **list/array indices only**. Each validates against a
half-open range `[0, length)`, returns the checked value on success so you can inline it, and throws
`IndexOutOfBoundsException` on failure:

- `checkIndex(index, length)` — a single index is in bounds.
- `checkFromToIndex(from, to, length)` — the sub-range `[from, to)` is in bounds.
- `checkFromIndexSize(from, size, length)` — the sub-range `[from, from+size)` is in bounds.

```java
// method that takes a raw index from a caller — guard, then use
int get(int index) {
  Objects.checkIndex(index, size);   // throws if index < 0 or >= size; returns index
  return elements[index];
}
```

Limits vs. `requireNonNull`: no custom detail message, indices/sub-ranges only, and **no closed ranges** (both
endpoints included) — everything is half-open. Handy when it fits; otherwise write the check yourself.

Note: rarely needed if you never accept a raw index as input and only compute positions from `size()`.

</details>

### Guava Preconditions — what do they add?
<details><summary>Show answer</summary>

`com.google.common.base.Preconditions` turns a 3-line `if`/throw into a one-line assertion (static-imported). You
assert what *should* be true (inverted condition), with lazy `%s`-only formatting — the message is built only on
failure, so no happy-path cost. Unlike `assert`, they're **always on** — no `-ea` flag — so they protect production.

- `checkArgument(cond, "msg %s", x)` → `IllegalArgumentException` — bad argument.
- `checkState(cond, ...)` → `IllegalStateException` — receiver in wrong state.
- `checkNotNull(ref, ...)` → `NullPointerException`, returns `ref` so you can inline it.

What they're worth vs. the JDK:
- **`checkNotNull` adds nothing** — it's `Objects.requireNonNull` (JDK 7+), which you already have. Google's own docs
  say non-Guava projects should just use `requireNonNull`. No reason to pull in Guava for this.
- **`checkArgument` / `checkState` have no JDK twin** — that's the only real gain: a one-liner throwing
  `IllegalArgumentException` / `IllegalStateException` from a boolean, vs. writing the `if`/throw by hand.

Best-context alternative: validate at the **boundary**, not deep in the method. When a value comes from a lookup,
`Optional.orElseThrow` beats `checkArgument` — it fires where you still hold the search inputs, so the message names
*why* (the id you searched, the request), context a downstream `checkArgument` has already lost.

```java
User user = repo.findById(id)
  .orElseThrow(() -> new NoSuchUserException("no user for id=" + id));
```

Verdict: don't add Guava just for preconditions. Use `requireNonNull` for null, boundary `Optional.orElseThrow` for
lookups (richest context), and a hand-written `if`/throw or `checkArgument` only for the leftover in-method argument
checks.

</details>

### When can the defensive copy be skipped?
<details><summary>Show answer</summary>

Copying is the default for mutable components; three cases license skipping it:

- **Ownership handoff** — the method is *designed* to take over the object; the caller promises not to touch it
  again. Copying would defeat the point. Document the transfer.
- **Same-package trust** — class and callers won't misbehave. Document that the caller must not modify.
- **Prohibitive copy cost** — too expensive, and callers are trusted. Document the responsibility.

The two trust cases trade enforcement for speed: the class can only *ask* callers to behave, not *guarantee* it.

</details>

### Describe this interface — method count
<details><summary><strong>Show details</strong></summary>

<details><summary>Show code</summary>

```java
interface Report {
  String render();
  String renderAsHtml();
  String renderAsHtmlNoHeader();
  String renderAsHtmlNoFooter();
  String renderAsPlainText();
  void renderToFile(Path p);
}
```

</details>

<details><summary>Show answer</summary>

**The defect: convenience methods that don't pull their weight.** Every one of these is `render()` plus a format or
destination the caller could supply. The variants also head toward a combinatorial explosion (`NoHeader`, `NoFooter`,
`NoHeaderNoFooter`…). Too many methods bloat the type to learn, document, test, and maintain — and on an *interface*
each one also burdens every implementor.

**Fix: one fully-functional method per action; add a shorthand only if it'll be used often.** Collapse the format
variants into one parameterized method, drop what the caller can trivially do themselves:

```java
interface Report {
  String render(Format format);   // HTML / PLAIN_TEXT, header/footer as options on Format
}
// caller writes the file: Files.writeString(p, report.render(HTML));
```

When in doubt, leave it out.

</details>

</details>

### Long parameter list
<details><summary>Show answer</summary>

Aim for **four parameters or fewer**; long runs of the *same* type are worst — a transposition still compiles and
silently misbehaves. Three ways to shorten:

1. **Split into multiple methods**, each taking a subset — if it raises orthogonality, not method count. `List` has no
   "first index in a sublist" (3 params); instead `subList(from, to)` returns a view, combined with `indexOf` (1
   param). High power-to-weight.
2. **Group params into a helper type** (usually a static member class / record) when a recurring sequence *is* a
   distinct entity — e.g. `(rank, suit)` → one `Card`.
3. [**Builder-for-invocation**](#builder-pattern-for-method-invocation) — when many params, some optional.

</details>

### Builder pattern for method invocation?
<details><summary>Show answer</summary>

Use when there are many params and several are **optional**: instead of one huge call, 
define an object that collects the parameters via setter-style calls, then an `execute()` that does final
validation and runs the computation. Same shape as a construction Builder, aimed at a *method call*.

```java
// vs. query(index, "user", 0, 100, true, false, null, "name")
QueryResult r = search.newQuery("user")   // required arg up front
    .from(0).size(100)                     // optional, each defaulted if omitted
    .fuzzy(true)
    .sortBy("name")
    .execute();                            // validates, then performs the search
```

Wins: each argument is named at the call site (no positional guessing), optional ones simply omitted, and
`execute()` is the single validation point. Cost: a helper class and an object per call — worth it only when the
parameter list is genuinely long or optional-heavy.

</details>

### Analyze this signature — parameter type
<details><summary><strong>Show details</strong></summary>

<details><summary>Show code</summary>

```java
public double sum(ArrayList<Double> values) {
  double total = 0;
  for (double v : values) {
    total += v;
  }
  return total;
}
```

</details>

<details><summary>Show answer</summary>

**The defect: the parameter type is more specific than the method needs.** The body only *iterates* — it never
indexes, never calls an `ArrayList`-specific method. Yet the signature demands an `ArrayList`, so a caller holding a
`LinkedList`, a `Set`, a `List.of(...)`, or any other source must copy into an `ArrayList` first — an unnecessary,
possibly expensive conversion.

**Rule: favor interfaces over classes for parameter types.** Widen to the *least* specific type that supports what
the body does. Since this body only needs sequential access, the honest type is `Iterable<Double>`:

```java
public double sum(Iterable<Double> values) { ... }   // accepts List, Set, Collection, anything iterable
```

If it needed `size()` or `contains()`, `Collection` would be the floor; if it indexed, `List`. Never write `HashMap`
where `Map` works, or `ArrayList` where `Iterable` works — it locks callers to one implementation for no reason.

</details>

</details>

### Describe this method — boolean parameter
<details><summary><strong>Show details</strong></summary>

<details><summary>Show code</summary>

```java
void export(Report r, boolean asHtml) {
  if (asHtml) {
    // ...render and write HTML
  } else {
    // ...render and write plain text
  }
}
```

</details>

<details><summary>Show answer</summary>

**Two defects.** At the call site `export(r, true)` is opaque — `true` says nothing about what it selects. And the
body is a two-way branch on the flag, which is a sign the choice shouldn't be a parameter at all.

**Two fixes, for two situations:**

- **The flag switches between two behaviours** (this case). Don't pass the choice in — **split into two methods**,
  `exportHtml(r)` and `exportText(r)`. The branch disappears and each name states its intent.
- **The flag selects a mode that could grow.** Use a **two-element enum**, readable and extensible:

```java
enum Format { HTML, PLAIN_TEXT }                     // PDF can join later
void export(Report r, Format format);                // vs. export(r, true)
```

Exception: a bare `boolean` is fine when the method name already makes it obvious — `setVisible(true)`.

</details>

</details>
