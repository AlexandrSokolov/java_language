### Which overloading runs — decided when?
<details><summary>Show answer</summary>

Which overloaded method is called is fixed at **compile time**, from the **static (declared) type** of the argument.
The runtime type is never consulted. So a parameter declared `Collection<?>` always selects the
`classify(Collection<?>)` overload, no matter what object it actually holds.

Contrast with **overriding**, the opposite rule: which override runs is decided at **runtime** from the object's
actual type. One handle: **overload = static type, compile time; override = dynamic type, runtime.** Overloading is
dispatched on what the compiler sees; overriding on what the object is.

</details>

### Overloading — how to stay safe?
<details><summary>Show answer</summary>

Conservative rule: **never export two overloads with the same number of parameters.** Then no caller is ever unsure
which one applies to a given argument list — the argument *count* alone picks the method, so the tricky
compile-time-type selection never gets a chance to surprise anyone.

The cost is low: when you'd be tempted to overload, **give the methods different names instead.** Overloading is
never forced — `writeInt`/`writeLong`/`writeBoolean` beats three `write` overloads. Distinct names are free and
remove all doubt.

The exception where same-count overloads stay safe: when corresponding parameters can't be confused because no
argument is ever assignable to both types (e.g. `int` vs `Collection`) — radically different types, not related ones.

</details>

### Constructors — overloaded or overridden?
<details><summary>Show answer</summary>

Only **overloaded**, never overridden. A class's constructors always share the name of the class, so multiple
constructors are overloads by definition — you can't give them different names to sidestep it the way you can with
methods.

They can't be overridden because a constructor isn't inherited — a subclass constructor *calls* the superclass one
via `super(...)`, it doesn't replace it. So the overloading-vs-overriding confusion that bites methods never arises
for constructors: there's no dynamic dispatch to interact with.

</details>

### Constructors — how to avoid overload confusion?
<details><summary>Show answer</summary>

You can't rename constructors, so the "use different names instead" escape is gone. Two other levers remain:

**Prefer static factories.** A static factory *does* have a name, so `Point.ofPolar(r, theta)` and
`Point.ofCartesian(x, y)` replace two same-signature constructors with two clearly-named entry points — the
ambiguity disappears. This is the primary recommendation (Effective Java Item 1).

**When you must overload constructors** — and you often will, since same-parameter-count constructors are common —
apply the same safety rule as methods: keep corresponding parameter types radically different, so no argument is
assignable to more than one, and no call site is ambiguous.

</details>

### "Radically different" types — meaning?
<details><summary>Show answer</summary>

Two types are radically different when **neither is assignable to the other** — you can't pass an instance of one
where the other is expected, in either direction. When parameters are radically different, the argument alone forces
exactly one overload, so the compile-time-type selection can never surprise a caller.

`ArrayList<Object>` and `String[]` are radically different: no value is both, so `f(list)` and `f(array)` are never
in doubt.

`String` and `Object` are **not** — every `String` is assignable to `Object`, so `f(new Object())` vs `f("x")`
resolves by static type in ways callers misread.

Classic real failure: `List.remove(int index)` vs `List.remove(Object o)`. `int` and `Object` feel radically
different, but autoboxing bridges them — `list.remove(2)` removes *index* 2, `list.remove((Integer) 2)` removes the
*value*. Autoboxing and generics erase the gap that made older overloads look safe, which is why the modern advice
is stricter than "just use different types."

</details>

### Overloading with functional interfaces?
<details><summary>Show answer</summary>

Rule: **don't overload a method or constructor to take different functional interfaces in the same argument
position.** Different functional interfaces are **not** radically different — so same-position overloads on, say,
`Runnable` vs `Callable<T>` can fail to resolve even when a human sees an obvious intended match.

The cause is *inexact* method references and implicitly-typed lambdas: their meaning isn't fixed until a target type
is chosen, so overload resolution can't use them to pick. It's the combination that bites — the referenced method
being overloaded *and* the invoked method being overloaded — not either alone.

`-Xlint:overloads` warns about this. The clean fix is the same as always: distinct names, so no resolution contest
exists.

</details>

### When may you break the overloading rule?
<details><summary>Show answer</summary>

When the competing same-count overloads **do exactly the same thing**. The rule exists because an ambiguous call is
dangerous *when the two behave differently*; if results are identical, the caller not knowing which fires costs
nothing.

Guarantee the sameness mechanically: **the more specific overload forwards to the more general one**, so a single
real implementation exists and the two can't drift apart.

```java
// specific forwards to general — one real body, no divergence
public boolean method(SpecificType x) {
  return method((GeneralType) x);
}
```

This comes up most when **evolving an existing class**, where you can't rename methods and must add an overload to
an already-shipped API.

</details>

### JDK overload that breaks the rule — but is fine?
<details><summary>Show answer</summary>

`String.contentEquals`. It had `contentEquals(StringBuffer)` since Java 4. Java 5 added `CharSequence` as a common
supertype for `String`, `StringBuffer`, `StringBuilder`, `CharBuffer`… — and at the same time added the overload
`contentEquals(CharSequence)`.

Now a `StringBuffer` argument matches **both** overloads (a `StringBuffer` *is* a `CharSequence`), so the call is
technically ambiguous — exactly what the rule warns against. It's harmless because the specific overload just
forwards to the general one:

```java
public boolean contentEquals(StringBuffer sb) {
  return contentEquals((CharSequence) sb);
}
```

The language change — adding the `CharSequence` interface — is what forced the second overload onto an existing
method. One real body, so which overload runs is invisible. Sanctioned.

</details>

### JDK overload that breaks the rule — and shouldn't?
<details><summary>Show answer</summary>

`String.valueOf`, the two static factories `valueOf(char[])` and `valueOf(Object)`. Passed the *same* object
reference, they do **completely different things** — one builds a string from the character array's contents, the
other calls `toString()` on the object.

That's the exact failure the rule guards against: same-count overloads whose behavior diverges, so which one runs
actually changes the result. Unlike `contentEquals`, there's no forwarding and no shared body — no justification for
it. Treat it as an anomaly, not a pattern to copy.

</details>

### How does varargs work — mechanically?
<details><summary>Show answer</summary>

On each call, the runtime **builds a fresh array** sized to the number of arguments at that call site, fills it with
the argument values, and hands that array to the method. The method body just iterates the array.

```java
static int sum(int... args) {   // args is an int[]
  int sum = 0;
  for (int arg : args) sum += arg;
  return sum;
}
// sum(1, 2, 3) -> 6 ;  sum() -> 0 (empty array, not null)
```

So a no-argument call is legal and yields an **empty** array, not `null`. That "zero or more" nature is the whole
point — and also the source of both pitfalls: the empty-call case, and one allocation per invocation.

</details>

### Varargs for "one or more" — how?
<details><summary>Show details</summary>
<details><summary>Show code</summary>

```java
static int min(int... args) {
  if (args.length == 0)
    throw new IllegalArgumentException("Too few arguments");
  int min = args[0];
  for (int i = 1; i < args.length; i++)
    if (args[i] < min) min = args[i];
  return min;
}
```

</details>
<details><summary>Show answer</summary>

Put a **required parameter first**, then the varargs parameter for the rest: `min(int firstArg, int... rest)`.

```java
// RIGHT: one required param + varargs for the rest
static int min(int firstArg, int... remainingArgs) {
  int min = firstArg;
  for (int arg : remainingArgs)
    if (arg < min) min = arg;
  return min;
}
```

The naive version — `min(int... args)` plus a runtime `args.length == 0` check — has two faults. An empty call
`min()` compiles and blows up only at **runtime**, and the body is ugly (explicit length check, or a
`MAX_VALUE` seed to allow a for-each). The leading required parameter moves the "too few arguments" error to
**compile time** and lets the loop start clean from `firstArg`.

</details>
</details>

### Varargs performance cost — and the fix?
<details><summary>Show answer</summary>

Every varargs call **allocates and initializes a new array** — cheap normally, but a real cost on a hot path.

When you've measured that cost and still want varargs' flexibility, guard the common cases with fixed-arity
overloads and let only the rare long calls hit the array:

```java
public void foo() { }
public void foo(int a1) { }
public void foo(int a1, int a2) { }
public void foo(int a1, int a2, int a3) { }
public void foo(int a1, int a2, int a3, int... rest) { }   // only this one allocates
```

If ~95% of calls have three or fewer args, only the other ~5% pay for array creation. `EnumSet`'s static factories
use exactly this, so enum sets stay competitive with bit fields. Like most micro-optimizations, rarely worth it —
but a lifesaver when it is.

</details>

### Empty collection return — null or empty?
<details><summary>Show answer</summary>

Return an **empty** collection/array, never `null`. `null` forces every caller to add a `!= null` guard before
using the result; forget it once and you get an NPE that may hide for years, since the method usually returns
non-empty. The "avoid allocation" defense is bogus — you can return a shared immutable empty (`Collections.emptyList`,
a static zero-length array) with no allocation at all.

</details>

### What governs returning Optional?
<details><summary>Show answer</summary>

Spine: **return `Optional<T>` when a method may have no result *and* you want to force the caller to handle that
case.** Optional is like a checked exception in spirit — it makes absence impossible to ignore, unlike `null` or an
unchecked exception, which the caller can silently skip until an NPE surfaces far away.

The concerns fan out:

- Core rule — return it when "no value" is a real outcome the client must confront → this card.
- Creating one — `Optional.of` / `ofNullable` / `empty`; never return `null` from an Optional method →
  [Creating an Optional](#creating-an-optional).
- Consuming one — `orElse` / `orElseGet` / `orElseThrow` / `get` let the *caller* pick the fallback →
  [Consuming an Optional](#consuming-an-optional).
- `isPresent` vs `map` — prefer the functional form → [isPresent or map?](#ispresent-or-map--which).
- Where Optional is wrong — containers, boxed primitives, collection members →
  [Where NOT to use Optional](#where-not-to-use-optional).
- Cost — not free: allocation + indirection → [Optional's cost?](#optionals-cost).
- In a field — usually a smell, sometimes right → [Optional in a field?](#optional-in-a-field--ok).

</details>

### Creating an Optional
<details><summary>Show answer</summary>

Three static factories:

- `Optional.empty()` — no value.
- `Optional.of(v)` — value present; **throws NPE if `v` is null** (use when null is a bug).
- `Optional.ofNullable(v)` — value present, or empty if `v` is null (use when null is expected).

**Never return `null` from an Optional-returning method** — it defeats the entire point; the caller now has to
null-check the Optional itself.

Stream terminal ops often build the Optional for you, so you rarely write these by hand:

```java
public static <E extends Comparable<E>> Optional<E> max(Collection<E> c) {
  return c.stream().max(Comparator.naturalOrder());   // returns Optional<E>
}
```

</details>

### Where NOT to use Optional
<details><summary>Show answer</summary>

**Wrong `T` inside the Optional:**

- **Never wrap a container** — collection, map, stream, array, or another Optional. Return an *empty*
  `List<T>`, not an empty `Optional<List<T>>`; the empty container already signals "nothing," with no Optional to
  unwrap (Item 54). `ProcessHandle.arguments()` returning `Optional<String[]>` is a mistake, not a model.
- **Never box a primitive** — `Optional<Integer>` has two layers of boxing. Use `OptionalInt`, `OptionalLong`,
  `OptionalDouble` instead. (The minor types `Boolean`/`Byte`/`Short`/`Character`/`Float` have no such analogue, so
  they're the only tolerable exception.)

**Wrong place to put the Optional itself:**

- **Never as a key, value, or element** in a collection/array. An Optional map *value* gives two ways to say
  "absent" — key missing, or key present mapping to empty — needless confusion.
- **Rarely as a field** — usually a smell → [Optional in a field?](#optional-in-a-field--ok).
- **Rarely as a method parameter** — pushes wrapping onto the caller →
  [Optional as a method parameter?](#optional-as-a-method-parameter--ok).

Rule of thumb: Optional is for **return values**, and rarely anything else.

</details>

### Consuming an Optional
<details><summary>Show answer</summary>

The caller — not the method — picks what happens when empty:

- `orElse(default)` — supply a fallback value. The default is **always evaluated**, even when a value is present.
- `orElseGet(supplier)` — same, but the supplier runs **only when empty**; use it when the default is expensive to
  build.
- `orElseThrow(exceptionFactory)` — throw a chosen exception. Pass a *factory* (`Ex::new`), so the exception is
  built only if actually thrown.
- `get()` — return the value directly; throws `NoSuchElementException` if empty. Only when you can prove it's
  present.

```java
String last = max(words).orElse("No words...");
Toy toy     = max(toys).orElseThrow(TemperTantrumException::new);
Element gas = max(NOBLE_GASES).get();               // provably non-empty
```

`orElse` vs `orElseGet` is the trap: eager value vs lazy supplier.

</details>

### Optional's cost?
<details><summary>Show answer</summary>

Not free. An Optional is an **object that's allocated and initialized**, and reading its value adds **one level of
indirection**. Fine almost always, but real enough that in a measured performance-critical method, returning `null`
or throwing may beat returning an Optional.

This is also *why* the primitive analogues exist: `Optional<Integer>` pays double boxing on top of that, so
`OptionalInt`/`OptionalLong`/`OptionalDouble` avoid it. Only measurement (Item 67) tells you if the cost matters —
don't assume it does.

</details>

### Optional in a field — OK?
<details><summary>Show answer</summary>

Usually a **bad smell** — it often hints the class should be split into a subclass that has the field and one that
doesn't.

But sometimes right: when a class has **many independent optional fields** so subclassing every combination is
impractical, *and* the fields are primitives (where absence is otherwise awkward to express). Then a getter
returning `Optional` per optional field is the cleanest API, so storing those optionals as fields is justified —
the `NutritionFacts` builder case (Item 2) is the example.

</details>

### Optional as a method parameter — OK?
<details><summary>Show answer</summary>

Almost never. Passing `Optional<T>` in forces every caller to wrap the argument —
`f(Optional.of(x))` instead of `f(x)` — which is noise, and the method still has to handle the empty case inside.

Same smell as an Optional field: the "maybe absent" belongs on the *outside* of the call, not inside the parameter
type. Model the two cases directly instead:

- an overload with the parameter and one without it, or
- a plain nullable parameter the method null-checks, or
- a default value the caller passes.

Reserve `Optional` for what it's for — **return values** — where it forces the *caller* to confront absence. As a
parameter it does the reverse: it pushes wrapping work onto the caller for no gain.

</details>
